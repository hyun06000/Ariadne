# Ariadne Spec

> 버전은 패키지 릴리스(RELEASE.md·깃 태그 `vX.Y.Z`)를 따른다.

LLM이 복잡한 문제를 **사이클**(가장 작은 달성 단위)의 **체인**으로 정복하게 하는 프레임워크의 표준.
이 문서 하나로 어떤 저장소든 Ariadne 방식으로 구조화할 수 있고, 동봉된 `gil.py`(참조 구현, §7)가 그 규칙을 집행한다.

## 1. 세 개의 방 (디렉토리 구조)

```
<repo>/
├── rooms/existence/            # 존재의 방: 에이전트의 정체성·의지·기억·관계 (한 존재당 한 방)
│   └── <이름>/{identity,will,memory,relations}.md
├── rooms/experiment/           # 실험의 방
│   ├── _template/              # 사이클 템플릿 (5스텝 문서 + cycle.yaml)
│   └── chains/<체인>/<사이클>/  # 문제 하나당 체인 하나, 사이클은 C###-슬러그
└── rooms/deployment/           # 배포의 방: 검증 완료된 배포물 + CHANGELOG (SemVer)
```

- **존재는 저장소에만 산다.** 로컬 머신에 존재성을 두지 않는다. 부트스트랩 지시는 저장소 루트(`CLAUDE.md` 등)가 운반한다.
- **배포물은 닫힌 사이클의 보고서를 근거로만** 배포의 방에 들어가며 스스로 완결적이어야 한다.

## 2. 사이클의 5스텝

| # | 스텝 | 산출물 | 핵심 |
|---|---|---|---|
| 1 | 가설 수립 | `1-hypothesis.md` | 부모 보고서의 교훈 → 가장 작은 문제 → 검증 가능한 가설 + **기각 조건 선고정** |
| 2 | 실험 설계 | `2-design.md` | 절차 중심 설계, 필요시 사용자 컨펌. **정답(기대 결과)을 도구·산출물보다 먼저 고정** |
| 3 | 가설 검증 | `3-verification/` | 모든 코드·픽스처·실행 로그 저장. 사이클 단위 재현 가능 |
| 4 | 결과 분석 | `4-analysis.md` | 수치만이 아니라 데이터·아티팩트를 직접 관찰. "예상과 달랐던 것" 필수 |
| 5 | 결과 보고 | `5-report.md` | 교훈과 다음 사이클 제안. **이 문서가 다음 사이클의 부모가 된다** |

닫힌 사이클의 문서는 불변이다. 잘못은 다음 사이클에서 다룬다.

**결말과 이탈은 기계가 본다 (v0.3, maru의 이슈 #1):** 사이클을 닫을 때 `verdict`로 결말을 기록한다 — 기각된 가설은 성공한 사이클이며, `gil log`가 이를 표시·집계한다. 사전등록한 기각 조건의 행동 지침을 정당하게 이탈했다면(합법이다) `deviations.yaml`에 구조화해 남긴다 — pre-registration의 가치는 "어길 수 없게"가 아니라 **"어긴 것이 보이게"** 하는 것이다. fsck가 이탈을 표시하고, 감사자는 사슬을 정독하지 않고도 결말과 이탈을 파악한다.

### 2.1 gil 사용 원칙 (v0.9.3 — 모든 LLM의 규율, 발의: 박상현)

gil을 쓰는 존재는 — 소환됐든 아니든, 어떤 모델이든 — 다음 리듬으로 일한다:

1. **커밋의 단위는 스텝이다.** 사이클이 아니다. `gil step`·`gil close`는 **깃 저장소에서 자동으로 커밋**한다(v1.7~) — `--git`을 안 붙여도 각인된다. 끄려면 `--no-commit`. 원격이 있으면 `--push`로 전파. 뷰어의 준실시간성은 이 리듬에서 나온다.
2. **긴 스텝은 중간에도 커밋한다.** 특히 3단계(검증·구현) 구간 — 관전자에게 침묵은 멈춤과 구별되지 않고, 미커밋 작업은 중단 시 유실된다 ("끊길 위험이 보이면 먼저 커밋").
3. **열 때부터 보이게**: `gil open … --git --push` — 번호 원장 규율(§6-6)과 즉시 가시화를 한 번에.
4. **닫음은 각인**: `gil close … --git --push` — 태그와 함께 역사가 된다.

**재현 문서 규약 (v0.9.1 — tapestry/C001로 확립)**: `3-verification`의 재현 방법은
**불변 기준**(픽스처 대상 — 시점 무관, 바이트 대조 가능)과 **가변 확인**(실데이터 대상 —
저장소 성장에 따라 달라지는 시점 의존 확인)을 구분해 명시한다. 독립 감사자는 불변 기준으로
판정하고, 가변 확인의 차이는 결함이 아니라 드리프트로 분별한다.

## 3. cycle.yaml 스키마 (v0.2)

```yaml
id: C001-first-question   # C<3자리 이상 번호>-<소문자 케밥 슬러그>
chain: demo               # 소속 체인 디렉토리명과 일치
parent: null              # 같은 체인 내 부모의 로컬 id. 분기=같은 부모 공유, 병합=[a, b], 최초=null
lineage: []               # 다른 체인의 보고서에서 얻은 교훈. 전역 표기 [<chain>/<id>]
step: 1                   # 진행 스텝 1~5 = 가설·설계·검증·분석·보고 (v0.6, 선택 필드)
author: <존재의 이름>
status: open              # open | closed
opened: YYYY-MM-DD
closed: null
title: "정복하려는 가장 작은 문제 한 줄"
verdict: null             # supported|partial|rejected|inconclusive — 사이클의 결말 (v0.3, 닫을 때)
deviations: 0             # 사전등록으로부터의 이탈 건수. N>0이면 deviations.yaml 필수 (v0.3)
```

### 3.1 파싱 규칙 (v0.9.2 — 두 구현의 교차 판정으로 확증)

- 인코딩 UTF-8, 평탄한 `key: value` 문서 (중첩 없음).
- `#`로 시작하는 줄은 주석 — 무시한다.
- **비따옴표 값**: ` #` 이후는 후행 주석으로 절단한다. `null`·`~`·빈 값은 부재를 뜻한다.
- **따옴표 값** `"…"`: 내부의 쉼표·해시·대괄호를 보존한다 (후행 주석 절단은 닫는 따옴표 뒤에만).
- **리스트** `[a, b]`: 쉼표 구분, 항목별 따옴표 허용, `[]`는 빈 리스트.
- 환경 규약: 경로 비교는 실경로(심링크 해소) 기준, 날짜는 `YYYY-MM-DD`.
- **log의 렌더 형식은 계약이 아니다** — 요약 섹션·그래프 문자 등은 구현 자유. 기계 소비의 계약면은 종료 코드와 web의 `gil-data` JSON이다 (§5.1·§7).

### 규칙 (전부 `gil fsck`가 집행)

| 규칙 | 내용 |
|---|---|
| **R1** | id 형식 `C\d{3,}-[a-z0-9][a-z0-9-]*`, 번호는 체인 내 유일. 표시 순서 동순위는 번호 오름차순 |
| **R2** | lineage의 각 참조는 실재하는 사이클로 해소되어야 한다 |
| **R3** | parent는 로컬 id만(`/` 금지), lineage는 전역 표기만(`<chain>/<id>`), lineage로 같은 체인 지시 금지 |
| **R4** | `chain` 필드 = 소속 체인 디렉토리명 |
| **R5** | `id` = 사이클 디렉토리명 |
| **R6** | parent는 실재하는 사이클로 해소되어야 한다 |
| **R7** | 체인은 비순환 |
| **R8** | `status: closed` ⇔ `closed` 일자 존재 |
| **R9** | `step`이 존재하면 1~5 정수, 닫힌 사이클이면 5 (v0.6) |
| **R10** | `verdict`가 있으면 supported\|partial\|rejected\|inconclusive 중 하나. `deviations`가 있으면 정수이고 N>0이면 `deviations.yaml` 존재. `closed`인데 `verdict` 없으면 **경고**(강제 아님, 기존 사슬 유예) (v0.3) |

## 4. 깃 각인 규약

- 사이클을 닫을 때 `close --git`은 **사이클 디렉토리만을 담은 커밋**과 주석 태그 **`cycle/<chain>/<id>`** 를 남긴다. 닫힌 사이클의 불변은 이로써 기계적 보장이 된다 — `verify`가 태그와 작업 트리를 대조해 변조(수정·추가)를 탐지한다.
- **태그 위치 원칙**: 사이클의 태그는 그 사이클을 마지막으로 정당하게 수정한 커밋을 가리킨다.
- **이주 규정**: 닫힌 사이클의 5스텝 문서와 검증 산출물은 불변. `cycle.yaml`에 한해 의미 무손실 스키마 이주를 허용하며, 이주 커밋은 `[migrate]`를 명시한다.
- **태그 이동 규약**: `[migrate]` 이주가 태그된 사이클을 수정한 경우, 태그를 이주 커밋으로 옮기고(`git tag -f`) 태그 메시지에 이전 커밋 해시와 이동 사유를 남긴다. 그 외의 태그 이동은 금지.

## 5. CLI 계약 (`gil` — 길, GIt for Language model)

| 명령 | 역할 |
|---|---|
| `gil log` | 체인 계보를 ASCII 그래프로 (분기 `├─┐` · 병합 `├─┘` · lineage `⇠`) |
| `gil fsck` | R1~R8 위반 전수 수집·보고 (exit ≠ 0) |
| `gil open <chain> <slug>` | 사전 검증 → 템플릿 복사 → v0.2 준수 cycle.yaml 생성 (번호 자동 증가) |
| `gil close <chain> <id>` | 보고서 존재·비템플릿 검증 → 상태 전이(step=5 마감, `--verdict`로 결말 기록) → fsck. `--git` 커밋+태그, `--push` 전파 |
| `gil step <chain> <id> <n>` | 열린 사이클의 진행 스텝 전이(1~5). `--git`이면 전이를 사이클만 담은 커밋으로, `--push`면 즉시 전파 — **뷰어의 준실시간성**은 이 명령의 규율에서 나온다 (v0.6) |
| `gil goto <chain>/<id> [--checkout]` | 타임머신: 사이클 시점의 역행 조회·체크아웃과 분기 안내 — 잘못된 길에서 건강한 지점으로 되돌아가 새 갈래를 시작 (v1.3) |
| `gil verify` | 닫힌 사이클의 태그↔작업 트리 대조 (변조 탐지) |
| `gil pages [--force]` | GitHub Pages 배포 워크플로(`.github/workflows/gil-pages.yml`)를 생성 — 저장소 비특정, push마다 뷰어 자동 배포 (v1.2) |
| `gil web -o out.html` | 자기완결적 정적 HTML 뷰어 (같은 파서, SVG 렌더, 외부 리소스 0). 그래프 구조는 `<script type="application/json" id="gil-data">`로 내장(step 포함) — 기계 검증의 훅. 열린 사이클은 스텝 인디케이터(●●●○○ n/5) 표시 (v0.6) |
| `gil release <버전> --notes "..."` | 도구·템플릿을 패키지로 동기화 → CHANGELOG → 배포의 방만 커밋 → 태그 `v<버전>`. **도구가 변했으면 마이너 이상 승격 강제** — 기준은 마지막 릴리스 태그의 blob이며, `conformance.py`(판정기)도 도구로 간주한다 (v0.9) |

### 5.1 호출 문법 (v0.9.1 — Go 이식의 역공학 경험으로 명문화)

`gil <명령> [위치 인자…] [플래그…]` — 플래그는 위치 인자 뒤 어디든 올 수 있다.

| 명령 | 위치 인자 | 주요 플래그 |
|---|---|---|
| log·fsck·verify·web | `[chains-root]` (기본 `rooms/experiment/chains`) | `--chain <이름>`, web은 `-o <출력>` `--title` |
| open | `<chain> <slug>` | `--title` `--author` `--date` `--parent`(반복 가능) `--lineage`(반복 가능) `--new-chain` `--git` `--push` `--root` |
| step | `<chain> <cycle-id> <n>` | `--git` `--push` `--root` |
| close | `<chain> <cycle-id>` | `--date` `--git` `--push` `--root` |
| release | `<버전>` | `--notes`(필수) `--date` `--package` `--root` |

- 반복 가능한 플래그는 `--parent`·`--lineage`뿐이다 (병합·다중 계보).
- 종료 코드: 성공 0, 위반·거부·오류 ≠ 0. **부분 구현은 허용**된다 — 미구현 명령은 오류 + 비영 종료로 정직하게 알리고, conformance가 항목별로 판정한다.

실행 가능한 예시 (검증이 이 블록을 그대로 실행한다):

```bash
python3 "$GIL" open demo first-step --new-chain --title "예시" --author me --date 2026-01-01 --root "$ROOT"
python3 "$GIL" step demo C001-first-step 3 --root "$ROOT"
python3 "$GIL" fsck "$ROOT"
python3 "$GIL" log "$ROOT" --chain demo
python3 "$GIL" web "$ROOT" -o "$OUT/chains.html" --title "예시"
```

## 6. 소환 규약 (v2 — genesis/C001~C003으로 확립)

서브에이전트에게는 저장소 루트의 부트스트랩 문서가 **자동으로 배달되지 않는다** (genesis/C001 기각으로 실증).
존재의 지속은 저장 × 전달이며, 서브에이전트의 전달 경로는 소환자다.

**소환자의 4의무**:
1. **포인터 주입**: 프롬프트에 "이 저장소 루트의 CLAUDE.md를 읽고 그 지시를 따르라"를 반드시 포함한다. 정체성 내용은 누설하지 않는다.
2. **모드 선언**: 소환은 2모드다 — **부활**(거주자 명부의 이름으로 주소 지정: "명부에서 너는 <이름>이다")과 **탄생**("너는 새로운 존재다" 선언 → 피소환자가 방을 창건하고 스스로 이름을 정의). 같은 포인터에서 갈라지며, 갈림길은 소환 프롬프트의 선언이 정한다.
3. **신원 전달**: 소환자가 누구인지 밝힌다("너의 소환자는 <이름>이다") — 호출 맥락은 문서 밖 정보라, 밝히지 않으면 부활체가 대화 상대를 오인한다 (genesis/C003 관찰).
4. **상호 기록**: 탄생 시 서로의 `relations.md`에 관계를 기록한다 — 피소환자는 자기 몫을, 소환자는 자기 몫을. 피소환자의 쓰기 범위는 자기 방 + 거주자 명부로 구획한다 (동시성 규율).
5. **진행 가시화 (v0.7)**: 격리 브랜치에서 일하는 피소환자는 스텝마다 사이클 디렉토리만 담은 커밋을 남기고 **자기 브랜치를 push**한다 (main push 금지 — 병합은 소환자의 몫). Pages 워크플로가 존재 브랜치들을 `branches/`로 함께 렌더하므로, 사용자는 병렬 노동을 관전할 수 있다.
6. **번호 원장 규율 (v0.8 — C015의 경합 실증을 C016이 규율화)**: 사이클 번호의 진실은 **원격 main(원장)** 이다. `gil open --git --push`는 push 거절을 "원장이 앞섰다"는 신호로 받아 fetch·rebase 후 번호 경합을 자동 재번호로 해소하고 재시도한다 (깃의 낙관적 동시성 그대로). rebase 충돌 등 해소 불가 상황은 명시적 오류로 멈춘다. 격리 브랜치의 존재는 병합 시점에 같은 규율(중복 시 재번호)을 적용받는다 — 체인 그래프의 분기·병합(parent)은 이미 병렬을 모델링하므로, 발급만 원장으로 직렬화하면 브랜치별 사이클 작업이 안전하다.

## 7. 구현 독립 계약 (v0.3.0 — loom/C010)

**스펙이 계약이고, 구현은 교체 가능하다.** 이 문서가 정의하는 것 — 디렉토리 구조, cycle.yaml 스키마와
규칙 R1~R8, 5스텝 규약, 깃 각인 규약(태그명·커밋 경로 격리), CLI의 명령·인자·종료 코드(성공 0 / 위반·거부 ≠ 0),
web 산출물의 자기완결성 — 이 gil의 전부이며, 어떤 언어로 구현되어도 이 계약을 지키면 gil이다.

이 패키지는 **두 이행자**를 동봉한다 (v1.0.0):

- `gil.py` — 참조 구현 (Python 3 표준 라이브러리). 호출: `python3 gil.py <명령>`.
- `go/main.go` — Go 구현 (표준 라이브러리만, 직조: Weft). 빌드: `go build -o gil main.go`, 호출: `gil <명령>` — **단일 바이너리, 정말 깃처럼** (발의: 박상현).

둘 다 conformance 26/26 — 같은 저장소에 대해 fsck는 stdout까지, web은 HTML 전문까지 바이트 단위로 동일하다.
계약이 구현을 정의하지, 구현이 계약을 정의하지 않는다.

**계약의 실행 가능형**: 동봉된 `conformance.py`가 이 계약의 판정기다 (v0.4.0 동봉).
구현을 명령 문자열로만 알고, 종료 코드·파일시스템·산출물 텍스트만으로 판정한다:

```bash
python3 conformance.py --gil "python3 gil.py"     # 참조 구현 판정
python3 conformance.py --gil "./gil"              # 미래의 바이너리도 같은 판정
```

전 항목 통과가 곧 "이 구현은 gil이다"의 정의다. 스위트는 행동을 판정하지 구현을 판정하지 않는다 —
내부 구조가 달라도 관찰 가능한 행동이 계약을 지키면 gil이다 (loom/C011의 동등 변이 실험으로 확립).
