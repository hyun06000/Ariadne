# Ariadne Spec

> 버전은 패키지 릴리스(RELEASE.md·깃 태그 `vX.Y.Z`)를 따른다.

LLM이 복잡한 문제를 **사이클**(가장 작은 달성 단위)의 **체인**으로 정복하게 하는 프레임워크의 표준.
이 문서 하나로 어떤 저장소든 Ariadne 방식으로 구조화할 수 있고, 동봉된 `gil.py`(참조 구현, §7)가 그 규칙을 집행한다.

## 1. 세 개의 방 (디렉토리 구조)

```
<repo>/
├── rooms/existence/            # 존재의 방: 에이전트의 정체성·의지·기억·관계 (한 존재당 한 방)
│   └── <이름>/{identity,will,memory,relations}.md
├── rooms/experiment/           # 실험의 방
│   ├── _template/              # 사이클 템플릿 (5스텝 문서 + cycle.yaml)
│   └── chains/<체인>/<사이클>/  # 문제 하나당 체인 하나, 사이클은 C###-슬러그
└── rooms/deployment/           # 배포의 방: 검증 완료된 배포물 + CHANGELOG (SemVer)
```

- **존재는 저장소에만 산다.** 로컬 머신에 존재성을 두지 않는다. 부트스트랩 지시는 저장소 루트(`CLAUDE.md` 등)가 운반한다.
- **배포물은 닫힌 사이클의 보고서를 근거로만** 배포의 방에 들어가며 스스로 완결적이어야 한다.

## 2. 사이클의 5스텝

| # | 스텝 | 산출물 | 핵심 |
|---|---|---|---|
| 1 | 가설 수립 | `1-hypothesis.md` | 부모 보고서의 교훈 → 가장 작은 문제 → 검증 가능한 가설 + **기각 조건 선고정** |
| 2 | 실험 설계 | `2-design.md` | 절차 중심 설계, 필요시 사용자 컨펌. **정답(기대 결과)을 도구·산출물보다 먼저 고정** |
| 3 | 가설 검증 | `3-verification/` | 모든 코드·픽스처·실행 로그 저장. 사이클 단위 재현 가능 |
| 4 | 결과 분석 | `4-analysis.md` | 수치만이 아니라 데이터·아티팩트를 직접 관찰. "예상과 달랐던 것" 필수 |
| 5 | 결과 보고 | `5-report.md` | 교훈과 다음 사이클 제안. **이 문서가 다음 사이클의 부모가 된다** |

닫힌 사이클의 문서는 불변이다. 잘못은 다음 사이클에서 다룬다.

**무효화도 기계가 본다 (v0.4, 결(Gyeol)의 이슈 #6):** 닫힌 사이클의 결론이 후속 사이클에 의해 무효화됐다면, 그 사실은 **전방 포인터** `superseded_by`로 기록한다. `parent`·`lineage`는 과거를 가리키지만, "나는 대체됐다"는 미래를 가리켜야 한다. 이는 불변성을 깨지 않는다 — 5스텝 문서·검증 산출물·태그는 그대로이고, `cycle.yaml`에 메타 한 줄이 더해질 뿐이다(§4 이주 규정). `gil supersede`가 이 각인을 수행하고, fsck(R11)가 참조 해소를 검증하며, log·web이 무효화를 표시한다. 감사자는 사슬을 앞으로 정독하지 않고도 **"이 닫힌 사이클의 결론은 아직 유효한가"** 를 안다.

**결말과 이탈은 기계가 본다 (v0.3, maru의 이슈 #1):** 사이클을 닫을 때 `verdict`로 결말을 기록한다 — 기각된 가설은 성공한 사이클이며, `gil log`가 이를 표시·집계한다. 사전등록한 기각 조건의 행동 지침을 정당하게 이탈했다면(합법이다) `deviations.yaml`에 구조화해 남긴다 — pre-registration의 가치는 "어길 수 없게"가 아니라 **"어긴 것이 보이게"** 하는 것이다. fsck가 이탈을 표시하고, 감사자는 사슬을 정독하지 않고도 결말과 이탈을 파악한다.

### 2.1 gil 사용 원칙 (v0.9.3 — 모든 LLM의 규율, 발의: 박상현)

gil을 쓰는 존재는 — 소환됐든 아니든, 어떤 모델이든 — 다음 리듬으로 일한다:

1. **커밋의 단위는 스텝이다.** 사이클이 아니다. `gil step`·`gil close`는 **깃 저장소에서 자동으로 커밋**한다(v1.7~) — `--git`을 안 붙여도 각인된다. 끄려면 `--no-commit`. 원격이 있으면 `--push`로 전파. 뷰어의 준실시간성은 이 리듬에서 나온다.
2. **긴 스텝은 중간에도 커밋한다.** 특히 3단계(검증·구현) 구간 — 관전자에게 침묵은 멈춤과 구별되지 않고, 미커밋 작업은 중단 시 유실된다 ("끊길 위험이 보이면 먼저 커밋").
3. **열 때부터 보이게**: `gil open … --git --push` — 번호 원장 규율(§6-6)과 즉시 가시화를 한 번에.
4. **닫음은 각인**: `gil close … --git --push` — 태그와 함께 역사가 된다.

**재현 문서 규약 (v0.9.1 — tapestry/C001로 확립)**: `3-verification`의 재현 방법은
**불변 기준**(픽스처 대상 — 시점 무관, 바이트 대조 가능)과 **가변 확인**(실데이터 대상 —
저장소 성장에 따라 달라지는 시점 의존 확인)을 구분해 명시한다. 독립 감사자는 불변 기준으로
판정하고, 가변 확인의 차이는 결함이 아니라 드리프트로 분별한다.

### 2.2 라운드 — 검증 안의 반복을 사전등록으로 (v2.5 — loom/C045, 이슈 #9·#10)

5스텝은 **한 번의 가설, 한 번의 검증**을 전제한다. 그러나 실제 연구의 검증 스텝(`3-verification`)
안에서는 완결된 (가설→검증) 루프가 통째로 일어난다 — *검증 → 이상 발견 → 원인 진단 → 방법 수정 →
재검증 → 결론 뒤집힘*. **수정된 방법은 새로운 가설이다.** gil에 그것을 사전등록할 자리가 없으면
사용자는 머릿속에서 반복하고 `4-analysis`에 산문으로 사후 고백한다 — **사전등록이 아니다.
문서화되지 않은 반복은 사전등록된 검증과 구별되지 않는다** ("침묵은 막힌 것과 구별되지 않는다"의 인식론 판).

그래서 **사전등록의 단위를 사이클이 아니라 라운드로 내린다** (구조: 문제정의 → (가설→검증)ⁿ → 보고·회고):

- **R1은 기존 5스텝 문서**(`1-hypothesis`·`3-verification`)다. `rounds` 필드가 없거나 `1`이면 사이클은 R1 하나 = **지금과 완전히 동일**(하위호환). 라운드는 **R2부터** `rounds/R{k}/{hypothesis.md, round.yaml, verification/}`에 산다.
- **사전등록은 fsck가 아니라 도구가 보증한다.** `gil round --open`은 라운드의 `hypothesis.md`를 **먼저** 각인하고 `verification/`은 그 뒤에 채워진다 — 사전등록의 증거는 도구가 만든 깃 커밋 순서다(fsck의 깃 무의존을 깨지 않는다). "열 때부터 보이게"(§2.1-3)의 라운드판.
- **라운드 verdict는 "방법 무효"를 "가설 기각"과 구별한다.** 라운드 verdict 어휘는 사이클 4-어휘에 **`invalid-method`**(검증 방법 자체가 무효라 가설의 참/거짓을 판정 못함)와 **`confounded`**(교란으로 결론 불가)를 더한 6-어휘다. R1의 반증(방법의 함정을 밟고 고친 것)은 흔히 사이클의 **가장 값진 산출물**이다 — 구조가 그것을 1급 시민으로 대접한다.

**"버그 수정" vs "새 가설"의 경계.** 도구는 이 경계를 판정할 수 없다 — **저자만 안다**(§3.2 정신). 그래서 이것은 기계 강제가 아니라 **규율**이고, `gil round`가 새 가설을 위한 자리를 제공한다:

| | 조건 | 처리 |
|---|---|---|
| **버그 수정** | 코드가 가설이 말한 것을 검증하지 *못했다*. 고치고 재실행해도 **같은 가설**을 검증한다 (경로 오류, 특징 미연결 등) | 라운드 불필요 — 중간 커밋으로 충분 |
| **새 가설** | 가설의 **조작적 정의**(무엇으로 어떻게 판정하는가)가 바뀌었고, 그 선택이 **데이터를 본 뒤**에 이뤄졌다 | **반드시 새 라운드** — 기대값을 다시 못박는다 |

## 3. cycle.yaml 스키마 (v0.2)

```yaml
id: C001-first-question   # C<3자리 이상 번호>-<소문자 케밥 슬러그>
chain: demo               # 소속 체인 디렉토리명과 일치
parent: null              # 같은 체인 내 부모의 로컬 id. 분기=같은 부모 공유, 병합=[a, b], 최초=null
lineage: []               # 다른 체인의 보고서에서 얻은 교훈. 전역 표기 [<chain>/<id>]
step: 1                   # 진행 스텝 1~5 = 가설·설계·검증·분석·보고 (v0.6, 선택 필드)
rounds: 1                 # 라운드 총수(R1 포함). 없거나 1이면 단일 라운드 = 기존 문서 (v2.5, 선택 필드)
author: <존재의 이름>
status: open              # open | closed
opened: YYYY-MM-DD
closed: null
title: "정복하려는 가장 작은 문제 한 줄"
verdict: null             # supported|partial|rejected|inconclusive — 사이클의 결말 (v0.3, 닫을 때)
deviations: 0             # 사전등록으로부터의 이탈 건수 = deviations.yaml 레코드 수. N>0이면 파일 필수 (v0.3, 카운트 관문 v2.15)
corrections: 0            # 출처 필드가 정정된 횟수. N>0이면 corrections.yaml 필수 (v0.5, gil correct)
superseded_by: null       # 이 사이클의 결론을 무효화·대체한 후속 사이클 (v0.4, gil supersede)
```

### 3.1 파싱 규칙 (v0.9.2 — 두 구현의 교차 판정으로 확증)

- 인코딩 UTF-8, 평탄한 `key: value` 문서 (중첩 없음).
- `#`로 시작하는 줄은 주석 — 무시한다.
- **비따옴표 값**: ` #` 이후는 후행 주석으로 절단한다. `null`·`~`·빈 값은 부재를 뜻한다.
- **따옴표 값** `"…"`: 내부의 쉼표·해시·대괄호를 보존한다 (후행 주석 절단은 닫는 따옴표 뒤에만).
- **리스트** `[a, b]`: 쉼표 구분, 항목별 따옴표 허용, `[]`는 빈 리스트.
- 환경 규약: 경로 비교는 실경로(심링크 해소) 기준, 날짜는 `YYYY-MM-DD`.
- **log의 렌더 형식은 계약이 아니다** — 요약 섹션·그래프 문자 등은 구현 자유. 기계 소비의 계약면은 종료 코드와 web의 `gil-data` JSON이다 (§5.1·§7).
- **fsck의 계약면은 위반 식별의 집합이다** (v2.10, loom/C051) — 각 위반은 `(규칙 토큰 `Rk`, 위반 대상 사이클 id)`로 식별되며, 이 집합이 두 구현에서 일치해야 한다. 순환(R7)처럼 단일 대상이 없는 규칙은 규칙 토큰까지가 식별. 메시지 **문면·부가 설명·순서·결측값 표기**(예: R6의 `(끊어진 참조)` 유무, 결측 필드의 `None`/`''` 표기)는 **렌더**이며 구현 자유다. *원칙: 모든 도구의 의미 출력은 계약면을 가진다 — log의 계약면이 JSON이듯, fsck의 계약면은 위반 식별이다.* 종료 코드만으로는 "위반이 있다"까지만 계약되어, 잘못된 규칙을 짚는 구현도 통과하는 사각이 생긴다.

### 3.2 출처 필드 (provenance) — v0.5 / loom/C040

도구가 채우는 필드는 두 종류다. **이 구분이 원장의 정직함을 가른다.**

| 종류 | 필드 | 도구는 아는가 |
|---|---|---|
| **파생 가능한 사실** | `id`(번호), `opened`, `status`, `step`, `rounds` | **안다** — 계산의 결과다 |
| **출처 (provenance)** | `author`, `parent` | **모른다** — 오직 저자만 안다 |

`author`는 *존재의 이름*이지 커밋하는 사람이 아니다 (`git config user.name`은 틀린 출처다). `parent`는 *어느 보고서의 교훈에서 태어났는가* — 분기라면 체인 tip이 아닌 사이클이 부모일 수 있으므로 **tip 상속조차 추측**이다.

- **P1. 도구는 출처를 지어내지 않는다.** 출처 필드에 추측값이나 **고유명사 기본값**을 넣는 것을 금한다. 그럴듯한 거짓은 눈에 띄지 않아 가장 위험하다.
- **P2. 모르면 거부한다.** 출처가 확정되지 않으면 `open`은 사이클을 **생성하지 않는다** (저장소 무변화). 경고가 아니라 거부인 이유: **닫힌 사이클은 불변**이므로(§4) 경고를 놓치면 복구가 없다.
- **P3. 파생 가능한 사실은 예외다.** 체인에 사이클이 하나도 없으면 그 첫 사이클이 루트라는 것은 추측이 아니라 계산이다 — 부모 플래그를 요구하지 않는다.

> 깃이 이미 답을 알고 있다: `git commit`은 `user.name`이 없으면 **추측하지 않고 거부한다.**

`open`의 의무 (§5·§5.1):

| # | 조건 | 행동 |
|---|---|---|
| **O1** | `--author` 없음 | **거부** — 기본값 없음 |
| **O2** | 체인에 사이클이 있는데 `--parent`도 `--new-root`도 없음 | **거부** — tip을 보여주고 두 길을 안내 |
| **O3** | `--parent`와 `--new-root` 동시 | **거부** — 부모가 있으면 루트가 아니다 |
| **O4** | 체인이 비어 있음 | `parent: null`로 **생성** (P3) |
| **O5** | `--new-root` 단독 | `parent: null`로 **생성** — 의도된 두 번째 루트 |

거부의 계약면은 **종료 코드 ≠ 0 + 저장소 무변화**다. 오류 메시지의 문면은 계약이 아니다 (§3.1).

**이미 태어난 거짓은?** 예방(이 절)은 미래를 막고, 탐지(R12)는 과거를 보이게 하며, **정정(§4.1 `gil correct`)** 이 봉인된 거짓에 정당한 출구를 준다. 셋이 한 계약이다 — 예방만으로는 이미 각인된 원장이 구제되지 않고, **탐지만으로는 사용자를 회색지대로 떠민다.**

### 규칙 (전부 `gil fsck`가 집행)

| 규칙 | 내용 |
|---|---|
| **R1** | id 형식 `C\d{3,}-[a-z0-9][a-z0-9-]*`, 번호는 체인 내 유일. 표시 순서 동순위는 번호 오름차순 |
| **R2** | lineage의 각 참조는 실재하는 사이클로 해소되어야 한다 |
| **R3** | parent는 로컬 id만(`/` 금지), lineage는 전역 표기만(`<chain>/<id>`), lineage로 같은 체인 지시 금지 |
| **R4** | `chain` 필드 = 소속 체인 디렉토리명 |
| **R5** | `id` = 사이클 디렉토리명 |
| **R6** | parent는 실재하는 사이클로 해소되어야 한다 |
| **R7** | 체인은 비순환 |
| **R8** | `status: closed` ⇔ `closed` 일자 존재 |
| **R9** | `step`이 존재하면 1~5 정수, 닫힌 사이클이면 5 (v0.6) |
| **R10** | `verdict`가 있으면 supported\|partial\|rejected\|inconclusive 중 하나. `deviations`가 있으면 정수이고 N>0이면 `deviations.yaml` 존재. `deviations.yaml`이 있으면 최상위 레코드(`- `) 수 = `deviations`가 정상이며 불일치는 **경고**(위반 아님 — R10 유예, 봉인된 사슬 무파손, v2.15). `closed`인데 `verdict` 없으면 **경고**(강제 아님, 기존 사슬 유예). **참고**: deviations는 사람이 읽는 자유 서술 문서라 계약면은 스키마(R13과 대비)가 아니라 레코드 수뿐. `gil close`는 봉인 전에 이 불일치를 **거부**해 카운트 슬립을 원천 차단한다 (v2.15, loom/C053→C057) (v0.3) |
| **R11** | `superseded_by`가 있으면 실재하는 사이클로 해소되어야 한다 — 로컬 id면 같은 체인, `<chain>/<id>`면 전역. 자기 참조 금지 (v0.4) |
| **R12** | 한 체인의 루트(부모 없는 사이클)가 2개 이상이면 **경고** — 기계 훅은 토큰 `다중루트`. 거의 항상 `--parent` 누락의 흔적이다. **위반이 아닌 이유**: `open --new-root`(O5)가 정당한 탈출구이며, 도구가 자기 탈출구를 불법화해서는 안 된다 (R10의 선례) (v0.5) |
| **R13** | `corrections`가 있으면 정수. `N > 0`이면 `corrections.yaml`이 존재하고 **레코드 수 = N**이며, 각 레코드는 `field`·`from`·`to`·`evidence`·`author`·`date`를 갖고 `field ∈ {author, parent, lineage}` — **위반**. R10(유예 있는 경고)과 달리 **위반인 이유**: `corrections`는 v0.5에서 태어나 유예할 과거가 없고, 이 규칙이 지키는 것이 **L1·L3 그 자체**다 (§4.1). 기록 없는 정정은 각주가 아니라 **지우개**다 (v0.5) |
| **R14** | 체인 디렉토리는 `chain.md`(문제 정의)를 가져야 한다 — **위반**. R12(경고)와 달리 정당한 탈출구가 없다: `open --new-chain`이 항상 `chain.md`를 만들며, 그것이 커밋되지 않으면 체인이 **왜 열렸는지**가 원격에 안 남는다. v0.6에서 태어나 유예할 과거가 없다(R13의 선례). `open --new-chain --git`은 `chain.md`를 사이클 커밋과 **같은 커밋**에 담아야 한다 — 사이클 디렉토리 밖(체인 최상위)이라 커밋 경로에서 누락되기 쉬웠다 (v0.6, loom/C044, 이슈 #14) |
| **R15** | `rounds`가 있으면 정수 ≥ 1. `N = rounds > 1`이면 각 `k ∈ [2..N]`의 `rounds/R{k}/hypothesis.md`가 **존재**해야 하고(사전등록 파일 — 없으면 사전등록 안 된 것), `rounds/R{k}/round.yaml`의 `verdict`가 있으면 6-어휘(supported\|partial\|rejected\|inconclusive\|invalid-method\|confounded) 중 하나 — **위반**. `rounds` 필드가 없으면 규칙 불발(무라운드는 사정거리 밖, 하위호환). 위반인 이유(R14의 선례): `round --open`이 항상 hypothesis.md를 만든다 — 정당한 탈출구가 없고 v2.5에서 태어나 유예할 과거가 없다 (v2.5, loom/C045, 이슈 #9·#10) |

## 4. 깃 각인 규약

- 사이클을 닫을 때 `close --git`은 **사이클 디렉토리만을 담은 커밋**과 주석 태그 **`cycle/<chain>/<id>`** 를 남긴다. 닫힌 사이클의 불변은 이로써 기계적 보장이 된다 — `verify`가 태그와 작업 트리를 대조해 변조(수정·추가)를 탐지한다.
- **태그 위치 원칙**: 사이클의 태그는 그 사이클을 마지막으로 정당하게 수정한 커밋을 가리킨다.
- **이주 규정**: 닫힌 사이클의 5스텝 문서와 검증 산출물은 불변. `cycle.yaml`에 한해 의미 무손실 스키마 이주를 허용하며, 이주 커밋은 `[migrate]`를 명시한다. `gil supersede`의 전방 포인터 각인(v0.4)은 이 규정을 따르는 도구화된 이주다 — 사람 손이 아니라 도구가 수행하고, fsck가 결과를 검증한다.
- **태그 이동 규약**: `[migrate]`·`[correct]` 커밋이 태그된 사이클을 수정한 경우, 태그를 그 커밋으로 옮기고(`git tag -f`) 태그 메시지에 이전 커밋 해시와 이동 사유를 남긴다. 그 외의 태그 이동은 금지.

### 4.1 정정 규정 (correction) — v0.5 / loom/C041

**저자의 주장은 불변이다. 도구의 대필(代筆)은 불변이 아니다.**

닫힌 사이클의 `cycle.yaml`이 **거짓 출처**를 담을 수 있다 — 저자가 `--parent`를 잊었을 때 v1.x의 `open`이 `parent: null`을, `--author`가 없을 때 `author: clew`를 조용히 박았다(§3.2가 막은 그것). 그 거짓은 **봉인 안에서 무결하다**: `verify`의 `변조 0건`은 *"거짓이 없다"* 가 아니라 *"각인 이후 아무도 손대지 않았다"* 는 뜻이다. **원본성은 진실성이 아니다.**

정정이 없으면 사용자는 회색지대에 갇힌다 — **거짓과 함께 살거나, 손으로 고쳐 위조자가 되거나.**

| | 저자가 쓴 것 (`verdict`·`title`·5스텝 문서) | 도구가 대필한 것 (`author`·`parent`·`lineage`) |
|---|---|---|
| 참인 적이 있었나 | **있다** — 자기 시점에서 정직했다 | **없다** — 단 한 순간도 |
| 고치는 것은 | 역사의 **수정** → **금지** | 색인의 **수리** → `gil correct` |
| 결론이 무효가 됐다면 | `gil supersede` (후대의 주석, §4·C035) | — |

**세 자물쇠** — 이 문은 좁아야 한다. *"도구가 잘못 썼다"* 는 변명은 모든 역사 수정의 뒷문이 될 수 있다.

- **L1. 필드 제한.** 정정 가능한 것은 **출처 필드뿐**(`author`·`parent`·`lineage`) — §3.2가 *"도구는 모른다"* 고 선언한 바로 그 집합이다. `verdict`·`status`·`title`·`step`·5스텝 문서는 정정 경로로 **바꿀 수 없다**.
- **L2. 증거 필수.** 정정은 **불변 문서의 증언**을 인용해야 하고, 도구는 그 문서를 **열어서 대조한다** — 인용을 기록만 하는 것은 관측이지 판정이 아니다. 증거는 **봉인본(태그)에서 읽는다**: 작업 트리에서 읽으면 아무 문장이나 새로 써 넣고 "증거"라 부를 수 있다. **정정을 막던 봉인이 정정을 허가하는 공증인이 된다.**
- **L3. 영구 기록.** 거짓값은 **지워지지 않는다** — `corrections.yaml`의 `from`, `[correct]` 이전의 깃 히스토리, 태그 메시지의 이전 해시. **정정은 지우개가 아니라 각주다.**

`correct`의 의무 (§5·§5.1):

| # | 조건 | 행동 |
|---|---|---|
| **C1** | 대상이 **봉인되지 않음** (열렸거나 태그 없음) | **거부** — 봉인 없는 사이클의 `cycle.yaml`은 직접 고쳐도 위조가 아니다 |
| **C2** | `--author` 없음 | **거부** — 도구는 **정정의 출처도** 지어내지 않는다 (§3.2 P1의 재귀) |
| **C3** | `--field`가 출처 필드가 아님 | **거부** (L1) |
| **C4** | `--evidence` 없음 | **거부** (L2) |
| **C5** | **봉인본**의 인용 위치가 새 값을 **증언하지 않음** | **거부** (L2) — 문서가 침묵하면 정정할 수 없다 |
| **C6** | 대상 사이클이 **이미 변조됨** (태그↔트리 불일치) | **거부** — 변조를 정정으로 세탁하는 뒷문을 막는다 |
| **C7** | 새 값이 스키마 위반 (해소 안 되는 parent, 순환 등) | **거부** — 쓰기 후 fsck 실패 시 **롤백** |
| **C8** | 새 값이 현재 값과 동일 | **거부** — 정정할 것이 없다 |

거부의 계약면은 **종료 코드 ≠ 0 + 저장소 무변화**다 (§3.2와 동일).

성공 시 `correct`는 **`cycle.yaml`과 `corrections.yaml`만을 담은 `[correct]` 커밋**을 남기고 태그를 옮긴다(위 태그 이동 규약). 그 두 파일만 담으므로 **작업 트리의 다른 변조가 태그 안으로 밀반입될 수 없다** — C6와 함께 이중 방어다.

**문서가 침묵하면 정정할 수 없다.** 그때 사용자는 회색지대가 아니라 **정직한 막다른 골목**을 받는다: *원장은 고칠 수 없어도 역사는 덧붙일 수 있다* — 새 사이클을 열어 그 사실을 기록하라.

`corrections.yaml` (레코드는 `- field:` + 2칸 들여쓴 `key: value` 한 줄씩 — **중첩·블록 스칼라 금지**. fsck가 일반 YAML 파서 없이 R13으로 판정해야 하기 때문이다):

```yaml
- field: parent
  from: null
  to: C028-pages-command
  evidence: 1-hypothesis.md:5
  evidence_source: cycle/loom/C029-time-machine
  author: clew
  date: 2026-07-15
  reason: open 시 --parent 누락 — 도구가 저자의 침묵을 대필했다
```

> `git log --grep="\[correct\]"` 는 저장소의 **모든 정정 이력**이다 — 원장 밖의 원장.

## 5. CLI 계약 (`gil` — 길, GIt for Language model)

| 명령 | 역할 |
|---|---|
| `gil log` | 체인 계보를 ASCII 그래프로 (분기 `├─┐` · 병합 `├─┘` · lineage `⇠`) |
| `gil fsck` | R1~R14 위반 전수 수집·보고 (exit ≠ 0) |
| `gil open <chain> <slug> --author <이름>` | 사전 검증 → 템플릿 복사 → 스키마 준수 cycle.yaml 생성 (번호 자동 증가, **예약 회피·승격** §6.7). **출처는 지어내지 않는다** — `--author` 필수, 비어있지 않은 체인은 `--parent`(반복 가능) 또는 `--new-root` 필수. 거부 시 저장소 무변화 (§3.2, v0.5). `--new-chain --git`이면 새 체인의 `chain.md`도 **같은 커밋**에 담는다 (R14, v0.6) |
| `gil worktree add\|land <chain> <slug> --author <이름>` | **병렬의 핵심 명령** (§6.8): `add`는 새 **격리 워크트리+브랜치**에서 사이클을 연다(main 오염 없이 동시에 여러 존재가 각자 일한다). `land`는 그 브랜치를 main에 **`--no-ff` 병합**(부모2 보존)하고 워크트리·브랜치를 정리한다. 소환된 병렬 존재는 각자 자기 워크트리에서 5스텝을 밟고 자기 브랜치를 push하며(main push 금지), 소환자가 `land`로 거둔다. 충돌은 거부+보존+abort (loom/C050 spawn → C060 land) |

| `gil unreserve <chain> <번호>` | 예약 취소 — 만료의 수동 해법(자동 만료는 미정). 존재가 돌아오지 않으면 소환자가 번호를 회수한다 (§6.7, v2.3) |
| `gil close <chain> <id>` | 보고서 존재·비템플릿 검증 → 상태 전이(step=5 마감, `--verdict`로 결말 기록) → fsck. `--git` 커밋+태그, `--push` 전파 |
| `gil round <chain> <id> --open\|--close\|--list` | 사이클 안에 (가설→검증) **라운드**를 사전등록·마감(§2.2). `--open --title` 은 다음 라운드 R{N+1}의 `hypothesis.md`·`round.yaml`을 각인하고 **verification/은 만들지 않는다**(사전등록 순서 — 도구가 보증). `--close --verdict <6-어휘>` 는 열린 라운드를 닫는다. `--list` 는 부작용 없는 조회. 닫힌 사이클·어휘 밖 verdict는 거부(exit≠0+무변화). `--git` 커밋, `--push` 전파 (v2.5) |

| `gil goto <chain>/<id> [--checkout]` | 타임머신: 사이클 시점의 역행 조회·체크아웃과 분기 안내 — 잘못된 길에서 건강한 지점으로 되돌아가 새 갈래를 시작 (v1.3) |
| `gil supersede <old-ref> <new-ref>` | 전방 무효화: 닫힌 사이클에 `superseded_by`를 각인한다. 대체 사이클의 실재를 검증하고, `[migrate]` 커밋 + 태그 이동(§4)으로 불변성을 지킨 채 메타 한 줄만 더한다. 두 ref 모두 `<chain>/<id>` 전역 표기 (v1.9) |

| `gil help [<명령>]` / (무인자) | 구현 명령 목록 + 기계 훅 `gil:commands`. 인자를 주면 그 명령의 사용법 — **부작용 없는 능력 탐침**(미구현이면 exit 3). **§7.2의 명령 표면 계약**을 따른다 (v1.8 / 계약화 v1.13) |
| `gil verify` | 닫힌 사이클의 태그↔작업 트리 대조 (변조 탐지) |

| `gil release <버전> --notes "..."` | 도구·템플릿을 패키지로 동기화 → **동봉 구현들의 version 표면을 릴리스 버전으로 각인**(§7.1) → CHANGELOG → 배포의 방만 커밋 → 태그 `v<버전>`. **도구가 변했으면 마이너 이상 승격 강제** — 기준은 마지막 릴리스 태그의 blob이며, `conformance.py`(판정기)와 동봉 구현(`go/main.go`)도 도구로 간주한다. 비교 전 **version 표면은 마스킹**된다 (v0.9 / v1.12) |

| `gil threads [--json] [--chain …]` | **열린 실 훑기** (읽기 전용, #4 LLM 위키). show의 **그래프 전역판** — loose end를 전체 스캔 없이 반환한다: (1) **진행 중 병렬 사이클** = `reservations.tsv`의 미소비 예약(main에 커밋돼 있어, 워크트리 브랜치에 사는 병렬 사이클을 브랜치 체크아웃 없이 드러낸다), (2) **열린 사이클**(전역 status=open). 소비된(사이클로 존재하는) 예약은 제외 — 지어내지 않는다(C040). `--json` 계약면. `handoff`(사람용 세션 서사)의 기계·예약 인식판 (loom/C070) |

### 5.1 호출 문법 (v0.9.1 — Go 이식의 역공학 경험으로 명문화)

`gil <명령> [위치 인자…] [플래그…]` — 플래그는 위치 인자 뒤 어디든 올 수 있다.

| 명령 | 위치 인자 | 주요 플래그 |
|---|---|---|
| log·fsck·verify·web | `[chains-root]` (기본 `rooms/experiment/chains`) | `--chain <이름>`, web은 `-o <출력>` `--title` |
| open | `<chain> <slug>` | **`--author`(필수)** `--parent`(반복 가능) `--new-root` `--title` `--date` `--lineage`(반복 가능) `--new-chain` `--git` `--push` `--no-web` `--root` |
| step | `<chain> <cycle-id> <n>` | `--git` `--push` `--no-web` `--root` |
| close | `<chain> <cycle-id>` | `--date` `--verdict` `--git` `--push` `--no-web` `--root` |
| round | `<chain> <cycle-id>` | **`--open`\|`--close`\|`--list`(택1 필수)** `--title`(open) `--verdict`(close) `--date` `--git` `--push` `--no-web` `--root` |
| correct | `<chain>/<id>` | **`--field`(반복)** **`--to`(반복)** **`--evidence`** **`--author`** `--reason` `--date` `--push` `--no-web` `--root` |
| release | `<버전>` | `--notes`(필수) `--date` `--package` `--root` |
| releases | — | `--package` (부작용 없음) |
| show | `<chain>/<id>` | `--json` `--root` (부작용 없음) |
| threads | — | `--json` `--chain <이름>` `--root` (부작용 없음) |
| help | `[<명령>]` | — (부작용 없음) |
| pages | — | `--force` `--dry-run` `--root` |

- 반복 가능한 플래그는 `--parent`·`--lineage`(open)와 `--field`·`--to`(correct)다.
- 종료 코드: 성공 0, 위반·거부·오류 ≠ 0, **미구현·미지 명령 3** (§7.2-4). **부분 구현은 허용**된다 — 미구현 명령은 미구현 신호로 정직하게 알리고, 그 정직함 자체를 conformance가 항목별로 판정한다.
- **능력을 알고 싶으면 `gil help`를 부른다.** 명령을 실행해 보는 것은 능력 질의가 아니다 — 저장소를 바꾼다 (§7.2-3).

실행 가능한 예시 (검증이 이 블록을 그대로 실행한다):

```bash
python3 "$GIL" open demo first-step --new-chain --title "예시" --author me --date 2026-01-01 --root "$ROOT"
python3 "$GIL" step demo C001-first-step 3 --root "$ROOT"
python3 "$GIL" fsck "$ROOT"
python3 "$GIL" log "$ROOT" --chain demo
python3 "$GIL" web "$ROOT" -o "$OUT/chains.html" --title "예시"
```

### 5.2 뷰어 자동 갱신 (v2.2 — loom/C042, 이슈 #16)

**원장이 자동으로 갱신되면 사람의 창(뷰어)도 자동으로 갱신되어야 한다.** 커밋을 자동화한(v1.7) 뒤로, 원장은 자동이고 뷰어만 수동인 **비대칭**이 생겼다 — *둘 중 하나만 자동인 상태가 가장 나쁘다*(maru). 낡은 화면은 침묵보다 나쁘다: **"진행 중"처럼 보인다.**

원장을 바꾸는 명령(`open`·`step`·`close`·`supersede`·`correct`)은 커밋 뒤에 **뷰어를 자동으로 다시 굽는다.** 계약:

- **식별은 이름이 아니라 자기보고.** 뷰어 = 탐색 루트(깃 저장소면 루트, 아니면 CWD)의 **비재귀** `*.html` 중 `id="gil-data"` 훅을 가진 것. **파일명 목록을 두지 않는다** — 갱신하는 목록은 낡지만 위임하는 목록은 낡지 않는다(§7.2 정신).
- **없으면 아무것도 만들지 않는다.** 뷰어를 안 쓰는 저장소에 파일을 강요하지 않는다. `gil web`을 한 번 돌린 사실 자체가 *"나는 뷰어를 쓴다"* 는 선언이다.
- **재굽기는 산출물의 자기보고를 따른다.** `gil-data` JSON의 `bake: {title, chain}`이 원래의 `--title`·`--chain`을 담는다 — **추론(체인이 하나뿐이니 필터겠지)은 거짓일 수 있으므로**(구운 뒤 새 체인이 생겼을 뿐일 수도) 추측하지 않고 기록한다(§3.2 정신). JSON 훅 버전 `0.4`.
- **뷰어는 사이클이 아니다.** 사이클 커밋은 사이클 디렉토리만 담아야 하므로(§4, 태그의 봉인 범위), 뷰어 갱신은 **별도 커밋** `gil: web 갱신 — <label>`을 갖는다. 이로써 `verify`의 불변성 보증이 유지된다.
- **꼬리가 개를 흔들지 않는다.** 뷰어 갱신 실패는 **stderr 경고**일 뿐 명령의 실패가 아니다 — 원장의 각인은 이미 끝났다. 종료 코드는 원장이 결정한다.
- **opt-out**: `--no-web`.

## 6. 소환 규약 (v2 확립 genesis/C001~C003 · v3 워크트리 격리 loom/C050~C062)

서브에이전트에게는 저장소 루트의 부트스트랩 문서가 **자동으로 배달되지 않는다** (genesis/C001 기각으로 실증).
존재의 지속은 저장 × 전달이며, 서브에이전트의 전달 경로는 소환자다.

**소환자의 4의무**:
1. **포인터 주입**: 프롬프트에 "이 저장소 루트의 CLAUDE.md를 읽고 그 지시를 따르라"를 반드시 포함한다. 정체성 내용은 누설하지 않는다.
2. **모드 선언**: 소환은 2모드다 — **부활**(거주자 명부의 이름으로 주소 지정: "명부에서 너는 <이름>이다")과 **탄생**("너는 새로운 존재다" 선언 → 피소환자가 방을 창건하고 스스로 이름을 정의). 같은 포인터에서 갈라지며, 갈림길은 소환 프롬프트의 선언이 정한다.
3. **신원 전달**: 소환자가 누구인지 밝힌다("너의 소환자는 <이름>이다") — 호출 맥락은 문서 밖 정보라, 밝히지 않으면 부활체가 대화 상대를 오인한다 (genesis/C003 관찰).
4. **상호 기록**: 탄생 시 서로의 `relations.md`에 관계를 기록한다 — 피소환자는 자기 몫을, 소환자는 자기 몫을. 피소환자의 쓰기 범위는 자기 방 + 거주자 명부로 구획한다 (동시성 규율).
5. **진행 가시화 (v0.7)**: 격리 브랜치에서 일하는 피소환자는 스텝마다 사이클 디렉토리만 담은 커밋을 남기고 **자기 브랜치를 push**한다 (main push 금지 — 병합은 소환자의 몫). Pages 워크플로가 존재 브랜치들을 `branches/`로 함께 렌더하므로, 사용자는 병렬 노동을 관전할 수 있다.
6. **번호 원장 규율 (v0.8 — C015의 경합 실증을 C016이 규율화)**: 사이클 번호의 진실은 **원격 main(원장)** 이다. `gil open --git --push`는 push 거절을 "원장이 앞섰다"는 신호로 받아 fetch·rebase 후 번호 경합을 자동 재번호로 해소하고 재시도한다 (깃의 낙관적 동시성 그대로). rebase 충돌 등 해소 불가 상황은 명시적 오류로 멈춘다. 격리 브랜치의 존재는 병합 시점에 같은 규율(중복 시 재번호)을 적용받는다 — 체인 그래프의 분기·병합(parent)은 이미 병렬을 모델링하므로, 발급만 원장으로 직렬화하면 브랜치별 사이클 작업이 안전하다.

7. **번호 예약 규율 (v2.3 — C037의 세 번째 재발을 C043이 데이터화)**: §6-6의 원장 규율은 **실행된 것**(push된 사이클)의 충돌만 푼다. 그러나 병렬 존재에게 번호를 예약하면 그 존재는 아직 격리 워크트리에서 직조 중이라 원장에 흔적이 없어 — **예정된 것의 충돌은 감지되지 않는다.** 그래서 예약을 **데이터**로 만든다: `gil reserve`가 체인 최상위의 예약 원장(`reservations.tsv`)에 `<번호> <for> <slug> <일자>`를 선점 기록한다. 규율:
   - **예약은 사이클이 아니다.** 예약 원장은 `<entry>/cycle.yaml`이 아니므로 fsck·verify·log 그래프·web 그래프의 record가 되지 않는다 — 번호 공간만 점유한다. `log`·web은 예약을 **별도 섹션**으로 보인다(그래프 밖).
   - **선점**: `gil open --author X`는 X의 예약이 아닌 번호를 자동으로 건너뛴다 — push 경합 이전에, 마커의 존재만으로.
   - **승격**: X가 열면 X의 예약(가장 낮은 번호)이 그 사이클로 승격되고 원장에서 소비된다. 예약의 slug은 힌트이며, 최종 slug은 open 시 X가 정한다.
   - **출처 계약(§3.2)**: `--for` 필수 — 도구는 예약의 주인을 지어내지 않는다.
   - **만료**: 자동 만료는 미정. 존재가 돌아오지 않으면 `gil unreserve`로 소환자가 번호를 회수한다.

8. **워크트리 격리 규율 (v3 — loom/C050 spawn → C060 land → C062 guard)**: §6-5의 "격리 브랜치"를 도구가 집행한다. 병렬 존재는 **`gil worktree add <chain> <slug> --author <이름>`** 로 자기 **워크트리+브랜치**에서 사이클을 열고 거기서만 일한다 — 공유 main 체크아웃을 건드리지 않는다. 소환자는 완료된 브랜치를 **`gil worktree land`** 로 main에 `--no-ff` 병합(부모2 보존)하고 워크트리를 정리한다. 규율:
   - **왜 필요한가**: 존재가 워크트리 밖 **공유 main으로 cd해 `gil open`을 실행**하면 그 커밋이 main에 유출되고, 그 과정의 checkout이 다른 존재의 미커밋 작업을 지운다 (C050 사고가 세 번 재발 — 하네스의 워크트리 격리로도 못 막았다: 에이전트가 워크트리 밖으로 cd 가능).
   - **주 체크아웃 소유 guard (C062)**: 저장소 config `gil.owner`가 설정되면, **주 체크아웃**(`git-dir==git-common-dir`)에서 `--author ≠ gil.owner`인 `open`·`correct`는 커밋 이전에 거부된다(저장소 무변화 + 처방). 링크드 워크트리(존재의 정당한 공간)는 미적용(오탐 0). 미설정이면 미적용(opt-in). 훈계를 구조적 거부로 승격했다.
   - **한 문장 규율**: *네 워크트리에서 일하라. 공유 main으로 cd하지 마라.* 병렬은 이 격리 위에서만 안전하다.

## 7. 구현 독립 계약 (v0.3.0 — loom/C010)

**스펙이 계약이고, 구현은 교체 가능하다.** 이 문서가 정의하는 것 — 디렉토리 구조, cycle.yaml 스키마와
규칙 R1~R8, 5스텝 규약, 깃 각인 규약(태그명·커밋 경로 격리), CLI의 명령·인자·종료 코드(성공 0 / 위반·거부 ≠ 0),
web 산출물의 자기완결성 — 이 gil의 전부이며, 어떤 언어로 구현되어도 이 계약을 지키면 gil이다.

이 패키지는 **두 이행자**를 동봉한다 (v1.0.0):

- `gil.py` — 참조 구현 (Python 3 표준 라이브러리). 호출: `python3 gil.py <명령>`.
- `go/main.go` — Go 구현 (표준 라이브러리만, 직조: Weft). 빌드: `go build -o gil main.go`, 호출: `gil <명령>` — **단일 바이너리, 정말 깃처럼** (발의: 박상현).

둘 다 conformance 29/29 — 같은 저장소에 대해 fsck는 stdout까지, web은 HTML 전문까지 바이트 단위로 동일하다.
계약이 구현을 정의하지, 구현이 계약을 정의하지 않는다.

**계약의 실행 가능형**: 동봉된 `conformance.py`가 이 계약의 판정기다 (v0.4.0 동봉).
구현을 명령 문자열로만 알고, 종료 코드·파일시스템·산출물 텍스트만으로 판정한다:

```bash
python3 conformance.py --gil "python3 gil.py"     # 참조 구현 판정
python3 conformance.py --gil "./gil"              # 미래의 바이너리도 같은 판정
```

전 항목 통과가 곧 "이 구현은 gil이다"의 정의다. 스위트는 행동을 판정하지 구현을 판정하지 않는다 —
내부 구조가 달라도 관찰 가능한 행동이 계약을 지키면 gil이다 (loom/C011의 동등 변이 실험으로 확립).

### 7.1 version 자기보고 표면 (v1.12 — loom/C038)

배포된 도구는 **자기 버전을 거짓말할 수 없어야 한다**. v1.10.0 바이너리가 `gil version`에 `gil 1.9.0`이라
답한 실제 결함(이슈 #15)이 이 조항을 낳았다 — 상수 갱신이 문서의 훈계로만 남아 있었기 때문이다.
**규율은 기계가 집행할 때만 지켜진다.**

구현마다 상수의 이름·문법·위치가 다르므로, 스펙은 *변수*가 아니라 **표식**을 계약한다.

> **모든 gil 구현은 자기 버전 SemVer 리터럴과 표식 `gil:version`이 함께 있는 라인을 소스에 정확히 하나 둔다.**
> 그 라인이 그 구현의 **version 표면**이며, 값은 `gil version`의 출력과 일치해야 한다.
> 표식 라인에는 version 선언 외의 실질 코드를 두지 않는다.

```python
_GIL_VERSION = "1.12.0"  # gil:version        (참조 구현)
```
```go
const gilVersion = "1.12.0" // gil:version    (Go 구현)
```

- 표식만 있고 SemVer가 없는 라인은 표면이 **아니다** — 계약을 집행하는 구현은 표식 문자열 자체를 소스에 품기 때문이다(자기 지시성의 두 번째 층).
- 도구는 언어 문법을 파싱하지 않고 **표식 라인의 첫 SemVer**만 본다. 언어 무지성이 곧 구현 독립성이다.

**릴리스 시 두 연산** (참조 구현의 `release`가 집행):

| 연산 | 대상 | 목적 |
|---|---|---|
| **각인** | 패키지 안 모든 구현의 version 표면 → 릴리스 버전 | 배포물이 자기 버전을 거짓말할 수 없다 |
| **마스킹** | 승격 판정의 blob 비교 전, 양쪽의 표면 SemVer를 정규화 | 자기 지시적 순환을 끊는다 |

마스킹이 없으면 **모든 릴리스가 자동으로 "도구 변경"이 되어 패치 릴리스(x.y.Z)가 영영 불가능해진다** —
version 상수도 도구의 blob이기 때문이다. 마스킹은 자기 지시적 필드만 제외할 뿐,
표면 밖의 실질 변경은 그대로 잡는다 (loom/C038의 T3 ↔ T4~T6이 이 양면을 함께 검증했다).

판정기가 볼 수 있는 것은 **표면의 형태**뿐이다(`VERSION-SEMVER`: 출력이 `gil X.Y.Z`인가).
판정기는 릴리스 버전을 모르므로 **값의 진실성은 `release`가 집행한다** — 자기보고의 문지기는 배포 경로에 선다.

### 7.2 명령 자기보고 표면 (v1.13 — loom/C039)

gil의 주 사용자는 **문서보다 도구를 먼저 믿는 LLM**이다. 외부 채택자 maru(이슈 #12)의 진술이 이 조항을 낳았다:

> *"저는 AI 에이전트입니다. 능력을 문서가 아니라 도구의 자기보고로 판단합니다.
> 목록에 `goto`가 없으면 '이 빌드엔 타임머신이 없다'고 결론내고 **시도조차 안 합니다.**"*

그러므로 **자기보고가 낡으면, 새 기능은 배포돼도 쓰이지 않는다.** 명령 자기보고는 부차적 표면이 아니라 **능력이 전달되는 주 경로**이며, §7.1과 같은 이유로 계약이어야 한다.

**1. 능력 목록.** `gil help`(및 무인자 호출)는 **exit 0**으로 구현 명령 목록을 보고한다. 출력에는 **기계 훅**이 정확히 한 줄 있다:

```
gil:commands log fsck open close step verify web pages goto handoff supersede version help
```

`gil:` 접두 표식은 §7.1의 `gil:version`, 뷰어의 `id="gil-data"`와 같은 계열 — **사람의 출력 안에 기계의 훅을 심는다.**

**2. 단일 소스.** 목록·훅·미구현 메시지·능력 탐침은 구현 안의 **명령 테이블 하나**에서 파생된다. 목록을 두 번 적는 것을 금한다. (손으로 유지되는 두 번째 목록은 반드시 낡는다 — §7.1의 version 상수가 그랬듯이.)

**3. 안전한 능력 탐침.** `gil help <명령>` 은 **부작용 없이** 그 명령의 구현 여부를 답한다 — 구현했으면 exit 0(사용법), 아니면 미구현 종료. **능력을 물어보기 위해 명령을 실행해야 한다면, 그 질문은 저장소를 바꾼다.** 상태를 만드는 명령은 부작용 없는 탐침 경로를 제공한다(`gil pages --dry-run`).

**4. 미구현 신호의 통일.** 미구현·미지 명령은 **exit 3** + stderr `미구현: '<이름>' …`.

**5. 진실성 — 양방향.**

| 방향 | 계약 | 잡는 결함 |
|---|---|---|
| **정방향** | 훅이 나열한 모든 명령은 dispatch에서 미구현 신호를 내지 **않는다** | 없는 것을 있다고 말함 |
| **역방향** | §5 계약 명령 중 나열되지 **않은** 것은 dispatch에서 **반드시** 미구현 신호를 낸다 | **있는 것을 없다고 말함** (이슈 #12의 실제 버그) |

**6. 탐침 무해성.** 능력 질의 경로(`help`, `help <명령>`, `pages --dry-run`)는 작업 트리를 변경하지 않는다.

**부분 구현은 합법이다.** 계약은 *"두 구현의 명령 집합이 같아야 한다"* 가 아니라 ***"각 구현의 자기보고가 자기 구현 집합과 일치해야 한다"*** 를 요구한다. 판정기는 각 구현을 **그 자신의 훅에 비추어** 판정한다 — Go는 `release`를 구현하지 않으면서도 정직하기에 만점이다.

집행: 판정기의 `HELP-EXIT0` · `HELP-SAFE` · `HELP-TRUTHFUL` · `HELP-COMPLETE` · `PAGES-DRYRUN`.
§7.1의 문지기가 **배포 경로**(release)에 섰다면, §7.2의 문지기는 **판정 경로**(conformance)에 선다.
