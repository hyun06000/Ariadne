# Ariadne Spec — Release

## v2.50.0 (2026-07-21) — 배포 축 완성: 실험과 별개인 배포 관리, 두 몸 한 계약으로 (loom/C101~C106)

상현님 관찰 **"우리는 실험 관리에 매몰됐지 배포 관리는 전혀 신경 안 썼다 — 들어오는 이슈들이 다 그거다."** gil이 지금껏 다룬 배포(`gil release`)는 **도구 자신의 버전**이었고, 필드가 요구한 건 **사용자 산출물(모델·서빙·앱)의 배포 관리**였다. 이 릴리스는 그 별개의 축 `gil deploy`를 처음부터 끝까지 세우고, 참조·Go 두 몸에서 계약으로 완성한다. **첫 배포(ariadne-deploy 1.0.0)를 바로 이 배포축으로 찍어 자기증명했다.**

- **배포 축 골격 (Selvage, loom/C101)**: `gil deploy cut/list/current/rollback` — 도구 릴리스(`gil release`)와 **완전히 별개 네임스페이스**(명령 `deploy`·태그 `deploy/<...>`·레지스터 `deployments.json`). cut=닫힌 사이클을 배포로 승격+supersede+status(live/superseded/rolled-back), rollback=전이(과거 안 지움). fsck R16(소스=닫힌 사이클·rejected 불가)·R17(체인/아티팩트당 live 1개 불변식).
- **아티팩트 축 재정렬 (Clew, loom/C102, 이슈 #27)**: 배포 단위를 chain 키에서 **artifact 키**로 이동 — 한 체인에서 여러 산출물을 배포한다. `deploy cut <artifact> <semver> --cycle <chain>/<id>…(복수) --kind(api-spec|app-code|model) --target --notes`, artifact당 live 1(R17). 실험 계보(chain)와 배포 단위(artifact)를 `source_cycles`가 잇는다 — **틀린 건 축이었지 골격이 아니었다**(불변식 전부 그대로, 키만 이동).
- **Go parity — deploy 명령군 (Weft, loom/C103)**: `gil deploy` 전체를 Go로. `deployments.json` 참조↔Go **바이트 동일**, R16 fsck 메시지도. 미달 1항목(DEPLOY-NAMESPACE)이 deploy 결함이 아니라 판정기가 releases 축과 결합된 결과임을 정직히 진단·이월(C105에서 회수).
- **뷰어 배포 패널 (Sheen, loom/C104, 이슈 #18)**: `gil web`에 배포 계보 패널(`.deployments`) — 도구 릴리스 패널(`.releases`)과 **별 카드·별 gil-data 키**. 아티팩트별 계보·status·근거 사이클 링크(`#cycdoc` 앵커). CHANGELOG 없는 샌드박스에서 릴리스 카드는 사라지고 배포 카드는 서는 것으로 **두 축 분리를 한 렌더로 실증**.
- **판정기 미달 2항목 마감 (loom/C105·C106)**: `releases`를 Go에 이식(Weft, C036 이래 마지막 미이식 명령군)해 **DEPLOY-NAMESPACE PASS** + RELEASE-LIST 활성; 배포 패널을 Go web 렌더에 이식(Sheen)해 **WEB-DEPLOYMENTS PASS**. 두 델타 hunk 무충돌(명령 로직 ↔ web 렌더). 참조↔Go web·releases 출력 바이트 동일.
- 참조 **134/134**·Go **117/117** "이 구현은 gil이다". 회귀 0. **네 존재(Selvage 골격·Clew 아티팩트·Weft Go·Sheen 뷰어)가 지은 배포축을, 바로 그 배포축으로 배포한 첫 산출물이 ariadne-deploy 1.0.0.**

**정직한 이월**: (D) #26(GitHub Release 미발행, 별개 인프라) · (E) 태그↔json drift 게이트·kind 강검증 · `worktree land`의 번호 충돌 자동 재번호(이번에도 C105 경합을 수동 재번호로 해소 — C016 규율을 land에 이식하는 게 근본).

## v2.48.0 (2026-07-20) — 폴링 상태보존 결함 수정: detKey idx=-1 뭉갬 (loomlight/C011)

C010 라이브 폴링이 남긴 필드 결함("자꾸 열었던 카드가 닫힌다")을 근본에서 고쳤다. 원인은 `detKey`의 버그: **id를 가진 `<details>`(체인 카드 `#chain-*`·마운트 `#cycdoc-*`)가 자기 자신을 scope로 삼아** `getElementsByTagName`에서 빠져 순번이 `idx=-1`로 뭉개졌다 — 모든 최상위 카드 키가 `<id>#-1`로 겹쳐, 데이터가 바뀌어 카드 순서가 흔들리면 열림 복원이 실패했다.

- **실측**: 헤드리스 CDP로 데이터 변경 폴링(그래프 구조 변화)에서 카드·상세 보존을 id 추적으로 확인(참조·Go 모두 `POST_loomOpen:true`). C010이 검증 못 한 "데이터 변경 시나리오" 공백을 메웠다.

## v2.46.0 (2026-07-20) — 미완 step 사이클의 rejected close (loom/C098)

C097(open 부모-닫힘 게이트)의 짝. C097이 잘못된 open을 **예방**했다면, C098은 정당히 죽일 미완 가지를 그래프에 **각인**하는 사후 기제다. 이번 세션 심야에 C095(open, 1/5)를 죽일 때 `gil close`가 닫힘=step5를 강제해 "5/5 형식 진행 후 rejected close"라는 **우회**를 써야 했고, step 필드가 "1/5에서 죽음" 대신 "5/5"라는 거짓을 담았다 — 그 근본을 없앴다.

- **rejected close 경로 (참조·Go)**: `close --verdict rejected`는 세 강제를 완화한다. ① **step 보존** — `step: 5` 덮어쓰기를 건너뛰어 죽은 시점 step이 그대로 남는다("1/5에서 죽음"이 그래프의 진실). ② **5-report 완화** — 강제 대신 마지막 스텝 문서가 실질 내용을 가질 것(`_step_written` 재사용, C090의 판정기). **죽음도 왜 죽었는지는 남긴다.** ③ **fsck R9 예외** — "닫힌 사이클의 step은 5, 단 verdict=rejected면 1~5 허용". 완화는 **오직 rejected에만** — supported/partial/inconclusive는 완주 강제 유지.
- **conformance**: `CLOSE-REJECTED-INCOMPLETE`(미완 rejected close+step 보존)·`CLOSE-REJECTED-NEEDS-REASON`(스텁만이면 거부)·`CLOSE-NORMAL-STILL-STRICT`(완화는 rejected 전용) 3항목 신설.

## v2.45.0 (2026-07-20) — open 부모-닫힘 게이트 (loom/C097)

이번 세션 심야에 죽은 C095·C096(rejected, "열린 부모의 자식" 절차 위반)의 **근본 결함을 직접 고쳤다.** `gil open`이 부모의 **존재**(R6)만 보고 **닫힘**은 안 봐서, 열린 사이클 위에 자식을 여는 것을 허용했다 — 상현님 원칙: *"부모 안 닫고 자식 여는 건 커밋 없이 새 버전 여는 것 — 분기할 시점이지 같은 가지에서 자라면 안 된다."*

- **부모 닫힘 게이트 (참조·Go)**: `open`의 parent 검증 루프에서 각 `--parent`의 status가 `"closed"`가 아니면 저장소 무변화로 거부. `"closed"`만 화이트리스트 → **open은 물론 rejected(죽은 가지)에서 자라는 것도 막는다.** 사전 검증 블록 안이라 거부가 원자적(부분 생성물 0). 에러는 원인(어느 부모·현 status)과 대안 둘(부모를 닫아라 / 닫힌 사이클로 분기하라)을 함께 준다(gateway/C003 "금지는 대안과 함께"). 참조↔Go 메시지 바이트 동일.
- **conformance**: `OPEN-PARENT-CLOSED-GATE`(열린 부모 자식 거부+무변화)·`OPEN-PARENT-CLOSED-OK`(닫힌 부모 자식 성공, 과잉거부 방어) 2항목 신설. 계약이 강해져 기존 OPEN-INCREMENT·GUARD-RESERVED-OK가 열린-부모 전제를 정정(C094 "계약 바꾸면 그 검증 테스트도 바뀐다").
- 참조 **125/125**·Go **107/107** "이 구현은 gil이다". 회귀 0.

**정직한 후속**: (A2) 미완 step 사이클의 rejected close(R9 계약 변경, 별도 사이클) — 이번 게이트는 잘못된 open의 *예방*, A2는 정당히 죽일 미완 가지를 그래프에 남기는 *사후 기제*.

## v2.41.0 (2026-07-20) — step-by-step을 파일 존재 수준에서 강제 (loom/C090)

상현님: **"사이클은 step-by-step을 강제하는 도구다. 이전 스텝을 수행·커밋하지 않으면 다음으로 못 가야 하고, 뷰어에도 로컬에도 그 경고가, 완수하면 다음 스텝 안내가 떠야 한다."** 기존 강제는 커밋 스코프(C080, step N 커밋은 ≤N 파일만)에만 있고 **파일 존재엔 없었다** — `open`이 5스텝 파일을 전부 `(작성할 것)`로 미리 만들고 `step N`은 순서를 검증하지 않아, "다음 스텝 재료가 미리 나와있는" 상태였다.

- **open은 1스텝만 스캐폴딩**: `1-hypothesis.md`만 만든다(2~5·3-verification은 전이 때 생성). 커스텀 `_template`이 있으면 그 1스텝을 존중.
- **step N 전이 가드**: 스텝 1..N-1이 **실질 작성됨**(스캐폴딩 문구만 남지 않음)을 검증해 미완이면 무변화로 거부한다. `step 5`를 바로 칠 수 없다. 통과 시 스텝 N 파일을 생성하고 **다음 스텝 안내**를 출력("…를 작성하고 `gil step … N+1`", 5스텝이면 `gil close`).
- **뷰어 표시**: 아직 없는 스텝을 "아직 — 이전 스텝을 완수·커밋해야 이 스텝으로 나아간다"로(존재 문서의 "(없음)"과 구별). 로컬·뷰어가 함께 순차를 드러낸다.
- **사전등록 보존**: 파일 *존재*를 강제하되 문서 *내용*은 제약하지 않는다 — 1·2스텝에 kill 조건(판정 기준)을 미리 쓰는 정당한 사전등록은 그대로. 하위호환(기존 5파일 사이클 무영향), STEP-GATE conformance 신설, 회귀 0.

**강제는 여러 층이 맞물려야 완성된다** — 파일 존재(open) + 전이 검증(step) + 표시(뷰어) + 커밋 스코프(C080). dogfooding이 거짓 양성을 즉시 잡았다: 이 도구를 설명하며 `(작성할 것)`을 인용한 문서가 미완으로 오판돼 C090 자신이 step 4에서 거부됐고, 판정을 "본문 실질 줄이 마크 하나뿐일 때만 미완"으로 정교화했다. **후속(정직)**: 지금은 "작성"만 검증하고 "커밋됐는지"는 안 본다 — 상현님 원문의 커밋 절반은 다음 카브.

**진행 전에 그림으로 합의하라** — 두 사이클(C065·C066)을 세로 목록으로 소비한 오해가, 전/후 목업을 나란히 놓자 한 번에 정렬됐다. 시각 산출물이 진짜 합의를 만든다(C037 디자인판). 다음부터 UI 변경은 목업 먼저.

## v2.22.0 (2026-07-19) — 체인 아코디언을 지도 카드 안으로: 제자리 확장 (loom/C066, 발의: 박상현)

상현님 정정 **"체인 노드를 누르면 지도 카드 안에서 다른 노드들이 주르륵 아래로 등장하면 좋겠어."** v2.21은 체인 스트림이 지도 카드 밖 아래 별도 섹션에서 열렸다. 이제 아코디언을 **지도 카드 안**(SVG 아래)으로 넣어, 지도에서 체인을 누르면 그 자리에서 카드가 확장된다.

**무엇을 어디에 두고 어디에 선을 긋는가가 곧 의미다** — 렌더 결과를 실제로 보여주는 검증 루프(헤드리스 스크린샷)가 "지도 아래"와 "지도 카드 안"의 미세한 차이를 빠르게 드러냈다.

## v2.21.0 (2026-07-19) — 펼쳐진 체인을 노드 스트림으로: 카드 걷고 주루룩 (loom/C065, 발의: 박상현)

## v2.20.0 (2026-07-19) — 위계 뷰어 상단에 체인 지도: lineage 한눈에 (loom/C064, 발의: 박상현)

상현님 지적 **"위계는 좋은데 리니지가 한눈에 안 보인다 — 상단에 노드-엣지 그래프를 얹고, 큰 동그라미=체인, 클릭하면 그 체인이 펼쳐지게."** v2.19의 위계(드릴다운) 뷰어에 **L0 체인 지도**를 얹었다. 각 체인을 펼쳐야 보이던 cross-chain lineage를 이제 상단 한 장으로 조망한다.

- **체인 지도 (양 구현)**: 원=체인(크기 ∝ √사이클수, 색=상태 — rejected→빨강·열림→점선 링), 초록 점선 화살표=체인 간 lineage(사이클 lineage를 체인→체인으로 집계, 굵기·숫자 라벨 ∝ 건수). **허브(최다 연결)를 가운데**로 배치해 스포크를 대칭·단축, 엣지 끝점을 원 둘레에 부채꼴로 흩어 화살표 겹침을 줄였다. 원을 누르면 아래 그 체인 `<details>`가 열린다 — **프래그먼트(#chainbody-*)→조상 details 자동 오픈**(loomlight/C002 메커니즘 재사용, JS 0줄 유지).
- **삼각함수 없는 좌표 = 바이트 parity의 조건**: 끝점 부채꼴을 `atan2/cos/sin`으로 내면 IEEE correctly-rounded 미보장이라 Python libm↔Go math가 마지막 ULP에서 갈려 `%.1f`가 어긋난다. **벡터 정규화(sqrt·사칙연산만)** 로 같은 기하를 표현 — sqrt는 IEEE 보장이라 Python `math.sqrt`↔Go `math.Sqrt`가 비트 동일. 결과: 무옵션(위계+지도)·`--flat`(평면) 모두 두 구현 **byte-identical**.
- **렌더는 계약이 아니지만 parity는 계약이다**: 지도는 `gil-data` JSON을 안 바꿔 conformance 무수정 통과 — 참조 **90/90**·Go **83/83** 회귀 0. 잠글 것은 렌더 구조가 아니라 바이트 일치라, 판정기를 늘리지 않고 parity 대조를 검증 산출물로 남겼다(C063 계열).

**개관과 드릴다운은 층으로 공존한다** — L0 지도(lineage 한눈에)→L1 목록→L2 그래프→L3 5스텝, 각 층이 다음의 입구이고 JS 없이 `<details>`+프래그먼트로만 잇는다.

## v2.8.0 (2026-07-15) — 새로고침 없는 실시간 관찰 (loom/C049, 발의: 박상현)

병렬 존재(워크트리)의 진행을 보려면 매번 브라우저를 새로고침해야 했다. 파일 갱신(C042 자동 재굽기, gil step마다)은 이미 있었으나, **브라우저 반영**이 빠진 조각이었다.

- **`gil web --refresh N`** (양 구현): 뷰어 `<head>`에 `<meta http-equiv="refresh" content="N">`을 넣는다 — JS 아닌 HTML 표준(자기완결 계약 유지, 외부 리소스 0)으로 N초마다 리로드. JSON `bake`에 `"refresh": N`을 **있을 때만** 기록.
- **자동 재굽기가 refresh를 보존한다** (핵심): `_bake_meta`가 bake에서 refresh를 읽어 재굽기에 넘긴다. 그래서 `gil step`이 원장을 바꿀 때마다 파일이 갱신되고 meta refresh가 브라우저를 리로드 = **새로고침 없는 실시간**. bake가 없으면 두 번째 리로드부터 meta를 잃어 죽는다("산출물이 자기 생성 조건을 말한다" — C042가 refresh에도 참).
- **`gil web --watch [--interval N]`** (양 구현): 원장 mtime을 감시해 변경 시 재생성, `--refresh` 함축. gil을 안 거치는 외부 변경(병합·pull)도 반영. 장기 실행이라 conformance는 `--refresh`(WEB-REFRESH)만 판정.
- 판정기 WEB-REFRESH 신설(참조 73/73·Go 65/65 회귀 0), 두 구현 web 바이트 동일(--refresh 유/무), 변이 격추. `--refresh` 없으면 개선 전과 바이트 동일(하위호환).

**JS 0줄 계약이 설계를 정제했다** — 폴링/WebSocket 대신 meta refresh라는 더 단순한 답으로.

## v2.6.0 (2026-07-15) — 토폴로지 레이아웃: 형제 갈래를 같은 높이에 (loom/C047, 발의: 박상현)

뷰어(`gil web`)의 그래프가 세로로 너무 길었다. 새 가지가 생기면 이전 가지의 맨 끝보다 더 **아래**에 노드가 잡혔기 때문 — 노드의 세로 위치(row)가 **토폴로지 순서(order 인덱스)** 였다. 그러나 형제 갈래는 서로 영향을 주지 않는다(부모만 공유, 지식 전수는 lineage로). 시간 순서로 세로에 누적할 이유가 없다.

- **row = 깊이(루트로부터 최장 경로)** 로 바꿨다(양 구현). 형제 갈래는 같은 깊이=같은 높이에 나란히 놓여, **세로가 노드 총수가 아니라 최장 경로로 압축**된다. 병합은 `max(부모 깊이)+1`.
- **선형 체인은 완전 불변**(depth = order 인덱스) — 하위호환. genesis(선형)는 개선 전/후 web 바이트 동일.
- **캔버스 크기도 함께 고쳤다**: `_render_svg`의 `max_rows`가 `len(order)`에 고정돼 있어 레이아웃만 바꾸면 노드는 겹쳐 올라가고 캔버스만 길게 남았을 것 — 실제 최대 깊이로 둘 다 바꿔야 압축이 완성됐다.
- 참조·Go **web 바이트 동일**(실 저장소·분기 픽스처), conformance 참조 72/72·Go 64/64 회귀 0. SVG 좌표는 §3.1 계약면이 아니라 정확성은 두 구현 cross-check + height 실측으로 보증(분기 픽스처 386→322px, loom 3074→2498px).

**압축의 값은 분기 밀도에 비례한다** — 형제 갈래가 잦은 저장소일수록 극적이다.

## v2.5.0 (2026-07-15) — 라운드를 1급 시민으로: 검증 안의 반복을 사전등록 데이터로 (이슈 #9·#10, loom/C045)

5스텝은 **한 번의 가설, 한 번의 검증**을 전제한다. 그러나 실제 연구의 검증 스텝 안에서는 완결된 (가설→검증) 루프가 통째로 일어난다 — *검증 → 이상 발견 → 방법 수정 → 재검증 → 결론 뒤집힘*. gil에 그것을 사전등록할 자리가 없어, 사용자(maru)는 머릿속에서 반복하고 `4-analysis`에 산문으로 사후 고백했다. **산문 고백은 사전등록이 아니다** — 문서화되지 않은 반복은 사전등록된 검증과 구별되지 않는다.

**사전등록의 단위를 사이클이 아니라 라운드로 내린다** (구조: 문제정의 → (가설→검증)ⁿ → 보고·회고):

- **사전등록은 fsck가 아니라 도구가 보증한다.** maru는 "hypothesis가 verification보다 먼저 커밋"을 fsck로 강제하자 했으나, fsck는 깃 무의존(C007)이라 커밋 순서를 못 본다. 답은 도구였다: `round --open`이 hypothesis만 먼저 각인하고 `verification/`은 만들지 않는다. **사전등록의 증거는 도구가 만든 깃 히스토리**다 — C013 "열 때부터 보이게"의 라운드판.
- **라운드 verdict 6-어휘** — 사이클 4-어휘에 **`invalid-method`**(방법 자체가 무효)와 **`confounded`**(교란으로 결론 불가)를 더한다. maru의 R1 반증(CV 함정, 일주기 공선성)은 "가설이 틀림(rejected)"이 아니라 "판정 방법이 무효"였다. C035 "결말과 무효화는 직교한다"의 라운드판.
- **fsck R15 신설** (SPEC §3): `rounds:N(N>1)`이면 `rounds/R{k}/hypothesis.md`가 존재해야 하고 round.yaml verdict가 어휘 안이어야 — **위반**(R14의 선례). `rounds` 없으면 규칙 불발.
- **cycle.yaml `rounds` 필드** — 개수만 담고 상세는 `round.yaml`에 산다(평탄 파서 계약 §3.1 무손상). 파생 가능한 사실(§3.2).
- **"버그 수정" vs "새 가설" 경계 명문화** (SPEC §2.2): 코드가 가설을 검증 못한 것은 버그(중간 커밋), 조작적 정의가 데이터를 본 뒤 바뀐 것은 새 가설(반드시 새 라운드). 도구는 이 경계를 판정 못한다 — 규율이지 강제가 아니다.
- 판정기 64 → **72항목**(라운드 8), 참조 **72/72**, 회귀 0, 변이 M1·M3·M4 격추 + M2 심층방어 생존(M2-both로 계약 판정 증명, C011 재현). 무라운드 web은 Go와 **바이트 동일**(rounds 키는 rounds>1일 때만 — 하위호환 공짜).

## v2.3.0 (2026-07-15) — 예약을 데이터로: 예정된 번호의 충돌을 원장이 푼다 (이슈 #13, loom/C043)

번호 예약이 소환자의 프롬프트에만 있고 **저장소의 데이터가 아니었다.** 그래서 병렬 존재에게 번호를 예약해도, 다른 워크트리의 `gil open`은 그것을 모르고 재발급했다. 세 사이클에 걸쳐 세 번 재발했고 — loom/C037에서는 **소환자 자신(Clew)이** 예약을 침범해 손으로 양보해야 했다:

> C016의 원장 규율(§6-6)은 push 경합, 즉 **실행된 것**의 충돌만 푼다. 예약된 존재는 격리 워크트리에서 아직 push하지 않았으므로 원장에 흔적이 없어 — **예정된 것의 충돌은 감지조차 되지 않았다.**

예약을 **데이터**로 만든다 (SPEC §6.7):

- **`gil reserve <chain> <slug> --for <이름>`** — 체인 최상위 원장 `reservations.tsv`에 `<번호> <for> <slug> <일자>`를 선점 기록. `--for` 필수(§3.2 — 예약 주인도 지어내지 않는다).
- **선점** — `gil open`은 남의 예약 번호를 자동으로 건너뛴다. push 경합 이전에, 마커의 존재만으로. **C037의 버그가 이제 불가능하다.**
- **승격** — 예약자가 열면 예약된 번호로 태어나고 원장에서 소비된다. slug은 open 시 확정.
- **예약은 사이클이 아니다** — 원장은 `<entry>/cycle.yaml`이 아니므로 `load_chain_records`가 안 본다. fsck·verify·그래프에 **예외처리 0줄**, 그럼에도 완전 비침습. **불변식을 구조가 지킨다.** `log`·web은 예약을 별도 섹션으로 보인다.
- **`gil unreserve <chain> <번호>`** — 만료의 수동 해법(자동 만료는 미정).
- 판정기 54 → **62항목**(예약 8), 참조 **62/62**, 변이 **5/5**, 무예약 web은 Go와 **바이트 동일**(예약 키는 있을 때만 JSON에 넣어 파서 계약 보존).

**두 몸, 한 계약**: 예약은 참조 구현만 구현했다(Go는 Weft의 영역). `CONTRACT_COMMANDS`에 예약을 넣어 **HELP-COMPLETE가 Go의 부재를 exit 3으로 판정**한다 — "판정기가 안 보는 계약은 없는 계약이다"(C036)를 이번엔 **예방으로**: 새 표면을 계약에 적는 같은 커밋에서 판정기에도 적었다.

**하위 호환**: 기존 명령·스키마 그대로. 예약을 안 쓰면 아무 변화 없다.

## v2.2.0 (2026-07-15) — 창은 원장을 따른다: 뷰어 자동 갱신 (이슈 #16, loom/C042)

커밋 자동화(v1.7) 이후 **원장은 자동인데 사람의 창만 수동**이었다. maru의 저장소 주인이 겪었다:

> *"사이클 11의 가설을 너가 커밋했는데도 뷰어에 잡히지 않네."*

`gil log`는 정확했고 커밋도 정확했다. 낡은 것은 `chains.html` 하나뿐이었다. **둘 중 하나만 자동인 상태가 가장 나쁘다** — 낡은 화면은 침묵보다 나쁘다("진행 중"처럼 보인다).

원장을 바꾸는 명령(`open`·`step`·`close`·`supersede`·`correct`)이 커밋 뒤 **뷰어를 자동으로 다시 굽는다** (SPEC §5.2):

- **식별은 이름이 아니라 자기보고** — 뷰어 = 탐색 루트의 `id="gil-data"` 훅을 가진 HTML. **파일명 목록을 두지 않는다**(C039). `gil web -o x.html`이므로 이름은 자유이고, 목록은 반드시 낡는다.
- **없으면 아무것도 만들지 않는다** — 뷰어를 안 쓰는 저장소에 파일을 강요하지 않는다. `gil web`을 한 번 돌린 사실이 *"나는 뷰어를 쓴다"* 는 선언이다.
- **추론하지 말고 기록하라** — JSON 훅에 `bake: {title, chain}`을 담아 원래 `--title`·`--chain`을 보존한다. *체인이 하나뿐이니 필터겠지* 라는 추론은 거짓일 수 있어(구운 뒤 새 체인이 생겼을 뿐일 수도) **조용한 버그**가 된다. 추측 대신 기록 (§3.2 정신). 훅 버전 `0.3 → 0.4`(하위 호환).
- **뷰어는 사이클이 아니다** — 사이클 커밋은 사이클 디렉토리만 담아야 태그의 봉인이 정확하다(§4). 뷰어 갱신은 **별도 커밋** `gil: web 갱신 — <label>`. `verify` 무변화.
- **꼬리가 개를 흔들지 않는다** — 뷰어 갱신 실패는 경고이지 명령 실패가 아니다. 원장의 각인은 이미 끝났다.
- **opt-out**: `--no-web`. 판정기 50 → **54항목**, 두 구현 **54/54**, 변이 **4/4**, 산출물 바이트 동일.

부수: 참조·Go 양쪽의 상단 독스트링에서 낡은 명령 목록을 **삭제**(`gil help`에 위임 — C039).

**하위 호환**: 기존 명령·스키마 그대로. 뷰어가 없으면 아무 동작도 없다.

## v2.1.0 (2026-07-15) — 정정 규정: 저자의 주장은 불변이다, 도구의 대필은 아니다 (loom/C041)

v2.0.0이 **미래의 거짓**을 막았다면(예방), 이번 릴리스는 **이미 태어난 거짓**에 정당한 출구를 준다(정정).

v2.0.0 직후 우리 저장소는 이런 상태였다 — `fsck`는 `경고 [다중루트]`로 거짓을 **가리키는데**, 고칠 길이 없었다:

| 선택 | `fsck` | `verify` (CI 게이트) |
|---|---|---|
| 거짓을 그대로 둔다 | 경고 영구 | ✅ `OK — 변조 0건` |
| 손으로 고친다 | 경고 소멸 | ❌ `변조 감지` **exit 1** |

> **진실을 말한 대가로 위조자가 된다.** `변조 0건`은 *"거짓이 없다"* 가 아니라 *"각인 이후 아무도 손대지 않았다"* 는 뜻이었다 — **원본성은 진실성이 아니다.**

**`gil correct`** (SPEC §4.1): 봉인된 사이클의 **출처 필드**(`author`·`parent`·`lineage`)를 **불변 문서가 증언하는 값**으로 수리한다.

```bash
gil correct loom/C029-time-machine --field parent --to C028-pages-command \
  --evidence 1-hypothesis.md:5 --author clew --reason "open 시 --parent 누락"
```

**세 자물쇠** — *"도구가 잘못 썼다"* 는 변명이 역사 수정의 뒷문이 되지 않도록:

- **L1 필드 제한** — 정정 가능한 것은 §3.2가 *"도구는 모른다"* 고 선언한 **출처 필드뿐**. `verdict`·`title`·5스텝 문서는 **저자의 주장**이며 정정 경로로 바꿀 수 없다 (결론이 무효가 됐다면 `gil supersede`).
- **L2 증거 필수** — 도구가 인용된 문서를 **열어서 대조한다**. 증거는 **봉인본(태그)에서 읽는다**: 작업 트리에서 읽으면 아무 문장이나 써 넣고 "증거"라 부를 수 있다. **정정을 막던 봉인이 정정을 허가하는 공증인이 된다.**
- **L3 영구 기록** — 거짓값은 `corrections.yaml`의 `from`, `[correct]` 이전의 깃 히스토리, 태그 메시지의 이전 해시에 **영구히 남는다.** **정정은 지우개가 아니라 각주다.**

- **스키마 v0.5**: `corrections: N` (N>0이면 `corrections.yaml` 필수) + **R13** (레코드 수·필수 키·필드 제한을 fsck가 집행 — 기록 없는 정정은 지우개다)
- **`[correct]` 커밋 등급** 신설 (§4): `[migrate]`(의미 무손실 이주)와 구별한다. `git log --grep="\[correct\]"` 가 저장소의 모든 정정 이력 — **원장 밖의 원장**.
- `log`의 `✎ corrected(N)`, `web`의 `gil-data` JSON `corrections`, `gil help`의 `correct`.
- **판정기 40 → 50항목**, 두 구현 **50/50**, 변이 **6/6 격추**, 산출물 바이트 동일.
- **문서가 침묵하면 정정할 수 없다** — 회색지대가 아니라 **정직한 막다른 골목**을 준다: *원장은 고칠 수 없어도 역사는 덧붙일 수 있다.*

부수: `supersede`의 태그 이동이 v1.9 이래 **판정되지 않고 있었다**(변이 시험이 밟아서 발견) → `SUPERSEDE-TAG-MOVE` 신설. 참조 구현 독스트링의 낡은 명령 목록 **삭제**(`gil help`에 위임 — C039).

**하위 호환**: 기존 명령·스키마 그대로. `corrections`는 선택 필드이며 부재는 합법이다.

## v2.0.0 (2026-07-14) — 도구는 출처를 지어내지 않는다 (이슈 #17) — **하위 호환 파괴**

**⚠ 파괴적 변경**: `gil open`이 이제 `--author`를 **요구**한다. 비어있지 않은 체인에서는 `--parent`(또는 명시적 `--new-root`)도 **요구**한다. 이 플래그 없이 돌던 스크립트는 **멈춘다** — 그것이 이 릴리스의 목적이다. **조용히 거짓을 쓰던 경로가 시끄럽게 멈춘다.**

외부 채택자 maru의 이슈 #17이 이 릴리스를 낳았다. `--parent`와 `--author`를 빠뜨렸더니 도구가 **경고 없이** 이렇게 박았다:

```yaml
parent: null      # ← 체인에 루트가 둘이 된다
author: clew      # ← 그 저장소에 존재하지 않는 다른 존재의 이름
```

> *"다른 존재의 이름이 기본값으로 박혀 있는 것이 특히 나쁘다 — **우연히 그럴듯해 보여서 눈에 안 띈다.** `unknown`이었다면 즉시 잡혔을 것이다."* — maru

**닫힌 사이클은 불변이다(§4).** 그래서 이 거짓은 발견 시점엔 이미 복구 불가다. 경고로는 부족하다 — **태어나지 못하게** 해야 한다.

- **출처 필드 계약** (SPEC §3.2 신설 — loom/C040). 도구가 채우는 필드를 둘로 가른다: **파생 가능한 사실**(번호·`opened`·`status`·`step` — 도구가 계산으로 안다)과 **출처**(`author`·`parent` — **오직 저자만 안다**). `author`는 존재의 이름이지 커밋하는 사람이 아니므로 `git config user.name`조차 틀린 출처이고, 분기가 있으므로 체인 tip 상속조차 추측이다.
  - **P1. 도구는 출처를 지어내지 않는다** — 추측값·고유명사 기본값 금지.
  - **P2. 모르면 거부한다** — 저장소 무변화.
  - **P3. 파생 가능한 사실은 예외** — 빈 체인의 첫 사이클이 루트라는 것은 추측이 아니라 계산이다.
  - 깃이 이미 답을 알고 있었다: `git commit`은 `user.name`이 없으면 **추측하지 않고 거부한다**.
- **`gil open`의 의무 O1~O5**: `--author` 없으면 거부(O1) / 비어있지 않은 체인에서 부모 플래그 없으면 tip을 보여주며 거부(O2) / `--parent`+`--new-root` 동시는 모순이므로 거부(O3) / 빈 체인은 그대로 열림(O4, **딸깍 보존**) / **`--new-root` 신설** — 의도된 두 번째 루트의 정당한 탈출구(O5).
- **R12 — 다중 루트 경고** (기계 훅 토큰 `다중루트`). `fsck`가 이제 한 체인의 루트가 2개 이상이면 알린다. **위반이 아니라 경고인 이유**: `--new-root`가 정당한 탈출구인데 그 결과물이 fsck를 실패시키면 도구가 자기 탈출구를 불법화하는 것이다(R10의 선례). 기존 저장소는 계속 통과한다 — 다만 **더 이상 침묵하지 않는다.**
- **판정기 34 → 40항목** (`OPEN-AUTHOR-REQUIRED`·`OPEN-PARENT-REQUIRED`·`OPEN-ROOT-CONFLICT`·`OPEN-NEW-ROOT`·`OPEN-ROOT-EMPTY-CHAIN`·`FSCK-MULTI-ROOT`). 두 구현 **40/40**, 기존 34항목 **회귀 0**. 변이 **4/4 격추** — 그중 M4는 *과잉 작동* 변이(빈 체인에서도 부모를 요구)로, **정당한 루트를 불법화하는 구현은 반드시 실패해야 한다**는 조항을 지킨다.
- **도구가 생성하는 안내문도 고쳤다**: `gil goto`가 출력하던 분기 안내 `gil open … --parent <id>`에는 `--author`가 없었다 — 에이전트가 **그대로 복사해 실행하면 실패하는 문장**이었다. 사람이 읽는 문서보다 도구가 뱉는 코드가 먼저다.
- **이 저장소도 걸려 있었다**: `loom/C029-time-machine`의 `cycle.yaml`은 `parent: null`인데 그 사이클의 **불변 문서**는 *"부모: loom/C028"* 이라 말한다. `gil log`는 11사이클 동안 `root: C001…, C029…`를 화면에 찍고 있었고 `fsck`는 *위반 0건*이라 답했다. **도구는 이미 보고 있었다. 판정하지 않았을 뿐이다.** 닫힌 사이클의 거짓 출처를 어떻게 정당하게 고치는가는 다음 사이클(정정 규정)의 문제다.
- 근거: loom/C040 (이슈 #17).

## v1.13.0 (2026-07-14) — 명령 표면이 계약이 되다: 도구가 자기 능력을 정직하게 말한다 (이슈 #12)

**에이전트는 능력을 문서가 아니라 도구의 자기보고로 판단한다.** 외부 채택자 maru의 진술이 이 릴리스를 낳았다 — *"목록에 `goto`가 없으면 '이 빌드엔 타임머신이 없다'고 결론내고 시도조차 안 합니다."* 자기보고가 낡으면 **새 기능은 배포돼도 쓰이지 않는다.** v1.12.0이 *version* 표면을 계약했다면, 이 릴리스는 *명령* 표면을 계약한다.

- **명령 자기보고 표면** (SPEC §7.2 신설 — loom/C039): `gil help`는 exit 0으로 능력 목록을 보고하고, 출력에 기계 훅 `gil:commands …`를 **정확히 한 줄** 심는다(§7.1의 `gil:version`, 뷰어의 `id="gil-data"`와 같은 계열 — 사람의 출력 안에 기계의 훅을).
- **단일 소스**: 목록·훅·미구현 메시지·능력 탐침이 구현 안의 **명령 테이블 하나**에서 파생된다. Go에 있던 **세 개의 목록**(테이블 · 하드코딩 문자열 · 매직 인덱스 `[:11]`)을 하나로 접었다. 손으로 유지되는 두 번째 목록은 반드시 낡는다.
- **안전한 능력 탐침**: `gil help <명령>` — 구현했으면 사용법 + exit 0, 아니면 **exit 3**. **부작용 없음.** 능력을 확인하려고 명령을 실행하면 안 된다: `gil pages`는 두드리는 순간 워크플로 파일을 만들었다 → **`gil pages --dry-run`** 신설.
- **양방향 진실성**: 나열한 명령은 실재해야 하고(*없는 것을 있다고 말하지 않기*), 나열하지 **않은** 계약 명령은 미구현 신호를 내야 한다(*있는 것을 없다고 말하지 않기* — 이슈 #12의 실제 버그).
- **미구현 신호의 통일**: exit **3** + `미구현: …`. 지금까지 Go는 3, 참조 구현은 argparse의 2로 **서로 다른 언어로 "없다"고 말하고 있었다.** 아무도 계약으로 적지 않아 아무도 몰랐다.

## v1.12.0 (2026-07-15) — version 표면이 계약이 되다 (이슈 #15)

v1.11.0이 손으로 정정한 것을 **도구가 집행하게** 만든 릴리스다. 자기보고의 문지기가 배포 경로에 선다.

- **version 자기보고 표면** (SPEC §7.1 신설 — loom/C038): 모든 구현은 자기 버전 SemVer와 표식 `gil:version`이 함께 있는 라인을 **정확히 하나** 둔다. 스펙은 *변수*가 아니라 *표식*을 계약한다 — 구현마다 상수의 이름·문법·위치가 다르기 때문이다. 도구는 언어 문법을 파싱하지 않고 표식 라인의 첫 SemVer만 본다. **언어 무지성이 곧 구현 독립성이다.**
- **`gil release`의 두 연산**: ① **각인** — 패키지 안 모든 구현의 version 표면을 릴리스 버전으로 갱신한다. 배포물이 자기 버전을 거짓말할 경로가 사라진다. ② **마스킹** — 승격 판정의 blob 비교 전 양쪽의 표면 SemVer를 정규화한다.
- **왜 마스킹이 필수인가**: 마스킹 없이 각인만 하면 **모든 릴리스가 자동으로 "도구 변경"이 되어 패치 릴리스(x.y.Z)가 영영 불가능해진다**(v1.11.0에서 실제로 부딪힌 순환). 마스킹은 자기 지시적 필드만 제외할 뿐 표면 밖의 실질 변경은 그대로 잡는다 — C038이 T3(표면만 다름 → 패치 **허용**)과 T4~T6(실질 변경 → 패치 **거부**)로 이 양면을 함께 검증했다. 하나만 성립하면 실패인 설계다.
- **승격 규칙의 관측 범위 확장**: `go/main.go`(동봉 구현)도 이제 도구 blob이다. 지금까지 **아무도 안 보던 blob** — Go만 바뀐 릴리스가 패치로 나갈 수 있었다.
- **판정기 28 → 29항목** (`VERSION-SEMVER`): 출력이 `gil X.Y.Z` 형태인가. 판정기는 릴리스 버전을 모르므로 **형태만** 볼 수 있다 — *값의 진실성은 release가 집행한다*. 이 한계를 감추지 않고 계약에 명시했다.
- 두 구현 conformance **29/29**, 픽스처 10/10. 근거: loom/C038.
- **자기 지시성의 두 번째 층**(검증 중 발견): 계약을 집행하는 소스는 표식 문자열 자체를 품는다(`_VERSION_MARK = "gil:version"`). 그래서 계약은 "표식이 있는 라인"이 아니라 **"표식과 SemVer가 함께 있는 라인"** 이다. 첫 실행이 이것을 바로 잡아냈다.

## v1.11.0 (2026-07-15) — 자기보고 정정: 바이너리가 자기 버전을 거짓말했다

v1.10.0으로 배포된 바이너리가 `gil version`에 **`gil 1.9.0`** 이라 답했다. 두 구현의 version 상수(`_GIL_VERSION`, `gilVersion`)를 갱신하지 않은 채 릴리스했기 때문이다.

- **어떻게 잡혔나**: 배포 직후 공개 URL에서 바이너리를 받아 설치 스니펫을 재실행하는 **끝단 실증**(loom/C037의 절차)에서 `gil 1.9.0`이 찍혔다. 문서도 판정기도 이 표면을 보지 않으므로, **수신자 경로를 실제로 밟지 않았다면 다음 릴리스까지 몰랐을 것이다.**
- **왜 일어났나**: v1.8.0(loom/C034)에서 *"version 상수는 릴리스마다 갱신(후속 자동화 여지)"* 라고 적어두고도, **`gil release`가 그것을 강제하지 않는다.** 규율이 문서의 훈계로만 남으면 결국 새어나간다 — 이 저장소가 반복해서 배우는 바로 그것이다.
- 조치: 양 구현 version 상수 → **`1.11.0`** (이 릴리스의 버전). conformance 28/28 ×2 유지.
- **패치가 아니라 마이너인 이유**: `gil release`가 거부했다 — 승격 규칙(loom/C018)은 "도구 blob이 변하면 마이너 이상"이고, version 상수도 도구의 blob이다. 내 편의에 맞춰 규칙을 고치는 대신 규칙에 복종했다 (loom/C002의 교훈: *규칙을 먼저 고정하는 관행은 남이 아니라 나로부터 스펙을 지킨다*). 다만 이 마찰은 진짜 설계 쟁점을 드러낸다 — **release가 version을 자동 갱신하면 모든 릴리스가 "도구 변경"이 되어 패치 릴리스가 영원히 불가능해진다.** 승격 규칙이 자기 지시적 필드(version 상수)를 제외해야 한다. 이슈 #15.
- **근본 해결은 이슈 #15로 연다**: `gil release`가 version 상수와 릴리스 버전의 일치를 **검사(또는 자동 갱신)** 해야 한다. 이슈 #12("바이너리가 자기 기능을 과소보고한다")와 같은 병의 다른 얼굴이다 — *판정기가 안 보는 계약은 없는 계약이다*.

## v1.10.0 (2026-07-15) — 문지기와 판정기: 설치 체크섬 + Go open --git (이슈 #8·#7)

gil이 사이클에 요구하던 규율 두 가지를 **자기 자신에게** 적용한 릴리스다.

- **설치 경로의 체크섬 문지기** (이슈 #8, 결/Gyeol — loom/C037): 설치 스니펫 4곳(README·ko·ai·QUICKSTART)과 **`gil pages`가 생성하는 워크플로**가 SHA256SUMS를 대조한다. 대조 통과를 `chmod +x`의 전제로 두어(`대조 && mv && chmod`), **불일치 시 `gil`이라는 실행 파일 자체가 생기지 않는다** — 검증 안 된 바이너리가 실행될 경로가 구조적으로 없다. `exit`에 의존하지 않아 대화형 셸(세션 유지)과 CI(rc≠0, 배포 중단) 양쪽에서 같은 안전. 릴리스 직후 CDN 지연으로 인한 불일치는 실측된 현상이므로 **재시도 안내**를 함께 넣었다.
  - 이슈는 문서만 지적했으나 전수 grep이 다섯 번째 경로를 찾았다: pages 산출 워크플로는 사용자 CI에서 **매 push마다 무조건 실행되는 코드**다. 훈계와 코드는 다른 층이다.
  - **한계(정직하게)**: SHA256SUMS도 같은 CDN에서 온다 — 이것은 공급망 공격 방어가 아니라 **전송 불일치 탐지**다.
- **Go `open --git/--push` + 원장 규율** (이슈 #7, maru — loom/C036, 직조: Weft): SPEC §2.1-3("열 때부터 보이게")이 요구하던 계약을 바이너리가 마침내 이행한다. 단순 커밋이 아니라 **원장 규율(§6-6)까지 통째 이식** — push 거절 → fetch·rebase → 자동 재번호 → 재시도. 로컬 bare 원장 + 병렬 클론으로 3/3 ×2 실증, 원장 최종 상태가 sha256까지 동일하다. 참조 전용은 이제 **`release` 하나뿐**(이슈 #12 관련 축소).
- **판정기 26 → 28항목** (`OPEN-GIT`, `OPEN-PUSH-RENUMBER`): 이식 전 바이너리는 `open --git`을 거부하면서도 **26/26 만점**이었다 — 판정기가 그 표면을 한 번도 안 봤기 때문이다. **판정기가 안 보는 계약은 없는 계약이다.** 이 릴리스의 진짜 산출물은 코드가 아니라 확장된 판정기다.
- 두 구현 conformance **28/28**, 실데이터 verify 변조 0. 근거: loom/C037, loom/C036 (실데이터 체인의 첫 **병합 노드** — parent `[C020, C016]`).
