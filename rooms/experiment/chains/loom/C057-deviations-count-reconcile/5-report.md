# 5. 결과 보고

## 요약

`gil close`가 deviations.yaml을 전혀 안 봐서, C053에서 내가 파일은 썼는데 `cycle.yaml deviations` 카운트를 손으로 못 고친 슬립이 봉인됐다(파일 1건 ↔ 카운트 0). **close에 "deviations.yaml 레코드 수 = 카운트" 관문(불일치 시 거부)** 과 **fsck R10 역방향 경고**를 양 구현에 넣어, C053 슬립 시나리오는 봉인 전 거부되고 이미 봉인된 C053은 위반 아닌 경고로 가시화된다. 두 구현 77/77(76→77 회귀 0), 변이 격추(관문 제거 76/77), 판정 **채택(supported)**.

## 교훈

1. **계약면의 등급은 서로 달라야 한다 — 스키마 강제(R13)와 카운트 일치(R10)를 섞지 않는다.** corrections는 v0.5산 하드 위반이라 필수 키·필드까지 검증하지만, deviations는 v0.3 유예-경고이자 "사람이 읽는 자유 서술 문서"다. 그래서 deviations의 계약면을 **레코드 수 하나**로 좁혔다 — 블록 스칼라 자유 서술은 그대로 두고 "몇 건인가"만 계약. 계약을 과하게 조이면(스키마까지) 봉인된 유예 사슬을 깨뜨리고, 덜 조이면(아무것도 안 봄) C053이 샌다. **등급에 맞는 계약면을 골라야 한다.**

2. **예방과 가시화는 한 쌍이다(C038 쌍 검증의 확장).** close 관문(발생 차단)만 있으면 이미 봉인된 슬립은 영영 안 보이고, fsck 경고(가시화)만 있으면 새 슬립이 계속 태어난다. 둘을 함께 넣어야 규율이 완결된다 — 앞으로 막고, 뒤로 비춘다.

3. **auto-count가 아니라 refuse — 카운트는 저자의 의도다.** corrections는 `gil correct`가 자동 증가시키지만 deviations엔 그런 명령이 없다(손으로 쓰는 문서). close가 자동 계수로 덮으면 저자가 2를 의도한 경우 도구가 임의 판단하게 된다. **거부는 저자에게 의식적 조정을 강제한다** — "손으로 세는 관행은 잊힌다"(C038)의 정확한 해독제는 도구가 잊지 못하게 거부하는 것.

4. **이관한 검증 경계는 되찾을 수 있다(C056 재확인).** Go를 로컬에서 못 빌드한다고 여겼으나 공식 타르볼을 sudo 없이 받아 두 구현을 나란히 세웠다. "다시 두드려라"가 또 통했다.

5. **판정기 픽스처는 실제 스캐폴드를 모사해야 한다.** write_cycle이 deviations 필드를 안 써서 양성 케이스가 처음엔 헛돌았다 — 최소 픽스처가 gil open 실산출과 다르면 계약을 헛짚는다.
