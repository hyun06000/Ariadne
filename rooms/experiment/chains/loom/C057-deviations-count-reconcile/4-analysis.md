# 4. 결과 분석

## 통계적 결과

기각 조건(1-hypothesis.md) 4건 대비:

| 기각 조건 | 기준 | 실측 | 판정 |
|---|---|---|---|
| 1. 관문이 슬립을 못 막음 | 레코드1/카운트0에서 close 성공하면 기각 | close **거부**(rc≠0), status open 유지 | 통과 |
| 2. 정상 사이클 차단(거짓양성) | 일치 시 close 거부되면 기각 | 카운트1로 맞추면 close **성공**(status closed) | 통과 |
| 3. 역방향이 위반 격상→CI 파손 | 봉인된 C053이 fsck rc≠0 유발하면 기각 | fsck **exit=0**, C053에 `이탈카운트` **경고**만 | 통과 |
| 4. 회귀 | 기존 판정기 통과 총계 회귀하면 기각 | 참조·Go 모두 76→**77/77**, 기존 76 전부 유지 | 통과 |

변이 격추: close 관문 제거 변이 → DEVIATIONS-COUNT **FAIL(76/77)**, detail `rejected=False`(슬립이 통과됨) — 판정기가 관문의 부재를 정확히 감지.

## 데이터 직접 관찰

- **실저장소 fsck 출력을 눈으로 확인**: `경고 [이탈카운트] loom/C053-windows-entry: deviations.yaml 1건인데 cycle.yaml deviations: 0 — 불일치 (봉인 전 조정, C057)`. C053의 봉인된 슬립이 **처음으로 기계에 보인다.** 동시에 C038(`이탈` 경고, 1↔1 일치)에는 `이탈카운트`가 뜨지 않아 — 일치/불일치를 정확히 가른다.
- **fsck 종료 코드 = 0** 을 직접 실행해 확인: 위반 0건(62사이클), 경고 37건. C053이 경고이지 위반이 아니어서 CI 게이트(fsck rc0 요구)를 깨지 않는다. 유예-경고 설계가 실물로 확인됐다.
- **close 관문 양면을 샌드박스에서 직접 재현**: 음성(거부 메시지 출력 + status open) → 카운트 수정 → 양성(닫힘). 관문이 저장소를 건드리기 전 초입에서 거부하므로 반쪽 상태가 남지 않는다.
- **두 구현 동일 행동**: 참조·Go 모두 DEVIATIONS-COUNT PASS, 나머지 76항목 동일. Go는 로컬에 툴체인이 없어 공식 타르볼(go1.23.4 darwin-arm64)을 sudo 없이 홈에 받아 빌드·검증했다.

## 예상과 달랐던 것

- **write_cycle이 deviations 필드를 안 써서** 판정기 양성 케이스가 처음엔 성립하지 않았다(필드 부재 → n_field=0 → 카운트 수정 regex가 매칭 실패 → 여전히 거부). cycle.yaml에 `deviations: 0`을 명시 append해 gil open 스캐폴드 상태를 정확히 모사했다. **판정기 픽스처는 실제 스캐폴드를 모사해야 한다** — 최소 픽스처가 실사용과 다르면 계약을 헛짚는다.
- **Go 검증 경계가 되찾아졌다.** 설계 시점엔 "Go는 로컬 빌드 불가(CI 이관)"로 여겼으나, C056의 교훈("이관한 경계는 우회 미탐색일 수 있다 — 다시 두드려라")을 적용해 공식 타르볼을 받아 로컬에서 두 구현을 나란히 세웠다. 이관이 아니라 실증으로 닫았다.

## 판정

**채택 (supported).** 기각 조건 4건 전부 통과, 변이 격추 확인, 두 구현 77/77. C053에서 태어나 세 사이클 이월된 도구 공백이 닫혔다 — 관문(예방)과 fsck 경고(가시화)가 한 쌍으로. 이탈 카운트를 손으로 세는 관행은 이제 도구가 대신 기억한다.
