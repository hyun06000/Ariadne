# 5. 결과 보고

## 요약

C012의 Go 단일 바이너리에 **open·close**(쓰기 porcelain)를 추가했고, 무수정
conformance.py(v0.5.0 배포본)가 목표 17항목(FSCK 9 + OPEN 3 + CLOSE 3 + LOG 2)을
전부 PASS로 판정했다 — **가설 채택**. 실데이터 교차 검증에서 같은 인자의 open이
만드는 cycle.yaml은 파이썬 참조 구현과 **바이트 단위 동일**, close 전이·fsck 출력도
동일. 총계는 15/22 → 18/22. Go 이식 로드맵의 잔여는 ② 깃 바인딩(close --git·verify),
③ web, ④ release다.

수행: Weft (씨실, 두 번째 폭). 판정기·참조 구현 무수정, 외부 의존성 0 유지 (739줄).

## 교훈

1. **쓰기 규율은 언어를 건넌다.** 참조 구현의 순서 — 사전 검증 전부 → 변경 →
   사후 fsck → 실패 시 원상 복구 — 를 구조 그대로 이식하니 판정이 따라왔다.
   범위 밖 `close --git`도 "어떤 변경보다 먼저 거부"(트리 해시 무변화 확인)로
   같은 규율을 지켰다.
2. **거부형 검사의 공허 통과는 수락형이 도착해야 실질이 된다** (C012 교훈 3의 완결).
   OPEN-REJECT-SLUG 등 세 거부형의 PASS 줄은 C012의 run과 글자가 같지만 이제
   의미가 다르다 — 스위트 출력만으로는 이 차이가 보이지 않는다는 것까지가 교훈이다.
3. **스펙의 공백: CLI 문법.** 계약(§7)은 명령 문자열 인터페이스만 정의하고,
   플래그의 위치·반복(`--parent` 다중) 규칙은 conformance의 호출 형태가 사실상
   정의하고 있다. Go `flag` 패키지가 이 형태를 못 받아 소형 파서를 손으로 썼다 —
   구현이 늘어날수록 이 암묵 문법은 스펙 §4에 명문화될 필요가 있다.
4. 판정 절차 자체도 틀릴 수 있다 — run3 최초 스크립트의 zsh 결함으로 전체 재실행
   (로그는 재실행본, 결함과 재실행 사실은 3-verification에 기록).

  도구 의존은 깃뿐)를 스펙 어휘로 정리할 것.
- **(B) CLI 문법의 스펙 명문화** (교훈 3) — §4에 플래그 문법·반복 규칙을 적시.

  들어오면 병합 지점이 생긴다.
