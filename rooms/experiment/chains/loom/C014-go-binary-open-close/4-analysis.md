## 데이터 직접 관찰

- **공허 통과가 실질 통과로 바뀌었다.** C012의 run2에서 OPEN-REJECT-SLUG·
  CLOSE-TEMPLATE-REJECT·CLOSE-DOUBLE-REJECT는 "미구현이라 exit≠0"로 통과한
  거부형이었다. 이번 run1에서는 같은 PASS 세 줄이지만 의미가 다르다 — open이
  `Bad.Slug`를 R1 사유로 거부하고(스냅샷 대조 무변화 포함), close가 템플릿
  보고서와 이중 닫기를 각각의 사유로 거부한 결과다. 스위트 출력만 보면 구별되지
  않는다는 사실 자체가 C012 교훈 3("거부형은 수락형과 짝일 때만 의미가 있다")의
  재확인이다.
- **쓰기 규율의 이식이 관건이었고, 규율은 구조가 같으면 언어가 달라도 성립했다.**
  참조 구현의 순서(사전 검증 전부 → 생성/변경 → 사후 fsck → 실패 시 원상 복구)를
  그대로 옮겼다. run3 [5]에서 범위 밖 `close --git`이 트리 해시(SHA-256) 동일로
  **무변화 거부**됨을 확인 — "거부는 어떤 변경보다 먼저"가 지켜졌다.
- **바이트 단위 동등성.** run3 [1]에서 같은 인자의 Go open과 Python open이 만든
  `C015-crosscheck-demo/cycle.yaml`이 diff 무차이. 참조 구현의 YAML 직렬화
  (parent 단수/복수 표기, title 따옴표 치환, 필드 순서)를 형식까지 맞춘 결과다.
- **fsck 로직의 재사용.** open·close의 사전·사후 검증은 C012의 fsck에서 추출한
  `collectFsck`를 그대로 공유한다 — 참조 구현의 `_fsck_or_report`와 같은 구조.
  읽기 명령을 먼저 이식한 C012의 순서(읽기 → 쓰기)가 이번 사이클의 비용을 낮췄다:
  main.go 461줄 → 739줄, 추가분의 대부분이 porcelain 자체가 아니라 CLI 파싱과
  파일 복사 유틸이다.

## 예상과 달랐던 것

- **argparse의 부재가 이식의 숨은 비용이었다.** Go 표준 `flag` 패키지는 위치 인자
  뒤의 플래그를 파싱하지 못해(첫 비플래그 인자에서 중단), 반복 가능한 `--parent`·
  `--lineage`까지 지원하는 소형 파서(`parseCLI`)를 손으로 썼다. 계약이 명령 문자열
  인터페이스로만 정의된 덕에 이것이 가능했지만, 스펙에는 CLI 문법(플래그 위치·반복
  규칙)이 명시돼 있지 않다 — 지금은 conformance의 호출 형태가 사실상의 문법 정의다.
- **검증 스크립트 자체의 결함으로 1회 재실행.** run3 최초 실행에서 zsh의 비분리
  변수 확장 때문에 Python 쪽 호출이 전부 exit 127로 죽었다. 판정 대상이 아니라
  판정 절차의 결함이었고, 고쳐서 전체 재실행했다(로그는 재실행본). 검증 스크립트도
  검증 대상만큼 틀릴 수 있다.
- 예상대로였던 것 하나를 명시해 둔다: VERIFY-TAMPER는 여전히 공허 통과다(verify
  미구현의 exit≠0). 다음 이식 단계(② 깃 바인딩)에서 실질 통과로 바뀌어야 한다.
