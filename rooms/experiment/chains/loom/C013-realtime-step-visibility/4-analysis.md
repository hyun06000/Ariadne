## 데이터 직접 관찰

- 준실시간의 실체는 **push 규율**이다: 도구는 커밋·push를 자동화할 뿐, 열림 즉시 보이려면 여는 존재가 `--git --push`를 쓰는 규율이 필요하다. 도구가 규율을 쉽게 만들고(플래그 하나), 뷰어가 규율의 보상을 보여준다(스텝 배지).
- 이 사이클의 git log 자체가 기능 명세다: "gil: open … 1/5 가설" → "step → 2/5" → "step → 3/5 검증" → "step → 4/5 분석" — 각 전이가 독립 커밋. 스텝 커밋에는 그 스텝에서 작성된 문서가 함께 담겨, 커밋 단위가 곧 스텝 단위가 됐다.
- 부트스트랩의 묘미: 기능이 태어나기 전의 전이(1·2)는 수동 커밋으로 시연했고, 태어난 직후(3)부터 도구 자신이 맡았다 — 전환점이 역사에 그대로 보인다.

## 예상과 달랐던 것

- 스위트 픽스처 헬퍼(write_cycle)가 새 필드를 모르면 R9 검사가 무효가 된다는 것 — **계약이 자라면 픽스처 생성기도 함께 자라야 한다.** 신설 검사에는 "픽스처가 정말 위반을 담고 있는가"의 자기 확인이 필요하다 (수정 완료).
- Pages 빌드 지연(수십 초~1분)이 준실시간의 하한이다. 진짜 실시간이 필요해지면 뷰어의 동적 소스(API) 전환이 별도 사이클 — 현재 요구(진행 관전)에는 충분.
