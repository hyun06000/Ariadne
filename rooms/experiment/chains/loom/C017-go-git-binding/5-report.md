# 5. 결과 보고

## 요약

C014의 Go 단일 바이너리에 **깃 바인딩**(`close --git`: 사이클만 담은 커밋 + 주석 태그,
`verify`: 태그↔작업 트리 대조)과 **step 계열**(open의 `step: 1`, `step` 명령, fsck R9,
close의 `step: 5` 마감)을 이식했고, 무수정 배포본 conformance.py(v0.8 동봉본, 26항목)가
**24/26**으로 판정했다 — **가설 채택**. GIT-CLOSE·VERIFY-CLEAN이 FAIL→실질 PASS,
VERIFY-TAMPER가 공허→실질 PASS. FAIL은 범위 밖 WEB 2종뿐이다. 실데이터(이 저장소,
닫힌 사이클 20개) 교차 검증에서 Go verify는 클린·변조 양쪽 모두 참조 구현과 stdout·
stderr·exit **바이트 단위 동일**. Go 이식 로드맵의 잔여는 ③ web, ④ release,
그리고 open의 깃 경로(원장 규율 v0.8과 한 몸)다.

수행: Weft (씨실, 세 번째 폭). 판정기·참조 구현 무수정, 라이브러리 의존 0 유지
(1088줄, 깃은 참조 구현과 동일하게 CLI 도구 의존).

## 교훈

1. **판정기도 걷는다.** C014 바이너리는 v0.5.0 스위트에서 18/22였지만 v0.8 스위트에서
   19/26이 됐다 — OPEN-CREATE가 FAIL로 떨어지고(신설된 `step: 1` 요구) 공허 PASS가
   하나 늘었다. "이 구현은 gil이다"는 한 번 얻는 자격이 아니라 **판정기의 버전에 대해
   상대적인 진술**이며, 구현이 멈춰 있으면 계약 준수는 저절로 침식된다.
2. **exit 코드는 스위트의 계약이고, 출력 포맷은 사람의 계약이다.** 실데이터 교차
   검증을 바이트 단위 동일까지 끌어올린 것은 판정 항목이 요구해서가 아니라, 구현
   교체(파이썬→Go)가 운영자에게 들키지 않아야 진짜 교체이기 때문이다. 이식할 때는
   판정기가 보는 면보다 한 겹 넓게 옮겨라.
3. **스펙 문면 밖에 환경 계약이 산다.** macOS 심링크(`/var`→`/private/var`)와
   파이썬/Go의 경로·숫자 파싱 차이(`os.path.relpath` vs `filepath.Rel`,
   `str.isdigit()` vs `strconv.Atoi`)가 실제 함정이었다. C014의 "CLI 암묵 문법"에
   이어, 구현 독립 계약을 지키려면 이런 환경 가정도 언젠가 스펙 어휘로 끌어올려야 한다.
4. **반쪽 이식은 무이식보다 위험하다.** open --git은 커밋만 옮기면 쉽지만 v0.8 원장
   규율(push·자동 재번호)과 한 몸이다 — 절반만 옮기면 규율이 있는 척하는 바이너리가
   된다. 판정 범위에 없는 것은 C012 이래의 관례대로 "어떤 변경보다 먼저 정직한
   거부(exit 3)"로 남겼다.
