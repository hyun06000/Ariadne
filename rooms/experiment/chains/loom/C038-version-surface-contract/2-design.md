# 2. 실험 설계

가설: version 자기보고 표면을 계약으로 정의하고 release가 그것을 **갱신**하되 승격 판정에서 **마스킹**하면, (A) 도구가 자기 버전을 거짓말할 수 없고 (B) 패치 릴리스가 되살아나며 (C) 진짜 변경은 여전히 잡힌다.

## 1. 계약의 정의 (SPEC §7 개정안)

**version 자기보고 표면 (version surface)** — 구현마다 상수의 이름·위치·문법이 다르므로, 스펙은 *변수*가 아니라 *표식*을 계약한다.

> 모든 gil 구현은 자기 버전을 보고하는 **SemVer 리터럴을 정확히 한 줄**에 두고, 그 줄에 표식 **`gil:version`** 을 (해당 언어의 주석으로) 단다. 그 줄의 SemVer는 `gil version`의 출력과 일치해야 한다. 표식 라인에는 version 선언 외의 실질 코드를 두지 않는다.

| 구현 | version 표면 |
|---|---|
| `gil.py` | `_GIL_VERSION = "1.11.0"  # gil:version` |
| `go/main.go` | `const gilVersion = "1.11.0" // gil:version` |
| `conformance.py` | (판정기도 도구 blob이다 — 표식을 단다) |

이 계약이 P3(구현마다 다른 위치)을 푼다. 도구는 언어 문법을 파싱하지 않고 **표식 라인의 첫 SemVer**만 본다 — 언어 무지성이 곧 구현 독립성이다.

## 2. 두 연산 (참조 구현에만 — `release`는 참조 전용 명령)

```
_set_version(text, v)  : 표식 라인의 첫 SemVer → v 로 치환 (치환 건수 반환)
_mask_version(text)    : 표식 라인의 첫 SemVer → "X.Y.Z" 로 정규화 (비교 전용)
```

**도구 blob 목록** (승격 규칙의 관측 범위 — P4):

| blob | 지금 | 이후 |
|---|---|---|
| `gil.py` | 관측함 | 관측 (마스킹 후 비교) |
| `conformance.py` | 관측함 | 관측 (마스킹 후 비교) |
| `go/main.go` | **안 봄** | 관측 (마스킹 후 비교) |

### `gil release <v>` 의 개정된 절차

**사전 검증** (저장소를 건드리기 전):

1. (기존) SemVer·단조증가·태그 중복·패키지 존재·fsck·verify·RELEASE.md 서술·CHANGELOG.
2. **(신규) 표식 계약 검사** — 도구 blob 3개 각각에 표식 라인이 **정확히 1개**이고 그 줄에 SemVer가 있어야 한다. 아니면 거부.
3. **(개정) 승격 판정** — 마지막 릴리스 태그의 blob과 현재 blob을 **둘 다 `_mask_version` 한 뒤** 비교. 차이 나는 blob만 `changed_parts`. 변경이 있는데 패치 승격이면 거부(기존 규칙 유지).

**실행**:

4. 도구를 패키지로 동기화(기존) → **(신규) 패키지 안 도구 blob 3개의 표면을 `_set_version(v)`** → template 동기화 → CHANGELOG → 배포의 방만 커밋 → 태그.

순환이 끊기는 지점: 4에서 쓴 version은 3에서 마스킹되어 **다음 릴리스의 변경 감지에 보이지 않는다.**

### 판정기 (conformance)

판정기는 릴리스 버전을 모른다(구현만 본다) — 그러므로 "1.9.0이라는 거짓말"은 판정기가 잡을 수 없다. 잡을 수 있는 것은 **표면의 형태**뿐이다:

- **`VERSION-SEMVER`** (신규, 29번째 항목): `gil version` 의 출력이 `gil X.Y.Z` (SemVer) 형태여야 한다.

이 한계는 정직하게 보고서에 남긴다. 진짜 문지기는 release다.

## 3. 검증 절차

C018의 픽스처 하네스(`fresh(tool_at_tag, tool_now, …)` — 태그 `v0.1.0` 시점의 패키지와 현재 작업 트리를 구분 구성)를 **재사용·확장**한다. 확장점: `go_at_tag` / `go_now` 인자.

산출물은 `3-verification/`에 격리 저장한다: `tests.py`, 사용한 `gil/`, `runs/` 로그.

### 기대 행동 (실행 전 선고정)

| # | 항목 | 조건 | 기대 |
|---|---|---|---|
| T1 | **A-py** | 실저장소 릴리스 후 `python3 gil.py version` | `gil <새버전>` |
| T2 | **A-go** | 갱신된 `go/main.go` 빌드 후 `./gil version` | `gil <새버전>` |
| T3 | **B (핵심 회귀)** | 태그 대비 도구가 **version 표면만** 다름 → 패치 승격 `0.1.1` | **허용** (rc 0), CHANGELOG "도구 변경: 없음" |
| T4 | **C-py** | `gil.py` 실질 코드 변경 → 패치 승격 | 거부 (rc≠0, `gil` 지목) |
| T5 | **C-conf** | `conformance.py` 실질 변경 → 패치 승격 | 거부 (`conformance` 지목) |
| T6 | **C-go** | `go/main.go` 실질 변경 → 패치 승격 | 거부 (`go` 지목) — **신규 관측 범위** |
| T7 | 표식 계약 | 도구 blob에 표식 0개 → 릴리스 | 거부 (계약 위반 명시) |
| T8 | 표식 계약 | 표식 2개(중복) → 릴리스 | 거부 |
| T9 | 무부작용 | T4·T7의 거부 후 저장소 스냅샷 | 거부 전과 해시 동일 |
| T10 | 마이너 정상 | 실질 변경 + 마이너 승격 `0.2.0` | 허용, 패키지의 version 표면 3개 전부 `0.2.0` |
| T11 | 회귀 | 실저장소 릴리스 후 두 구현 conformance | 29/29 ×2 |

T3과 T4~T6은 한 규칙의 양면이다 — **T3만 통과하고 T4~T6 중 하나라도 실패하면(= 마스킹이 진짜 변경을 삼키면) 가설 기각**이다.

### 끝단 (수신자 경로 — C037의 절차)

기계 검증이 끝나면 실릴리스(v1.12.0)를 내고, **공개 URL에서 바이너리를 받아 체크섬 대조 후 `gil version`** 을 실행한다. v1.10.0의 거짓말이 잡혔던 바로 그 경로다. 이것이 통과해야 사이클을 닫는다.

## 4. 예상 쟁점 (미리 적어둔다)

- **작업 트리의 정직성**: 릴리스 사이의 작업 트리는 "마지막 릴리스 버전"을 보고한다. 이건 거짓말이 아니라 *스냅샷의 진실*이다(그 blob은 아직 릴리스되지 않았다). 다만 `gil version`이 "미배포 변경분 있음"을 알릴 수는 없다 — 한계로 기록.
- **마스킹의 사각지대**: 표식 라인에 version 외의 실질 코드를 함께 두면 그 변경이 숨는다. 계약이 "표식 라인은 version 선언만"을 요구하는 이유다.
- **이 릴리스 자신**: v1.11.0 태그의 blob에는 표식이 없다 → 이번 릴리스는 정당하게 "도구 변경"으로 판정되어 **마이너 승격(v1.12.0)** 이 강제된다. 규칙을 만든 릴리스가 그 규칙의 첫 준수자가 되는 모양은 C018과 같다.
