# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C037 — 체크섬 설치 문지기](../C037-checksum-install-gate/5-report.md)

C037이 남긴 두 교훈이 이 사이클을 낳았다.

1. **문지기의 증거는 "없는 파일"이다** — 규율을 문서의 훈계로 두면 새어나가고, 기계가 집행할 때만 지켜진다. C037은 체크섬 대조를 `chmod +x`의 선행 조건으로 만들어, 검증 안 된 바이너리가 *태어나지 못하게* 했다.
2. **검증은 수신자 경로로 끝까지** (C024·C027·C028·C037의 반복 확인) — 공개 URL에서 받아 실행하는 그 경로가 아니면 안 보이는 결함이 있다.

그리고 C037 직후의 v1.10.0 배포에서 교훈 ②가 교훈 ①의 미이행을 잡아냈다.

**배포된 v1.10.0 바이너리가 `gil version`에 `gil 1.9.0`이라 답했다.**

두 구현의 version 상수(`gil.py`의 `_GIL_VERSION`, `go/main.go`의 `gilVersion`)를 갱신하지 않은 채 릴리스했기 때문이다. 그런데 이 갱신 의무는 v1.8.0(loom/C034) 보고서에 이미 적혀 있었다:

> *"version 상수는 릴리스마다 갱신(후속 자동화 여지)"*

**적어두고도 새어나갔다.** `gil release`가 강제하지 않기 때문이다. 이것이 교훈 ①의 정확한 반례다 — 우리는 남의 설치 경로에는 문지기를 세워놓고, 우리 자신의 릴리스 경로는 훈계로 두었다.

정정하려 v1.10.1(패치)을 내려다 **도구에게 거부당했다**: 승격 규칙(C018)은 "도구 blob이 마지막 릴리스 태그 대비 변했으면 마이너 이상"이고, version 상수도 도구의 blob이다. 규칙에 복종해 v1.11.0으로 냈다(C002의 교훈). 그 마찰이 규칙의 진짜 구멍을 드러냈다 — 이슈 [#15](https://github.com/hyun06000/Ariadne/issues/15).

## 문제 분할

"배포된 도구가 자기 버전을 정직하게 보고한다"를 최소 단위로 쪼개면:

| # | 문제 | 상태 |
|---|---|---|
| P1 | 릴리스 때 version 상수를 **갱신하는 주체**가 사람(존재)이다 → 새어나간다 | 이번 사이클 |
| P2 | version 상수를 자동 갱신하면 **모든 릴리스가 "도구 변경"**이 되어 패치 릴리스가 영원히 불가능해진다 (자기 지시적 순환) | 이번 사이클 |
| P3 | version 표면의 **위치·이름이 구현마다 다르다** — §7 구현 독립 계약이 이걸 정의하지 않는다 | 이번 사이클 |
| P4 | 승격 규칙의 관측 범위가 `gil.py`·`conformance.py`뿐 — **`go/main.go`는 아무도 안 본다** (Go만 바뀐 릴리스가 패치로 나갈 수 있다) | 이번 사이클 |
| P5 | 바이너리의 자기보고가 진실과 어긋나는 다른 표면 (help가 goto·pages를 '없다'고 말함) | 이슈 #12, 별도 사이클 |

P1~P4는 **하나의 매듭**이다. P1을 자동화하면 P2가 터지고, P2를 풀려면 P3(무엇이 version 표면인가)의 정의가 필요하며, P3의 정의를 도구가 스캔하는 순간 P4(스캔 범위 = 패키지 안의 모든 도구 blob)가 같이 풀린다. 넷을 따로 떼면 어느 하나도 완결되지 않는다.

P5는 자기보고의 다른 표면이고 마스킹·승격과 무관하므로 분리한다.

## 가설

> **가설**: **version 자기보고 표면**을 §7 구현 독립 계약의 일부로 정의하고(표식 `gil:version`을 단 한 줄), `gil release`가 ① 패키지 안의 모든 도구 blob에서 그 표면을 릴리스 버전으로 **자동 갱신**하고 ② 승격 규칙의 변경 감지에서 그 표면을 **마스킹**한 뒤 비교하면 —
> **(A)** 배포된 도구는 자기 버전을 거짓말할 수 없고(양 구현),
> **(B)** version만 바뀐 릴리스는 "도구 미변경"으로 판정되어 **패치 승격이 되살아나며**,
> **(C)** version 외의 도구 blob 변경(`go/main.go` 포함)은 여전히 패치 승격을 거부한다.

핵심은 **마스킹이 진짜 변경을 숨기지 않으면서 자기 지시적 순환만 끊는가**다. (B)와 (C)는 한 규칙의 양면이고, 둘 중 하나만 성립하면 이 설계는 실패다.

## 기각 조건

미리 고정한다. 다음 중 **하나라도** 발생하면 가설을 기각한다.

1. **(A) 실패**: 릴리스 후 패키지의 두 구현 중 하나라도 `gil version`이 릴리스 버전과 다른 값을 보고한다 (파이썬은 직접 실행, Go는 갱신된 소스를 빌드해 실행).
2. **(B) 실패**: 도구의 코드가 version 표면 외에 변하지 않은 상태에서 패치 승격(x.y.Z)이 거부된다.
3. **(C) 실패**: 도구의 실질 코드(`gil.py`·`conformance.py`·`go/main.go` 중 하나)가 version 표면 외에서 변했는데 패치 승격이 **허용**된다. 즉 마스킹이 진짜 변경을 삼킨다.
4. **회귀**: 릴리스 후 두 구현의 conformance가 28/28을 유지하지 못한다.
5. **거부 시 부작용**: 승격 거부 경로에서 저장소가 변경된다 (기존 규율 위반).

> 기각 조건은 문면이 아니라 **의도**로 읽는다 (C011의 교훈). 위 5건의 의도는 하나다 — *"도구가 자기 버전에 대해 하는 말과, 도구가 자기 변경에 대해 하는 말이 둘 다 참인가."*
