# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C010](../C010-rename-to-gil/5-report.md) (채택).

- SPEC §7이 선언됐다: "스펙이 계약이고 구현은 교체 가능하다." 그러나 선언은 아직 **기계가 아니다** — 어떤 구현이 계약을 지키는지 판정하는 실행 가능한 것이 없다. 바이너리 이식(상현님 방향)의 전제 조건.
- C010의 잔재 재발: 커밋 접두어 `ari:`가 남았고(C010 보고에서 자백), 전수 grep으로 추가 확인 결과 CSS 클래스 `.ari`, JSON 훅 `id="ari-data"`, footer 문구까지 4종. "치환 스크립트는 치환한 것만 안다"의 두 번째 물림 — 이번엔 절차(전수 grep)를 먼저 실행했다.
- 방법론의 기존 교훈(C008): "테스트는 기대 행동 문서만큼만 본다" — 계약 스위트가 곧 계약의 기대 행동 문서가 된다.

## 문제 분할

(a) 스위트 — 구현을 **명령 문자열로만** 알고(내부 import 금지), 자기 완결적 샌드박스에서 계약 항목(fsck R1~R8, open/close 규율, log, web 자기완결·JSON 훅, 깃 각인·변조 탐지)을 판정, (b) 스위트의 스위트 — 스위트가 진짜 뭔가를 검사함을 증명하는 **변이 테스트**(고의 결함 구현이 잡히는가), (c) 잔재 정리 + JSON 훅의 계약 승격(`gil-data`), (d) v0.4.0 릴리스(도구 변경 → 마이너, 스위트 동봉).

## 가설

> **가설**: 구현을 명령 문자열로만 아는 자기 완결적 계약 준수 스위트(`conformance.py`)를 만들면, ① 참조 구현 gil.py v0.4.0(잔재 4종 정리 + JSON 훅 `gil-data` 계약화)은 전 항목을 통과하고, ② 계약을 고의로 어긴 변이 구현 3종(fsck 무력화 · 슬러그 검증 제거 · 외부 리소스 주입)은 각각 해당 항목에서 탐지되며, ③ 스위트는 패키지에 동봉되어 어떤 미래 구현(바이너리 포함)도 같은 판정을 받을 수 있다.

## 기각 조건

1. **참조 구현 불통과**: v0.4.0이 스위트의 어떤 항목이라도 실패한다.
2. **변이 미탐지**: 변이 3종 중 스위트를 전부 통과하는 것이 있다 — 스위트가 장식이라는 뜻.
3. **구현 의존**: 스위트가 명령 실행·파일 관찰·종료 코드 외의 수단(구현 내부 import, 특정 파일명 가정)을 쓴다.
4. **잔재 잔존**: v0.4.0의 gil.py에 `ari` 잔재(표준 용어 `aria-*` 제외)가 남는다.
