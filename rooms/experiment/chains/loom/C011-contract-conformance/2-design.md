# 2. 실험 설계

## 절차 (판정 표를 겸해 선고정)

1. **v0.4.0 gil.py** — 전수 grep으로 확인된 잔재 4종 정리: 커밋 접두어 `ari:`→`gil:`, CSS `.ari`→`.gil`, JSON 훅 `id="ari-data"`→`id="gil-data"`, footer `ari web`→`gil web`. (`aria-label`은 웹 표준 — 보존.) 로직 변경 없음.
2. **conformance.py** — 계약 준수 스위트. 원칙: 구현은 `--gil "<명령>"` 문자열로만 주입, 자체 생성 샌드박스(템플릿 포함), 판정은 종료 코드·파일시스템·산출물 텍스트로만. 검사 항목:
   - fsck: 깨끗한 저장소 exit 0 + **R1~R8 위반 8종 각각** exit ≠ 0
   - open: 생성(필수 필드)·번호 증가·형식 위반 슬러그 거부(무변화)
   - close: 템플릿 보고서 거부 · 정상 닫기(상태 전이) · 이중 닫기 거부
   - log: 정상 렌더 exit 0 · 끊어진 참조 exit ≠ 0
   - web: 외부 리소스 0 · 내장 JSON `id="gil-data"` 파싱·노드 집합 일치
   - 깃(가용 시): close --git의 태그 생성·경로 격리 · verify의 변조 탐지
3. **참조 구현 판정** — 스위트 vs v0.4.0 → 전 항목 통과해야 함.
4. **변이 테스트** — 결함 구현 3종을 만들어 스위트가 잡는지 확인: M1(fsck가 항상 빈 위반 반환), M2(슬러그 검증 제거), M3(web에 외부 스크립트 주입). 각각 해당 항목에서 FAIL 나야 함.
5. **계약 문서화** — SPEC §5 web 행에 JSON 훅 계약 추가, §7에 "계약의 실행 가능형 = conformance.py" 명시. RELEASE.md 0.4.0 서술.
6. **v0.4.0 릴리스** — conformance.py를 패키지에 동봉하고 `gil release 0.4.0` (도구 변경 → 마이너). 릴리스 후 **배포된 gil.py**를 스위트로 재판정.
