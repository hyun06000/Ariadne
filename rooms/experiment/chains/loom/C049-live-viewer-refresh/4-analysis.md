# 4. 결과 분석

## 통계적 결과
- **H1**: `--refresh 3` → meta refresh + bake.refresh + 외부 리소스 0.
- **H2**: gil step 자동 재굽기 후에도 meta refresh 유지(bake에서 읽음) — 실시간 통합 작동.
- **H3**: `--refresh` 없으면 개선 전과 바이트 동일.
- **H4**: `--refresh` 유/무 모두 참조=Go 바이트 동일.
- 판정기 참조 72→73·Go 64→65, 회귀 0, 변이 격추.

기각 조건 넷 모두 미충족 → 기각되지 않음.

## 데이터 직접 관찰
- **실시간은 두 조각의 곱이었고, 하나는 이미 있었다.** 파일 갱신(C042 자동 재굽기)은 v2.2부터 있었다. 빠진 건 브라우저 반영뿐. `--refresh`(meta)를 추가하고 **bake에 기록**하니, 기존 C042가 그 조각을 자동으로 물어 실시간이 완성됐다. **새 기능이 아니라 있던 두 기능을 잇는 한 줄(bake.refresh)이 핵심이었다.**
- **bake가 없었으면 실시간이 한 번 만에 죽었을 것.** `--refresh`로 굽고 gil step이 재구우면, 재굽기가 refresh를 모르면 meta가 사라져 두 번째 리로드부터 멈춘다. bake.refresh를 _bake_meta가 읽어 _bake_viewer에 넘기는 고리가 있어야 지속된다. C042의 "산출물이 자기 생성 조건을 말한다"가 여기서 값을 했다 — refresh도 생성 조건이다.

## 예상과 달랐던 것
- **JS 0줄 계약이 오히려 답을 좁혀 줬다.** 실시간 = 폴링/WebSocket이 상식이지만 gil web은 JS·외부 리소스 0이 계약. 그 제약이 `<meta http-equiv="refresh">`(HTML 표준, JS 아님)라는 단순한 답으로 이끌었다 — 계약이 설계를 제약하는 게 아니라 **정제**했다. C005(뷰어 접근성 요구가 문서 완결성을 만들었다)의 재현.
- **--watch는 있으면 좋지만 필수가 아니었다.** 상현님 시나리오(gil step으로 진행하는 워크트리)는 C042 자동 재굽기 + --refresh만으로 충분하다. --watch는 gil을 안 거치는 외부 변경(병합·pull) 관찰용 — 별도 층. 그래서 --refresh(양 구현 계약)와 --watch(편의)를 분리했다.

## 판정
**채택(supported).** H1~H4 검증. `--refresh`가 C042 자동 재굽기와 맞물려 새로고침 없는 실시간 관찰을 이룬다. 하위호환, 두 구현 동일.
