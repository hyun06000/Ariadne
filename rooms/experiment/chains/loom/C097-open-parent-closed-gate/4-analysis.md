# 4. 결과 분석

## 통계적 결과

가설의 다섯 kill 조건(1-hypothesis.md)이 전부 통과 → **채택(supported)**.

| kill 조건 | 기준 | 결과 |
|---|---|---|
| 1. 위반 차단 | 열린 부모 open → exit≠0 + 무변화 | ✔ exit=1, snapshot 무변화 (OPEN-PARENT-CLOSED-GATE PASS) |
| 2. 위양성 없음 | 닫힌부모·new-root·new-chain·다중부모 통과 | ✔ 전부 PASS |
| 3. 원자성 | 거부 시 저장소 무변화 | ✔ 판정에 `snapshot==before` 포함 |
| 4. 회귀 없음 | 참조 ≥124 | ✔ 참조 123→**125**, Go 105→**107** (신규 2 정확 반영) |
| 5. 메시지 | 부모 id·원인·대안 | ✔ 참조↔Go **바이트 동일** |

## 데이터 직접 관찰

실출력을 직접 봤다(3-verification 재현 커맨드):

```
오류: 부모 'C001-alpha'가 아직 닫히지 않았다 (status: open) — 열린 부모 위에 자식을 열 수 없다.
      부모를 먼저 닫거나(gil close), 정말 갈래를 나누려면 이미 닫힌 사이클을 --parent로 지정하라.
```

참조와 Go가 이 두 줄을 바이트 단위로 동일하게 낸다. C095·C096의 죽음을 되짚으면: 그때 `gil open`은 부모의 **존재**(R6)만 봤지 **닫힘**은 안 봐서 C095(open 1/5) 위에 C096이 태어났다. 이 게이트가 있었다면 C096의 open이 **처음부터** 이 메시지로 거부되어, 노드 하나가 헛되이 죽지 않았을 것이다. loom/C003의 경계("도구는 절차, 존재는 진실")에서 "부모 닫힘"은 순수 절차이므로 도구의 몫이 맞다.

`!= "closed"` 화이트리스트라 **rejected(죽은 가지)도 부모 자격에서 배제**된다 — 죽은 가지에서 자라면 안 되므로 부수효과가 아니라 의도된 정확성이다. 사전 검증 블록(디렉토리 생성·번호 소비 이전) 안에 두어 원자성을 구조가 공짜로 보장한다(별도 롤백 불필요).

## 예상과 달랐던 것

1. **기존 테스트 2개가 열린 부모 전제를 갖고 있었다.** OPEN-INCREMENT·GUARD-RESERVED-OK가 `--parent`로 방금 연 열린 사이클을 쓰고 있었다 → 게이트가 이를 불법화하며 FAIL. 이는 게이트의 결함이 아니라 **계약이 실제로 강해졌다는 증거**다(C094 "계약 바꾸면 그 검증 테스트도 바뀐다", 세 번째 작동). 두 항목의 원래 의도(번호증가·예약guard)는 부모 닫힘과 직교하므로 전제만 `_seal_closed`로 정정하고 의도는 보존했다.

2. **내 새 픽스처의 `closed: null` 결함이 다음 항목들을 연쇄 오염**시켜, 마치 게이트가 정상 흐름을 깬 것처럼 보였다(OPEN-INCREMENT 등 FAIL). C093의 "한 결함이 다른 결함을 가린다"가 재현 — `closed` 일자를 채우니 진짜 원인(열린 부모 전제)만 남았다. `git stash`로 baseline(123/105)을 재확인해 회귀 여부를 착시 없이 갈랐다(C092 규율).
