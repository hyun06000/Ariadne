# 5. 결과 보고

## 요약

상현님이 비개발자 Windows 친구에게 gil을 넘기려는 계기에서, **넘기기 전 우리 환경(비-Windows)에서
검증 가능한 최대치**를 확보했다. C053이 pwsh를 못 구해 수신자로 이관했던 **PowerShell 설치
게이트를 진짜 pwsh 7.7 런타임에서 검증**(정상 경로 gil.exe 생성, 위조 체크섬 throw+미생성) —
pwsh를 sudo 없이 .pkg 페이로드 추출로 확보해 우회했다. 그리고 README.ai.md Step C에
**Windows 명령형(`.\gil.exe`) 치환 규칙 한 줄**을 더해 에이전트 경로를 OS-완전화했다.
세 가설 모두 지지, 이탈 0. 남은 것(네이티브 .exe 실행, Windows 경로·git 실환경)은
흉내 없이 수신자 로그 대기.

## 교훈

1. **이관한 경계는 고정이 아니다 — 다음 사이클이 되찾을 수 있다.** C053의 "pwsh 설치 실패"는
   설치의 실패가 아니라 **sudo의 실패**였다(.pkg 시스템 설치가 암호 요구). 바이너리는 .pkg
   안에 멀쩡했고 `pkgutil --expand`+cpio로 페이로드만 뽑으면 됐다. C007("전제는 확인하라")의
   재판: *못 하는 줄 알았던 검증이 다른 경로로 가능했다.* C050(다섯 번 이월된 매듭을 되찾음)과
   같은 결 — **수신자 경계는 우리 능력의 한계가 아니라 그 시점의 우회 미탐색일 수 있다.**
   못박되, 주기적으로 다시 두드려라.

2. **"코드가 옳은가"와 "이 바이너리가 이 OS에서 뜨는가"는 다른 질문이다.** 게이트는 pwsh에서
   돌고(우리 몫), gil.exe는 유효 PE32+이며 그 코드는 동일 소스가 darwin 76/76이다(우리 몫).
   `.\gil.exe`의 실제 기동만 `kLSExecutableIncorrectFormat`으로 막혔다(수신자 몫). 이 선을
   에러 메시지가 물리적으로 그어 줬다 — **경계를 흉내로 뭉개지 말고 정확한 지점에 그어라**
   (C053 "못 하는 검증을 흉내내지 말라"의 좌표 확정).

3. **검증돼야만 태어난다 — PowerShell 실증.** 위조 체크섬에서 `throw`가 실제로 던져지고
   gil.exe가 생기지 않았다. C037의 게이트 원리가 이식 셸 동치(C053)를 넘어 실제 cmdlet
   런타임에서 사실이 됐다. 안전속성은 이제 "동치일 것"이 아니라 "그렇다".

4. **에이전트용 문서는 규칙을 일반화하는 독자를 가진다.** Step C 전체 이중 표기 대신
   "`./gil`을 `.\gil.exe`로 바꿔라" 한 줄로 족했다 — README.ai.md의 독자는 지시를 일반화한다
   (C025 "AI에겐 명령형 작전 문서"의 실용판). 사람용이라면 매 줄 이중 표기가 필요했을 것.
