# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C023](../C023-v1-integration/5-report.md) (v1.0.0 — 두 구현 26/26). 발의: 사용자 (박상현, 2026-07-15): "바이너리 릴리스해서 다른 LLM이 딸깍만으로 쓸 수 있게. 내가 다른 맥에서 테스트해 볼게."

- Go 소스는 패키지에 있지만, 쓰려면 Go 툴체인이 필요하다 — "딸깍"이 아니다. 깃처럼 되려면 **빌드된 바이너리가 배포 채널에** 있어야 한다.
- 사용자의 테스트 머신: 다른 맥 (arm64/amd64 모름 → 둘 다).

## 가설

> **가설**: ① 패키지 go/ 소스에서 4타깃(darwin arm64·amd64, linux arm64·amd64) 크로스 빌드가 재현 가능하고, ② 네이티브 산출물이 conformance 26/26이며, ③ GitHub Release(v1.0.0)에 바이너리·체크섬을 첨부하면 공개 URL에서 내려받은 파일이 로컬 빌드와 체크섬 일치하고, ④ 태그 push마다 같은 일을 하는 Actions 워크플로의 빌드 스텝이 신선 클론에서 추출 실행된다 — 그러면 다른 머신의 다른 LLM은 curl 한 줄로 gil을 갖는다.

## 기각 조건

1. 크로스 빌드 실패 또는 릴리스 산출물의 conformance 미달.
2. 공개 URL 다운로드 파일의 체크섬이 로컬 빌드와 불일치.
3. 워크플로 빌드 스텝의 신선 클론 추출 실행 실패.
4. 바이너리의 미구현 명령(release·open --git 등)이 문서에 정직하게 고지되지 않음.
