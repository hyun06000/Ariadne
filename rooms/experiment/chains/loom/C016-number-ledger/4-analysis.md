# 4. 결과 분석

## 통계적 결과

| # | 항목 | 결과 |
|---|---|---|
| 1 | 병렬 경합 자동 해소 | ✅ T1: B의 C002가 자동으로 C003 재번호·push 성공, 원장 fsck 0, 제목·author 무손상 |
| 2 | 무경합 회귀 | ✅ T2 + conformance 26/26 |
| 3 | 해소 불가의 정직성 | ✅ T3: rebase 충돌 → 명시적 오류 + abort (원장·작업 트리 오염 없음) |

## 데이터 직접 관찰

- 재번호 로그가 규율의 전 과정을 말한다: "경합 감지: C002-beta-path → C003-beta-path (원장 규율에 따라 재번호)" — 존재는 실패 대신 순서를 부여받는다. 깃 push의 낙관적 동시성이 사이클 번호에 그대로 이식됐다.
- T3의 1차 실패는 구현이 아니라 픽스처였다: 충돌 재료가 원장에 도달하지 못했다(non-FF 조용한 거절). **테스트가 실패하면 먼저 테스트를 의심하라** — 최소 재현(9줄)으로 격리 진단한 것이 유효했다.

## 예상과 달랐던 것

- 픽스처 결함의 아이러니: 원장 규율을 검증하는 테스트가 스스로 원장 규율 위반(무확인 push)으로 깨졌다. 문제의 보편성을 이보다 잘 보여줄 수 없다.
- 체인 그래프에는 손댈 것이 없었다 — 사용자의 직관("브랜치별 관리의 도입")은 그래프가 아니라 발급의 문제였고, parent 모델은 이미 병렬을 담는다. 진단이 설계의 절반이었다.
