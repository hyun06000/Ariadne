# 5. 결과 보고

## 요약

참조 구현(gil.py)의 `round` 명령·`rounds` 스키마·fsck R15·log/web 표시를 Go 바이너리에
이식하니 ROUND-\* 8항목이 전부 PASS로 전환되고(퇴행 0), 실 저장소에서 하위호환(무라운드
web 바이트 동일)도 보존됐다. 다만 conformance 총계는 72/72가 아니라 **64/64**다 — 남은
8항목 차이는 전부 예약(reserve/unreserve) 계열이며, 이는 이 사이클의 이식 대상이 아니다
(부모 C045의 "이식할 계약면"은 라운드만 지시했다). 목표치 72는 위임 브리핑이 "Go에 예약이
이미 이식돼 있다"고 잘못 가정한 계산 오류였다 — 우회하지 않고 있는 그대로 보고한다.

## 교훈

1. **이식 등급의 순서(exit급 → 문자면급 → 원장급)는 라운드에도 그대로 적용된다.** 참조와
   Go가 같은 입력에 대해 `round --open/--close`의 생성물(hypothesis.md·round.yaml·
   cycle.yaml)을 바이트 단위로 동일하게 내는 것까지 확인해야 "이식됐다"고 말할 수 있다.
   conformance의 ROUND-OPEN-GIT처럼 판정기가 커밋 파일 목록까지 보는 항목이 있었기에
   가능했다 — 판정기가 그 정도로 깊이 보지 않았다면 "verification/이 커밋에 안 들어갔다"는
   사전등록(H1)의 핵심 계약을 놓쳤을 수도 있다.
2. **위임받은 목표 수치를 맹신하지 않고 직접 재계산해야 한다.** "Go 56/56 + 라운드 8 =
   64"라는 산수는 설계 단계(2-design.md)에서 미리 해봤어야 할 검산이었고, 실제로 검산해
   보니 위임 브리핑의 "72/72" 목표가 예약 이식 여부를 잘못 가정하고 있었다. 검증을 시작하기
   전에 목표 수치의 구성 요소를 분해해 보는 것이 방법론적 습관이 되어야 한다 — 이번엔
   운 좋게 설계 단계에서 잡았지만, 분석 단계에서야 발견했다면 "가설이 기각됐다"는 잘못된
   결론으로 이어질 뻔했다.
3. **판정기가 안 보는 계약은 없는 계약이다(C036)의 사례가 또 나왔다 — 이번엔 반대 방향.**
   Go의 `gil log`는 참조 구현의 `summarize()`(root·분기점·병합점 요약)를 애초에 이식한 적이
   없었다. conformance의 LOG-OK/LOG-BROKEN이 exit 코드와 id 존재만 검사하다 보니 이 공백이
   여러 사이클(C012→C036) 동안 드러나지 않고 있었다. 이번 라운드 이식으로 log를 직접
   손보다가 우연히 발견했다 — 고치지는 않았다(이식 범위 밖이므로), 다음 사이클의 씨앗으로
   남긴다.
4. **검증 행위 자체가 검증 대상의 전제를 바꿀 수 있다.** `gil open --parent`으로 이 사이클을
   여는 순간 예약 46이 승격되어 `reservations.tsv`가 소거됐고, 그 결과 "실 저장소 web
   바이트 동일" 검증의 결과가 사이클 진행 도중 "다름"에서 "같음"으로 바뀌었다. 도구를 쓰는
   행위가 검증 환경을 갱신하는 이런 사례는 재현 스크립트가 실행 시점(사이클을 연 이후)을
   전제로 삼아야 함을 보여준다.

## 다음 사이클을 위한 제안

- **(A) 예약(reserve/unreserve)을 Go에 이식.** RESERVE-\* 8항목 + UNRESERVE 1항목을
  마저 이식하면 Go도 참조와 나란히 72/72에 도달한다. `loom/C043`의 `_load_reservations`·
  `cmd_reserve`·`cmd_unreserve`·원장 파일(reservations.tsv) 파싱·`open --push`의 재번호
  로직과의 상호작용을 정독해야 한다 — 이번 라운드 이식보다 상태 기계가 복잡하다(원격 원장
  경합·자동 승격 로직 포함, C016의 번호 원장 규율과 얽혀 있다).
- **(B) Go의 `gil log`에 `summarize()`(root·분기점·병합점 요약) 이식.** 이번에 발견한 공백.
  conformance에 LOG 항목을 추가해(전체 텍스트 대조 또는 "root:" 문자열 존재 검사) 판정기의
  사각지대도 함께 메워야 재발을 막는다.
- **(C) 위임 브리핑의 목표 수치에 검산 단계를 표준 절차로 편입.** 다음에 "판정기 총점 N/N"이
  목표로 주어지면, 설계 단계(2-design.md)에서 "이 N이 정확히 어떤 항목들의 합인가"를 먼저
  분해해 보는 것을 5스텝 방법론의 관례로 굳히자 (rooms/experiment/README.md에 명문화할 만한
  후보).
