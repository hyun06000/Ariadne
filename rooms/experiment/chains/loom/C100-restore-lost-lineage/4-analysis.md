## 데이터 직접 관찰

`gil log`가 복원을 그대로 비춘다:

```
● │ │  C018-release-baseline [closed] …  ✎ corrected(1)  ◀ 병합: C016-number-ledger + C012-go-binary-log-fsck
● C057-deviations-count-reconcile [closed · supported] …  ✎ corrected(1)  ◀ 병합: C056-… + C053-windows-entry
병합점: C018-release-baseline ← C016-number-ledger, C012-go-binary-log-fsck
분기점: C012-go-binary-log-fsck → C013…, C014…, C018-release-baseline
분기점: C053-windows-entry → C054…, C056…, C057-deviations-count-reconcile
```

C012는 원래 C013·C014로만 분기했는데, 이제 **C018로도 분기**한다 — C018이 태어날 때 실제로 물려받은 두 근원(C016의 번호원장 + C012의 승격규칙 공백)이 계보에 복원됐다. C053도 마찬가지로 C057을 자식으로 얻었다. **거짓(단일부모)이 지워진 게 아니라, 진실(다중부모)이 corrections.yaml에 증거와 함께 덧대어졌다** — §4.1의 "원장은 고칠 수 없어도 역사는 덧붙인다"가 계보 복원에서 작동했다.

## 예상과 달랐던 것

1. **풀슬러그가 경로 덕에 매칭됐다.** parent 값은 로컬 id 풀슬러그(`C016-number-ledger`)여야 하는데, 문서엔 번호(`C016`)나 링크텍스트로만 보였다. 그런데 마크다운 **경로**(`../C016-number-ledger/5-report.md`)에 풀슬러그가 부분문자열로 들어 있어 correct C5가 통과했다. memory가 "미검"이라 남긴 매칭 문제가, **문서가 상대경로 링크를 쓰는 관행 덕에 우연히 풀렸다.** 단 이건 문서 작성 스타일 의존이라 신뢰할 계약은 아니다.
2. **C041은 그 우연이 안 통했다.** 2-design.md가 C035·C040을 순수 번호(`C035`·`C040`, 경로 없이)로만 적어 풀슬러그가 어디에도 없었다. → correct로는 복원 불가. 이것이 memory가 예견한 진짜 제약의 실물이다: **correct는 evidence에 값이 문자 그대로 있어야 하는데, 문서가 번호로만 조상을 부르면 복원 경로가 없다.**
3. **C043 재검토가 대상을 하나 줄였다.** 설계 전엔 4건이었으나, "재발이력≠부모"(상현님 기준)를 엄격 적용하니 C043은 이미 옳았다. 복원은 "빠진 것을 넣는" 것이지 "관계를 새로 만드는" 게 아니다 — 재발이력은 계보가 아니다.

## 판정

**채택(supported).** 문서가 부모/근원/만난다로 명시한 잃은 계보 2건(C018·C057)을 증거 기반으로 복원했고, fsck·conformance 무손상, C043 불변, Sheen 무충돌. C041은 도구 제약(번호-only 언급)으로 정직히 이월 — 되는 것을 하고 안 되는 것을 명시하는 절제(C036·C046 리듬).
