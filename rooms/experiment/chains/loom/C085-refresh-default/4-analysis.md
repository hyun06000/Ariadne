# 4. 결과 분석

## 통계적 결과

| 기준 | 기대 | 관측 | 판정 |
|---|---|---|---|
| 무옵션 실시간 | `content="5"` | `content="5"` | ✅ |
| 옵트아웃 | meta 없음 + bake `"refresh":0` | 동일 | ✅ |
| 명시값 존중 | `content="10"` | `content="10"` | ✅ |
| 옵트아웃 재굽기 보존 | `_bake_meta`→0 | 0 | ✅ |
| 구버전 뷰어 재굽기 | `_bake_meta`→5 | 5 | ✅ |
| WEB-REFRESH-DEFAULT | PASS | PASS | ✅ |
| 회귀 | 0 | baseline과 동일 5 FAIL(전부 gil 미설치로 인한 러너 환경 문제, 변경 무관) | ✅ |

## 데이터 직접 관찰

- `_bake_meta`의 fallback을 `_WEB_DEFAULT_REFRESH`로 바꾼 뒤, bake에 refresh 키가 없는 뷰어(C049 이전 방식으로 구워진 것)를 재굽으면 실시간을 **획득**한다 — 기존 뷰어도 gil step 한 번이면 살아난다.
- bake 조건을 `if refresh`(truthy)에서 `if refresh is not None`으로 바꾼 게 핵심. 이게 없으면 `--refresh 0`(옵트아웃)이 bake에 기록되지 않아 재굽기가 옵트아웃을 5초로 되돌려버린다("명시적 끔"과 "말 안 함"의 구별).

## 예상과 달랐던 것

- **필드 사용자(#24)가 이미 이 마찰의 다음 단계를 진단해 두었다.** meta refresh 기본화는 "통째 리로드(상태 리셋)·고정 주기"라는 정적 파일 방식의 본질적 한계를 남긴다. 이슈 #24는 로컬 HTTP 서버 + SSE로 "변경 시에만 push"하는 `--live`를 프로토타입까지 만들어 제안했다. C085는 그 상위호환(서버)이 오기 전까지, 그리고 서버 없이 정적 파일만 볼 때의 기본 실시간으로 유효하다.

## 판정

**채택(supported).** 기각 조건 셋(옵트아웃 불가·회귀·명시값 무시) 어디에도 걸리지 않았다. C063의 "위계 기본화 + --flat 옵트아웃" 패턴을 refresh에 그대로 재현했다. 후속: #24(`--live` 서버)를 별도 사이클로.
