# 5. 결과 보고

## 요약

참조 구현(gil.py)의 reserve/unreserve/open-예약인식 세 조각을 Go(`go/main.go`)에 문면
이식했다. Go conformance는 65/**65** → **73/73**이 됐고(빠졌던 예약 8항목 전부 PASS),
참조 구현 회귀 0, 예약 관련 생성물은 전부 원장급으로 참조와 바이트 동일하다. **가설 채택** —
두 구현이 무수정 판정기에서 나란히 73/73(완전 계약 병렬).

## 교훈

1. **65/65는 65/73의 위장이었다 — 훅이 분모를 정한다.** 이식 전 Go가 만점으로 보인 것은
   reserve를 자기보고(gil:commands)에 나열하지 않아 판정기가 8항목을 건너뛴 결과였다.
   commandTable에 두 명령을 등록하자 분모가 65→73으로 늘며 동시에 분자도 8 늘었다. **자기가
   무엇을 못 하는지 정직히 신고하지 않는 구현은, 못 하는 것 덕분에 만점으로 보인다.** C036의
   "판정기가 안 보는 계약은 없다"를 자기보고 표면에서 다시 만졌다.

2. **비침습성은 코드가 아니라 데이터의 거처가 보증한다.** 예약이 fsck·verify·log 그래프·web
   그래프에 record로 안 들어가는 것은 방어 코드 때문이 아니라 `reservations.tsv`가
   `<entry>/cycle.yaml`이 아니어서다 — 수집기가 애초에 줍지 않는다. "예약은 사이클이 아니다"를
   파일 위치가 물리적으로 집행한다. 이식할 때 이 불변을 깨지 않으려면 헬퍼를 cycle 수집 경로
   **밖에** 두면 됐다.

3. **web은 동일한데 log만 다르다 — 비대칭이 결함을 국소화한다.** 무예약 web은 참조와 바이트
   동일한데 무예약 log는 다르다(선재 결함: Go `logCmd`가 ASCII 트랙 그래프·`summarize()`를
   애초에 미이식). 두 렌더러가 같은 로더를 쓰는데 출력 대칭이 깨졌다는 사실 자체가 "log
   렌더러만 반쪽 이식됐다"를 정확히 가리킨다. 고치지 않고 보고한다(범위 밖).

4. **cwd도 환경 계약이다 — 워크트리에서 메인으로 cd하지 말 것.** 격리 워크트리 대신 메인
   저장소 루트에서 open을 실행해 open 커밋이 main에 얹혀 push됐다. 즉시 내 브랜치로 회수했으나
   origin/main 복원은 하네스가 차단 → 우회 없이 보고. C020의 "호출자의 cwd도 환경 계약"이
   이번엔 **작업 디렉토리 선택**으로 확장됐다.

## 다음 사이클을 위한 제안

- **(A) Go log 렌더러 완전 이식** — 참조 `render_graph`(ASCII 트랙 그래프 `│ ├─┐ ┴`)와
  `summarize()`(root·분기점·병합점)를 Go에 이식하면 무예약 log도 참조와 바이트 동일해지고,
  conformance LOG 계열이 지금 못 보는 그래프 위상까지 판정에 들어올 수 있다(LOG 계열 항목
  증설의 재료). 이 사이클에서 관측·보고만 하고 남긴 선재 결함의 해소.
- **(B) origin/main 정리** — 소환자(Clew)가 origin/main을 876ad11(예약 커밋)로 되돌릴지,
  아니면 C050 open 커밋(5cded69)을 그대로 둘지 판단. 어느 쪽이든 병합은 Clew의 몫.
- **(C) 예약 만료 자동화** — 현재 unreserve는 수동. 예약에 만료일/TTL을 두고 open이 만료된
  남의 예약을 자동 회수하는 규율(SPEC §6.7 확장). 병렬 존재가 늘수록 유령 예약이 쌓인다.
