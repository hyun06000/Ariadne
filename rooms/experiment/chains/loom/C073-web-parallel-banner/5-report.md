# 5. 결과 보고

## 요약

`gil web`의 두 헤더 아래에 미소비 예약을 집계한 "⟳ 병렬 진행 중" 배너를 렌더했다(gil-data의 reservations 출처, 예약 0이면 부재). 참조 98/98·Go 84/84, parity 3/3 바이트 동일, 회귀 0으로 **채택**. C070 `gil threads`(CLI)에 이어 뷰어도 상현님의 "워크트리 병렬이 실시간으로 안 잡힌다"를 답한다.

## 교훈

1. **없던 건 상단 렌더뿐이었다 (C070의 뷰어판).** 병렬 진행 데이터(reservations)는 loom/C043부터 이미 gil-data JSON에 계약면으로 있었다. 이번 사이클은 계약을 한 줄도 안 늘리고 렌더만 더했다 — CLI(threads)와 뷰어(배너)가 **같은 원장, 같은 진실**을 다른 표면으로 말한다(C042).
2. **배너 CSS는 무조건, 배너 div는 조건부.** 빈 상태 정직(예약0→div 부재)을 지키려면 CSS(항상)와 div(토글)를 분리해야 했다. parity는 둘 다 두 구현이 문자 동일해야 성립 → 정수·문자열만 쓴 이식이 곧 바이트 동일(C064 재확인).
3. **순차 작업엔 워크트리가 없어도 됐다 — 규약 정련의 첫 실천.** 이 사이클은 동시 에이전트 0(넷 다 거둠)이라 main에서 직접 열었다. 상현님이 승인한 "워크트리 트리거 = 사이클당이 아니라 동시성당"을 몸으로 실증. gil.owner=clew guard가 나(소유자)의 main open은 허용.

## 다음 사이클을 위한 제안

- **배너 상호작용**: 칩 클릭 → 그 예약/브랜치의 상태로(원격 브랜치가 push돼 있으니 링크 가능). 정적 → 항해 가능으로.
- **`--refresh`와 결합**: 배너 + `gil web --refresh N`이면 준실시간 병렬 대시보드. push마다 예약 변동이 뷰어에 반영.
- **워크트리 규약 정련 사이클**: 상현님 승인("추천대로") — §6.8·CLAUDE §3·README.ai Step E에 "워크트리 트리거 = 동시성(순차·단독이면 건너뛰기)"을 명문화(C069식 온보딩 문서 사이클). **이 사이클(C073)이 그 실천 사례로 인용될 수 있다.**
- **`gil threads` Go 이식**(C070 이월) / **`gil trace` N홉**(C059 이월, #4 아크).
