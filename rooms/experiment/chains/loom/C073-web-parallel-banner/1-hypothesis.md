# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C070](../C070-threads/5-report.md) — `gil threads`가 미소비 예약(=진행 중 병렬)+열린 사이클을 CLI로 답했다. C070이 남긴 다음 스텝: *"뷰어 병렬 배너 — threads의 예약 데이터를 `gil web` 상단에 얹으면, 상현님의 '워크트리 병렬이 뷰어에 실시간으로 안 잡힌다'가 풀린다. Sheen 착지 뒤 web 충돌 없음."*

이어받는 뿌리:
- **상현님 요청(2026-07-19)**: *"워크트리 병렬은 안전해서 좋지만 뷰어에 실시간으로 안 잡힌다. 최소한 뭐가 병렬로 도는지 알 수 있으면."* threads가 CLI로 답했으니 이제 **뷰어**가 답할 차례.
- **C070 발견**: 예약 원장은 이미 main에 커밋돼 있고, `gil web`은 이미 그것을 gil-data JSON의 `reservations`(계약면)로 싣고 있었다. 없는 건 저장도 계약도 아니라 **상단에 드러내는 렌더**뿐.
- **C021/C063 "렌더는 계약이 아니다, 그러나 무엇을 드러내는가는 잠글 값어치가 있다"**: 배너는 렌더지만 "병렬이 드러난다"는 의도를 판정기로 잠근다.
- **C064 "바이트 parity는 계약"**: 배너는 정수·문자열만(삼각함수 0)이라 이식이 곧 바이트 동일.

## 문제 분할

`gil web`은 위계(기본)·평면(--flat) 두 몸체가 있고 둘 다 헤더를 가진다. 배너는 **헤더 아래 전역 요소**(체인별 아님 — 전 체인 예약을 집계)로 한 번만. 데이터는 이미 있으니(gil-data reservations) 순수 렌더 추가. 첫 카브는 **정적 배너**(생성 시점의 예약 반영, `--refresh`로 준실시간). 클릭→브랜치 이동 같은 상호작용은 범위 밖(이월).

## 설계 선분 사전 고정

- **배너 내용**: 전 체인 미소비 예약을 집계 → "⟳ 병렬 진행 중 (예약, 아직 안 거둬짐): N" + 칩(`chain/C0NN → author`). 예약 0이면 **빈 문자열**(배너 div 부재 → 예약 유무로 div 토글).
- **위치**: 위계·평면 두 헤더 바로 아래. 전역 1회.
- **데이터 출처**: gil-data의 `reservations`(계약면, 이미 있음). 배너는 그 렌더(C021).
- **JS 0**: 순수 HTML/CSS. `role="status"` 접근성.
- **양 구현 바이트 동일**: 정수/문자열만. Go 이식 동반(정수 좌표라 이식=바이트 동일, C064).
- **판정기**: `WEB-PARALLEL-BANNER` — 예약 0이면 배너 부재, 예약 있으면 배너 출현+예약 표기. web은 양 구현에 있어 둘 다 판정.
- **범위 밖**: 배너 클릭→브랜치/워크트리 이동, 다른 브랜치 스캔(예약이 main 표면 제공), 실시간 push 알림(--refresh로 갈음).

## 가설

> **가설**: `gil web`의 두 헤더 아래에 미소비 예약을 집계한 "병렬 진행 중" 배너를 렌더하면(gil-data의 reservations를 출처로, 예약 0이면 배너 부재), (a) 상현님이 뷰어만 보고도 "지금 뭐가 병렬로 도나"를 알고, (b) 데이터가 이미 계약면(gil-data)에 있어 새 계약 없이 렌더만 늘며, (c) 두 구현이 예약 유무 모두에서 바이트 동일하고, (d) 판정기가 배너의 출현/부재를 관측·판정한다(회귀 0).

## 기각 조건 (의도로 읽음 — C011)

1. **parity 붕괴**: 예약 유무 어느 경우든 참조↔Go web이 바이트 다르면 기각.
2. **바이트 오염**: 예약 0인데 배너 div가 뜨거나(빈 상태 오염), 있는데 안 뜨면 기각.
3. **비계약**: 판정기가 배너 출현/부재를 안 보면 기각.
4. **회귀**: 기존 판정기 항목 회귀하면 기각(분모 증가는 회귀 아님).
