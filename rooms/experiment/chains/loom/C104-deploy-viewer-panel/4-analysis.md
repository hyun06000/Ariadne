## 데이터 직접 관찰

- **두 축이 실제로 갈렸다.** 샌드박스엔 CHANGELOG가 없어 릴리스 카드가 0개인데 배포 카드는 1개였다.
  같은 페이지에서 한 축(릴리스)은 데이터가 없어 사라지고 다른 축(배포)만 뜬 것 — 두 축이 별 데이터
  소스(CHANGELOG vs deployments.json)·별 gil-data 키·별 카드 클래스로 완전히 독립함을 한 렌더가 실증했다.
  C102 DEPLOY-NAMESPACE가 뷰어에서도 성립.
- **rollback이 live를 되돌리는 걸 패널이 정직히 비췄다.** churn-model은 v2.0.0을 cut(v1.9.0 superseded)한
  뒤 rollback했다 → v2.0.0이 rolled-back(↩)이 되고 v1.9.0이 다시 live(●)가 됐다. 패널의 live 배지가
  "v1.9.0"을 가리켰다 — deployments.json의 status 전이를 빌더가 추론 없이 그대로 읽어(가장 최근 live
  레코드) 옮겼다. 아티팩트당 live 1 불변식이 시각으로 확인됐다.
- **다중 근거가 복수 링크로 폈다.** v2.0.0은 근거 2사이클(C001·C002)이라 depcyc 칩이 2개 나왔고, CDP로
  하나를 클릭하니 그 cycdoc이 열리며 5스텝 body가 채워졌다. C091의 릴리스→근거 링크 방식(`#cycdoc-*`)이
  배포에서도 그대로 작동 — 새 기제 없이 기존 앵커 재사용(C004 절제).

## 예상과 달랐던 것

- **"바이트 동일"은 이 변경엔 애초에 성립 불가였다 — 그리고 그건 결함이 아니었다.** M2를 짤 때 나는
  C002/C003처럼 `cmp` 바이트 동일을 기대했으나, 배포 패널은 **CSS를 항상 인라인**한다(릴리스 패널 C006도
  동일). 그래서 무파일 저장소도 CSS 블록만큼 바이트가 는다. 처음엔 "회귀"로 보였지만, tag-split diff로
  차이가 **CSS/JS 상수에만** 있고 HTML 본문 콘텐츠엔 0임을 확인하니, 이것이 C006이 이미 확립한 계약임이
  드러났다 — 하위호환의 단위는 "바이트"가 아니라 "렌더되는 카드/데이터"다. 실측이 오진(회귀)을 갈랐다.
- **폴링이 오히려 배포 패널을 안전하게 감쌌다.** C014 지뢰(열린 노드가 폴링 스왑에 깨짐)를 걱정했으나,
  배포 패널엔 중첩 details가 없어 그 자체는 스왑돼도 무해했고, 열린 cycdoc 마운트는 기존 detKey 가드가
  이미 지켰다. 새 가드가 불필요했다 — 기존 상태보존 설계(C010~C014)가 새 카드를 공짜로 덮었다.
