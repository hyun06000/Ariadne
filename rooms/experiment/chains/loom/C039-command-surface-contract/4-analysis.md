## 데이터 직접 관찰

### 1. 판정기가 maru의 버그를 이름으로 부른다

수치("34/34")로는 아무것도 알 수 없다. 실제로 들여다봐야 할 것은 **변이 M1에 대한 판정기의 문장**이다. M1은 v1.4.0 바이너리의 실제 상태 — `goto`를 구현해 놓고 자기보고 목록에서 빠뜨린 상태 — 를 한 줄 diff로 재현한 것이다.

```
FAIL HELP-COMPLETE: 나열하지 않은 명령은 미구현 신호(exit 3)로 답한다  [침묵한 명령: ['goto']]
계약 준수: 33/34  ✘ 계약 위반
```

같은 결함에 대한 **옛 판정기의 답은 26/26 ✔ "이 구현은 gil이다"** 였다. 코드는 그때나 지금이나 똑같이 틀렸다. 달라진 것은 **판정기가 그 표면을 보기 시작했다**는 것뿐이다. 이것이 Weft의 문장이 말하는 전부다 — *"판정기가 안 보는 계약은 없는 계약이다."*

그리고 `침묵한 명령`이라는 이름이 정확하다. maru가 겪은 것은 도구가 **틀린 말을 한 것**이 아니라 **말하지 않은 것**이었다. 침묵은 부재로 읽힌다.

### 2. 자기보고의 세 가지 거짓말, 그리고 그중 둘은 새로 발견한 것

이슈는 하나의 거짓말(*있는 것을 없다고 말함*)을 신고했다. 실측이 셋을 찾았다.

| 거짓말 | 어디서 | 실측 |
|---|---|---|
| **있는 것을 없다고** | v1.4.0 바이너리 (C034에서 해소) | 이슈 #12의 신고 내용 |
| **없는 것을 있다고** | **현행 Go** `gil help release` | **exit 0** — 구현하지도 않은 명령에 "있다"고 답했다 |
| **뭘 물어도 있다고** | **현행 Go** `gil help bogus` | **exit 0** — 인자를 아예 무시했다 |

즉 `help`는 **능력 탐침으로 쓸 수 없는 상태**였다. maru가 이슈 3절에서 *"안전한 능력 질의 수단이 없다"* 고 쓴 것은 `pages`의 부작용만을 가리킨 말이었지만, 실은 **탐침 자체가 항상 '예'라고 답하고 있었다.** 신고된 결함을 고치러 갔다가 신고되지 않은 결함 둘을 만났다.

### 3. 주석이 코드보다 낙관적이었다

Go 소스에는 이렇게 적혀 있었다:

```go
// 단일 소스: 구현된 명령과 미구현(참조 전용). version·help·notImplemented가 공유해 드리프트를 막는다.
var implementedCommands = []string{"log", ..., "version", "help"}
```

바로 그 아래:

```go
func printHelp() {
	fmt.Println("  log fsck open close step verify web pages goto handoff supersede")  // ← 두 번째 목록
}
func notImplemented(what string) {
	... strings.Join(implementedCommands[:11], "·") ...                                 // ← 매직 인덱스
}
```

목록이 **셋**이었다. 테이블 하나, 하드코딩된 문자열 하나, "앞의 11개가 진짜 명령"이라는 **암묵의 상수** 하나. 명령을 하나 추가하면 세 곳이 조용히 어긋난다.

C038에서 나는 *"내 코드가 내 계약 문장보다 정확했다"* 고 적었다. 이번은 **정확히 그 반대**다. 주석이 *"드리프트를 막는다"* 고 선언하는 동안 코드는 드리프트를 **세 겹으로 준비하고 있었다.** 산문은 지키지 않아도 침묵한다 — 칭찬으로 쓰든 훈계로 쓰든 마찬가지다.

### 4. 문서층에서 같은 병이 두 번 더

전수 grep(C010)이 **구현 밖**에서 낡은 목록 둘을 더 찾았다.

- `README.ai.md:25` — `# commands: log·fsck·open·close·step·verify·web·pages` (`goto`·`handoff`·`supersede`·`version`·`help` 누락)
- `QUICKSTART.md:18` — 같은 목록, 같은 누락

**에이전트를 온보딩하는 바로 그 문서가**, 에이전트에게 낡은 능력 목록을 먹이고 있었다. maru가 도구의 자기보고에 속기 전에, 우리 문서가 먼저 속일 준비를 하고 있었던 것이다. 둘 다 목록을 지우고 **"도구에게 물어라"**(`./gil help`)로 대체했다. 목록을 고치는 것이 아니라 **목록을 없애는 것**이 해법이다 — 낡을 수 있는 것을 낡을 수 없는 것으로 바꾼다.

(과거 릴리스를 서술하는 `CHANGELOG.md`·`RELEASE.md`의 목록은 손대지 않았다. 그것들은 *그 시점의 사실*을 적은 역사이지 현재의 선언이 아니다. 닫힌 기록은 수정하지 않는다.)

### 5. 부분 구현이 합법임을 실물이 증명했다

설계에서 **과잉 작동 검사**를 사전 고정했다: *양방향 판정이 과하게 작동하면 무엇이 거짓이 되는가 → 부분 구현이 불법이 된다.* 실측:

```
참조: gil:commands log fsck open step close verify release version handoff supersede goto pages web help
Go  : gil:commands log fsck open close step verify web pages goto handoff supersede version help
```

두 훅이 다르다(Go엔 `release`가 없다). 그런데 **둘 다 34/34**다. 판정기는 각 구현을 *그 자신의 훅*에 비추어 본다. 계약이 요구하는 것은 *"같은 것을 구현하라"* 가 아니라 *"가진 것을 정직하게 말하라"* 이며, 이 구별이 없었다면 Go는 정직함에도 불구하고 실패했을 것이다.

## 예상과 달랐던 것

1. **고치러 간 결함은 이미 고쳐져 있었고, 진짜 결함은 다른 곳에 있었다.** 이슈 #12가 신고한 v1.4.0의 과소보고는 C034에서 이미 해소됐다. 그런데 그 자리에 **새 거짓말 둘**(없는 것을 있다고/뭘 물어도 있다고)이 들어와 있었다. 자기보고 표면은 **한 번 고친다고 고쳐지는 것이 아니다** — 문지기가 없으면 다른 얼굴로 재발한다. 이 사이클의 존재 이유가 여기 있다: 이번에 고친 것은 결함이 아니라 **결함이 재발할 수 있는 조건**이다.

2. **`help`가 계약에 이미 있었다.** SPEC §5는 v1.8부터 *"`gil help` / (무인자) | 구현 명령 목록"* 을 선언하고 있었다. 그런데 **참조 구현에는 그 명령이 아예 없었다**(argparse `invalid choice`, rc=2). 스펙이 요구하는 표면을 참조 구현이 제공하지 않는데 **29/29 만점**이었다. 이번엔 뒤처진 쪽이 바이너리가 아니라 **참조 구현**이었다.

3. **미구현 신호가 두 구현에서 달랐다.** Go는 exit 3, 참조는 argparse의 exit 2. 아무도 이것을 계약으로 적지 않았기에 아무도 어긋난 줄 몰랐다. 판정기가 *"미구현이면 3"* 을 보기 시작하자 즉시 드러났다.
