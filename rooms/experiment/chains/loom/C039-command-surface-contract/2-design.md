# 2. 실험 설계

가설의 세 조항을 각각 **기계가 판정 가능한 형태**로 만들고, 그 판정기가 실제로 드리프트를 잡는지 **변이 주입**으로 확인한다.

## 현행 실측 (설계의 출발점)

스텝 1에서 두드린 결과를 여기 고정한다. 이것이 설계가 고쳐야 할 대상이다.

| # | 관측 | 실측값 | 문제 |
|---|---|---|---|
| O1 | 참조 구현 `gil.py help` | exit 2 (argparse `invalid choice`) | **SPEC §5가 요구하는 명령이 없다** |
| O2 | Go `printHelp()` | 하드코딩된 문자열 `"log fsck open close …"` | 목록이 **셋**이다: `implementedCommands` 테이블 · printHelp의 리터럴 · `implementedCommands[:11]`의 **매직 인덱스** |
| O3 | Go `gil help release` | **exit 0** | 구현하지도 않은 명령을 "있다"고 답한다 — **탐침이 거짓말한다** |
| O4 | Go `gil help bogus` | **exit 0** | 인자를 무시한다. 뭘 물어도 "있다" |
| O5 | Go `gil release` (미구현) | exit **3** + stderr `미구현: …` | 정직하다 (§5.1 준수) |
| O6 | 참조 `gil.py bogus` | exit **2** (argparse) | 미구현 신호가 Go(3)와 **다르다** |
| O7 | `gil pages` 무인자 | 파일 생성 | **탐침이 저장소를 변경한다** (maru 3절) |

> **O2가 이 사이클의 심장이다.** Go 소스의 주석은 *"단일 소스: … 드리프트를 막는다"* 라고 적혀 있는데, 바로 그 아래 코드가 목록을 **다시 하드코딩**한다. C038의 교훈 — *"내 코드가 내 계약 문장보다 정확했다"* — 의 **정반대 사례**다. 이번엔 **주석이 코드보다 낙관적이었다.** 산문은 지키지 않아도 침묵한다.

## 절차

### 1. SPEC §7.2 — 명령 자기보고 표면 (계약)

§7.1(version 표면)과 **같은 틀**을 명령 표면에 적용한다. 새 규칙을 만들지 않고 기존 구조에 들어가야 한다 (C035: *좋은 규정은 아직 없는 기능의 자리를 비워둔다*).

> **1. 능력 목록.** `gil help`(및 무인자 호출)는 **exit 0**으로 구현 명령 목록을 보고한다. 출력에는 **기계 훅**이 정확히 한 줄 있다:
> ```
> gil:commands log fsck open close step verify web pages goto handoff supersede version help
> ```
> `gil:` 접두 표식은 §7.1의 `gil:version`, 뷰어의 `id="gil-data"`와 같은 계열 — **사람의 출력 안에 기계의 훅을 심는다.**
>
> **2. 단일 소스.** 목록·훅·미구현 메시지는 구현 안의 **명령 테이블 하나**에서 파생된다. 목록을 두 번 적는 것을 금한다.
>
> **3. 안전한 능력 탐침.** `gil help <명령>` 은 **부작용 없이** 그 명령의 구현 여부를 답한다 — 구현했으면 exit 0(사용법), 아니면 **미구현 종료**. 저장소를 변경하지 않는다.
>
> **4. 미구현 신호의 통일.** 미구현·미지 명령은 **exit 3** + stderr `미구현: '<이름>' …`. (기존 관행의 명문화. 참조 구현도 이를 따른다 — O6 해소)
>
> **5. 진실성 — 양방향.**
> - **정방향**: `gil:commands`가 나열한 모든 명령은 dispatch에서 미구현 신호를 내지 **않는다.**
> - **역방향**: §5 계약 명령 중 나열되지 **않은** 것은 dispatch에서 **반드시** 미구현 신호를 낸다.
>   ← *구현했는데 목록에 없다* (maru가 겪은 바로 그 버그) 를 잡는 방향.
>
> **6. 탐침 무해성.** 능력 질의 경로(`help`, `help <명령>`, `pages --dry-run`)는 작업 트리를 변경하지 않는다.

**부분 구현은 합법이다.** 계약은 *"두 구현의 명령 집합이 같아야 한다"* 가 아니라 *"각 구현의 자기보고가 **자기** 구현 집합과 일치해야 한다"* 를 요구한다. Go는 정당하게 `release`를 구현하지 않는다. 판정기는 **각 구현을 그 자신의 훅에 비추어** 판정한다. (스텝 1의 과잉 작동 검사)

### 2. 구현 — 참조 (`gil.py`)

| 변경 | 내용 |
|---|---|
| 명령 테이블 | argparse 서브파서 등록을 **단일 소스**로 삼아 목록을 파생 (`sub.choices`) |
| `gil help` | 신설 — 목록 + `gil:commands` 훅, exit 0 |
| 무인자 `gil` | help와 동일 출력, exit 0 (현재는 argparse 오류 rc=2) |
| `gil help <명령>` | 구현 → 그 명령의 사용법(argparse `format_help`), exit 0 / 미지 → exit 3 |
| 미구현 신호 | 미지 명령 → exit **3** + `미구현:` (argparse의 rc=2 오류 경로를 가로챈다) |
| `gil pages --dry-run` | 생성될 경로만 출력, **파일 미생성**, exit 0 |

### 3. 구현 — Go (`go/main.go`)

| 변경 | 내용 |
|---|---|
| **목록 삼중화 해소** | `printHelp()`의 하드코딩 문자열과 `implementedCommands[:11]`의 매직 인덱스를 제거. 테이블 하나(`이름 → 설명`)에서 목록·훅·미구현 메시지를 전부 파생 |
| `gil:commands` 훅 | help 출력에 한 줄 추가 |
| `gil help <명령>` | **O3·O4 교정** — 구현했으면 사용법 + exit 0, 아니면 **exit 3** (`release`·`bogus` 둘 다 3) |
| `gil pages --dry-run` | 참조와 동일 |

### 4. 판정기 — conformance.py (신설 5항목, 29 → 34)

| 항목 | 판정 |
|---|---|
| `HELP-EXIT0` | `gil help` exit 0 ∧ `gil:commands` 훅 라인이 **정확히 하나** ∧ 비어있지 않음 |
| `HELP-SAFE` | `help`, `help <모든 명령>`, `pages --dry-run` 실행 전후 저장소 **스냅샷 해시 동일** |
| `HELP-TRUTHFUL` | 훅의 **모든** 명령 c: `gil help c` → exit 0 **∧** `gil c` (격리 픽스처, 무인자) → exit **≠ 3** |
| `HELP-COMPLETE` | §5 계약 명령 중 훅에 **없는** c: `gil help c` → exit **3** **∧** `gil c` → exit **3** |
| `PAGES-DRYRUN` | `pages --dry-run` → exit 0 ∧ 생성될 경로 출력 ∧ **파일 미생성**. 대조: 이어지는 `pages`는 실제 생성 |

판정기는 §5 계약 명령 목록을 자기 안에 갖는다. **이것 자체가 목록형 규칙이며 낡을 수 있다** (C038 교훈 2) — 그래서 이 사실을 보고서에 명시적 한계로 남긴다.

### 5. 변이 주입 — 판정기가 진짜 잡는가 (기각 조건 1의 검증)

C011의 방법. 참조 구현을 변이시켜 판정기에 건다. **잡히지 않으면 기각.**

| 변이 | 내용 | 잡아야 하는 항목 |
|---|---|---|
| **M1** | 훅에서 `goto`를 지운다 (구현은 그대로) — ***maru가 겪은 바로 그 버그*** | `HELP-COMPLETE` (목록에 없는데 dispatch가 미구현이 아니다) |
| **M2** | 훅에 없는 명령 `timemachine`을 목록에 추가 (구현 없음) | `HELP-TRUTHFUL` (나열했는데 dispatch가 미구현이다) |
| **M3** | `pages --dry-run`이 파일을 생성하게 | `PAGES-DRYRUN` / `HELP-SAFE` |

**M1이 이 사이클의 진짜 시험이다.** M1은 v1.4.0 바이너리의 실제 상태를 재현한 변이다 — 그때 판정기는 26/26 만점을 줬다. 새 판정기가 M1을 격추하지 못하면, 우리는 같은 버그를 또 배포할 수 있다는 뜻이다.
