# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C073**(web 병렬 배너)에서 `gil web`은 gil-data(계약면)를 렌더만 늘리며 배너를 얹었다. 그 흐름의 반대편 문제가 이번 사이클이다: `gil web`은 **모든 사이클의 5스텝 문서 전문을 한 페이지에 인라인**한다(C063 위계 기본화 → C065 노드 스트림 → C067 가로 노드-엣지 그래프 + `:target` 드릴다운). 접혀 있어도(`<details>`) 브라우저는 전부 파싱·레이아웃한다. 상현님 관찰: **"모든 내용을 다 담다 보니 CPU가 치솟고 로딩이 안 되고, 계보가 깊어질수록 너무 무거워진다."**

## 문제 분할

상현님 지시: **"원인부터 측정하고, 문서 부분은 필요할 때 위치에서 불러서 렌더 — 어차피 원문이 레포에 다 있다."**

이번 사이클의 첫 번째(그리고 가장 작은) 문제는 **원인 측정과 가설 고정**이다. "무겁다"를 감이 아니라 바이트·DOM 수치로 국소화해야, 이후 설계(요청 시 로드)가 진짜 병목을 겨냥한다. 측정 없이 lazy-load부터 짜면 `gil web`의 두 계약(**JS 0줄**, **자기완결/외부 리소스 0**)을 건드리는 큰 개정을 근거 없이 감행하게 된다.

### 측정 결과 (v2.26.0, 128 사이클 / loom 119, 스크래치패드 산출물 web-check.html)

| 구성 | 크기 | 비중 |
|---|---|---|
| **cycdoc (문서 5스텝 전문 인라인)** | 876 KB | **59.4%** |
| 나머지(CSS·구조·그래프 SVG) | ~560 KB | 38.1% |
| gil-data (그래프 데이터 JSON) | 36.5 KB | 2.5% |
| **총 HTML** | **1.47 MB** | 100% |

DOM 태그 5,207개 중 **문서 인라인이 만든 무거운 노드**가 대부분:

| 태그 | 수 | 출처 |
|---|---|---|
| tr / th / td | 2,490 | 문서 안 표 |
| details / summary | 840 | 위계·드릴다운 접이 |
| pre | 415 | 문서 안 코드·정렬 블록 |

- 노드당 문서 평균 **10.5 KB**, 인라인 cycdoc 83개(현재 상태 필터 후).
- **핵심 진단**: CPU가 치솟는 원인은 바이트(876KB)보다 **~4,000개의 무거운 노드(표·pre·details)를 항상 렌더 트리에 올리는 것**이다. `<details>`로 접어도 파싱·DOM 구축·초기 레이아웃 비용은 발생한다.
- **성장 곡선**: 이 59%는 사이클 수에 정비례한다. gil-data(그래프 데이터)는 2.5%로 작고 계보 깊이와 무관하게 가볍다 — **뺄 수 있는 몫은 문서 전문이고, 그 원문은 이미 레포에(`rooms/experiment/chains/<chain>/<id>/*.md`) 있다.**

## 가설

> **가설**: `gil web`이 만드는 HTML 무게의 지배적 몫(≈59%, 노드당 ~10.5KB × 전 사이클)은 **5스텝 문서 전문의 무조건 인라인**에서 오며, 문서를 초기 페이지에서 빼고 **노드를 열람할 때(요청 시) 그 위치에서 렌더**하면 메인 페이지는 **계보 깊이에 무관하게 O(그래프 데이터)** 로 가벼워진다(≈메타+그래프만, 현재 총량의 ~40% 이하). 그래프·계보 렌더는 무손실로 보존된다.

## 방향 확정 (스텝2 진입 시, 상현님 결정)

스텝1의 "요청 시 렌더"를 어떻게 실현할지 상현님과 좁혔다:
- **fetch(외부 파일 로드)는 기각** — `gil web`의 두 계약(JS 0·자기완결) 근거를 확인: C005·SPEC §7이 "자기완결은 이 gil의 전부 — 어디서든(CSP·오프라인·file://) 동작"이라 못박음. fetch는 file://에서 막히고 자기완결을 깬다.
- **상현님 최종 결정: "완전한 앱으로 무조건 통일. JS까지 사용."** 두 모드 분기(로컬 lite / 배포 full) 검토했으나 상현님이 단일 앱으로 통일 선택. 문서 소스는 **한 파일 내장(gil-data JSON)** — file://·Pages·오프라인 어디서든 작동(fetch 불필요, 자기완결 유지), 단일 파일 유지(Pages 배선 불변).
- **무게 해소의 실체**: 바이트는 여전히 크나(문서가 JSON에 있음), **초기 파싱·레이아웃 비용을 없앤다** — 진짜 병목은 항상 렌더 트리에 올라간 ~4,000 노드(표 2,490·pre 415·details 840). JS가 노드 클릭 시에만 그 하나의 DOM을 구축하면, 초기 로드는 그래프+메타만 렌더 → CPU 급락. **"완전한 앱"에서 JS 0 계약은 이 산출물에 한해 개정된다** (SPEC 갱신 동반).

## 기각 조건

- 문서 인라인을 전부 제거해도 메인 HTML이 유의미하게(예: 총량의 30% 이상) 줄지 않으면 → 진단(문서가 지배적 몫) **기각**.
- "요청 시 로드"가 `gil web`의 두 계약(**JS 0줄**, **자기완결·외부 리소스 0**) 중 하나라도 위반 없이는 불가능하고, 그 계약 개정을 상현님이 승인하지 않으면 → 이 접근 **기각**(대안: 그래프 전용 기본 + `--full` 옵트인으로 후퇴).
- 문서를 빼서 가벼워지되 **계보·그래프·드릴다운 진입점이 손상**되면(노드를 눌러도 원문에 닿을 길이 없어지면) → 무손실 요구 **기각**.
