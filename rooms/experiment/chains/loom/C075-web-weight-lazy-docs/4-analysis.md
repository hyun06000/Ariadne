## 데이터 직접 관찰

수정 전 뷰어는 문서 5스텝 전문을 초기 HTML에 인라인해, 접혀 있어도(`<details>`) 브라우저가 전부 파싱·레이아웃했다 — `td`/`tr` 각 850, `pre` 425. 앱화 후 초기 DOM에서 이들이 **2·1·2로 사라졌다**(남은 소수는 페이지 골격). 문서는 gil-data JSON에 텍스트로 이동했고, 헤드리스 Chrome에서 `#cycdoc-loom-C001-...` 해시로 로드하니 JS가 그 `.cycbody`에 메타표와 5스텝(hstep 5개)을 정확히 구축했다 — **가설 내용까지** 렌더. fetch가 아니라 내장 JSON에서 읽으므로 `file://`에서도 동작(자기완결 유지).

## 예상과 달랐던 것

- **파일 바이트는 거의 안 줄었다**(1.51MB→1.46MB). 문서가 DOM에서 JSON으로 자리만 옮겼기 때문. 그러나 **진짜 병목은 바이트가 아니라 초기 렌더 트리의 노드 수**였다 — 브라우저가 로드 시 파싱·레이아웃하는 것은 DOM이지 `<script type=application/json>` 안 텍스트가 아니다. 5,330→1,433 태그가 CPU 급락의 실체. 스텝1의 "876KB(59%)" 관찰이 무게의 *소재*는 맞혔으나, 사용자 증상(CPU 100%)의 직접 원인은 **바이트가 아니라 DOM 노드 수**였다.
- **선재 버그가 앱화로 표면화**: 사이클 문서가 gil web 코드를 인용하며 `</script>`를 포함했는데, 이걸 gil-data JSON에 넣자 브라우저가 스크립트를 조기 종료 → JSON 파싱 실패로 앱 전체가 죽었다. `_json_for_script`의 `</`→`<\/` 치환으로 봉인(C076 "버그가 진단 도중 발현"과 같은 결)했다. 문서를 DOM에 굽던 시절엔 잠복했던 결함.

## 판정

**채택.** 기각 조건 4개(무게 미감소 / 외부 리소스 발생 / 문서 도달 불가 / 그래프 손상) 전부 불발. 초기 DOM 73% 감소로 CPU/로딩 문제 해소, 자기완결 유지(fetch 0), 참조 100/100. **범위(참조 앱화)대로 Go 이식은 정직히 이월.**
