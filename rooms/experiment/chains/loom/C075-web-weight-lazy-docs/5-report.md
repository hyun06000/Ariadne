# 5. 결과 보고

## 요약

`gil web`(위계)이 모든 사이클 5스텝 문서 전문을 초기 HTML에 인라인해, 계보가 깊을수록 브라우저가 항상 파싱·레이아웃하는 DOM이 수천 노드로 불어 **CPU가 치솟고 로딩이 안 됐다**(상현님 신고). 문서를 초기 DOM에서 빼 **gil-data JSON에 내장**하고, **인라인 앱 스크립트**가 노드 클릭 시 그 하나의 DOM을 구축하게 했다(fetch 0 → 자기완결 유지, file://·오프라인·Pages 어디서든 동작). 초기 DOM **5,330→1,433 태그(73%↓)**, `<pre>`·`<td>` 100% 제거. 참조 100/100(WEB-DOCS-EMBEDDED 신설), flat 바이트 동일, 그래프·계보 무손실. SPEC §7 자기완결 정의 명문화(JS 허용, 외부 의존 0). **채택.**

## 교훈

1. **무게의 소재와 증상의 원인은 다르다.** 스텝1은 "문서 876KB(59%)"로 소재를 맞혔으나, 사용자 증상(CPU 100%)의 직접 원인은 **바이트가 아니라 초기 렌더 트리의 노드 수**였다(5,330 태그). 브라우저는 DOM을 파싱·레이아웃하지, `<script type=application/json>` 텍스트는 안 그린다 — 문서를 JSON으로 옮기니 바이트는 그대로여도 초기 DOM이 73% 줄어 증상이 풀렸다. **병목은 "무엇이 크냐"가 아니라 "무엇이 매 로드 렌더되냐".**
2. **계약은 수단이 아니라 목적이다.** "JS 0줄"은 계약처럼 보였으나 SPEC §7의 실제 계약은 "자기완결(어디서든 열림)"이었다. fetch를 기각하고 JS를 허용하되 문서를 내장한 것 — 목적(외부 의존 0)을 지키며 수단(JS 없음)을 놓았다. 상현님의 "왜 지켜야?"라는 반문이 계약의 본질을 캐게 했다.
3. **문서를 데이터로 옮기면 잠복 결함이 표면화된다.** 사이클이 gil web 코드를 인용해 담은 `</script>`가, JSON에 들어가자 브라우저 스크립트를 조기 종료시켰다(`_json_for_script` 봉인). 표현(DOM)에서 데이터(JSON)로의 이동은 이스케이프 경계를 새로 만든다.
