# 2. 실험 설계

## 절차

1. **SPEC §6.8** 항목 8에 새 불릿 "**언제 워크트리인가 — 트리거는 '사이클당'이 아니라 '동시성당'**"을 "왜 필요한가" 앞에 추가: 동시 다중 에이전트+공유 도구 편집일 때 필수, 순차·단독이면 main 직접, 사이클 디렉토리는 산출물만 격리. C073(순차→main)·이 세션 4트랙(동시→워크트리) 예시.
2. **CLAUDE.md §3** "병렬로 일하라" 문단 끝에 한 문장 추가: 워크트리는 동시성일 때만, 순차·단독이면 main 직접(사이클 디렉토리는 산출물만 격리).
3. **README.ai Step E**에 문단 추가: "The trigger is concurrency, not per cycle" — 순차·단독이면 worktree 건너뛰고 main.
4. 도구 코드 무변경(안내-only, C069 패턴). 검증: conformance 불변, fsck 0, web ref↔Go parity 불변.

## 준비물

- SPEC.md(패키지)·CLAUDE.md·README.ai.md(루트 거버넌스). gil.py·go/main.go·conformance.py 무변경.
- main에서 직접(순차·단독 — 이 사이클 자체가 정련하는 규약의 실천).

## 측정 방법

- **계약 불변**: 참조 conformance 98/98 유지(변화 0), fsck 위반 0.
- **자기무모순**: 정련 문구가 §6.8 "네 워크트리에서 일하라"(동시 시 규율)와 충돌하지 않음 — 트리거(동시성)와 규율(동시일 때 격리)은 층위가 다르므로 양립. 문서 정독으로 확인.
- **안전 무손실**: "순차면 건너뛰기"가 "동시에도 생략 가능"으로 읽히지 않게 — 조건을 "동시 다중 에이전트"로 명시(오해 차단).

기준: conformance 98/98·fsck 0 유지 ∧ 세 문서에 트리거 문구 존재 ∧ 자기모순 없음.
