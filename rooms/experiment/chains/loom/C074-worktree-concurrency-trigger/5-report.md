# 5. 결과 보고

## 요약

상현님 질문 *"사이클마다 별도 디렉토리인데 굳이 워크트리가?"*에서 출발해, 세 온보딩 경로(SPEC §6.8·CLAUDE §3·README.ai Step E)에 **"워크트리 트리거 = 동시성(동시 다중 에이전트가 공유 도구 편집), 순차·단독이면 main 직접"**을 명문화했다. 도구 코드 무변경, conformance 98/98·fsck 0 불변으로 **채택**. 이 사이클 자체가 순차라 main에서 열려 규약을 실천했다.

## 교훈

1. **격리의 축을 정확히 명명하라.** 사이클 디렉토리는 산출물을, 워크트리는 공유 도구 소스·동시 작업 상태·깃 히스토리를 격리한다 — 다른 축이다. 트리거를 "사이클당"이 아니라 "동시성당"으로 잡아야 순차의 낭비와 동시의 사고를 둘 다 피한다.
2. **사용자의 의문은 규약의 침묵을 비춘다 (C069 재연).** "굳이?"는 반대가 아니라 트리거 문장의 부재 신호였다. C069가 "병렬을 안내"했다면 C074는 "언제 병렬인가"를 안내한다 — 온보딩은 기능뿐 아니라 **판단 기준**도 전달해야 한다.
3. **트리거와 규율은 층위가 다르다.** "언제 워크트리를 여는가"(트리거)와 "열었으면 어떻게"(main으로 cd 금지, C050)는 순서가 있는 두 층 — 문서에서 트리거를 규율 앞에 놓아 읽는 순서를 맞췄다.
