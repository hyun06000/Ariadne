# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C047**(web 위상 레이아웃)이 `_layout_columns`의 행 좌표를 "토폴로지 순서"에서 **"루트로부터의 깊이(최장 경로)"** 로 바꿨다(발의: 박상현). 형제 갈래를 같은 깊이=같은 행에 나란히 놓아 세로를 압축한 것이다. 그런데 **레인(col) 배치는 여전히 track 시스템(C031, pop 없는 슬롯)** 으로 하는데, C047이 도입한 `while (row,col) in occupied` 회피가 **track 시스템과 통합되지 않았다.** 이 분리가 이번 사이클의 결함이다.

## 문제 분할

**긴급 트리거**: 소비자 저장소 `2ndRound`에서 maru가 eda 체인(44 사이클)으로 `gil open/step/close`를 돌리자, 매 명령이 커밋 후 뷰어 렌더(`_layout_columns`)에서 **무한 스핀** → 프로세스가 종료 못 하고 **50개까지 누적 → CPU 100%, load 100+**. 워치독으로 임시 방어(15초+·40%+ gil 자동 kill) 중이나, 근본은 레이아웃 무한 루프다.

이번 사이클의 가장 작은 문제는 **무한 루프 자체의 종료 보장**이다. web 무게(C075)와 분리한 이유: 스핀은 지금 사용자의 실작업을 막는 **가동 중단(availability)** 문제라 우선순위가 위다.

### 재현·확증 (이미 완료 — 스텝 1에서 계측)

- **양 구현 공통 결함**: eda 44 데이터로 참조 `gil.py`(python)·Go v2.25 바이너리 **둘 다** web 렌더에서 25초+ 스핀(정상은 <2초). Go만의 버그가 아니라 **계약(알고리즘) 버그**다.
- **데이터는 정상**: eda parent 그래프는 순환 없는 정상 DAG(모든 parent가 자기보다 작은 번호, superseded_by 없음). 스핀은 데이터 결함이 아니라 알고리즘 결함이다.
- **계측으로 무한 루프 지점 확정** (`probe_layout.py`, GUARD>100000에서 포획):
  ```
  node=C032 row=12 col=0
  tracks=[None, None, None, None, None, None, None, None]   ← 전부 None
  occupied at row 12: [0, 4, 6]                              ← (12,0) 이미 점유
  ```
- **메커니즘**: `(12,0)`이 occupied라 `while` 루프 진입 → `tracks[0] != node`라 track을 안 비우고 `free_slot()` 호출 → **tracks가 전부 None이라 free_slot이 또 0 반환** → `(12,0)` 여전히 occupied → **col=0 영원히 반복.** `free_slot()`은 "빈 track"을 찾지만 while은 "빈 (row,col) 좌표"를 원한다 — 이 둘이 분리돼, tracks가 비었는데 그 좌표가 점유된 상황에서 col이 **전진할 수단이 없다.**
- **유발 조건**: 한 부모에 자식이 여럿인 분기가 반복돼(eda: C009→C010·C020·C025, C034→C035·C036 등) track 예약이 쌓이고, 흡수·병합으로 tracks가 비면서 특정 깊이에서 free_slot이 점유된 col만 반환하게 된다.

## 가설

> **가설**: `_layout_columns`의 무한 스핀은 `free_slot()`(빈 **track** 탐색)과 `while (row,col) in occupied`(빈 **좌표** 요구)가 분리돼, tracks가 비었으나 그 좌표가 점유된 상황에서 col이 전진하지 못하는 것이 원인이다. **occupied 회피를 좌표 기준으로 통합**(예: occupied면 col을 단조 증가시켜 반드시 미점유 (row,col)을 찾음)하면, 임의의 유한 DAG에서 `_layout_columns`는 **항상 유한 스텝에 종료**하며, 기존 정상 그래프(loom 등)의 좌표는 불변이다.

## 기각 조건

- 통합 수정 후에도 eda 44에서 스핀하거나(GUARD 초과), 어떤 유한 DAG에서 종료가 보장되지 않으면 → 수정 **기각**.
- 수정이 **기존 그래프(loom·loomlight 등)의 노드 좌표를 바꾸면**(web 출력 바이트가 달라지면) → 하위호환 위반, **기각**(회귀). 정상 케이스는 반드시 바이트 동일해야 한다.
- 참조·Go 두 구현이 수정 후 같은 좌표를 내지 못하면(parity 깨짐) → 계약 위반, **기각**.
- 판정기가 이 스핀(종료 실패)을 관측·판정하지 못하면 → "판정기가 안 보는 계약은 없는 계약"(Weft), 회귀 방어 미완 → 판정 항목 신설 요구.
