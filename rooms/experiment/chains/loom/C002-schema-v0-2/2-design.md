# 2. 실험 설계

## 절차

1. **스펙 초안 고정** — `3-verification/schema-v0.2-draft.md`에 규칙 R1~R8과 이주 규정을 작성한다. *(C001 관행의 계승: 규칙을 도구보다 먼저 고정한다.)*
2. **기대 위반 목록 고정** — 위반 픽스처가 일으켜야 하는 위반의 전체 목록을 `fixtures/expected-fsck.md`에 먼저 기록한다.
3. **픽스처 생성** — (a) `fixtures/bad/chains/`: 규칙별 위반을 하나씩 담은 사이클들, (b) `fixtures/good/chains/`: 체인 간 lineage를 포함한 규칙 준수 체인 2개.
4. **구현** — C001의 ari.py를 계승·확장(`3-verification/ari/ari.py`): `lineage` 필드 파싱, log의 lineage 주석 표시, `fsck` 서브커맨드(위반을 전부 수집해 보고, 첫 위반에서 중단하지 않음). 표준 라이브러리 전용 유지.
5. **픽스처 검증** — fsck를 bad에 실행 → 기대 위반 목록과 **누락 0·거짓 양성 0**으로 일치해야 한다. good에 실행 → OK.
6. **실데이터 이주** — v0.2 규칙에 맞춰 저장소를 이주한다: `_template/cycle.yaml`에 lineage 필드 추가, loom/C001의 주석으로만 있던 계보(genesis에서 태어남)를 `lineage` 필드로 승격. 이주 전후 diff를 기록해 무손실을 확인한다.
7. **실데이터 검증** — 이주된 저장소에 fsck(→ OK여야 함)와 log(→ loom 체인의 첫 실제 간선 C001→C002와 lineage 주석이 보여야 함)를 실행한다.

## 준비물

- C001의 산출물: ari.py v1, 모호성 5건 목록 ([4-analysis.md](../C001-lineage-is-reconstructable/4-analysis.md))
- Python 3 표준 라이브러리만 (계승)

## 측정 방법

| # | 항목 | 통과 기준 |
|---|---|---|
| 1 | 위반 탐지 | fsck 출력이 `expected-fsck.md`와 일치 — 누락 0, 거짓 양성 0 |
| 2 | 정상 통과 | good 픽스처와 이주된 실데이터에서 fsck 위반 0 (exit 0) |
| 3 | 무손실 이주 | 이주 diff에 의미 변화 없음 — 주석으로만 있던 정보의 필드 승격과 스키마 필드 추가뿐 |
| 4 | log 동작 | 이주 후 log 정상 렌더 + loom의 실제 간선 + lineage 주석 표시 |
| 5 | 규칙 전환 완전성 | 모호성 5건 각각이 R1~R8 중 최소 하나에 대응됨을 스펙 문서의 대응표로 확인 |

전부 통과 → 채택. 하나라도 실패 → 기각, 실패 양상이 다음 사이클 재료.
