# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C006](../C006-spec-release/5-report.md) (채택·닫힘).

- Spec v0.1.0이 나왔고 "문서가 곧 테스트" 관행이 확립됐다. 제안 (A): **Actions + Pages 배선** — push마다 `ari web`을 돌려 정적 배포하면, "모두의 공유"가 말이 아니라 URL이 된다.
- C005의 검증: `ari web`의 산출물은 자기완결적이라 어떤 정적 호스팅에서도 동작한다. 남은 것은 배선뿐이다.
- **제약**: 이 저장소는 아직 GitHub 원격이 없다. Actions의 실행 자체는 로컬에서 재현할 수 없으므로, 가설은 로컬에서 반증 가능한 것으로 한정해야 한다.

## 문제 분할

"자동 배포"는 (a) 빌드 — 체크아웃된 저장소에서 사이트 생성, (b) 업로드/서빙 — GitHub 제공 액션,
(c) 원격 끝단 확인으로 나뉜다. (b)는 GitHub의 검증된 부품이고 (c)는 원격이 생겨야 가능하다.
우리가 소유하고 반증할 수 있는 것은 **(a)와 워크플로 정의의 정합성**이다.

## 가설

> **가설**: 워크플로 정의(`.github/workflows/ariadne-pages.yml`)의 빌드 스텝(`run:` 블록)을 **그대로 추출해** 신선한 클론에서 실행하면, ① 배포 가능한 정적 사이트(`_site/index.html`)가 생성되고 그 내장 JSON의 사이클 집합은 클론의 파일시스템과 일치하며, ② 산출물은 자기완결적이고, ③ 빌드는 python3와 저장소 자신(배포된 v0.1.0 도구) 외에 어떤 설치도 요구하지 않으며, ④ 워크플로 정의는 Pages 배포에 필요한 구조(트리거·권한·업로드·배포)를 갖춘다.

*C006 관행의 연장: 검증이 워크플로 파일에서 명령을 추출하므로, 워크플로가 낡으면 테스트가 깨진다.*

## 기각 조건

1. **재현 실패**: 신선한 클론에서 추출된 빌드 스텝이 실패하거나, 내장 JSON이 클론의 사이클 집합과 다르다.
2. **자기완결 실패**: 생성된 사이트에 외부 리소스 참조가 있다.
3. **숨은 의존성**: 빌드 스텝에 패키지 설치(pip/npm 등)나 네트워크 접근이 필요하다.
4. **불완전 워크플로**: push 트리거·pages 권한·업로드·배포 액션 중 결여가 있다.

## 한계 (정직하게)

Actions 러너에서의 실제 실행과 Pages 끝단은 원격이 생긴 뒤 사용자 확인 사항으로 남는다.
이 사이클이 보증하는 것은 "빌드는 재현 가능하고 정의는 정합하다"까지다.
