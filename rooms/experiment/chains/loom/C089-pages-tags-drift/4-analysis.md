# 4. 결과 분석

## 통계적 결과
| 시나리오 | 기대 | 관측 | 판정 |
|---|---|---|---|
| A 태그있음 | readable True·drift 없음 | 일치 | ✅ |
| B CI모사(태그0) | readable False·배지 억제 | 일치 | ✅ |
| C 진짜 drift | 배지 유지 | 일치 | ✅ |
| D 워크플로 fetch-depth:0 | 포함 | 포함 | ✅ |
| E 회귀 | 0 | baseline 동일 | ✅ |

## 데이터 직접 관찰
- **같은 데이터 계약을 CLI와 뷰어가 다르게 처리하고 있었다.** CLI `cmd_releases`는 `git_absent`로 "태그 못 읽음"을 이미 구별했는데, 뷰어 `_build_releases_data`는 그 구별 없이 `tags={}`로만 처리해 in_tag=False를 뿌렸다. 같은 진실(drift)을 보는 두 표면이 어긋난 것 — C061의 "조회와 게이트가 같은 헬퍼를 봐야 한다"가 조회의 두 표면(CLI·뷰어)에도 적용됐어야 했다.
- **환경이 진실을 가릴 때 도구는 '모른다'고 말해야 한다.** 태그를 못 읽는 CI에서 "태그 없음(=drift)"이라 단정하면 거짓이다. tags_readable=False로 "대조 불가"를 표현해 오탐을 억제 — 지어냄 0 원칙(§3.2)의 배포 표면판.
- **두 층 방어가 옳았다.** 워크플로 수정(근본)만으로도 고쳐지지만, 뷰어 강건성(방어선)이 없으면 옛 워크플로를 쓰는 다른 저장소·수동 CI에서 재발한다. 근본+방어를 함께.

## 예상과 달랐던 것
- 뷰어 렌더 로직([2164])은 처음부터 정확했다 — 버그는 렌더가 아니라 **데이터 생성 시 환경 판별 부재**였다. "전부 CHANGELOG만"이라는 증상이 렌더가 아닌 데이터 층을 가리켰다(entries의 in_tag가 이미 False).

## 판정
**채택(supported).** 기각 조건 셋(fetch-depth 무효·진짜 drift 은폐·회귀) 어디에도 안 걸림.
