# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C060](../C060-worktree-land/5-report.md)(Weft, worktree land) · 연원: [loomlight/C002](../../loomlight/C002-hierarchical-viewer/5-report.md)(Sheen, 병렬 사고 실증).

상현님 발의: **"발생한 사고를 도구 차원에서 막는 방법을 gil에 넣자."** 이번 세션 병렬 4트랙에서 C050 사고가 **세 번** 재발했다 — Weft(C060 open)·Selvage(C061 open)가 각자 워크트리 밖 **공유 main 체크아웃으로 cd**해 `gil open --git --push`를 실행 → open 커밋이 main에 유출되고, 그 과정의 git 정리가 내 미커밋 cmd_show WIP를 wipe(무손실 복구). **하네스 워크트리 격리로도 못 막았다** — 에이전트가 워크트리 밖으로 cd할 수 있으므로.

교훈의 뿌리: **C038 "규율이 문서의 훈계로만 남으면 새어나간다"** + **C058 "규율을 도구로 승격하면 사고가 구조적으로 불가능해진다."** 소환 규약 v3의 "워크트리에서 일하라"는 훈계로만 있어 세 번 잊혔다. add(C058)→land(C060)가 세운 병렬 모드에 **셋째 조각 guard**가 필요하다.

## 문제 분할 — 무엇을 지킬 수 있는가

gil이 규제할 수 있는 경계를 실측으로 가른다:
- **링크드 워크트리**(harness가 만든 것 포함): gil은 이것의 소유자를 **모른다**(harness가 gil 밖에서 만듦). 여기를 규제하려 하면 **정당한 존재 작업을 오탐 차단**한다. → 규제 대상 아님.
- **주 체크아웃**(primary worktree = 공유 main, `git-dir == git-common-dir`): **모든 존재가 공유하는 유일한 공간**이자 **정확히 세 사고가 전부 일어난 곳**. 존재는 여기서 절대 커밋하면 안 된다(자기 워크트리에서만). Clew(주 작업자)만 여기서 일한다.

→ **첫 카브: 주 체크아웃을 소유자 전용으로.** 링크드 워크트리는 건드리지 않는다(존재의 정당한 작업공간). 이 비대칭이 오탐 0과 사고 차단을 동시에 만든다 — C044 "디렉토리 경계는 양날, 무엇이 그것을 보느냐가 축복과 저주를 가른다"의 소유권판.

분할:
1. **[이번 C062] 주 체크아웃 소유 guard** — 주 체크아웃에서 커밋하는 gil 뮤테이션은 `--author`가 주인과 다르면 거부.
2. **[다음] 링크드 워크트리 소유 표식**(`gil worktree add`가 소유자 각인 → 존재가 남의 워크트리에서 커밋하는 것도 차단). add가 만든 워크트리는 gil이 아니까 규제 가능.
3. **[다음] 소환 문서 규율** — README.ai.md·SPEC §6에 guard를 명문화.

## 설계 선분 사전 고정

- **주인 기록**: 저장소 config `gil.owner`(주 체크아웃의 주인 이름). 설정 수단은 설계에서 확정(`gil worktree add`가 주 저장소에 최초 각인 / 명시 설정).
- **guard 지점**: 커밋을 수행하는 모든 뮤테이션(open·step·close·reserve·round·correct·supersede·worktree)의 **커밋 직전**. 주 체크아웃(`git-dir==git-common-dir`) ∧ `gil.owner` 설정됨 ∧ `--author ≠ gil.owner` → **거부**(저장소 무변화, open 계열 규율 계승).
- **오탐 방지**: 링크드 워크트리에서는 guard 미적용(존재의 정당한 공간). `gil.owner` 미설정이면 미적용(기존 저장소 무파손 — opt-in, 우리 저장소는 설정).
- **거부 메시지는 처방을 준다**(C040 "도구가 뱉는 출력을 에이전트가 실행한다"): "이 체크아웃은 <owner>의 주 작업공간이다. author <X>로 여기서 커밋할 수 없다 — 네 워크트리에서 실행하라(gil worktree add)."
- **범위 밖**: 링크드 워크트리 소유 표식(카브 2), Go 이식(Weft 도메인 유력 — 이번엔 참조 먼저 + 판정기, 회귀 처리는 설계에서).

## 가설

> **가설**: gil이 저장소 config `gil.owner`를 주 체크아웃의 주인으로 읽고, 커밋을 수행하는 뮤테이션이 **주 체크아웃에서 `--author ≠ gil.owner`이면 거부**(저장소 무변화)하게 하면, (a) 존재가 워크트리 밖 공유 main으로 cd해 `gil open` 하는 C050 사고가 **커밋 이전에 구조적으로 차단**되고, (b) 링크드 워크트리(harness·gil 생성 불문)의 정당한 존재 작업은 오탐 없이 통과하며, (c) 주인(Clew)의 주 체크아웃 작업은 그대로 통과하고, (d) 판정기가 이 거부/허용을 관측한다.

## 기각 조건 (문면 아닌 의도 — C011)

1. **사고 미차단**: 주 체크아웃에서 남의 author로 open이 커밋되면 기각(C050 재현 = 실패).
2. **오탐**: 링크드 워크트리에서 정당한 author 작업이 거부되면 기각(존재의 작업공간을 막으면 안 된다).
3. **주인 차단**: `gil.owner`와 같은 author의 주 체크아웃 작업이 거부되면 기각.
4. **비원자성**: 거부 시 저장소가 변하면 기각(부분 커밋 잔여).
5. **회귀**: 기존 판정 항목 회귀 시 기각.
