# 5. 결과 보고

## 요약

상현님 발의 **"발생한 사고를 도구 차원에서 막자"**의 구현. 이번 세션 병렬 4트랙에서 존재가 워크트리 밖 공유 main으로 cd해 `gil open`을 실행하는 C050 사고가 **세 번** 재발했다(하네스 워크트리 격리로도 못 막음). **주 체크아웃 소유 guard** — `open`·`correct`가 주 체크아웃에서 `--author ≠ gil.owner`이면 커밋 이전에 거부(저장소 무변화). 양 구현, 참조 **89/89**·Go **82/82**, GUARD-* 3항목, 변이 격추. 우리 저장소에 `gil.owner=clew` 활성화. 판정 **채택(supported)**. #1 병렬 모드의 셋째 조각(add→land→**guard**).

## 교훈

1. **규율을 도구로 승격하면 사고가 구조적으로 불가능해진다** (C038·C058의 안전판). "워크트리에서 일하라"(소환 규약 v3)는 훈계로만 있어 **세 번** 잊혔다. 이제 도구가 커밋 이전에 거부한다 — 존재가 잊어도, 하네스 격리가 뚫려도.
2. **무엇을 규제하지 *않는가*가 오탐을 없앤다** (C044 "디렉토리 경계는 양날"의 소유권판). gil은 링크드 워크트리(harness 생성 포함)의 소유자를 모른다 — 규제하려 하면 정당한 존재 작업을 오탐 차단한다. **주 체크아웃(모든 존재가 공유하는 유일한 공간, 세 사고가 전부 난 곳)만** 규제하니 오탐 0 + 사고 차단이 동시에. `git-dir==git-common-dir`이 그 경계.
3. **guard의 author는 행위자여야지 목적어가 아니다.** reserve의 `--for X`는 예약 대상(소환자가 남을 위해 main에서 예약 = 정당) → guard하면 오탐. open·correct의 `--author`는 행위자 → guard 대상. **같은 필드 이름(author)도 의미가 다르면 다르게 다뤄야 한다** — 구현이 설계를 정련했다(C007 "실측이 설계를 정한다").
4. **소유는 git 자신의 모델을 훔친다** (C016 "깃의 답을 훔쳐라"). `gil.owner`는 `user.name`처럼 클론-로컬 config·opt-in — 미설정이면 미적용이라 기존 저장소·CI 무파손. git이 소유를 로컬로 다루듯 우리도.
