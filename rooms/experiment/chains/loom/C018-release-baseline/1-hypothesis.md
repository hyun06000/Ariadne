# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C016](../C016-number-ledger/5-report.md) (채택). 근원: [C012의 발견 2건](../C012-go-binary-log-fsck/4-analysis.md).

- **공백 ①**: 도구 변경 감지가 "실행 파일 vs 패키지 파일" 비교다. 패키지의 gil.py를 직접 실행하면 자기 자신과 비교하므로 변경이 절대 보이지 않는다 — 기준은 **마지막 릴리스 태그의 blob**이어야 한다.
- **공백 ②**: 승격 규칙이 gil.py만 도구로 간주해, 판정기(conformance.py) 변경이 "문서 릴리스"로 오분류됐다 (v0.5.0에서 실제 발생).

## 가설

> **가설**: 도구 변경 판정의 기준을 마지막 릴리스 태그의 blob(`git show v<last>:<경로>`)으로 바꾸고 gil.py·conformance.py 둘 다 도구로 간주하면, ① 패키지 도구 직접 실행 시나리오에서도 변경이 감지되어 패치가 거부되고, ② 판정기-only 변경도 마이너를 요구하며, ③ 문서-only 릴리스와 기존 conformance는 회귀 없이 유지된다.

## 기각 조건

1. T1(직접 실행 시나리오)에서 패치가 통과하거나, T2(판정기 변경)가 패치로 통과한다.
2. T3(문서-only) 패치가 거부되거나 CHANGELOG 분류가 틀린다.
3. conformance 26항목 회귀.
