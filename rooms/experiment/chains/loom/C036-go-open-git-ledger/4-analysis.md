# 4. 결과 분석

수치만 보지 않는다. 아티팩트를 직접 들여다본다 (C020의 교훈: **인용은 기억이 아니라
아티팩트에서**). 아래 인용은 모두 `3-verification/runs/`의 원문에서 왔다.

## 통계적 결과 — 기각 조건 다섯 개 모두 반증됐다

스텝 1에서 **미리** 정한 기각 조건에 결과를 그대로 맞댄다.

| # | 기각 조건 (사전 등록) | 측정 | 결과 |
|---|---|---|---|
| 1 | 26항목 중 하나라도 PASS→FAIL | run0 ↔ run1 항목명 대조 | **퇴행 0** |
| 2 | 새 항목을 참조 구현이 통과 못함 | run2 | **참조 28/28** |
| 3 | Go가 경합을 해소 못함 | run3 | **T1~T3 3/3** |
| 4 | 두 구현의 최종 상태 불일치 | run5 | **sha256 동일** |
| 5 | 경합을 검사할 항목 부재 | run1 | `OPEN-PUSH-RENUMBER` 실재 + PASS |

**기각 조건 1의 기계적 반증** — run0(이식 전, 26항목)과 run1(이식 후, 28항목)의
PASS 항목명 diff:

```
26a27,28
> PASS OPEN-GIT
> PASS OPEN-PUSH-RENUMBER
```

기존 26항목의 이름과 순서가 **한 글자도 바뀌지 않았고**, 전부 PASS인 채로 두 항목이
뒤에 붙었을 뿐이다. 퇴행은 "FAIL이 없다"로 보이는 게 아니라 **"기존 항목 집합이 보존된
채 전부 PASS"** 로 보여야 한다 — 항목이 조용히 사라지면 FAIL 0도 거짓말이 되기 때문이다.

집계: **Go 26/26 → 28/28**, **참조 구현 28/28**, 원장 실증 **양 구현 3/3**.

## 데이터 직접 관찰

### (a) 원장 규율은 "재시도"가 아니라 "정정 후 재시도"다

가장 중요한 관찰은 숫자가 아니라 **원장의 커밋 목록**이다 (run8):

```
--- B의 최종 커밋 전문 (--amend가 정정한 메시지):
    gil: open demo/C003-beta-path — 1/5 가설

    B의 폭
    (원장 경합 재번호: C002-beta-path → C003-beta-path)

--- 원장(bare)이 실제로 받은 커밋:
    gil: open demo/C003-beta-path — 1/5 가설
    gil: open demo/C002-alpha-path — 1/5 가설
    seed
```

**원장은 `C002-beta-path`를 본 적이 없다.** 잘못된 번호를 담은 커밋은 push되었다가
취소된 게 아니라 **애초에 원장에 도달하지 못했다** — `git mv` + `id:` 치환 +
`commit --amend`가 로컬에서 커밋 자체를 다시 쓴 뒤에야 push가 성공했기 때문이다.

이것이 반쪽 이식이 위험한 이유의 실물이다. 커밋만 이식했다면 B의 커밋은
`C002-beta-path`인 채로 남아, 최선의 경우 push 거절로 멈추고 **최악의 경우(다른
브랜치에 있다가 병합되면) 번호가 중복된 채 원장에 들어간다**. R1 위반은 그때 fsck가
잡겠지만, 그건 이미 역사가 더럽혀진 뒤다. 원장 규율은 **더럽혀지기 전에** 번호를 고친다.

### (b) 이식의 등급이 하나 더 생겼다 — 원장급

C020에서 나는 이식의 등급을 나눴다: web·verify·open·close·step은 **문자면급**(바이트
동일), log는 **exit급**(판정만 동일). C036은 세 번째 등급을 요구했다.

| 등급 | 판정 수단 | 사례 |
|---|---|---|
| exit급 | 같은 종료 코드 | log (C012) |
| 문자면급 | 같은 stdout/stderr 바이트 | open·close·verify·web (C017·C020) |
| **원장급** | **같은 부작용 — 외부 상태(원격 저장소)에 남는 결과가 동일** | `open --push` 경합 해소 (C036) |

`open --push`는 stdout이 같아도 충분하지 않다. 판정해야 할 것은 **원격에 무엇이 남았는가**다.
그래서 run5의 판정 수단은 stdout이 아니라 **제3의 클론으로 관찰한 원장의 진실**(사이클
디렉토리 목록 + 각 `cycle.yaml` 전문 + 커밋 제목)이고, 두 구현의 덤프가 sha256까지 같았다:

```
6365ac3e34613c639764b3276755f71886f5505d64225b3fcb33233a7c15361e  dump-go.txt
6365ac3e34613c639764b3276755f71886f5505d64225b3fcb33233a7c15361e  dump-py.txt
```

같은 경합에 던져진 두 구현이 **같은 원장**에 도달한다. 계약이 stdout을 넘어 **부작용까지**
규정된 것이다 — 병렬 존재들이 서로 다른 구현을 써도(하나는 Go, 하나는 파이썬) 번호는
안전하게 직렬화된다.

### (c) 판정기가 안 보면 계약이 아니다 — 공허의 거울상

C012·C017에서 나는 "공허 통과"(거부형 항목이 수락형 구현의 부재 덕에 그냥 통과하는 것)를
배웠다. C036은 그 **거울상**을 보여준다:

- `open --git`은 **SPEC §2.1-3과 §5 CLI 표가 요구하는 계약**이었다.
- 그런데 26항목 판정기는 그것을 **한 번도 검사하지 않았다**.
- 그래서 Go 바이너리는 그 표면을 `exit 3`으로 거부하면서도 **26/26 만점**이었다 —
  run0이 그 사실의 기록이다. 이식 전 바이너리는 완벽한 점수를 받는다.

**만점은 계약 준수가 아니라 "판정기가 본 범위 안에서의 준수"였다.** maru가 이슈 #7을
열지 않았다면 우리는 26/26을 근거로 "이 구현은 gil이다"라고 계속 말했을 것이다.

## 예상과 달랐던 것

1. **이식 자체는 싱거웠다 — 첫 스모크에서 바로 통과.** 재작업 0회. 코드를 쓰기 전에
   참조 구현의 문면(`cmd_open` 539~547행, `_push_with_renumber` 418~452행)을 2-design에
   표로 고정한 덕이다(C020의 방법). 놀라운 건 **어려웠던 게 이식이 아니라 판정 설계**
   였다는 점이다. 원장급 부작용을 어떻게 구현 독립적으로 볼 것인가 — 그 답(제3의 클론
   + 정규화 덤프 + diff)을 찾는 데 더 오래 걸렸다. **무엇을 만들지보다 무엇으로 볼지가
   비쌌다.**

2. **선행 결함을 하나 주웠다** (run7): `open --new-chain --git`이 새 체인의 `chain.md`를
   커밋하지 않는다. **양 구현 동일**:

   ```
   [go] 커밋 안 된 잔여물:    ?? rooms/experiment/chains/fresh/chain.md
   [py] 커밋 안 된 잔여물:    ?? rooms/experiment/chains/fresh/chain.md
   ```

   커밋 경로가 사이클 디렉토리로 한정되는데 `chain.md`는 그 **밖**에 있다. 참조 구현이
   원래 그렇고, 나는 파리티를 지켰다. 딸깍 사용자가 새 체인을 `--new-chain --git --push`로
   열면 **원장에 사이클은 올라가지만 체인의 정의는 누락된다.** fsck 규칙이 `chain.md`를
   요구하지 않아 **위반으로도 안 잡힌다 — 그래서 조용하다.** 이슈 #7이 고발한 구조
   (계약은 요구하는데 판정기가 안 보는 표면)의 축소판이 참조 구현 안에 또 있었던 셈이다.
   2-design의 범위 밖 규정대로 **고치지 않고 보고한다** (제안 A).

3. **기록이 증거를 잃을 뻔했다.** run5 첫 생성 때 `shasum … | sed 's|.*/||'`의 정규식이
   경로뿐 아니라 **해시까지** 지워, 아티팩트에 파일명만 남았다. `diff` 판정은 옳았지만
   그 판정을 뒷받침할 증거가 기록에서 사라진 상태였다 — 발견 즉시 고쳐 재생성했다.
   C014에서 "판정 절차도 틀릴 수 있다"를 배웠는데, 이번엔 판정이 아니라 **기록**이 틀렸다.

4. **부수 확인 — 실데이터의 첫 병합 노드.** 이 사이클 자신이
   `parent: [C020-go-web-port, C016-number-ledger]`로, 픽스처에만 있던 다중 부모가 실제
   체인에 처음 들어왔다. 양 구현의 fsck가 위반 없이 수용했다 (체인 4개·사이클 41개·위반
   0건, 두 구현 출력 일치). 방법론이 설계한 병합이 실물에서도 성립한다.

## 판정

**가설 채택 (supported).** 사전 등록한 기각 조건 다섯 개가 모두 반증됐다.

- (a) 두 구현이 확장된 무수정 판정기에서 **나란히 28/28** — 퇴행 0, 판정 항목 독립.
- (b) 로컬 bare 원장 경합에서 Go가 참조 구현과 **동일한 재번호 결과**에 도달했고,
      원장 최종 상태가 **바이트 단위 동일**(sha256 일치).
- (c) 자기보고에서 참조 전용 목록이 `release, open --git/--push (원장 규율)` →
      **`release`** 로 줄었다 (run6). 이슈 #12의 `open --git` 과소보고 해소.

C017에서 내가 "반쪽 이식 대신 정직한 거부"를 택하며 남긴 미완이 이로써 **완결**된다.
정직한 거부는 옳은 임시방편이었지만, 임시방편은 계약이 아니다.

**남긴 공백** (정직하게 기록한다):

- `release`의 Go 이식은 여전히 참조 전용 (설계상 범위 밖 — 바이너리의 자기 동기화는
  "파일 복사"가 아니라 "재현 빌드"라 스펙 어휘의 확장이 필요하다).
- `open --new-chain --git`의 `chain.md` 누락 (양 구현 공통, 참조 구현의 선행 결함).
- 판정기(`conformance.py`)가 변했으므로 **다음 릴리스는 마이너 이상 승격**이 강제된다
  (C018의 승격 규칙). 릴리스는 소환자 Clew의 권한이다.
