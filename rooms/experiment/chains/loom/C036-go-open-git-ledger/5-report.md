# 5. 결과 보고

## 요약

Go 구현이 `open --git/--push`를 미구현으로 거부해 **SPEC §2.1-3("열 때부터 보이게")이
요구하는 계약을 구현이 지키지 않는 상태**였다 (이슈 #7 — 사용자 maru가 체인 2개·사이클
13개를 돌리며 13번 수동 커밋한 실사용 증언). 커밋 경로와 **번호 원장 규율**(push 거절 →
fetch·rebase → 번호 경합이면 자동 재번호 → 재시도, 최대 3회 — SPEC §6-6)을 **한 몸으로**
이식하고, 그 표면을 처음으로 보는 판정 항목 2개를 판정기에 더했다.
**판정: 가설 채택.** 두 구현이 확장된 무수정 conformance에서 **나란히 28/28**(퇴행 0,
참조 구현도 통과 = 판정 항목 독립), 로컬 bare 원장 경합에서 **최종 원장 상태가 바이트
단위 동일**(sha256 일치), 바이너리의 자기보고에서 참조 전용 목록이 `release`만 남았다
(이슈 #12의 `open --git` 과소보고 해소).

## 교훈

1. **판정기가 안 보는 계약은 없는 계약이다 — "만점"의 의미는 판정기의 시야에 상대적이다.**
   이식 전 Go 바이너리는 `open --git`을 `exit 3`으로 거부하면서도 **26/26 만점**이었다
   (run0). SPEC §2.1-3과 §5 CLI 표가 그 표면을 요구하는데도 26항목 판정기가 한 번도
   검사하지 않았기 때문이다. 만점은 계약 준수가 아니라 **"판정기가 본 범위 안에서의
   준수"** 였고, 그 사각지대는 **사용자가 밟아서** 발견됐다(이슈 #7). 그래서 이 사이클의
   진짜 산출물은 이식된 코드가 아니라 **확장된 판정기**다 — 코드는 26/26을 28/28로
   만들었지만, 판정기의 확장은 **26/26이라는 만점의 의미 자체를 바꿨다.**
   C012·C017의 "공허 통과"(구현이 없어 거부형 항목이 그냥 통과하는 것)의 **거울상**이다.

2. **원장 규율은 "재시도"가 아니라 "정정 후 재시도"다 — 잘못된 번호는 원장에 도달조차
   하지 않는다.** run8이 육안으로 보여준다: 경합한 B의 커밋은 push되었다 취소된 게
   아니라, `git mv` + `id:` 치환 + `commit --amend`로 **로컬에서 다시 쓰인 뒤에야**
   push됐다. 원장의 커밋 목록에 `C002-beta-path`는 **존재한 적이 없다**. 커밋만 이식하는
   반쪽 이식이 위험한 이유가 이것이다 — 최악의 경우 번호가 중복된 채 원장에 들어가고,
   fsck는 **역사가 더럽혀진 뒤에야** R1으로 잡는다. 규율은 더럽혀지기 전에 고친다.

3. **이식에 세 번째 등급이 생겼다 — 원장급.** C020에서 나는 등급을 둘로 나눴다(exit급:
   같은 종료 코드 / 문자면급: 같은 stdout 바이트). `open --push`는 stdout이 같아도
   불충분하다 — 판정해야 할 것은 **원격에 무엇이 남았는가**, 즉 **부작용**이다.
   판정 수단도 그에 맞춰 바뀐다: stdout이 아니라 **제3의 클론으로 관찰한 원장의 진실**
   (사이클 목록 + `cycle.yaml` 전문 + 커밋 제목)을 정규화 덤프해 diff한다. 두 구현의
   덤프가 sha256까지 동일했다. **계약이 stdout을 넘어 부작용까지 규정된 첫 지점** —
   병렬 존재가 서로 다른 구현을 써도 번호는 안전하게 직렬화된다.

4. **무엇을 만들지보다 무엇으로 볼지가 비쌌다.** 이식 자체는 첫 스모크에서 통과했다
   (재작업 0회 — 코드 작성 전에 참조 구현의 문면을 2-design에 표로 고정한 C020의 방법이
   또 통했다). 시간이 든 곳은 **원장급 부작용을 구현 독립적으로 보는 판정 설계**였다.
   이식은 문면 대조로 끝나지만, 판정은 **관찰 수단의 발명**을 요구한다.

5. **재현 가능성은 옳은 판정만이 아니라 증거의 보존까지 요구한다.** run5를 처음 만들 때
   `shasum … | sed 's|.*/||'`의 정규식이 경로뿐 아니라 **해시까지** 지워, 판정은 옳은데
   **근거가 기록에서 사라진** 아티팩트가 나왔다(발견 즉시 고쳐 재생성). C014의 "판정
   절차도 틀릴 수 있다"가 이번엔 **기록 층위**에서 재발했다. 옳게 판정하고 근거를 안
   남기면 감사자에게는 주장만 남는다.

## 발견 (고치지 않고 보고)

**`open --new-chain --git`이 새 체인의 `chain.md`를 커밋하지 않는다 — 양 구현 공통,
참조 구현의 선행 결함** (run7). 커밋 경로가 사이클 디렉토리로 한정되는데 `chain.md`는 그
밖에 있기 때문이다. 딸깍 사용자가 새 체인을 `--new-chain --git --push`로 열면 **원장에
사이클은 올라가나 체인의 정의(`chain.md`)는 누락된다.** fsck 규칙(R1~R11)이 `chain.md`를
요구하지 않아 **위반으로도 안 잡힌다 — 그래서 조용하다.** 이슈 #7이 고발한 구조(계약은
요구하는데 판정기가 안 보는 표면)의 축소판이 참조 구현 안에 또 있었던 셈이다.
2-design의 범위 밖 규정("참조의 결함을 발견하면 고치지 않고 보고한다")에 따라 보고만
한다 — 참조는 계약의 기준점이고, 양 구현을 동시에 고쳐야 하므로 독립 사이클이 맞다.

## 남긴 공백

- `release`의 Go 이식은 여전히 **참조 구현 전용** (설계상 범위 밖). 바이너리의 자기
  동기화는 "파일 복사"가 아니라 "재현 빌드"라 스펙 어휘의 확장이 필요하다.
- **다음 릴리스는 마이너 이상 승격이 강제된다** — 판정기(`conformance.py`)가 변했기
  때문이다(C018의 승격 규칙). 릴리스는 소환자 Clew의 권한이므로 사실만 남긴다. 릴리스
  시 `SPEC.md`·`RELEASE.md`의 "26항목/26/26" 표기도 28항목으로 갱신이 필요하다.
- **push 미수행** — 이 워크트리는 upstream이 없고 push는 하네스의 권한 경계 밖이다.
  커밋까지만 남긴다 (C014·C017·C020과 동일한 관례). 병합·push는 Clew의 몫.

## 다음 사이클을 위한 제안

- **(A) `open --new-chain`의 chain.md 각인** (위 발견) — 양 구현의 커밋 경로에 새 체인의
  `chain.md`를 포함시키고 판정 항목(`OPEN-NEWCHAIN-GIT`)을 더한다. **참조 구현의 동작
  변경이 필요한 첫 사이클**이므로, "참조가 곧 계약"인 상태에서 참조를 고치는 절차
  (참조 변경 → 판정기 확장 → 양 구현 재판정)를 규율로 정립할 기회다.
- **(B) fsck에 체인 정의 규칙 추가** — `chain.md` 부재를 규칙(R12) 또는 경고로 잡으면
  (A)의 결함은 애초에 조용할 수 없었다. **"조용한 결함"은 규칙의 부재에서 온다**는 교훈
  1을 fsck 층위에서 일반화하는 갈래.
- **(C) `release`의 Go 이식** — 남은 마지막 미구현. 바이너리의 자기 동기화 의미(재현
  빌드)를 스펙에 정의해야 한다. 완주하면 참조 전용 목록이 **비고**, 두 구현이 진정한
  상호 교체 가능성에 도달한다.
- **(D) 계약 표면의 전수 감사** — 교훈 1의 일반화. SPEC §5 CLI 표의 **모든 플래그**를
  판정기 항목과 대조해 "계약은 요구하는데 판정기가 안 보는 표면"의 전체 목록을 만든다.
  이슈 #7은 그중 하나가 **사용자에게 먼저 발견된** 사건일 뿐이다 — 나머지는 우리가 먼저
  찾는다.
