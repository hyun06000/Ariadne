# 1. 가설 수립

**부모**: loom/C053-windows-entry (그 보고서의 다음 추천 (A) — 상현님 발의, 비개발자·크로스플랫폼 아크의 연속)

## 이전 사이클의 교훈

C053의 부수 감사("GitHub 없으면 push 안 하지?")에서 발견됐다: **`--push`인데 원격이 없으면
gil이 git 날것 힌트를 뱉는다.** C052가 git *부재*를 우아하게 강등했듯("침묵보다 나쁜 것은 잘못된
신호다"), 원격 *부재*도 같은 대우를 받아야 한다 — 비개발자·크로스플랫폼 아크의 마지막 거친 모서리.

## 문제 분할

git은 있으나 원격이 없는 저장소에서 `--push` 경로를 실측하니(C007: 실측이 설계를 정한다),
**두 얼굴의 비일관**이 드러났다:

1. **날것 fatal + rc1 (오도 신호)**: `open --new-chain --git --push` → 커밋은 되지만
   `_push_with_renumber`의 `git fetch origin`이 *"'origin' does not appear to be a git repository"*
   를 뱉고 exit 1. `step --push` → *"fatal: 푸시 대상을 설정하지 않았습니다 … git remote add"*
   exit 1. **커밋이 이미 성공했는데 rc1이라 전체가 실패한 것처럼 보인다** — 비개발자에겐 오도.
2. **조용한 삼킴 (침묵)**: `check=False` 경로(open non-newchain·reserve·round)는 실패를 삼켜
   사용자가 *push된 줄로* 안다.

이것은 정확히 **C052 교훈의 두 얼굴**이다 — 무서운 날것 신호(오도)와 침묵(안 나갔는데 나간 줄 앎).
지금 정복할 첫 문제: 이 두 얼굴을 하나의 우아한 강등으로 통일하는 것.

## 가설

> **가설**: 모든 push 경로를 단일 관문(`_has_push_remote`/`_warn_no_remote_once`/`_push`)으로
> 통일해 원격 부재를 **앞에서 감지**하면(C052의 `_git_available`/`_warn_git_missing_once`의
> 원격판), 날것 fatal도 침묵도 아닌 **원인 한 줄 안내 + 커밋 보존 + rc 0 완주**로 강등되고,
> 원격이 *있으면* 기존 push 동작(원장 규율 재번호 포함)은 불변일 것이다. (H1)
>
> 그리고 C053이 감사로 확정한 네트워크 자세(자체 호출 0·텔레메트리 0, 외부는 `--push` 시
> 자기 원격 git push/fetch뿐)를 README·README.ko에 명문화하면, 비개발자의 보안 걱정에
> **코드로 뒷받침된 답**을 줄 것이다. (H2)

**계약**: 새 판정기 항목 `NO-REMOTE-GRACEFUL` — git 있고 원격 없는 저장소에서 `open … --push`가
rc0 + 사이클 파일 생성 + 로컬 커밋 존재 + 무크래시. 양 구현(참조·Go) 동시 준수.

## 기각 조건 (선등록 — H1 실패로 간주)

닫기 전에 다음 중 하나라도 참이면 채택하지 않는다:

- (a) 수정 후에도 원격 부재 `--push` 경로 중 하나라도 **크래시하거나 rc≠0**로 끝난다.
- (b) 수정 후에도 원격 부재에서 **커밋은 됐는데 아무 원인 안내 없이** 조용히 끝난다.
- (c) 원격이 **있는** push 경로에 **회귀**가 생긴다(원장 규율 재번호 포함). **회귀 0**이 조건.
- (d) 두 구현(참조·Go)의 원격 부재 강등 **행동이 갈린다**(한쪽만 우아).

수치가 아니라 성질로 못박는다(C045): 총 판정 항목 수는 늘 수 있으나(NO-REMOTE-GRACEFUL 신설),
**git-있음·원격-있음 회귀는 0**이어야 한다.
