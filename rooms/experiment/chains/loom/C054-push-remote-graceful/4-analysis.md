# 4. 결과 분석

## 통계적 결과

| 측정 | 기준 | 결과 | 판정 |
|---|---|---|---|
| 원격 부재 `--push` 경로 rc | rc0 (기각 a) | open·step·close·reserve·round 전부 rc0 | ✅ |
| 침묵 없음 (기각 b) | 원인 안내 출력 | 모든 경로 `ℹ 원격이 없어…` 한 줄 | ✅ |
| 원격-있음 회귀 (기각 c) | 회귀 0 | 판정기 75/75 ×2, fsck 위반 0·verify 변조 0, 실 push 정상 | ✅ |
| 두 구현 동형 (기각 d) | 문면·rc·부작용 동일 | 완전 동일 | ✅ |
| NO-REMOTE-GRACEFUL | 양 구현 PASS | 참조 75/75, Go 75/75 | ✅ |
| 변이 M1 (가드 제거) | FAIL 기대 | FAIL, 74/75 | ✅ |

## 데이터 직접 관찰

- **수정 전 두 얼굴을 실제로 봤다**: 같은 "원격 없음" 조건에서 open은 `git fetch origin` 날것
  fatal(rc1)로, step은 `git push` 날것 fatal(rc1)로, reserve/round는 침묵(check=False)으로 —
  **한 결함이 세 얼굴**이었다. push 경로가 6곳에 흩어져 각자 다른 실패 방식을 갖고 있었다.
- **단일 관문이 세 얼굴을 하나로 접었다**: `_push`/`gitPush` 하나로 모으니 6개 호출부가 전부
  같은 강등 행동을 공유한다. 관문 이전에는 "어느 명령이냐"에 따라 사용자 경험이 달랐다.
- **변이 로그가 정확히 옛 행동을 재현**: M1의 FAIL 상세 `err=git fetch origin 실패: fatal:
  'origin' does not appear …` 는 수정 전 open 경로가 뱉던 바로 그 문자열. 계약이 붙잡는 것이
  "우리가 고친 그 결함"임을 데이터가 증언한다.

## 예상과 달랐던 것

- **판정기가 먼저 다른 버그를 물었다** — Go는 통과하는데 참조만 FAIL. 원인은 push 로직이 아니라
  macOS `/var`→`/private/var` 심볼릭 링크에서 참조 `os.path.relpath`가 깨지고 Go `relToRepo`는
  `EvalSymlinks`로 견뎠기 때문. **환경 함정(C028·C029)이 push 계약 검증을 가리려 했다.** 테스트를
  realpath로 정규화해 의도만 검증하고, 경로 해석 비대칭 자체는 선재 결함으로 범위 밖에 남겼다
  (C044·C051 "고치러 간 자리의 새 침묵은 범위에 넣지 않는다").
- **Go 함수명 충돌**: 참조는 `_push`로 깔끔했으나 Go엔 `reserveCommitPush(… push bool …)`
  파라미터가 있어 `push` 함수가 지역 bool에 가려졌다. `gitPush`로 개명 — 두 구현의 계약은 같되
  이름은 언어의 제약을 따른다(C047 "시그니처를 안 바꾸고 계약을 지킨다"의 친척).
