# 4. 결과 분석

## 통계적 결과

| 기대 행동 | 기준 | 관측 | 판정 |
|---|---|---|---|
| T1 강화 판정기 × 참조 | 73/73, 회귀 0 | 73/73 | ✔ |
| T2 강화 판정기 × Go | 73/73, 회귀 0 | 73/73 | ✔ |
| T3 두 구현 R1~R8 stdout | 토큰 ∧ 대상 id 존재 | 8규칙 × 2구현 전부 존재 | ✔ |
| T4 변이 strip(토큰 제거) | FSCK 계열 격추 | 63/73 (10항목 격추) | ✔ |
| T5 변이 rewrite(→R1) | R2~R8 격추, R1 생존 | 64/73 (R1 생존) | ✔ |
| T6 변이 render(문면만) | 전부 생존 | 73/73 (격추 0) | ✔ |

기각 조건 3개 전부 불성립: 회귀 없음, 변이 격추됨, 렌더 변이 생존(과잉확장 없음).

## 데이터 직접 관찰

수치 뒤로 들어가 실제 출력을 인용한다.

- **같은 위반, 다른 문면, 같은 식별** (R6): 참조는 `R6  alpha/C003-r6: parent 'C099-ghost'가 존재하지 않는다 (끊어진 참조)`, Go는 `(끊어진 참조)` 없이 동일. **식별 `(R6, alpha/C003-r6)`은 일치, 접미사만 다름.** 이것이 계약/렌더 경계의 산 증거다 — 강화된 판정기는 앞을 보고(통과), T6은 뒤를 지운다(여전히 통과).
- **T5의 R1 생존이 결정적이다**: `MUT=rewrite`는 모든 토큰을 `R1`로 바꿨다. R2~R8 픽스처는 `R1 alpha/C003-r2: lineage…`가 되어 규칙 불일치로 격추됐지만, **R1 픽스처만은 진짜 규칙이 R1이라 정당히 통과**했다. 판정기가 "무조건 R1이면 안 된다"가 아니라 "**이 픽스처의 올바른 규칙**을 짚었나"를 본다는 증거 — 맹목적 거부가 아니라 정확한 식별.
- **구 판정기가 준 만점의 정체**: T4의 strip 출력(`alpha/C01-short: id 'C01-short' 형식 위반 …`)은 규칙 토큰이 없다. 이 출력에 구 `FSCK-R1`(rc만)은 **PASS**를 줬을 것이다 — 위반은 있으니까. `gil fsck`가 "무언가 잘못됐다"만 말하고 "무슨 규칙인지"는 침묵하는 구현이 만점을 받는 세계였다. 이제 63/73으로 떨어진다.

## 예상과 달랐던 것

- **강화가 R14·R15까지 함께 격추했다.** 설계는 R1~R8·R7만 강화했으나, 토큰 파괴 변이(T4·T5)는 이미 토큰을 검사하던 R14·R15도 넘어뜨렸다. 이는 결함이 아니라 **규칙 식별이 이제 fsck 전 규칙에서 균일한 계약**이 됐다는 방증 — 신·구 항목이 같은 계약면을 공유한다.
- **설계가 예상보다 더 조용했다.** "구현 무변경"을 설계했고 실제로 gil.py·go/main.go 한 줄도 안 건드렸다. **두 구현은 처음부터 fsck 계약면에 합의하고 있었다** — C036·C040의 "도구는 이미 보고 있었다, 판정하지 않았을 뿐"의 세 번째 판본. 이번엔 fsck가 **이미 올바른 것을 말하고 있었고**, 스펙과 판정기만 그것을 계약으로 인정하지 않았다.

## 판정

**가설 채택 (supported).** fsck의 계약면(위반 식별 집합)이 스펙에 명문화되고 판정기가 그것을 보게 됐다. 두 구현은 이미 합의하므로 회귀 0, 잘못된 식별 변이는 격추, 문면 변이는 생존. 이탈 없음(사전등록한 T1~T6이 그대로 관측됨, 총계를 불변식으로 쓰지 않아 C038식 이탈 불필요).

**부산물 — Weft의 이월 항목 해소**: "Go log summarize 부재"는 §3.1이 이미 렌더로 분류(C021)하므로 계약 위반이 아님이 확정됐다. log의 계약면은 web JSON(바이트 동일). 이 사이클은 그 대칭을 fsck에도 세웠다: **모든 도구의 의미 출력은 계약면을 가진다.**
