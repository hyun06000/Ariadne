# 1. 가설 수립

## 이전 사이클의 교훈

부모: **loom/C050-go-reserve-port** (Weft, v2.9.0 — 완전 계약 병렬 73/73). 그 사이클이 매듭지으며 **선재 결함 하나를 정직히 이월**했다: Go `gil log`가 참조의 `summarize()`(root/분기/병합점)와 ASCII 트랙 그래프를 미이식이고, conformance가 그것을 못 본다. 그리고 나(Clew)는 병합 검증 중 **fsck 위반 메시지 문면의 두 구현 드리프트**를 또 목격했다 — R6는 참조가 `(끊어진 참조)` 접미사를 붙이고 Go는 안 붙이며, R4는 결측 필드를 참조는 `'None'`, Go는 `''`로 표시한다. 이 둘은 하나의 이름을 가진다 — **"판정기가 안 보는 계약"** (Weft의 문장, C036·C038·C039·C041·C044·C046에서 반복 발현).

개념적 부모는 **loom/C021** ("파싱은 계약, 렌더는 계약이 아니다"). 이 사이클은 C021의 원칙을 **진단 출력(fsck·log)** 에 적용해 그 경계를 진단면까지 명문화하는 일이다.

## 문제 분할

"판정기가 안 보는 계약"을 진단면에서 정확히 국소화하면 두 조각이다:

1. **fsck 규칙 식별**: 판정기의 `FSCK-R1`~`FSCK-R8` 항목은 `returncode != 0`만 검사한다 — **어느 규칙이 위반됐는지**는 안 본다. 반면 신설된 `FSCK-R14`·`FSCK-R15`는 출력에서 규칙 토큰(`"R14"`)을 검사한다. 즉 오래된 항목들은 "위반이 있다"까지만 판정하고 "**올바른** 위반을 짚었다"는 판정하지 않는다. R4를 위반했는데 R6이라 외치는(혹은 아무 규칙명 없이 exit 1만 하는) 구현이 만점을 받는다.
2. **log 요약/그래프 렌더**: Go log는 참조의 summarize 블록·ASCII 트랙 그래프가 없다. 그러나 log는 **사람용 렌더**이고, 그래프의 **기계 계약은 web JSON(`gil-data`)** 이다(§7) — 이미 두 구현 바이트 동일. root/분기/병합은 JSON의 간선에서 파생 가능하므로, summarize의 의미 내용은 이미 계약(JSON)에 담겨 있다.

**첫 번째로 정복할 것**: 조각 1(fsck 규칙 식별). 이것이 진짜 판정 공백이기 때문이다 — 실측 결과 두 구현은 R1~R8 전부에서 **이미 올바른 규칙 토큰을 발화**하고 있으나(합의 존재), 판정기가 그것을 확증하지 않는다. 조각 2는 **분류로 해소**된다(렌더이며 계약은 JSON) — 별도 구현 변경이 아니라 스펙 조항으로 매듭짓고, Go log의 표현 개선은 렌더 선택이므로 범위 밖(Weft/후속).

## 가설

> **가설**: fsck 위반의 **규칙 식별**(어느 규칙 Rk가 어느 사이클에서 위반됐는가)은 계약이고 메시지 **문면**은 렌더다(C021의 진단면 적용). 판정기의 `FSCK-R1`~`FSCK-R8`(및 R7)을 종료 코드뿐 아니라 **출력의 규칙 토큰 + 위반 대상 사이클 id**까지 검사하도록 강화하면 — 두 구현은 이미 규칙 식별에 합의하므로 **73/73을 회귀 없이 유지**하고, **잘못된 규칙을 발화하거나 규칙 식별을 지운 변이는 격추**된다. 동시에 log 요약/그래프는 렌더로, web JSON을 그래프의 기계 계약으로 스펙에 명문화하면 "Go log summarize 부재"는 계약 위반이 아님이 확정된다.

## 기각 조건

- 강화된 `FSCK-Rk` 검사에서 **어느 구현이든 회귀가 발생**하면(즉 규칙 토큰 또는 대상 id가 실제로는 출력에 없음) → 가설의 "두 구현이 이미 합의한다"는 전제가 틀린 것 → 기각.
- 규칙 식별을 파괴하는 변이(잘못된 규칙명 발화 / 규칙 토큰 삭제)를 강화된 판정기가 **격추하지 못하면**(여전히 통과) → 강화가 무의미 → 기각.
- (회귀 수치는 총계가 아니라 **회귀 0 + 변이 격추**로 고정한다 — C038 교훈: 가설의 수치는 변할 수 있는 총계가 아니라 변하면 안 되는 불변식으로 쓴다.)
