# 2. 실험 설계

## 설계를 바꾼 발견 — fsck에는 계약면이 없다

설계에 들어가며 §3.1을 읽고 그림이 날카로워졌다.

- §3.1은 **이미** 말한다: *"log의 렌더 형식은 계약이 아니다 — 요약 섹션·그래프 문자 등은 구현 자유. 기계 소비의 계약면은 종료 코드와 web의 `gil-data` JSON이다."* (C021, v0.9.2)
- 따라서 **조각 2(Go log summarize 부재)는 이미 스펙이 렌더로 분류**한다. Weft의 이월 항목은 계약 위반이 아니라 렌더 선택임이 기존 조항으로 확정된다. 이 사이클은 그 사실을 **보고에 명시**하기만 하면 된다(새 구현·새 조항 불필요).

그런데 같은 문장이 fsck에 대해서는 위험한 침묵을 드러낸다:

> fsck의 기계 계약면은 **종료 코드뿐이다.** log은 의미 출력(그래프)의 계약면으로 JSON을 얻었지만, **fsck의 의미 출력(위반 집합)은 어떤 계약면도 갖지 못했다.**

이것이 진짜 공백이다. 판정기 `FSCK-R1`~`FSCK-R8`이 `returncode != 0`만 보는 것은 결함이 아니라 **스펙에 충실한 것** — 스펙이 exit code만 계약했으니까. R4를 위반했는데 R6이라 외치는 구현, 혹은 규칙명 없이 exit 1만 하는 구현도 스펙을 어기지 않는다. 그러면 두 구현의 `gil fsck`는 "위반이 있다"에만 합의하고 "**무엇이** 위반인가"에는 합의를 강제받지 않는다 — 같은 도구가 아닐 수 있다.

## 절차

1. **실측 (T3)**: R1~R8 각 위반 픽스처를 만들어 두 구현의 `fsck` stdout에 규칙 토큰(`Rk`)과 위반 대상 사이클 id가 실제로 실리는지 확인한다. (강화가 회귀 없이 통과할 전제)
2. **SPEC §3.1 확장 (산출물 1)**: log 문장 옆에 fsck 대칭 조항 신설 — fsck의 계약면 = 위반 식별 집합 `{(규칙 토큰, 위반 대상 사이클 id)}`, 문면·순서·결측 표기는 렌더.
3. **conformance 강화 (산출물 2)**: `FSCK-R1`~`R8`, `FSCK-R7`을 rc뿐 아니라 규칙 토큰(+ 대상 id) 검사로 강화. R14/R15의 선례를 소급 적용.
4. **회귀 검증 (T1·T2)**: 강화된 판정기로 참조·Go 각각 재판정 → 73/73 유지 확인.
5. **변이 검증 (T4·T5·T6)**: 규칙 식별 파괴 변이 2종 격추 + 문면 변이 1종 생존 확인.
6. 5스텝 각인·릴리스.

## 준비물
- 참조 `gil.py`, Go 바이너리(`GO111MODULE=off go build -o /tmp/gil-go go/main.go`, go1.26).
- `conformance.py`(73항목), 위반 픽스처는 conformance.py의 R1~R8 정의를 그대로 재현.
- `--gil`에는 **절대 경로** (C028·C043·C045 세 번 밟은 상대경로 함정).

## 측정 방법 — 기대 행동

| # | 자극 | 기대 |
|---|---|---|
| T1 | 강화된 판정기 × 참조 gil.py | 73/73 (회귀 0) |
| T2 | 강화된 판정기 × Go | 73/73 (회귀 0) |
| T3 | 두 구현, R1~R8 각각 | stdout에 `Rk` 토큰 ∧ 대상 cid 존재 (실측) |
| T4 | 변이 M1: fsck가 규칙 토큰 없이 exit 1만 | 강화 판정기가 `FSCK-Rk` FAIL (구 판정기는 통과했음) |
| T5 | 변이 M2: fsck가 항상 `R1`이라 발화(잘못된 규칙) | R2~R8에서 규칙 토큰 불일치로 FAIL |
| T6 | 메시지 문면만 바꾼 변이(예: `(끊어진 참조)` 제거) | 여전히 통과 — 문면은 렌더, 계약 아님(C021 재확인) |

성공 기준: **T1·T2 회귀 0 ∧ T4·T5 격추 ∧ T6 생존.** T4·T5(강화가 문다)와 T6(과잉금지)이 함께 서지 않으면 설계 실패 (C038의 쌍 검증 규율). 수치 총계(73)는 불변식이 아니라 관측값 — 기각은 회귀·변이로 판정한다.

### 강화 코드 (설계 확정본)
```python
out = r.stdout
ok = r.returncode != 0 and rule in out and spec["cid"] in out
check(f"FSCK-{rule}", f"{rule} 식별 (exit≠0 ∧ 규칙토큰 ∧ 대상 id)", ok, out.strip()[:120])
```
R7(순환)은 단일 대상이 없으므로 규칙 토큰 `"R7"`까지만. 출력 소스는 `r.stdout`(R14 선례). 항목 수 불변(73) — 과소판정 교정이지 신설 아님.

## 구현 변경 없음
gil.py·go/main.go 무변경 — 두 구현이 이미 규칙 식별에 합의(1-hypothesis 실측). 판정기 + 스펙만 바뀐다. 순수 계약/판정 사이클이므로 릴리스는 도구 blob 불변 → **문서/판정기 변경**(conformance.py는 C018 이래 도구다 → 마이너 승격).
