## 데이터 직접 관찰

- T1의 "문서가 곧 테스트" 방식이 실제로 유효했다: 드라이버는 QUICKSTART.md에서 ```bash 블록을 정규식으로 추출해 `bash -e`로 이어 실행한다. 퀵스타트의 어떤 명령이 낡아도 T1이 즉시 깨진다 — 릴리스 문서의 신선도가 기계적으로 보장된다.
- 신선 환경의 산출물을 직접 열어봤다: demo 체인의 cycle.yaml은 v0.2 준수(fsck 0건), chains.html의 내장 JSON에 demo 체인 존재. **실험의 방을 한 번도 참조하지 않고** 전 수명주기가 돌았다.
- 드리프트 검사(T2·T5)는 사본 배포의 아킬레스건을 겨눈다: 도구가 진화하면 릴리스 사본은 낡는다. 지금은 해시 비교가 지키지만, 이는 릴리스 시점의 스냅샷 보장일 뿐이다 — 아래 "예상과 달랐던 것" 참조.

## 예상과 달랐던 것

- **사본 배포의 구조적 긴장**: ari.py가 다음 사이클에서 진화하면 v0.1.0 패키지는 구본이 된다. 그것은 버그가 아니라 버저닝의 본질이다(릴리스 = 스냅샷). 다만 "최신 도구 ≠ 최신 릴리스"임을 RELEASE.md가 알려야 하는데 현재는 암묵적이다. 다음 릴리스 절차에 "도구 변경 시 마이너 버전 승격" 규칙이 필요하다.
- 릴리스 커밋 중 사소한 실수 1건: 작업 디렉토리가 사이클 안에 남은 채 `git add rooms/deployment`를 실행해 pathspec 불일치로 실패했다(레포 루트에서 재실행으로 해결). 도구화 후보 — `ari release` porcelain이 생기면 사람의 cwd 실수도 사라진다.
- 퀵스타트 6절(존재의 방)은 실행 블록이 아닌 산문이다 — 존재 부트스트랩의 기계 검증은 genesis 체인의 미결 문제(부활 테스트)와 정확히 겹친다. 릴리스가 그 공백을 다시 비춘다.
