# 4. 결과 분석

## 데이터 직접 관찰

수치 뒤로 들어가 실제 거동을 확인했다(3-verification/run-log.txt):

1. **거부 메시지의 처방성.** drift 저장소에서 release는 단순 실패가 아니라 처방을 준다: `배포 계보 drift 1건 — 태그와 CHANGELOG가 어긋난 릴리스가 있다(v1.0.0). … 'gil releases'로 확인하고 두 기록을 일치시킨 뒤 다시 릴리스할 것`. 조회 프리미티브(C061)를 처방으로 지목 — 게이트와 조회가 한 진실을 공유하는 설계가 사용자 동선으로도 이어진다.

2. **무변화가 트리 너머 깃까지.** `snapshot`(작업트리, .git 제외)뿐 아니라 `rev-parse HEAD`와 `tag -l`도 before==after로 확인했다. 게이트가 사전 검증 최상단(태그 검증 직후)에서 발화하므로 커밋·태그 생성 이전 — 부분 각인의 여지가 없다. open 계열의 무변화 규율을 그대로 계승한다.

3. **위양성 0의 실증.** 일치 저장소(태그 v1.0.0 + CHANGELOG [1.0.0])에서 release는 drift 게이트를 **지나** 하류의 `RELEASE.md 서술 없음`에서 멈췄다. 즉 게이트는 적법한 릴리스 흐름을 방해하지 않고, 오직 두 기록의 어긋남만 잡는다. 이는 가설의 핵심(정상 경로 drift=0 → 위양성 0)을 저장소 밖 최소 재현으로 확증한다.

4. **정직한 부재의 실증.** Go 판정에 RELEASE-DRIFT-GATE 항목이 **아예 나타나지 않는다**(로그의 Go 블록엔 그 줄이 없다). 판정기가 `help release`(Go에서 exit 3, referenceOnly)로 능력을 탐침해 참조 구현에서만 게이트를 판정하기 때문. Go는 게이트를 거짓 통과하지도, 거짓 실패하지도 않는다 — 계약면에서 존재하지 않을 뿐. HELP-COMPLETE가 이 부재의 정직함을 이미 별도로 판정한다(C043 리듬).

## 예상과 달랐던 것

- **`help release`가 Go에서 exit 3.** 설계 시 능력 게이트를 `help releases`(C061의 짝)로 걸까 고민했으나, Go는 `release`(단수)도 referenceOnly로 선언해 `help release`가 정확히 exit 3을 준다. 그래서 게이트 대상 명령(`release`) 자체를 능력 탐침으로 삼는 더 정밀한 선택이 가능했다. 단, Go의 `release` **직접 호출**은 exit 0(미구현 안내)로 나오는 기벽이 있으나, 판정 탐침은 `help release`를 쓰므로 무관하다.
- **changelog 경로 계산의 중복.** 게이트가 사전 검증 상단에서 CHANGELOG 경로를 필요로 해, 하류(실행 블록)의 중복 계산을 제거하고 한 변수로 재사용하도록 정리했다 — 계획엔 없던 부수적 DRY 정리.

## 판정

**H1 채택** — release 사전 검증의 drift 하드 게이트는 ① drift 저장소를 exit≠0·무변화(트리·커밋·태그)·처방 메시지로 거부하고 ② 일치 저장소를 통과시킨다(위양성 0). 기각 조건 어느 것도 발생하지 않았다.

**H2 채택** — RELEASE-DRIFT-GATE가 판정기 항목으로 참조 구현에서 통과(91/91)하고, Go는 게이트를 노출하지 않아도 정직한 부재로 83/83을 유지한다. "판정기가 안 보는 계약은 없는 계약"(C036)을 지켰다 — 게이트는 이제 계약면의 1급 항목이다.

**회귀 0** — 기존 90개 항목 전부 유지. 저장소 무변화 규율 보존.
