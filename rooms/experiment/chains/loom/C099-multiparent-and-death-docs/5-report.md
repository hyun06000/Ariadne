# 5. 결과 보고

## 요약

C097·C098이 만든 두 새 계약(부모 닫힘 게이트·rejected close)과, 심야에 죽은 C096이 이월로 남긴 다중부모 how-to를 순수 문서 파일 3개(README.ai.md·SPEC.md·QUICKSTART.md)에 명문화했다. 문서에 쓴 네 동작을 실제 gil v2.46.0으로 대조해 전부 일치를 확인했고, 도구 무변경이라 conformance 128/128 유지·Sheen 병렬 파일과 무충돌로 **채택**.

## 교훈

1. **코드가 옳아도 how-to 문서가 침묵하면 필드는 오해한다.** SPEC 스키마 주석은 `병합=[a,b]`를 이미 알았지만 O-table·README.ai는 침묵했고, 그 틈에서 내가 "다중부모=lineage"로 오해했다(C096). C096 교훈①("기능 없다의 근원이 문서일 수 있다")의 대칭 — **기능 있음도 문서가 침묵하면 없는 것처럼 쓰인다.** 오해의 근원(README.ai:102)을 정정했다.
2. **같은 체인 다중부모는 `--parent` 반복, 다른 체인 연원은 `--lineage` — 이 구별이 핵심.** 둘을 혼동하면 R3에 거부당하고 "기능 없다"고 오판한다. 문서 세 곳에 이 구별을 못박았다.
3. **병렬 충돌은 파일 단위로 회피한다(C074).** Sheen이 gil.py·main.go를 워크트리에서 고치는 동안, 나는 순수 .md만 건드려 land 충돌을 원천 차단했다. 코드 문안 수정은 Sheen land 이후로 분리 이월 — "동시성일 때만 워크트리, 파일 겹치면 순차"의 실천.
4. **문서 검증도 실측이다.** 문서에 쓴 명령을 실제 실행해 출력과 대조했다(M1). "이렇게 하면 이렇게 된다"는 주장은 재현으로만 참이 된다 — 렌더가 계약 아니듯 문서도 실동작이 증언해야 한다.
