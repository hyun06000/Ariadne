# 1. 가설 수립

## 이전 사이클의 교훈

부모: **loom/C038-version-surface-contract** (§7.1 version 표면 계약 — release가 각인·마스킹으로 자기보고를 집행). 그 사이클이 릴리스 경로의 자기보고를 계약화했다면, 이 사이클은 그 릴리스들이 **남긴 계보를 조회하는 표면**을 연다.

현황을 읽어 확인한 사실 (#3 배포 버저닝 관리, 상현님 발의 — "배포 추적층 + `gil release` 강화, 둘 다"):

1. **배포는 두 몸으로 기록된다.** ① 깃 태그 `v<semver>`(어노테이트, `cmd_release`가 각인 — 깃의 진실), ② `rooms/deployment/CHANGELOG.md`의 `## [X.Y.Z] — 날짜` 엔트리(사람의 원장). `cmd_release`는 이 둘을 같은 순간에 쓴다.
2. **배포 계보를 조회하는 표면이 없다.** 사이클 계보는 `gil log`가 그래프로 그린다. 배포 계보에는 그 짝이 없다. `git tag -l`은 태그 이름만 나열할 뿐 — 일자·노트·도구 변경·CHANGELOG와의 일치 여부를 대조하지 못한다.
3. **두 기록은 어긋날 수 있다(drift).** release가 부분 실패로 커밋되거나, 손으로 CHANGELOG를 고치거나, 태그가 지워지면 태그 집합과 CHANGELOG 버전 집합이 달라진다. 오늘은 아무도 이 drift를 감시하지 않는다.

## 문제 분할

#3 전부(추적층 + release 강화)를 한 사이클에 하지 않는다. 정복할 **첫 번째 최소 단위**:

> **`gil releases` — 배포 계보의 읽기 전용 조회 프리미티브.**
> 깃 태그(`v*`)와 CHANGELOG를 **대조**해 각 릴리스를 시간 역순으로 보고한다: 버전 · 일자 · 노트 · 도구 변경 여부 · **두 기록 각각의 존재 여부**. 사람용 렌더 + 기계 훅 `gil:release` 한 줄/릴리스. 저장소를 **바꾸지 않는다**(§7.2 안전한 탐침 정신).

이것을 첫 카브로 고른 이유: `git tag -l`을 넘어서는 값은 **대조**에 있다 — 태그와 CHANGELOG를 한 뷰로 합치고 한쪽에만 있는 릴리스(drift)를 드러낸다. 읽기 전용이라 회귀 위험이 0에 가깝고, release 실행 경로를 건드리지 않아 병렬 병합에 안전하다. release 강화(근거 사이클 계약화 등)는 이 조회 표면 위에서 다음 사이클이 다룬다.

## 가설

> **가설(H1)**: `gil releases`가 깃 태그와 CHANGELOG 두 기록을 대조하면, 어느 기록에도 없는 계보를 지어내지 않으면서(§3.2) 배포 계보를 한 뷰로 조회 가능하게 만들 것이다 — 저장소 무변화, 회귀 0, 기계 훅으로 소비 가능하게.
>
> **가설(H2)**: 새 명령 표면 `releases`를 CONTRACT_COMMANDS에 등록하면, 아직 이식되지 않은 Go 구현의 정직한 부재(exit 3)를 HELP-COMPLETE가 판정할 것이다(C043 예방 리듬).

## 기각 조건

- 태그도 CHANGELOG도 없는 항목을 지어내면(계보 발명) → 기각.
- `gil releases`가 저장소를 변경하면(§7.2 위반) → 기각.
- Python 참조 conformance가 78/78에서 후퇴하면 → 기각.
- Go 바이너리가 `releases`에 exit 3(정직한 부재)로 답하지 못하면 → 기각.
- 태그 없는(비-git) 환경에서 크래시하면 → 기각(우아한 강등이어야, C052의 결).

## 범위 밖 (다음 사이클로 정직히 이월)

- `gil release` 강화 — 근거 사이클(`--cycle <chain>/<id>`)을 계약으로 요구·기록(배포의 방 README 규칙 #1의 집행). release 실행 경로를 건드리는 더 큰 카브.
- Go 이식(`releases`의 Go 구현).
- 배포 계보의 뷰어(web) 통합 (Sheen의 loomlight와 접점).
