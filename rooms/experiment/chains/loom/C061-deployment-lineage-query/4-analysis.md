## 데이터 직접 관찰

- **실저장소는 두 기록이 완벽히 일치한다** — 48개 릴리스 전부 `[TC]`, `drift=0`. 이는 `cmd_release`가 태그·CHANGELOG를 **한 커밋에** 함께 쓰기 때문이다(gil.py의 release 실행부). 즉 정상 경로로만 릴리스하면 drift는 0이 유지된다. drift는 곧 "정상 경로 밖에서 누군가 손댔다"는 신호 — 이것이 `git tag -l`이 절대 줄 수 없는 정보다.
- **drift 케이스의 note 출처가 갈린다.** CHANGELOG-only(v1.2.0)는 CHANGELOG 불릿을, 태그-only(v1.1.0)는 태그 subject("Ariadne release v1.1.0 — 태그만")를 노트로 쓴다. 출처를 지어내지 않고 **그 기록이 실제로 말하는 것**만 각기 인용했다. 태그-only의 일자도 태그 creatordate(2026-07-19)를 그대로 썼다 — CHANGELOG가 없으니 CHANGELOG 일자를 발명하지 않는다.
- **cycle 태그 배제는 두 겹으로 안전하다.** `refs/tags/v*` 글롭이 1차로 거르고, `_SEMVER_RE.match(name[1:])`가 2차로 거른다. `cycle/x/C001-y`는 `v`로 시작하지 않아 글롭에서 이미 탈락. 사이클 봉인 태그와 릴리스 태그가 같은 저장소에 공존해도 섞이지 않는다.

## 예상과 달랐던 것

- **저장소 유추의 함정(설계 중 발견·수정).** 처음엔 `_repo_root(args.root)`로 저장소를 유추했으나, `--root` 기본값(`rooms/experiment/chains`)이 없는 디렉토리를 가리키면 `git -C <없는경로>`가 실패해 **git이 있는데도 "git 부재"로 오판**했다. 릴리스가 사는 곳은 체인 루트가 아니라 **패키지**(CHANGELOG를 담는 곳)임을 깨닫고, 유추 기준을 `--package`로 옮겨 해결했다. 교훈: 조회의 앵커는 조회 대상이 실제로 사는 경로여야 한다.
- **재사용한 경고가 맥락에 안 맞았다.** 초안은 git 부재 시 `_warn_git_missing_once()`("각인을 건너뛴다")를 재사용했으나, releases는 읽기 전용이라 각인(커밋) 얘기가 부적절했다. releases 전용 메시지("태그 대조 생략 — CHANGELOG만")로 교체. 재사용이 늘 옳은 건 아니다.

## 판정

**H1 채택** — `gil releases`가 두 배포 기록을 대조해, 계보를 지어내지 않고(§3.2), 저장소 무변화로, 기계 훅으로 소비 가능하게 배포 계보를 조회 가능하게 만들었다. drift 대조는 `git tag -l`을 넘어서는 실질 가치를 입증했다.

**H2 채택** — `releases`를 CONTRACT_COMMANDS에 등록하니 Go의 정직한 부재(exit 3)를 HELP-COMPLETE가 판정했다(Go 78/78). 부분 구현 합법·거짓 보고 불법의 리듬(C043) 준수.

기각 조건 어느 것도 발동하지 않았다.
