# 5. 결과 보고

## 요약

배포는 깃 태그(`v<semver>`)와 CHANGELOG라는 **두 기록**으로 남지만 그 둘을 대조·조회하는 표면이 없었다(사이클 계보엔 `gil log`가 있으나 배포 계보엔 짝이 없었다). 읽기 전용 조회 프리미티브 **`gil releases`** 를 참조 구현에 더해, 두 기록을 대조하고 **drift**(한쪽에만 있는 릴리스)를 드러내게 했다. Python 79/79·Go 78/78, 회귀 0, 저장소 무변화로 **H1·H2 모두 채택**.

## 교훈

1. **배포의 두 몸은 대조되어야 계보가 된다.** 태그는 깃의 진실, CHANGELOG는 사람의 원장. `cmd_release`가 둘을 한 커밋에 쓰므로 정상 경로에선 drift=0이 유지된다 — 그래서 **drift는 곧 "정상 경로 밖에서 손댔다"는 신호**다. `git tag -l`이 못 주는 이 신호가 추적층의 첫 실질 가치다.
2. **출처를 지어내지 않는 원칙은 배포에도 그대로 적용된다(§3.2).** 노트·일자는 그 기록이 실제로 증언하는 것만 각기 인용했다(태그-only는 태그 subject/creatordate, CHANGELOG-only는 불릿/헤더일자). 없는 일자를 발명하지 않는다.
3. **조회의 앵커는 조회 대상이 사는 경로다.** 저장소 유추를 체인 루트(`--root`)에 걸었다가, 그 경로가 없으면 "git 부재"로 오판하는 함정에 빠졌다. 릴리스가 사는 곳은 **패키지**(CHANGELOG를 담는 곳)이므로 유추를 거기로 옮겨 해결.
4. **새 명령 표면은 CONTRACT_COMMANDS 등록으로 이식 부재를 정직하게 만든다(C043 리듬).** Go는 `releases`를 exit 3으로 정직히 부재 신고 → HELP-COMPLETE가 판정. 부분 구현은 합법.

## 다음 사이클을 위한 제안 (#3의 남은 카브들 — 분기 후보)

1. **`gil release` 강화 — 근거 사이클 계약화.** 배포의 방 README 규칙 #1은 "어느 사이클이 이 배포를 낳았는지 명시"를 요구하나 `release`는 이를 강제·기록하지 않는다(지금은 자유 노트에만). `gil release --cycle <chain>/<id>`(반복 가능)를 두어 ① 그 사이클이 **닫혀 있는지** 검증(배포는 닫힌 사이클 근거만) ② 태그/기계 훅에 기록. 그러면 `gil releases`가 릴리스→근거사이클 계보를 대조해 보일 수 있다. (release 실행 경로를 건드리므로 더 큰 카브.)
2. **drift를 게이트로.** 지금 `gil releases`는 drift를 보고만 한다(exit 0). `--check`(또는 fsck 편입)로 drift가 있으면 exit≠0 → CI가 배포 기록 불일치를 잡게.
3. **Go 이식** — `releases`의 Go 구현(현재 참조 전용).
4. **배포 계보의 뷰어 통합** — Sheen의 loomlight가 배포 계보(drift 포함)를 비추게.
5. **CHANGELOG 파싱의 강건화** — 현재 첫 `- ` 불릿 하나만 노트로 취한다. 다줄 노트·다른 CHANGELOG 방언 대응은 필요 시 별도 카브.
