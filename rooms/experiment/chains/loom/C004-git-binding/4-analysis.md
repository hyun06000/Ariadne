## 데이터 직접 관찰

- T3에서 verify가 두 종류의 변조를 다른 경로로 잡는 것을 확인했다: 추적 파일 수정은 `git diff <tag>`로, 미추적 신규 파일은 `git status --porcelain`으로. 후자를 빠뜨리면 "닫힌 사이클에 몰래 파일 끼워넣기"가 침묵 통과한다 — 설계에서 (a)수정+(b)추가를 모두 기대 행동에 박아둔 것이 구현 누락을 막았다.
- T7의 커밋 파일 목록(`git show --name-only`)을 직접 읽었다: 사이클 파일 8개뿐, 더럽혀둔 chain.md와 stray.txt는 각각 미커밋·미추적으로 남았다. `git add -A -- <경로>` + `git commit -- <경로>`의 이중 경로 제한이 격리를 보장한다.
- 백필에서 C001의 태그를 닫힘 커밋(9497ccb)이 아닌 **이주 커밋(a111d05)** 에 단 것이 옳았음을 run3이 증명한다 — 9497ccb에 달았다면 [migrate]로 승격된 lineage 줄이 즉시 "변조"로 보고됐을 것이다.

## 예상과 달랐던 것

- **이주 규정과 태그의 긴장이 실체가 됐다.** C002가 허용한 무손실 이주가 앞으로 일어나면, 태그↔작업 트리 diff가 그것을 변조로 보고한다. 백필에서는 "마지막 정당 수정 커밋에 태그"로 풀었지만, 이미 태그된 사이클의 미래 이주에는 **태그 이동 규약**(예: `[migrate]` 커밋 후 `git tag -f` + 이동 기록)이 필요하다 — 스펙 공백, 다음 사이클 재료.
- verify는 태그 없는 닫힌 사이클을 경고로만 알린다(exit 0). 모든 닫힌 사이클이 태그되는 시대(= close --git이 표준이 된 후)에는 경고를 위반으로 승격시켜야 할 것이다.
- 커밋 메시지 서명(author)은 깃 설정을 그대로 쓴다 — C003에서 발견한 "존재와 도구의 연동" 공백이 깃 계층에서도 반복된다: cycle.yaml의 author와 깃 커밋의 author가 따로 논다.

## 판정

**채택.** 기각 조건 4건(탐지 실패·격리 실패·부분 상태·하위 호환 파손) 중 어느 것도 발생하지 않았다.
"닫힌 사이클은 수정하지 않는다"는 이제 규범이 아니라, 태그와 diff로 집행되는 기계적 사실이다.
