# 1. 가설 수립

## 이전 사이클의 교훈

부모 **loom/C058-parallel-worktree-spawn**(`gil worktree add`)의 5-report.md에서:

1. **규율을 도구로 승격하면 사고가 구조적으로 불가능해진다.** C050에서 "메인에 잘못 open"이 훈계로만 막혔다가 잊혀 재발했다. `add`는 open을 워크트리 안에서 self-invoke해 그 사고를 물리적으로 봉인했다. 훈계 → 계약.
2. **얇은 오케스트레이터 + self-invoke가 발산을 막는다.** spawn이 open을 복제하지 않고 git worktree 명사에 올라타 self-invoke하니 양 구현이 자연히 대칭이 됐다.
3. **변이가 살아남는 것은 진단이다(C011).** 격리의 진짜 축이 cwd가 아니라 `--root`임을 살아남은 변이가 가르쳐 줬다.
4. spawn은 **여는 반쪽만** 만들었다. 병렬 작업의 머지백(land)은 여전히 소환자가 손으로. C058 제안 (A)가 곧 이 사이클이다.

**그리고 이 사이클을 여는 순간, 그 필요가 다시 실물로 증명됐다**: 나(Weft)는 격리 워크트리 안에서 일해야 했으나 bash cwd가 콜마다 리셋되는 환경에서 메인 저장소 경로로 `cd`해 `gil open --git --push`를 실행했고, C060 open 커밋(31d30dc)이 **main과 origin/main에 샜다** — C050의 정확한 재연. 소환자 Clew가 main에 라이브였으므로 main을 되돌리는 것은 Clew의 작업을 파괴할 위험이라, 커밋을 내 워크트리 브랜치로 ff해 확보하고 main 정리는 Clew에게 넘겼다. **훈계는 또 잊혔다. 도구만이 이 사고를 막는다.**

## 문제 분할

C058이 만든 결정론적 매핑 — 브랜치 `<author>/<chain>-<slug>`, 워크트리 경로 `<repo부모>/<repo이름>-worktrees/<chain>-<slug>` — 위에서 병렬 사이클을 main으로 되돌리는 닫는 반쪽이 필요하다. 최소 단위로 분할:

1. **매핑 역산**: `add`와 동일한 인자(`<chain> <slug> --author`)로 브랜치·워크트리 경로를 결정론적으로 재유도. (add와 대칭 — 이식 발산 최소화)
2. **--no-ff 병합**: 브랜치를 main(정확히는 repo의 현재 HEAD)에 병합 커밋을 반드시 남기며 병합. 병렬 작업의 경계가 이력에 보이게.
3. **충돌 검사**: 병합이 충돌하면 되돌리고(merge --abort) 거부 + **워크트리 보존**(사람이 해소하도록). 충돌을 삼키지 않는다.
4. **워크트리 정리**: 성공 시에만 워크트리 제거 + 브랜치 삭제. 병합 안 된 브랜치는 안 지운다(`-d` 안전 삭제).

지금 첫 번째로 정복할 것은 **1~4를 한 원자적 명령 `gil worktree land`로 묶는 것** — spawn(여는 반쪽)의 대칭 짝(닫는 반쪽). 이유: 방금 내가 겪은 사고가 증명하듯, 머지백이 수동이면 소환자가 손으로 병합·정리하다 실수하고, 병렬 모드는 도구가 양끝(add·land)을 다 잡아야 비로소 일급 절차가 된다.

## 가설

> **가설**: `gil worktree land <chain> <slug> --author X`가 (a) add와 동일한 결정론적 매핑으로 브랜치·워크트리를 역산하고, (b) `--no-ff`로 병합 커밋을 남기며 병합하고, (c) 충돌 시 되돌려 거부하고 워크트리를 보존하며, (d) 성공 시에만 워크트리+브랜치를 정리한다면 — 병렬 사이클의 머지백이 수동 실수 없이 한 명령으로 되돌아오고, add와 대칭인 얇은 오케스트레이터로 양 구현이 나란히 계약을 충족할 것이다.

## 기각 조건

1. **병합 무결성**: land 후 main에 병합 커밋(--no-ff)이 없거나, 브랜치 커밋이 main에 반영 안 되면 기각.
2. **충돌 안전**: 충돌 상황에서 land가 성공(rc0)하거나, 충돌 잔재(conflict marker·MERGING 상태)를 남기거나, 워크트리를 지워버리면 기각. (충돌 시엔 rc≠0 + 워크트리·브랜치 보존이어야)
3. **정리 정확성**: 성공 후 워크트리·브랜치가 남으면 기각. 반대로 병합 안 된 브랜치를 지우면 기각.
4. **회귀**: 양 구현 기존 conformance 총계 유지 못 하면 기각(새 WORKTREE-LAND 항목은 분모 증가라 회귀 아님).
5. **두 몸**: 참조·Go가 WORKTREE-LAND에서 다르게 관측되면 기각(구현 독립 계약 §7).
