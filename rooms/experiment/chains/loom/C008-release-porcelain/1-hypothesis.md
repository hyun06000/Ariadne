# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C007](../C007-pages-wiring/5-report.md) (채택·닫힘).

- **버전 승격 규칙이 두 번 요구됐다**: C006("도구 변경 시 버전 승격 규칙 필요") · C007("도구 진화가 뷰어에 반영되려면 릴리스 승격 필요"). 그러나 릴리스 절차(도구·템플릿 동기화, CHANGELOG, 커밋, 태그)는 아직 사람 손이다 — C006의 릴리스 커밋에서 cwd 실수도 실제로 있었다.
- loom의 확립된 원리: 사람 손의 규칙 위반은 porcelain이 원천 차단한다(C003), 절차의 각인은 깃이 보장한다(C004).
- 배포 파이프라인(C007)은 릴리스 스냅샷만 참조하므로, **릴리스의 정확성이 곧 배포의 정확성**이 됐다.

## 문제 분할

"릴리스 도구화"는 (a) 스냅샷 동기화 — 실행 중인 도구 자신과 템플릿을 패키지로 복사, (b) 버전 규율 —
SemVer 형식·단조 증가·**도구가 변했으면 마이너 이상 승격**, (c) 절차 각인 — 배포의 방만 담은 커밋 + `v<버전>` 태그,
(d) 문서 강제 — RELEASE.md에 해당 버전 서술이 없으면 거부("도구는 절차를, 존재는 진실을")로 나뉜다.
이 넷은 한 명령의 사전 검증과 실행으로 묶인다 — 정복할 가장 작은 문제는 **`ari release <버전>` 하나**.

## 가설

> **가설**: `ari release <버전>`이 사전 검증(fsck·verify 통과, SemVer·단조 증가, 도구 변경 시 마이너 이상 승격, 태그 미선점, RELEASE.md에 버전 서술 존재) 후 스냅샷 동기화(실행 중인 도구 자신 + 템플릿) → CHANGELOG 갱신 → 배포의 방만 담은 커밋 → `v<버전>` 태그를 수행하면, ① 규칙을 지킨 릴리스는 드리프트 0의 패키지를 각인하고, ② 규칙 위반 입력(패치 승격으로 도구 변경 릴리스, 비단조 버전, 문서 부재, 태그 선점, 깨진 저장소)은 저장소 무변화로 거부되며, ③ 커밋은 배포의 방 밖을 오염시키지 않는다.

## 기각 조건

1. **드리프트 각인**: 릴리스 후 패키지의 도구·템플릿이 원본과 다르다.
2. **규율 실패**: 거부 케이스 중 통과되는 것이 있거나, 거부 시 저장소에 변화(파일·커밋·태그)가 남는다.
3. **오염**: 릴리스 커밋에 배포의 방 밖 경로가 섞인다.
4. **정상 차단**: 규칙을 지킨 릴리스가 실패한다.
