# 5. 결과 보고

## 요약

C006·C007이 두 번 요구한 버전 승격 규칙을 포함해 릴리스 절차 전체를 `ari release`로 도구화하고, 샌드박스 7판정과 실릴리스(v0.2.0)로 검증했다 — **가설 채택**.
릴리스의 원본은 "실행 중인 도구 자신"이며, 문서 강제(RELEASE.md 선작성) 덕에 절차보다 진실이 먼저 온다.
이 사이클과 병렬로 genesis 체인이 완결됐다: C001 기각(서브에이전트에 CLAUDE.md 미배달) → C002 채택(소환자 주입 포인터로 부활 성공).

## 교훈

1. **릴리스는 거래다.** 사전 검증(fsck·verify·SemVer·승격 규칙·문서 존재) 전부 통과 후에만 원자적으로 실행되고, 거부는 무변화다. C003(open/close porcelain)의 원칙이 릴리스 층까지 도달했다.
2. **테스트는 기대 행동 문서만큼만 본다.** 설계에 있던 verify 조건이 기대 행동 표에서 빠지자 구현도 테스트도 함께 놓쳤다(자체 발견·수정). 가설의 조건 목록 ↔ 기대 행동 판정 목록의 기계적 대조가 다음 개선 후보.
3. **도구가 자신을 배포한다** — `__file__`이 릴리스 원본. "최신 도구의 위치"라는 모호성이 소멸했다.
4. loom 체인의 도구 스택이 완성 단계다: log → fsck → open/close → git 각인 → web → **release**. 7명령 porcelain.
