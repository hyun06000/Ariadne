# 2. 실험 설계

## 절차

1. **기대 행동 고정** — T1~T7을 `3-verification/fixtures/expected-release-tool.md`에 구현보다 먼저 기록한다.
2. **구현** — ari v6: `release <버전>` 서브커맨드.
   - 사전 검증(저장소를 건드리기 전에 전부): fsck·verify 통과 → SemVer 형식·기존 `v*` 태그 대비 단조 증가 → **도구 변경(실행 중인 자신 vs 패키지 사본) 시 마이너 이상 승격 요구** → 태그 미선점 → RELEASE.md에 해당 버전 문자열 존재(문서 강제).
   - 실행: 실행 중인 도구 자신(`__file__`)과 `_template/`을 패키지로 복사(동기화) → CHANGELOG의 `[Unreleased]` 아래에 `[버전] — 일자 — notes` 섹션 삽입 → **배포의 방만** `git add`·커밋 → 주석 태그 `v<버전>`.
   - 실패 시 원상 복구(C003·C004의 pre-flight 원칙 계승).
3. **테스트** — 샌드박스(git 저장소 + 구버전 패키지 + 태그 v0.1.0)에서 T1~T7. 거부 케이스는 저장소 무변화(스냅샷 해시 + 태그 목록) 확인.
4. **도그푸딩 — v0.2.0 실릴리스**: 이 사이클로 도구가 변했다(release 명령 추가) → 마이너 승격 의무 발동. SPEC.md CLI 표에 release 추가, RELEASE.md에 0.2.0 서술 작성(문서 강제 통과) 후 실제 레포에서 `ari release 0.2.0` 실행. push하면 Pages도 새 스냅샷으로 갱신된다.
