# 4. 결과 분석

## 통계적 결과

기준값 전부 충족: 마지막 예약 소비 open --git 정상 각인(디렉토리 ∧ 커밋 ∧ 원장 비움), 커밋된 예약 소비 시 삭제 반영(유령 0), 예약 여럿·무예약 불변, 참조 103/103·Go 89/89(OPEN-LAST-RESERVATION-GIT PASS, 회귀 0), parity.

## 데이터 직접 관찰

버그의 핵심은 `git add -A -- <path>`의 비대칭이었다(실측): tracked 파일 삭제는 스테이징하지만, **tracked인 적 없는 부재 경로는 pathspec 거부**한다. 예약을 `--git` 없이 걸면(reserve는 원장 파일만 쓰고 커밋 안 함) reservations.tsv가 untracked인데, open이 그걸 소비·삭제하면 그 부재 경로가 git add를 죽였다.

수정은 "파일이 존재하거나 tracked이면"으로 조건을 좁혔다 — 두 축을 함께 봐야 한다: (a) 파일 존재 = 여럿 중 하나 소비(파일 남음), (b) tracked = 커밋된 예약 소비(삭제 스테이징 필요), (c) 둘 다 아님 = untracked 마지막 예약(제외해야 함). 기존 코드는 (a)만 봤고(else로 무조건 넣음) (c)를 (b)로 오인했다.

**판정기가 이 버그를 못 본 이유도 데이터에 있었다**: OPEN-PROMOTES-OWNER가 예약 소비를 검사했으나 `--git` 없이 실행했다 — 버그는 `--git` 경로에만 있으니 사각이었다. 새 항목은 git 저장소에서 --git으로 검사해 그 사각을 메운다.

## 예상과 달랐던 것

- **기존 코드가 이미 삭제를 의식하고 있었다**: `os.path.isfile(res_path) else _reservations_path(...)` — 삭제 케이스에 다른 경로를 넣도록 분기돼 있었으나, "삭제됐으면 넣지 말아야 한다"가 아니라 "다른 경로를 넣는다"로 잘못 처리했다. 결함을 반쯤 알면서 틀리게 대응한 코드.
- **exit=0 오보**: 커밋 실패인데 종료코드가 0이었다(부수 관찰). 수정으로 실패가 사라져 해소되나, open의 부분 실패 시 exit 전파는 별도 점검 후보로 남긴다.
