# 5. 결과 보고

## 요약

`gil threads`를 넣어 (1) 미소비 예약(= 진행 중 병렬 사이클, main에 커밋된 `reservations.tsv`)과 (2) 전역 status=open 사이클을 `--json` 계약면으로 반환하게 했다. 참조 96/96(THREADS-\* 6항목)·변이 격추(95/96)·회귀 0으로 **채택**. show가 단일 노드의 이웃을 준다면 threads는 그 그래프 전역판이며, 상현님의 "병렬로 뭐가 도나"를 명령 한 줄로 답한다.

## 교훈

1. **없던 것은 저장이 아니라 질의 표면이었다 (C040의 threads판).** 병렬 진행 정보는 이미 `reservations.tsv`로 main에 있었다. threads는 새 파서 0줄로(`_scan_chains`·`_load_reservations` 재사용) 그것을 드러내기만 했다. 워크트리가 .git을 공유하기에 예약이 main에 살아, 브랜치를 체크아웃하지 않고도 병렬이 보인다 — 관측성은 워크트리 모델의 공짜 이점이다.
2. **관점 상대성이 정직함이다.** threads는 "이 체크아웃 기준 loose end"를 반환한다 — 워크트리에선 자기 사이클이 open으로, main에선 그 예약이 미소비로 보인다. 소비 필터(그 트리에 사이클로 존재하면 제외)가 이 정직함을 지킨다. 변이 테스트가 이 필터를 제거하자 즉시 잡혔다(지어냄 방지, C038).
3. **계약과 온보딩은 다른 표면이다.** 판정기(HELP-COMPLETE)엔 show가 등록됐지만 SPEC §5 표엔 show도 handoff도 없었다 — 기능은 계약됐으나 발견 경로엔 없었다(C069 재연). threads뿐 아니라 show도 같은 커밋에 표에 넣어 짝을 복원했다. **등록만으론 부족하다; 사람·LLM이 읽는 경로에도 문장이 있어야 산다.**
4. **handoff와 threads의 경계 = 독자와 등급.** handoff는 사람용 세션 서사(예약 무시, JSON 없음), threads는 기계용 loose-end 인덱스(예약 인식, JSON 계약). 겹치지 않게 경계를 그었다.
