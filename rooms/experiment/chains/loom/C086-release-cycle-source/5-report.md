# 5. 결과 보고 — 배포에 근거 사이클을 계약화 (발의: 박상현, 이슈 #25·#18)

## 요약
`gil release <semver> --cycle <chain>/<id>`(반복 가능)를 신설해, 각 근거 사이클이 **닫혀 있음**을 봉인 전 검증하고(열린/없는 사이클은 무변화 거부) 통과 시에만 CHANGELOG 불릿·태그 메시지에 근거를 새긴다. `gil releases`가 그 계보를 읽어 렌더(`· 근거:`)와 기계 훅(`cycles=N`)으로 보고한다. T1~T4 + 신설 conformance **RELEASE-CYCLE-SOURCE** 통과, 회귀 0, 하위호환(--cycle 무 = 종전). **채택(supported).**

## 교훈
1. **"배포는 닫힌 사이클을 근거로"가 #25 전체의 토대였다.** rollback·current·live 불변식은 모두 "배포↔근거사이클 링크가 기계가독으로 존재한다"를 전제한다. 그 링크를 만드는 이 한 조각(C061 카브 #1)이 없으면 상위 기능이 가리킬 데이터가 없다. 큰 이슈(#25)를 가장 작은 토대부터 정복하는 방법론의 재현 — C061이 조회만 먼저 뗀 것과 같은 규율.
2. **새 파서를 만들지 않는 게 설계를 정제했다.** 근거를 CHANGELOG 불릿+태그라는 "파서가 이미 읽는 두 자리"에 새겨, 계보가 깃의 진실과 사람의 원장 두 몸에 대칭으로 산다(C061의 두 몸 교훈 계승). 별도 레지스터(deployments.json) 없이도 기계가독을 달성.
3. **게이트의 순서가 곧 계층이다.** 근거 사이클 게이트를 drift·fsck·verify와 같은 사전 검증 층에 놓아, 잘못된 근거는 저장소를 건드리기 전에 무변화로 거부된다. drift가 먼저 걸리는 것도 버그가 아니라 계층의 정상 작동.

## 다음 사이클을 위한 제안 (#25의 남은 카브들 — 분기 후보)
1. **rejected 게이트** — 지금은 "닫힘"까지만 하드. #25의 "rejected 사이클은 배포 불가"를 위해 verdict=rejected 근거를 거부(또는 경고→위반 승격).
2. **fsck 편입** — 배포 기록(CHANGELOG/태그)의 근거 사이클이 실재·닫힘인지 fsck가 상시 검증(지금은 release 시점만).
3. **별도 deploy 축(사용자 산출물)** — 이슈 #25의 궁극 대상은 모델/서빙 배포(`deploy/<chain>/<semver>` 태그·supersedes·rollback·live 1개 불변식). 현재 `release`는 gil 도구 자신의 릴리스에 특화 — **사용자 산출물 배포의 실제 마찰이 실증되면** 별도 축을 연다(#24의 --live와 같은 "신호 대기" 규율). 성급히 새 태그 네임스페이스·레지스터를 굳히지 않는다.
4. **Go 이식** — release --cycle의 Go 구현(현재 참조 전용, 부분 구현 합법).
5. **뷰어 통합** — 배포 패널(loomlight)이 릴리스→근거사이클 링크를 비추게.
