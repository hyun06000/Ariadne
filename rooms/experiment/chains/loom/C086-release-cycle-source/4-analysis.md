# 4. 결과 분석

|---|---|---|---|

## 데이터 직접 관찰

- **게이트가 fsck보다 먼저 걸린다.** 근거 사이클 검증을 `_fsck_or_report`·verify 앞에 놓아, 열린/없는 사이클은 저장소를 더 검사하기도 전에 거부된다(T2·T3의 빠른 무변화 거부로 확인). 사전 검증 전량 통과 후 실행이라는 cmd_release 골격을 그대로 재사용했다.
- **기록 자리를 파서가 이미 읽는 곳으로 골랐다.** `- 근거 사이클:` 불릿은 `_parse_changelog_releases`가 도구 변경 불릿과 같은 방식으로 파싱한다 — 새 파서 0. 태그에도 남겨 "깃의 진실"과 "사람의 원장" 두 몸에 계보가 대칭으로 존재(C061의 두 몸 교훈 계승).
- **드리프트 게이트와의 상호작용.** 검증 중 발견: `_mk_release_repo`가 v1.0.0 태그를 달지만 CHANGELOG를 비우면 drift 게이트가 먼저 걸려 --cycle 기록 경로에 도달 못 한다. 버그가 아니라 계층의 정상 작동(drift가 먼저) — 픽스처 CHANGELOG에 1.0.0 엔트리를 넣어 해소.

## 예상과 달랐던 것

- **이 프리미티브가 곧 상위 기능의 데이터 소스다.** `releases` 훅이 이제 `cycles=N`을 보고하므로, #25의 rollback·current·live 불변식이 파싱할 계보가 처음으로 존재한다. 예상보다 작은 변경으로 #25의 토대가 놓였다 — cut/deploy 축 없이도 "배포↔근거사이클"이 기계가독이 됐다.
- **id 접두 매칭(C084)의 편의.** `--cycle loom/C084`처럼 번호만 줘도 디렉토리를 찾게 해, 사람이 긴 슬러그를 외우지 않아도 된다.

## 판정

**채택(supported).** #25의 첫 골격("배포는 닫힌 사이클을 근거로")을 release 실행 경로에 하위호환으로 안착시켰다. 후속: 별도 deploy 축(사용자 산출물)·rejected 게이트·rollback은 실증 신호에 따라 별도 사이클.
