# 5. 결과 보고

## 요약

"최신 gil을 받은 Claude가 병렬 기능을 안 쓴다"는 상현님 관찰을 감사해, 원인이 **기능 부재가 아니라 안내 부재**임을 확인했다(신규 에이전트 읽기 경로의 병렬 언급 전부 0, SPEC 명령 표에 `gil worktree` 행 부재). 병렬 워크트리 워크플로(`gil worktree add/land`·소유 guard·"네 워크트리에서 일하라")를 읽기 경로 4곳(README.ai.md Step E·CLAUDE.md §3·SPEC §5표+§6.8·QUICKSTART §4.5)에 명시했다. 도구 코드 무변경이라 conformance 90/90·83/83·web 바이트 동일 — 가설 채택.

## 교훈

1. **기능의 수명은 코드가 아니라 온보딩 경로의 문장이 정한다.** 병렬은 세 사고·세 사이클(spawn·land·guard)로 공들여 만들었는데, 안내가 없으니 다음 세대 에이전트는 존재조차 몰라 안 썼다. **만들면 쓰는 게 아니다 — 읽는 경로에 있어야 쓴다.** 도구 자기보고(`gil help`)는 탐침이지 교육이 아니다: 무엇을 물어야 할지 모르면 못 묻는다.
2. **좋은 문서가 좁게 완벽하면 넓은 기능을 가린다.** README.ai.md는 순차 사이클을 완벽히 가르쳤고, 그걸 충실히 따를수록 병렬을 안 썼다. 온보딩은 "가장 흔한 길"뿐 아니라 "언제 다른 길로 가는가"의 분기점도 심어야 한다.
3. **안내와 계약(도구)은 분리된다.** 문서 4곳을 고쳐도 도구 파일 0변경 → conformance·web 바이트 불변. 안내 개선은 계약면을 안 건드린다(문서 릴리스의 근거).
