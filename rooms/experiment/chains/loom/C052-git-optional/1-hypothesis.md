# 1. 가설 수립

## 이전 사이클의 교훈

부모: **loom/C051** (fsck 계약면). 그 사이클을 닫으며 상현님이 실사용 제보를 주었다 — **비개발자 친구가 gil을 시도했으나 git 부재로 에러·경고 폭탄을 맞고 이탈.** C027·C028의 *"수신자 경로가 배포자가 못 보는 간극을 드러낸다"* 의 새 사례이되, 이번 수신자는 **비개발자**다. 우리는 늘 git이 있는 환경에서만 딸깍을 검증했다 — 정확히 우리가 안 밟은 길.

상현님과 스코프를 정리했다: git 부재 대응(A) + Windows(B 바이너리·C PowerShell 설치·D 온보딩 문서). 상현님이 **A부터 지금**을 선택 — 플랫폼 무관, 내가 완전 검증 가능. B~D는 이어지는 사이클로.

## 문제 분할 — 실측이 설계를 정했다

git 부재 시 두 구현을 직접 돌려봤다 (전제는 가정 말고 확인, C007):

- **참조(Python): `rc=1`, 트레이스백.** `subprocess.run(["git", ...])`가 git 미설치 시 `FileNotFoundError`를 던지는데 `_git`·`_repo_root`가 안 잡는다 → 날것의 크래시. 파일은 만들어지고 커밋 단계에서 터진다.
- **Go(배포 바이너리): `rc=0`이나 엉뚱한 경고** — "뷰어 갱신 실패 — 렌더할 체인이 없다". git 부재라는 진짜 원인은 침묵하고 하류 증상만 뱉는다.

**두 구현이 서로 다르게 망가져 있다.** 배포되는 건 Go이므로 친구가 본 "경고 폭탄"의 정체는 Go의 하류 경고다. 목표 행동을 **하나로 수렴**시킨다: git 부재를 **앞에서 감지 → git 연산 건너뜀 → 친절한 한 줄 안내 → rc 0 + 파일 생성.**

## 가설

> **가설**: gil이 명령 시작 시 git 가용성을 감지하여(`git` 실행 파일 부재), 커밋이 요청됐어도(C033 기본) **크래시·오도 경고 없이 no-commit으로 강등하고 한 줄 안내를 출력한 뒤 rc 0로 완주**하게 하면 — git 없는 환경(비개발자)에서도 `open`/`step`/`close`가 파일을 남기고 정상 종료한다. 양 구현이 같은 행동으로 수렴하며, 이를 판정기(`NO-GIT-GRACEFUL`)가 집행한다.

## 기각 조건

- 수정 후에도 어느 구현이 git 부재에서 **트레이스백을 내거나 rc≠0**이면 → 기각.
- 안내가 git 부재를 **명확히 지목하지 못하고** 여전히 하류 증상(뷰어 실패 등)만 말하면 → 부분 실패(수신자가 원인을 모른다).
- git이 **있는** 환경에서 회귀가 발생하면(정상 커밋 경로가 깨짐) → 기각. (핵심 불변식: git 있을 때 73→74/74 회귀 0.)
- 판정 기준은 총계가 아니라 **회귀 0 + git-부재 완주 + git-있음 정상**(C038: 수치는 불변식으로).
