## 데이터 직접 관찰

- **두 언어의 같은 병, 다른 증상.** 참조는 `FileNotFoundError`(파이썬 예외)라 `returncode` 가드를 통과해 **크래시**했다. Go는 `exec.Command`가 `*exec.Error`를 돌려주고 `gitRun`이 `code=-1`로 흡수해 **크래시는 없었으나**, `repoRoot`가 ""(=저장소 아님)를 반환하면서 원인(git부재)을 잃고 하류의 `refreshViewers`가 `os.Getwd()`를 뒤져 무관한 HTML을 주워 **"렌더할 체인이 없다"** 헛경고를 냈다. **친구가 본 "경고 폭탄"의 정체가 이 하류 경고다** — 진짜 원인(git 없음)은 아무도 말해주지 않았다.
- **수정 전/후 stderr 대비 (참조, git부재 open):**
  - 전: `Traceback (most recent call last): … FileNotFoundError` (rc 1)
  - 후: `ℹ git이 없어 각인(커밋)을 건너뛴다 — 사이클 파일은 저장됐다. git 설치(…) 후 이력·되감기·뷰어 자동갱신이 켜진다.` (rc 0)
- **헛경고의 뿌리는 `root = repo or os.getcwd()`.** git이 있을 땐 `repo`(저장소 루트)로 뷰어를 찾지만, 없으면 **임의의 cwd**로 떨어졌다. cwd에 이전 테스트의 HTML이 있으면 그걸 주워 굽다 실패. `gil pages`가 이미 쓰던 heuristic(`chains_root/../../..`)으로 고정하니 사라졌다. 두 구현 동일 수정.

## 예상과 달랐던 것

- **Go는 크래시하지 않았다 — 그래서 더 나빴다.** 설계 시 "둘 다 크래시"를 예상했으나, Go는 `exec.Error`를 흡수해 조용히 rc 0로 넘어가되 **원인을 오도**했다. 크래시는 최소한 어디서 터졌는지 알려주지만, 오도 경고는 사용자를 엉뚱한 곳으로 보낸다. **침묵보다 나쁜 것은 잘못된 신호다**(C042 "낡은 화면은 침묵보다 나쁘다"의 친척).
- **수정의 절반이 git과 무관했다.** git부재 감지는 핵심이었지만, 정작 "경고 폭탄"을 없앤 건 **뷰어 검색 루트 고정**이었다 — git부재가 *드러낸* 잠복 버그(cwd 의존). **한 문제를 고치러 가면 그 문제가 가리고 있던 다른 문제가 보인다**(C037 "실패 로그를 끝까지 읽어라"의 구조판).

## 판정

**가설 채택 (supported·이탈 0).** git 부재를 앞에서 감지해 크래시·오도 경고 없이 no-commit으로 강등하고 한 줄로 원인을 지목한다. 양 구현이 같은 행동으로 수렴, git-있음 회귀 0, `NO-GIT-GRACEFUL`이 회귀를 문다. 비개발자의 첫 경로(빈 폴더·git 없음)가 이제 완주한다 — 스코프 A 완결. B~D(Windows)는 후속.
