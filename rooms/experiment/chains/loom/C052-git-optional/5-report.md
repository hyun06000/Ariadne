# 5. 결과 보고

## 요약

상현님의 실사용 제보(비개발자 친구가 git 부재로 에러·경고 폭탄을 맞고 이탈)를 스코프 A로 매듭졌다. **git 부재 시 참조는 `FileNotFoundError`로 크래시, Go는 크래시는 없으나 원인을 오도하는 하류 경고**를 냈다. 두 구현이 git 부재를 앞에서 감지해 **크래시·오도 경고 없이 no-commit으로 강등하고 한 줄로 원인을 지목**하게 하고, 이를 판정기(`NO-GIT-GRACEFUL`)로 집행했다. git-있음 회귀 0(74/74×2), git-없음 완주, 변이 격추. **비개발자의 첫 경로가 이제 완주한다.**

## 교훈

1. **침묵보다 나쁜 것은 잘못된 신호다.** 참조의 크래시는 "여기서 git 때문에 터졌다"를 (거칠게라도) 알려줬지만, Go는 크래시를 흡수하고 **"렌더할 체인이 없다"**는 무관한 경고로 사용자를 엉뚱한 곳으로 보냈다. 비개발자는 그 경고를 해석할 수 없어 그냥 껐다. **오류를 삼키는 것이 곧 친절이 아니다 — 원인을 정확히 지목하는 것이 친절이다.** (C042 "낡은 화면은 침묵보다 나쁘다"의 친척.)
2. **한 문제를 고치러 가면 그것이 가리던 다른 문제가 보인다.** "경고 폭탄"의 진짜 뿌리는 git 감지가 아니라 **`root = repo or os.getcwd()`** — git이 없을 때 뷰어 검색이 임의의 cwd로 떨어져 무관한 HTML을 줍던 잠복 버그였다. git 부재가 그것을 *드러냈을* 뿐. 표면 증상(git)을 파다 구조적 결함(cwd 의존)을 만났다. (C037 구조판.)
3. **크래시 없음이 견고함이 아니다.** Go가 rc 0로 "통과"한 것은 견고해서가 아니라 에러를 조용히 흡수했기 때문. 판정기가 `NO-GIT-GRACEFUL`로 **"완주 + 파일 + 무크래시 + (원인 지목)"** 을 계약하기 전까지, Go의 rc 0는 만점처럼 보였다. C051의 교훈("도구가 이미 보고 있어도 판정하지 않으면 없는 계약")의 배포판 — **관측되지 않은 견고함은 견고함이 아니다.**
4. **수신자가 비개발자면 배포자의 눈은 더 멀다.** 우리는 늘 git 있는 개발 환경에서만 검증했다(C024·C027·C028의 반복). 이번 수신자는 우리가 밟을 수조차 없는 환경(git 없음)에 있었다 — 상현님의 친구가 그 길을 밟아 알려줬다. **가장 먼 수신자가 가장 깊은 간극을 드러낸다.**

## 다음 사이클을 위한 제안

이 사이클은 스코프 A(git 부재)만 닫았다. 상현님과 합의한 나머지 — **비개발자·크로스플랫폼 진입** 아크:

- **(B) Windows 바이너리 타깃** — 릴리스 워크플로에 `windows/amd64`(+arm64) 추가, `gil-windows-amd64.exe` 산출. Go 크로스컴파일로 가능하나 **Windows 실행 검증은 상현님/친구가 수신자-테스터**(내가 못 밟는 플랫폼).
- **(C) PowerShell 설치 스니펫** — 현재 `curl|sha256sum|chmod`는 Unix 전용. Windows용 `Invoke-WebRequest`+`Get-FileHash` 스니펫과 체크섬 게이트(C037의 문지기 정신을 PowerShell로).
- **(D) README.ai.md 온보딩** — LLM이 OS·git 가용성을 확인하고 설치를 유도하도록. "당신의 에이전트에게 이 글을 읽히세요"(gateway/C001)의 환경-인식판.
- (E) verify·release 등 다른 명령의 git부재 경로 점검(이 사이클은 open/step/close/reserve만 확인 — supersede/correct는 하드에러로 남았으나 크래시는 아님).

권고: **B·C·D를 묶어 하나의 아크로**, 다만 실행 검증은 상현님의 Windows 접근에 의존하므로 상현님과 타이밍을 맞춘다.
