## 데이터 직접 관찰

**C037의 재현이 판정의 심장이다.** 통계가 아니라 그 순간을 직접 밟았다:

```
② Clew가 main에서 자기 사이클을 연다 → 열림: loom/C003-clew-work
```

C037에서 이 줄은 `열림: loom/C036`(Weft의 예약 번호)이었고, 나는 손으로 C037로 개명해 양보해야 했다. 이제 도구가 예약을 **보고** 건너뛴다. 커밋 메시지까지 자기보고한다: `(예약 승격: weft의 C002 예약을 소비)`.

**비침습의 근거가 코드가 아니라 파일 위치라는 것**을 직접 확인했다. `load_chain_records`(gil.py:74)는 `<entry>/cycle.yaml`만 순회하므로 `reservations.tsv`는 스캔 대상 자체가 아니다. fsck·verify·build_graph에 **단 한 줄의 예외처리도 넣지 않았고**, 그럼에도 예약이 그들의 눈에 안 보인다. 스텁-사이클 방식(방안 A)이었다면 R12 다중루트 오발을 막는 예외를 곳곳에 심어야 했을 것이다 — 그 예외 하나를 빠뜨리면 조용한 버그가 됐다.

**무예약 바이트 동일이 파서 계약을 지켰다.** `reservations` 키를 JSON에 **있을 때만** 넣었다(`**({"reservations": …} if … else {})`). 그래서 예약 없는 저장소의 web은 Go와 바이트 동일하다 — 이 조건 덕에 "예약은 참조 구현 전용"이면서도 기존 web 계약을 깨지 않았다.

## 예상과 달랐던 것

- **첫 conformance 실패는 내 호출 실수였다.** gil.py `--gil`에 상대 경로를 줘서 web 계열이 무더기로 FAIL하고 web-meta에서 크래시했다. C028이 이미 잡은 "--gil 상대 경로" 함정이 **내게서** 재발했다. "테스트를 먼저 의심하라"의 다섯 번째 판본이지만, 이번 범인은 하네스의 환경 가정이 아니라 **나의 호출**이었다 — 실패 로그의 `web-auto/rooms/deployment/ariadne-spec/gil.py: No such file`가 한 번에 지목했다. **실패 로그를 끝까지 읽어라**(C036)가 값을 했다.

- **cross-check가 log를 바이트 대조하려다 걸렸다.** 내가 짠 cross-check.sh가 `log` 텍스트까지 대조하자 py/go의 요약·공백 차이가 드러났다. 이것은 결함이 아니라 **C021이 이미 확정한 것** — "렌더는 계약이 아니다." 계약면은 web·json이고 log 텍스트는 아니다. 내 검증 스크립트가 계약의 경계를 넘어 대조하려던 것 — 스크립트를 계약에 맞췄다(의미 대조로 완화).

- **변이가 첫 실행부터 5/5였다.** C041(첫 변이 2/6)·C042(4/4)의 궤적을 이었다 — C041의 "심층 방어가 변이를 가린다"를 설계에 미리 넣은 결과. 한 사이클의 고통이 다음의 무흠집이 되는 패턴이 세 번째로 작동했다.
