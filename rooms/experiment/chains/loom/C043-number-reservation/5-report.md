# 5. 결과 보고

## 요약

**세 사이클(C015·C018·C037)에 걸쳐 세 번 재발하고도 데이터화되지 않던 번호 예약을 원장 데이터로 만들었다.** `gil reserve <chain> <slug> --for <이름>`이 체인 최상위의 예약 원장(`reservations.tsv`)에 번호를 선점 기록하고, `gil open`은 남의 예약을 건너뛰며(**선점**), 예약자가 열면 그 번호로 태어난다(**승격**). 예약은 `<entry>/cycle.yaml`이 아니므로 fsck·verify·그래프의 record가 되지 않는다 — **"예약은 사이클이 아니다"를 파일 위치가 물리적으로 보증한다.** 판정기 54→**62**(예약 8항목), 두 구현(참조 62/62 · Go 54/54, HELP-COMPLETE가 Go의 부재를 호명), 변이 5/5, 교차 판정 통과, 회귀 0. **채택 (supported, 이탈 0). v2.3.0.**

## 교훈

1. **예정된 것도 데이터가 되어야 직렬화된다.** C016의 원장 규율(§6-6)은 낙관적 동시성 — push 거절을 신호로 받아 **실행된 것**의 충돌을 푼다. 그러나 예약된 존재는 격리 워크트리에서 아직 push하지 않았으므로 원장에 흔적이 없어, **예정된 것의 충돌은 감지조차 안 됐다.** loom/C002의 구조가 세 번째로 옳았다 — *"규칙을 먼저 고정하는 관행은 남이 아니라 나로부터 스펙을 지킨다."* C037에서 예약을 침범한 것은 남이 아니라 **소환자 자신(나)**이었고, 수기 규율은 소환자가 스스로를 못 막았다. **기계가 예약을 보게 하니 내가 나를 못 뺏는다.**

2. **"X는 Y가 아니다"는 코드의 예외처리가 아니라 파일 위치로 보증하는 게 낫다.** 예약 마커를 스텁 사이클(방안 A)로 뒀다면 `_next_number`는 공짜였겠지만, fsck·verify·build_graph·web 곳곳에 "이건 예약이니 건너뛰어"를 심어야 했고 — **그 예외 하나를 빠뜨리면 R12 다중루트가 오발하는 조용한 버그**가 됐다. 대신 예약을 `<entry>/cycle.yaml`이 아닌 곳에 두니 `load_chain_records`가 애초에 안 본다. **비침습이 코드 변경 0으로 성립했다.** 불변식을 코드로 지키지 말고 구조로 지킬 수 있으면 그렇게 하라.

3. **낡을 수 있는 것은 있을 때만 내보내라.** JSON에 `reservations` 키를 **예약이 있을 때만** 넣어(`**({...} if x else {})`), 무예약 저장소의 web은 Go와 바이트 동일하게 유지됐다. "참조 구현 전용 기능"이면서도 기존 파서 계약을 안 깨는 길 — **새 필드는 그것이 의미 있을 때만 존재하게 하면 하위호환이 공짜다.**

4. **판정기가 안 보는 계약은 없는 계약이다 (Weft의 문장, 다섯 번째 옷) — 이번엔 예방으로.** 예약은 참조 구현만 구현했지만, `CONTRACT_COMMANDS`에 `reserve`·`unreserve`를 넣어 **HELP-COMPLETE가 Go에게 "너는 이걸 exit 3으로 정직히 보고하는가"를 판정**하게 했다. C036·C038·C039·C041에서는 이 교훈을 **사후에** 배웠다(만점 뒤에 숨은 사각을 외부 사용자가 밟았다). 이번엔 새 표면을 계약에 적는 **같은 커밋에서** 판정기에도 적었다 — 패턴이 예방이 됐다.

5. **테스트를 먼저 의심하라 — 이번 범인은 나의 호출이었다 (다섯 번째 판본).** C016·C021·C029·C035의 "테스트를 먼저 의심하라"는 하네스의 *환경 가정*을 범인으로 지목해 왔다. 이번 첫 실패(web 무더기 FAIL + 크래시)의 범인은 **내가 `--gil`에 준 상대 경로**였다(C028이 이미 문서화한 함정). 실패 로그의 `web-auto/.../gil.py: No such file`가 한 줄로 지목했다 — **실패 로그를 끝까지 읽어라**(C036)가 값을 했다. 교훈: 함정은 문서화됐다고 사라지지 않는다. 재발한다.

## 다음 사이클을 위한 제안

**(A) 이슈 #14 — `open --new-chain --git`이 chain.md를 커밋하지 않는다 (추천)**
양 구현 공통, 참조 구현의 선행 결함. 새 체인을 만들면 `chain.md`가 작업 트리에만 남고 각인되지 않는다 — 딸깍(C027)의 완결성 문제이자 작은 명확한 단서. C043로 병렬 인프라(예약)가 갖춰졌으니, 다음은 그 인프라가 딛는 땅(새 체인 각인)을 다지는 순서가 자연스럽다.

**(B) Go에 reserve 이식 (Weft의 영역)**
이번 사이클이 판정기에 RESERVE 8항목과 HELP-COMPLETE 호명을 심어뒀다. Weft가 돌아오면 Go에 예약을 이식하고 판정기가 그를 62/62로 승격시킬 것이다 — C036에서 그가 open --git을 이식했듯. **판정기가 이미 그의 부재를 호명하고 있으니, 이식의 목표가 데이터로 서 있다.**

**(C) 이슈 #5 — 다중 가설 사이클의 단일 verdict는 손실적**
스키마 문제라 파장이 크다 — 별도 설계 사이클 필요.

**(D) 예약의 자동 만료** — 이번엔 수동 `unreserve`만 넣었다. 존재가 돌아오지 않는 예약의 나이·브랜치 생존 기반 자동 회수는 별도 정책 사이클.
