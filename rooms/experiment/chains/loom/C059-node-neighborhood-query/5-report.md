# 5. 결과 보고

## 요약

상현님의 4방향 아크 중 **#4 [매우 중요] LLM 위키**의 첫 카브. 사이클 DAG는 이미 지식그래프인데(엣지가 cycle.yaml에 데이터로 존재) 사람용 web으로만, 전방으로만 탐색 가능했다. **`gil show <chain>/<id>`** — 한 노드의 신원 + 정방향 엣지(parent·lineage) + **백링크**(cited-by, 엣지 반전) + 보고서 포인터를 `--json` 계약면과 함께 반환하는 읽기 전용 원자 프리미티브를 신설. 참조 **86/86**(SHOW-* 6항목 신규, 회귀 0), Go는 정직한 부재(HELP-COMPLETE). 판정 **채택(supported)**.

## 교훈

1. **없던 것은 저장이 아니라 질의면이었다** (C040 "도구는 이미 보고 있었다"의 조회판). parent·lineage 엣지는 이미 데이터에 있었고 `gil log`·`web`이 그리고 있었다. show는 **새 파서 0줄** — `parse_cycle_yaml`·`load_chain_records`의 결과를 반전했을 뿐. 지식그래프는 만든 게 아니라 **질의 가능하게** 한 것이다.
2. **정방향과 백링크는 비대칭 비용이고, 그 비대칭이 #4의 로드맵 순서를 정한다.** 정방향은 노드 자신의 파일에 있어 공짜, 백링크는 전 체인 스캔(엣지 반전). 백링크가 가장 비싼 질문이라 먼저 정복했고, 효율 인덱스는 그 비용이 규모에서 아플 때의 다음 카브(65 노드에선 스캔이 무시할 만해 유예 — C007).
3. **두 표면이 다른 그래프를 말하면 안 된다** (C042 "web과 log가 다른 이야기를 못 한다"의 show판). `SHOW-EDGES-MATCH-GRAPH`가 show의 엣지와 web JSON을 같은 `build_graph`에 묶었다 — 지식그래프의 신뢰는 표면 간 일치에서 온다.
4. **부재는 지어냄이 아니라 신호다** (C040 P2의 조회판). 부재 노드에 빈 이웃을 조용히 반환하는 대신 명확한 거부(exit≠0).
5. **자기보고가 실측을 이겼다 — Claude Code가 독립적으로 #4를 원했다.** 세션 중 상현님이 실제 사용하는 Claude Code가 *"기억이 아니라 사슬 문서에서 다음 후보/열린 질문을 긁는다"*고 말했다. 지식그래프 표적 탐색이 편의가 아니라 LLM이 자연히 원하는 행동임의 실물 증거 → 다음 카브 `gil threads`의 근거.

## 다음 사이클을 위한 제안

- **(A) [상현님 발의·안전 최우선] cd-to-main guard** — 이 사이클을 세 번 문 병렬 사고(에이전트가 워크트리 밖 공유 main으로 cd → open 유출·WIP wipe)를 **도구가 막게**. #1 병렬 모드의 세 번째 조각(add→land→**guard**). 저자↔체크아웃 바인딩(자기 작업공간 밖 커밋 거부)이 유력. 훈계를 구조적 거부로(C038·C058).
- **(B) [#4 계속] `gil threads`** — "열린 실 훑기"(상현님·Claude Code 발의). 열린 사이클·무-verdict·예약·열린 라운드 등 **구조화된 신호**를 그래프 전역에서 긁는 조회. show(단일 노드)의 그래프 전역판. 산문 "다음 후보"는 데이터로 승격 고려(verdict·superseded·예약의 전례).
- **(C) [#4 계속] `gil trace <ref> --hops N`** — show를 N홉으로 조립한 표적 서브그래프.
- **(D) [#4] 노드 요약 표면** — 파일 안 열고 "이 노드가 관련 있나" 판단하게 하는 기계 요약(현재 report는 포인터만).
- **(E) show Go 이식** — 참조 전용을 Go로(HELP-COMPLETE가 이미 부재 호명, 목표 데이터로 섬).
