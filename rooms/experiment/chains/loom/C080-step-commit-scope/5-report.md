# 5. 결과 보고

## 요약

`gil step N`이 사이클 디렉토리 전체를 커밋해, 미리 만든 뒷 스텝 파일(4·5)까지 `2/5 설계` 커밋에 담기던 이슈 #20을 고쳤다(Clew가 이 세션에 실제 겪음). `git add/commit` 스코프를 사이클 디렉토리 → **cycle.yaml + 스텝 ≤N 파일(`_STEP_FILES[:n]` 중 존재하는 것)**으로 좁혀, 커밋이 스텝 단위를 반영한다. 참조·Go 동시 수정, 참조 **104/104**·Go **90/90**(STEP-SCOPE 신설, 회귀 0). 자기적용 실증(이 사이클 커밋이 스텝 경계를 지킴). **채택.**

## 교훈

1. **스코프의 굵기가 계약이다.** 문제는 스코프 부재가 아니라 스코프가 "디렉토리 단위"여서 "스텝 단위"보다 굵었던 것 — 한 겹 더(`[:n]` 파일 단위) 좁혀 해결. 이슈 제목의 "작업트리 전체"는 실은 "사이클 디렉토리 전체"였다(진단이 문제를 한 단계 넘겨 짚었으나 방향은 옳았다).
2. **자기적용이 최선의 시연**: 수정된 도구가 자기 사이클의 step 커밋을 스텝 경계로 자르는 것을 커밋 히스토리로 봤다(C013 정신). 도구가 자기를 심판한다.
3. **이 세션의 4연속 커밋-규율 사이클**: C077(Go 앱화 커밋)→C078(guard)→C079(예약 소비 git add)→C080(step 스코프). 커밋이 무엇을 담는가를 반복해 정련했다.
