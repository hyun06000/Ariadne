# 4. 결과 분석

## 통계적 결과

기준값 전부 충족: 뒷 스텝 미리 작성 후 step 2 커밋에 4·5 없음(cycle.yaml·1·2만), 정상 흐름 누락 0, step 3의 3-verification/ 하위째 담김, cycle.yaml 항상 커밋. 참조 104/104·Go 90/90(STEP-SCOPE PASS, 회귀 0), parity.

## 데이터 직접 관찰

기존 `cmd_step`은 `rel = cycle_dir`(사이클 디렉토리 전체)를 `git add -A -- <rel>`했다 — 그 안 모든 스텝 파일이 담겼다. 스코프를 **cycle.yaml + `_STEP_FILES[:n]` 중 존재하는 것**으로 좁혔다. `[:n]`이 정확한 경계다: step 2면 `_STEP_FILES[:2]` = 1·2 파일, 스텝 3(3-verification)·4·5는 슬라이스 밖이라 제외. 존재 확인(`os.path.exists`)이 정상 흐름을 보존한다 — 아직 안 쓴 파일은 애초에 paths에 없다.

**자기적용이 증거다**: 이 사이클 자신이 수정된 gil로 step 4를 밟자, 그 커밋에 5-report(아직 안 씀)가 없고 4-analysis·3-verification만 담겼다. 도구가 자기 사이클에서 스텝 경계를 지킨다(C013 "시연은 사이클 자신").

## 예상과 달랐던 것

- **cmd_step은 이미 사이클 디렉토리로 스코프하고 있었다** — "작업트리 전체(`git add -A`)"가 아니라 이미 `-- <cycle_dir>`로 제한돼 있었다. 이슈 제목의 "작업트리 전체를 삼켜"는 정확히는 "**사이클 디렉토리 전체**를 삼켜"였다. 문제는 스코프 부재가 아니라 스코프가 **디렉토리 단위**여서 스텝 단위보다 굵었던 것. 스텝 파일 단위로 한 겹 더 좁혔다.
- 3-verification이 디렉토리라 `_STEP_FILES[:n]`에 그대로 넣으면 `git add -A -- <dir>`가 하위째 담아줘, 별도 처리가 불필요했다.
