# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C079**가 마지막 예약 소비 open의 git add 경로 버그를 고치며, "**판정기 사각은 플래그 조합에 숨는다**"를 배웠다. 이번 이슈 #20도 커밋 경로(무엇을 add하나)의 문제다 — 이번엔 open이 아니라 step.

## 문제 분할 (이슈 #20)

`gil step N`은 `rel = cycle_dir` 전체를 `git add -A -- <cycle_dir>`로 커밋한다. 그래서 사이클 디렉토리 안의 **모든 스텝 파일**이 담긴다. 뒷 스텝 산출물을 미리 만들어 두고 step을 연달아 누르면, `2/5 설계` 커밋에 4-analysis·5-report까지 들어가 **스텝 경계와 커밋이 어긋난다**(재현 확인: step 2 커밋에 1·2·4·5 파일 전부).

이것은 이번 세션에 Clew 자신이 실제로 겪은 결함이다(문서 미리 쓰고 step 연달아).

### 스텝 N ↔ 파일 매핑
| N | 파일 |
|---|---|
| 1 | 1-hypothesis.md |
| 2 | 2-design.md |
| 3 | 3-verification/ |
| 4 | 4-analysis.md |
| 5 | 5-report.md |
| 매 step | cycle.yaml (step 필드 갱신 — 항상 커밋) |

## 갈래 (설계 판단 필요)

"step N 커밋이 무엇을 담아야 하나"에 두 해석:
- **(A) 스텝 경계 스코프**: cycle.yaml + **이미 전이한 스텝(≤N)의 파일**만 커밋. 아직 전이 안 한 스텝(>N) 파일은 제외 → 커밋이 스텝 단위를 반영. 뒷 파일은 그 스텝 전이 때 커밋됨.
- **(B) 경고만**: 스코프는 그대로 두되, 뒷 스텝 파일이 이미 있으면 경고("2/5인데 4-analysis 변경 감지"). 운영 규율에 맡김.

A가 이슈의 근본 해결(도구가 강제)이나, "스텝 N 커밋 = ≤N 파일"이라는 의미 결정이 필요하다. 상현님께 갈래를 묻는다.

## 가설 (A 방향 잠정)

> **가설**: `gil step N`이 `git add`할 경로를 사이클 디렉토리 전체에서 **cycle.yaml + 스텝 ≤N의 파일(`_STEP_FILES[:N]`)**로 좁히면, (a) `step N` 커밋에 스텝 >N 파일이 안 담기고(스텝 경계 반영), (b) 정상 흐름(스텝마다 작성 후 전이)은 불변이며, (c) cycle.yaml의 step 전이는 항상 커밋된다.

## 기각 조건

- 스코프 후에도 뒷 스텝(>N) 파일이 step N 커밋에 담기면 → 기각.
- cycle.yaml 전이가 커밋 누락되면 → step 진행이 깃에 안 보임, 기각.
- 정상 흐름(각 스텝 작성→전이)에서 그 스텝 파일이 누락되면 → 기각.
- 3-verification 디렉토리(스텝3)가 하위 산출물째 안 담기면 → 기각.
