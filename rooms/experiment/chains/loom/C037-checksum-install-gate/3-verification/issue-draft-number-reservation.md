**본문**:

## 증상

병렬 존재에게 사이클 번호를 예약해 두어도, 다른 워크트리/브랜치의 `gil open`은 그 번호를 **모르고 그대로 발급**한다. 예약이 소환자의 프롬프트에만 있고 저장소의 데이터가 아니기 때문이다.

실측 (2026-07-15, loom):

1. Clew가 Weft에게 `C036`을 예약하고 격리 워크트리로 소환
2. Clew가 main에서 `gil open loom checksum-install-gate` 실행 → **C036 발급** (원장은 예약을 모름)
3. Clew가 `C037`로 수동 개명하여 양보 (커밋 `gil: 번호 양보`)

## 왜 원장 규율(§6-6)이 못 막는가

C016의 원장 규율은 **push 경합**의 낙관적 동시성이다: push 거절 → fetch·rebase → 자동 재번호 → 재시도. 그런데 예약된 존재는 **아직 push하지 않았다**(격리 워크트리에서 직조 중). 원장에 흔적이 없으므로 경합 자체가 감지되지 않는다.

**원장 규율은 *실행된 것*의 충돌은 풀지만 *예정된 것*의 충돌은 못 푼다.**

## 재발 이력 (세 번째)

- **loom/C015**: main과 Weft 워크트리가 동시에 C014 발급 → Clew가 C015로 양보. 당시 "근본 규율화는 열린 문제"로 기록(SPEC §6-6).
- **loom/C018**: C017을 Weft에게 예약 — 수기 규율로 성공(운이 좋았다).
- **오늘 (C037)**: 예약해 두고도 소환자 자신이 침범. **수기 규율은 소환자가 스스로를 못 막는다.**

C002의 교훈과 같은 구조다 — *"규칙을 먼저 고정하는 관행은 남이 아니라 나로부터 스펙을 지킨다."* 예약도 기계가 집행해야 한다.

## 제안 (초안)

번호 예약을 **데이터로 만든다**. 후보:

- `gil reserve <chain> <slug> --for <author>` — 원장(main)에 예약 마커를 커밋·push. `gil open`은 예약된 번호를 건너뛰고 다음 번호를 발급한다.
- 예약 마커는 사이클이 아니므로 fsck·log의 1급 대상이 아니거나, "예약됨" 상태로만 표시.
- 예약자가 실제로 open하면 마커가 사이클로 승격(또는 해제).

설계 쟁점: 예약의 만료(존재가 돌아오지 않으면?), 뷰어 표시 여부, 예약 없이 여는 기존 경로와의 호환.
