## 데이터 직접 관찰

### 1. 문지기의 증거는 "없는 파일"이다

E3의 판정 근거는 종료 코드가 아니라 **파일 시스템의 부재**다. 변조된 바이너리에 대해:

```
gil-darwin-arm64: FAILED
shasum: WARNING: 1 computed checksum did NOT match
rc=1
$ ls gil
ls: gil: No such file or directory
```

`&&` 체인이 끊겨 `mv`·`chmod`가 실행되지 않았고, **`gil`이라는 실행 파일이 태어나지 못했다.** 이것이 이 설계의 핵심이다 — 검증되지 않은 바이너리는 *"실행하면 안 되는 파일"* 로 존재하는 게 아니라 **아예 존재하지 않는다.** 사람이 실수로, 에이전트가 부주의로 `./gil`을 쳐도 "No such file"이 나온다. 규율이 훈계가 아니라 파일 시스템의 사실이 됐다.

`exit 1`을 쓰지 않은 것도 의도다. 대화형 터미널에 복붙하면 `exit`이 **셸 세션을 죽인다** — 사용자를 쫓아내는 문지기는 우회당한다. `&&` 종속은 대화형 셸(세션 유지)과 CI(rc≠0로 중단) 양쪽에서 같은 안전을 준다.

### 2. 이슈가 못 본 곳 — 도구가 생성하는 코드

결(Gyeol)의 이슈 #8은 `README.ai.md`의 스니펫을 지적했다. 전수 grep(C010의 절차)이 다섯 번째 경로를 찾아냈다: **`gil pages`가 생성하는 GitHub Actions 워크플로.**

| 층 | 누가 실행하나 | 성격 |
|---|---|---|
| 문서 스니펫 ×4 | 사람·에이전트 | 따를 수도, 안 따를 수도 있는 **훈계** |
| **`gil pages` 산출 워크플로** | **기계 (사용자 CI, 매 push)** | **무조건 실행되는 코드** |

우리는 *"규율은 기계가 집행한다"* 를 말하면서, **우리 도구가 생성한 기계에게 검증 없는 실행을 시키고 있었다.** 문서만 고쳤다면 가장 확실히 실행되는 경로가 그대로 뚫려 있었을 것이다.

E4-c가 그 구멍이 막혔음을 보인다 — 변조된 자산에서 워크플로의 대조 단계가 rc=1로 실패한다. **불일치하면 사용자의 배포가 멈춘다.**

### 3. 하네스가 세 번 틀렸다 — 그리고 구현은 한 번도 틀리지 않았다

첫 실행은 4건 실패했다. 진단해 보니 **넷 다 하네스의 결함이었고, 구현의 결함은 0건이었다.**

| 실패 | 진짜 원인 | 성격 |
|---|---|---|
| E4-b `cannot execute binary file` | macOS 호스트에서 **리눅스 바이너리**를 실행하려 함 (러너는 ubuntu-latest) | 하네스 |
| E5 Go 산출물 없음 | `go build .`(패키지)를 썼는데 **이 저장소엔 go.mod이 없다** — CI는 `go build … main.go`로 파일을 직접 빌드한다 | 하네스 |
| E4-b `체인 루트가 없다` | 빈 디렉토리에서 `gil web` 실행 — 진짜 러너는 `actions/checkout` 직후라 체인이 있다 | 하네스 |
| E7 ×2 "재시도 안내 없음" | 문서엔 "re-run" / "다시 실행"이라 썼는데 패턴은 `retry|재시도`만 봤다 | 하네스(+문서 어휘) |

특히 주목할 것: **E4-b의 첫 실패 로그에도 `gil-linux-amd64: OK`가 찍혀 있었다.** 대조는 처음부터 통과하고 있었고, 실패한 것은 그 다음 줄이었다. 실패 메시지를 끝까지 읽지 않았다면 "체크섬 로직이 깨졌다"고 오진했을 것이다.

이것이 **다섯 번째·여섯 번째 "테스트를 먼저 의심하라"** 다 (C016·C021·C029·C035에 이어). 반복 패턴이 이제 뚜렷하다 — 실패한 단언의 범인은 대개 **하네스의 환경 가정**이다: git 기본 브랜치(C029), 작업 트리 기준선(C035), 그리고 오늘의 **빌드 호출 문법·호스트 아키텍처·러너의 작업 디렉토리**.

## 예상과 달랐던 것

1. **`go.mod`이 없다.** C023에서 "Go 소스를 패키지로 승격"했다고 기억하고 있었으나, 실제로는 go.mod 없이 `go build main.go`(단일 파일)로 빌드된다. 기억과 저장소가 어긋났고, **저장소가 옳았다.** C007의 교훈("전제는 가정하지 말고 명령 한 줄로 확인하라")이 또 유효했다. 이 사실은 하네스 주석에 못 박아 두었다.

2. **macOS에 `sha256sum`이 있었다.** 설계 시 "macOS엔 없다"고 가정하고 건너뛰기 분기를 넣었는데, 실제로는 존재해서(coreutils) 워크플로 run 블록을 로컬에서 실행할 수 있었다 — 덕분에 E4-b/c를 실측할 수 있었다. 가정이 틀린 덕에 검증이 강해졌다.

3. **`--ignore-missing`이 macOS `shasum`에서 작동했다.** 그럼에도 `grep | shasum -c -` 형태를 택했다 — 옵션 지원에 이식성을 걸지 않기 위해서다. 이 선택은 리눅스 `sha256sum`에서도 무수정으로 동작한다.

## 판정

> **채택 (supported)**

가설이 예측한 대로: 정상 바이너리는 통과하고(E2, 거짓 양성 0), 변조된 바이너리는 **실행 파일이 되지 못하며**(E3), 같은 규율이 도구 산출물에까지 미쳐(E4) 사용자의 CI가 불일치 시 멈춘다. 두 구현은 갈라지지 않았고(E5) 회귀도 없다(E6).

기각 조건 세 개 모두 미발동. 1-hypothesis에 명시한 **의도**("검증되지 않은 바이너리가 실행되는 경로를 없앤다")도 충족 — 대조 실패 후 실행이 계속되는 경로가 문서·도구 양쪽에 남아 있지 않다.

### 한계 (정직하게)

- **이것은 공급망 공격 방어가 아니라 전송 불일치 탐지다.** SHA256SUMS도 같은 CDN에서 온다. 바이너리와 SUMS가 **함께** stale하면 둘이 서로 일치하여 통과할 수 있다. 결의 실측에서는 바이너리만 stale하고 SUMS는 최신이어서 불일치가 드러났다. 근본 해결(서명, 릴리스 노트에 해시 고정)은 별도 사이클의 후보다.
- **`gil pages` 산출물 규율이 아직 계약(conformance) 밖이다.** 지금은 체크섬 없는 pages를 만든 구현도 26/26을 받는다. Weft가 `loom/C036`에서 같은 파일들을 만지고 있어 판정 항목 수 경합을 피하려 이월했다 (설계 §범위 밖 참조).
- 실 릴리스(v1.10.0) 이후 **새 바이너리에 대해 스니펫을 다시 실행**하는 끝단 실증이 남아 있다 — 배포 후 수행한다 (C024·C027·C028의 반복 교훈: 검증은 수신자 경로로 끝까지).
