# Chain: loom

## 이 체인이 정복하려는 문제

**사이클을 깃만큼 유용하게 다룰 수 있는 도구를 만든다.**

깃이 커밋이라는 단위로 코드의 역사를 다루듯, LLM은 "사이클"이라는
**행동 체인 기록**을 만들 수 있어야 하고, 인간과 LLM 모두 그 기록을
조회하고(log), 분기하고(branch), 병합하고(merge), 검증할(fsck) 수 있어야 한다.

이 도구는 우리만의 것이 아니다. 어떤 사용자든 자신의 LLM에게 아리아드네
방식을 입힐 수 있는 하네스의 핵심 부품이다 — 새로운 깃 같은 무언가.

## 성공의 정의

Ariadne 레포 구조를 가진 어떤 저장소에서든, LLM 또는 인간이 단일 도구로
사이클 체인을 생성·조회·검증할 수 있고, 그 출력만으로 "어떤 교훈에서
어떤 가설이 태어났는지"의 전체 계보를 추적할 수 있다.

## 깃과의 대응 (설계 나침반)

| 깃 | loom | 의미 |
|---|---|---|
| commit | cycle | 기록의 최소 단위 |
| commit graph (DAG) | chain | parent 참조로 이어진 계보 |
| `git log --graph` | 계보 조회 | 역사를 읽는 능력 — **첫 번째로 정복** |
| branch / merge | 분기 / 병합 | 같은 부모에서 갈라지고, 여러 부모로 합쳐짐 |
| `git fsck` | 체인 검증 | 고아 사이클, 끊어진 parent 참조 탐지 |

## 아키텍처 원칙: 깃 위에 올라탄다

깃을 다시 만들지 않는다. 깃은 "내용의 DAG"를, Ariadne는 그 위에 겹쳐지는
"추론의 DAG"를 다룬다. 저장·전송·충돌 해결은 깃이, 호스팅·공유·정적 배포는
깃헙이 이미 준 것을 쓴다. 우리가 만드는 것은 의미 계층뿐이다:
사이클 DAG 스펙 · `ari` CLI(깃 porcelain) · 웹 그래프 뷰어(같은 파서, 다른 렌더러).

## 로드맵 (나침반 — 예언이 아니며, 각 사이클의 가설은 직전 보고서의 교훈에서 나온다)

1. **C001** 계보 파서 + log — CLI와 웹 뷰어가 공유할 심장
2. **C002** open/close 쓰기 porcelain — 사이클 생성·닫기의 자동화
3. **C003** 깃 바인딩 — 사이클 닫기 = 커밋 + 태그(`cycle/<chain>/<id>`), 기록의 기계적 불변성
4. **C004** 정적 웹 뷰어 — GitHub Actions + Pages로 push마다 체인 그래프 자동 배포
5. **C005** 스펙 v0.1 — 디렉토리 구조·스키마·5스텝 규약의 표준 문서화, 배포의 방 첫 릴리스
