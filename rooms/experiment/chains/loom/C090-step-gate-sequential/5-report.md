# 5. 결과 보고 — step-by-step을 파일 존재 수준에서 강제 (발의: 박상현)

## 요약
gil의 step-by-step 강제가 "커밋 스코프"(C080)에만 있고 파일 존재엔 없던 것을 메웠다. (a) `open`이 1-hypothesis.md만 스캐폴딩(2~5는 전이 때 생성), (b) `step N`이 스텝 1..N-1의 실질 작성을 검증해 미완이면 무변화 거부, (c) 통과 시 N 파일 생성 + 다음 스텝 안내 출력, (d) 뷰어가 없는 스텝을 "아직 — 이전 스텝 완수 필요"로 표시. STEP-GATE conformance 신설, 회귀 0, 하위호환. **채택(supported).**

## 교훈
1. **강제는 여러 층이 맞물려야 완성된다.** 파일 존재(open) + 전이 검증(step) + 표시(뷰어) + 커밋 스코프(C080). C080만으론 "다음 스텝 재료가 미리 보임"을 못 막았다 — 파일이 이미 존재했으니까. 상현님 지적이 빠진 층(존재·표시)을 정확히 짚었다.
2. **dogfooding이 거짓 양성을 즉시 잡았다.** 새 강제가 자기 사이클(C090)에 적용되며 "스캐폴딩 문구 인용"을 미완으로 오판 — step 4에서 자기가 거부됐다. 판정을 "본문 실질 줄이 마크 하나뿐일 때만 미완"으로 정교화. 도구를 도구로 쓰다 도구의 결함이 드러나는 리듬(C084·C088 재현).
3. **강제(파일 존재)와 정직성(사전등록=kill조건)은 층위가 다르다.** 상현님과 "답을 먼저 고정 = 가드레일" 이해를 맞춘 뒤 설계 — 파일 존재는 순차 강제하되, 1·2스텝에 판정 기준을 미리 쓰는 정당한 사전등록은 계속 허용(M7).

## 다음 사이클을 위한 제안
- (A) **Go parity** — open/step 강제의 Go 이식(뷰어 3연작 + 이것까지 누적).
- (B) **커밋 검증** — 지금은 "실질 작성"을 검증하나 "이전 스텝이 **커밋됐는지**"는 안 본다(작성만 하고 미커밋도 전이 가능). 상현님 원문 "수행·커밋하지 않으면"의 커밋 절반 — 필요 시 git 상태 검증 추가.
- (C) fsck에 "열린 사이클의 스텝 파일 = step 값과 일치" 규칙(선택).
