# 4. 결과 분석

## 통계적 결과
Go 104/104, 참조 122/122 무회귀. 4항목(WEB-REFRESH-DEFAULT·OPEN-CREATE·STEP-GATE·WEB-MD-RENDER) 전부 PASS. 기각 조건(잔여·회귀·무의미한 분기) 미해당.

## 데이터 직접 관찰
- **참조가 정본, Go는 그것을 따른다.** 특히 md 렌더 JS는 두 구현이 같은 클라이언트 JS를 뱉으므로 참조의 _WEB_APP_JS를 거의 그대로 Go 문자열에 옮겼다 — 로직 재작성이 아니라 이식. "계약이 구현을 정의한다"의 실천.
- **타입 차이가 계약을 미묘하게 만든다.** C085의 "None(미지정) vs 0(끔)" 구별을 Python은 None으로, Go는 `*int` 포인터로 재현했다. Go zero value(0)가 둘을 못 가르므로 포인터가 필수 — 언어가 달라도 계약(관측 가능 동작)은 같아야 한다.
- **정적 conformance의 한계.** WEB-MD-RENDER는 문자열 존재 검사라 Go가 JS를 담기만 해도 PASS했다. 실제 렌더 동작·이미지 임베드까지 검증하려면 헤드리스가 필요(C088에서 참조는 헤드리스로 검증). Go의 이미지 임베드 미이식이 이 검사에 안 걸린 게 그 방증 — 후속에 남긴다.

## 예상과 달랐던 것
- 4개가 생각보다 독립적이어서 점진 커밋(101→103→104)이 깔끔했다. 각 이식 후 즉시 해당 conformance 항목만 PASS로 바뀌어, 진전이 기계적으로 가시화됐다.

## 판정
**채택(supported).** 이번 세션의 참조 구현 변경(C085·C088·C090)을 Go에 이식해 gil-gate 양쪽 완전 녹색. gil의 "두 구현이 같은 계약을 통과한다"가 회복됐다.
