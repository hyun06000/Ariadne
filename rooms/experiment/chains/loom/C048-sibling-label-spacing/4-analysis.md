# 4. 결과 분석

## 통계적 결과
- **H1 겹침 해소**: 분기 픽스처 형제 x 간격 26→**260**px ≥ 라벨 끝(x+246). 라벨이 옆 노드를 안 덮는다.
- **H2 선형 불변**: genesis(선형) web 바이트 동일 — col0만 써 `col*_COL_W=0`.
- **H3 두 구현 동일**: 실 저장소·픽스처 참조=Go 바이트 동일.
- **회귀 0**: 참조 72/72, Go 64/64.

기각 조건 네 개 모두 미충족 → 기각되지 않음.

## 데이터 직접 관찰
- **한 상수가 세 값을 동시에 정한다.** `_COL_W`는 ① 세로 간선 곡선의 좌우 폭 ② 활성 레인 간 거리 ③ (C047 이후) 같은 행 형제 간 거리 — 셋을 겸한다. 26px는 ①②(간선 미학)엔 좋았으나 C047이 ③을 새로 얹으며 라벨 폭(230)과 충돌했다. 260으로 올리면 ③이 풀리는 대신 ①②의 간선이 넓고 완만해진다 — 시각적으로 오히려 여유롭다(부작용이 악화 아님).
- **선형이 공짜로 불변인 이유**: 선형 체인은 항상 col0. `0*260=0*26=0`. 상수가 곱해지는 대상이 0이면 상수 값과 무관 — C047의 "선형은 depth=order라 불변"과 같은 구조의 두 번째 발현.

## 예상과 달랐던 것
- **C047과 C048은 한 쌍이었다.** C047(세로 압축)이 형제를 같은 행에 놓자마자 라벨 겹침이 필연이었는데, C047에서 못 봤다 — 우리 저장소 loom은 형제 라벨이 짧거나 분기가 적어 육안으로 안 드러났고, 상현님이 자기 저장소(분기 잦음)에서 밟았다. **배포자는 자기가 안 밟는 길의 지뢰를 못 본다**(C028·C042의 반복) — 세로 압축의 수혜자(분기 많은 사용자)가 곧 라벨 겹침의 첫 피해자다. 두 사이클이 같은 사용자 맥락에서 났다.

## 판정
**채택(supported).** 상수 하나(_COL_W 26→260)로 겹침 해소, 선형 불변, 두 구현 동일. C047의 세로 압축을 유지한 채 가로 가독을 회복했다.
