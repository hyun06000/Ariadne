# 1. 가설 수립

## 이전 사이클의 교훈

부모 = loom/C095-deploy-axis(1/5 홀드). C095를 여는 도중 실제 마찰이 이 사이클을 낳았다: 나(Clew)가 C095의 계보를 `--lineage loom/C086`으로 잇다가 **"같은 체인의 계보는 parent (R3)"** 로 거부당했고, 그 순간 **"같은 체인 내 다중 부모는 불가능하다"고 오해**했다. 상현님이 정정했다 — "리니지는 체인간 참조로 두되 여러 부모를 두는 기능을 추가해야 했어."

조사 결과 상현님의 진단은 절반만 맞았다(그리고 더 중요한 절반이 드러났다):

- **다중부모는 이미 gil에 완전 구현돼 있다.** `--parent`는 `action="append"`, 저장은 `parent: [A, B]`, 로더·그래프·토포소트·fsck·모든 렌더러가 이미 `parents` 리스트를 순회한다. 실제로 **C036이 `parent: [C020, C016]`으로 다중부모를 올바르게 기록**하고 있었다. 스크래치패드에서 `--parent A --parent B` 실증 완료.
- **그렇다면 나는 왜 불가능하다고 믿었나?** 상현님의 두 번째 지적 — "메뉴얼이 잘못됐을 수 있어" — 이 정곡이었다. 문서 감사 결과 **README.ai.md(LLM이 읽는 빠른 참조)** 가 단일 `--parent`만 보여주고 두 번째 조상을 `--lineage`로 유도한다. QUICKSTART·SPEC §3.2 O-table도 병합 케이스에 침묵하고, `gil open`의 부모 누락 에러 메시지(gil.py:694)는 "분기면 여러 번"만 말하고 "병합"을 빠뜨린다. **문서 갭이 나의(그리고 필드 모든 LLM의) 오해의 근원이다.**

## 문제 분할

발견된 세 갈래:
1. **문서 갭** — 다중부모 how-to가 오도/침묵. (이 사이클)
2. **과거 잃은 계보 4건** — C043·C057·C018·C041이 다중부모여야 하는데 단일부모로 봉인됨. (다음 사이클)
3. **correct의 한 줄 증거 제약** — `--evidence`가 단일이라 여러 줄에 흩어진 다중부모를 복원 못 함. (2번의 전제일 수 있음)

**이 사이클이 뗄 조각 = #1 문서 갭.** 근원을 먼저 막는다(계보를 복원해도 문서가 그대로면 다시 잃는다). 도구 행동은 이미 옳으므로 이건 **행동 변경 없는 문서·안내 정합** — 단, `gil open` 에러 메시지(gil.py:694)는 사용자 접점이라 그 한 줄은 코드다.

## 가설

> **가설**: 다중부모(같은 체인 병합)의 how-to를 (a) README.ai.md·QUICKSTART에 `--parent A --parent B` → `parent: [A, B]` 워크드 예제로 추가하고, (b) SPEC §3.2 open 의무 표에 병합 케이스를 명문화하고, (c) `gil open` 부모 누락 에러 메시지(gil.py:694)를 "분기면 여러 번"에서 "분기·병합이면 여러 번"으로 정합하며, (d) `--lineage` 플래그 help에 "체인 간 전용"을 명시하면 — 도구 행동을 바꾸지 않고도 LLM·사람 독자가 같은 체인 다중부모를 lineage에 잘못 넣지 않고 올바르게 기록하게 된다.

## 기각 조건

1. **행동 회귀**: 문서·메시지 변경이 conformance 회귀(참조 123 / Go 105)를 하나라도 깬다. 특히 gil.py:694 메시지 변경이 HELP-* 또는 open 관련 판정을 깬다.
2. **메시지가 여전히 오도**: 변경 후에도 "두 번째 같은 체인 조상은 어디에?"라는 독자 질문에 문서가 답하지 못한다(예제 없이 서술만).
3. **범위 침범**: 문서 정합이 아니라 도구 행동을 바꾼다(이 사이클은 문서 사이클이다 — 행동 변경은 별도 사이클/이슈).
4. **버전 표면 불일치**: 문서 릴리스인데 `_GIL_VERSION`·CHANGELOG·RELEASE 정합이 깨진다(C003 교훈: README.ai.md는 패키지 밖이라 별도 커밋).
