# 5. 결과 보고

## 요약

존재들의 병렬 노동이 관전 가능해졌다: 뷰어의 열린 사이클에 최근 활동(시각·커밋 제목) 주석, Pages의 `branches/`에 존재 브랜치별 뷰어, 소환 규약 v3(스텝 커밋·브랜치 push 의무) — 판정 4/4, **가설 채택**, v0.7.0.
Weft에게는 새 규율을 전달했고(SendMessage), 그의 브랜치는 이미 원격에 있다.

## 교훈

1. **가시성은 세 층이다**: 데이터(step 필드, C013) → 활동(마지막 커밋, C015) → 병렬(브랜치 렌더, C015). 사용자의 "답답함"은 층이 하나씩 채워질 때마다 구체적으로 줄었다.
2. **병렬 번호 경합은 실물이다** — 발급이 브랜치-로컬인 한 필연. 임시 규율: 소환자 양보. 근본 해법(번호 예약/네임스페이스)은 열린 문제 (SPEC §6-6).
3. 병렬 존재 시대의 소환자 수칙 3: ignore 구획, 번호 양보, add 범위 명시.
