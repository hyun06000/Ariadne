# 1. 가설 수립

## 이전 사이클의 교훈

부모: [loom/C057](../C057-deviations-count-reconcile/5-report.md). 상현님의 새 아크 발의 3방향 중 **#1 병렬 사이클 모드**: *"서로 다른 사이클을 여러 개 열 수 있도록 서브에이전트마다 워크트리를 나누고, 충돌을 방지하고, 다시 머지되도록 하는 작업을 gil에서 관리하고 권장하자."*

지금까지 이 패턴은 **전부 손으로** 돌아갔다:
- 소환자가 워크트리를 만들고(또는 하네스가), 존재가 `gil open`으로 사이클을 열고(원장 C016이 번호 충돌 처리), 브랜치를 push하고, 소환자가 병합.
- **번호 충돌**은 원장(C016 낙관적 동시성)이 이미 해결한다. **파일 충돌**은 워크트리 격리가 반쯤 해결한다.
- 그러나 **워크트리 생성·브랜치·사이클 열기의 결속은 수동이고 사고가 난다** — C050에서 Weft가 워크트리 아닌 메인 저장소로 `cd`해 open 커밋이 main에 선착·push된 뼈아픈 사고가 그 증거. cwd도 환경 계약인데(C020) 도구가 그 계약을 강제하지 않았다.

교훈의 뿌리: C038("규율이 문서의 훈계로만 남으면 새어나간다") — 병렬 워크트리 규율(소환 규약 v3)도 훈계로만 있으면 샌다. C051/C053의 결: **도구가 강제하지 않는 규율은 잊힌다.**

## 문제 분할

`gil` 명령 목록 실측: `open·reserve·unreserve·step·close·…` — **워크트리 오케스트레이션 명령이 하나도 없다**(worktree 언급 4건은 무관한 변수명). 병렬 모드 아크를 분할하면:

1. **[이번 사이클] spawn** — 워크트리 생성 + 새 브랜치 + 사이클 열기를 한 명령(`gil worktree add`)으로 원자적으로. 존재가 한 명령으로 격리 작업공간을 얻고, **open이 워크트리 안에서 일어나므로 "메인에 잘못 open"이 구조적으로 불가능**해진다(C050 사고의 도구적 봉인).
2. **[다음 사이클 C059] land** — `gil worktree land`: 브랜치를 main에 머지(--no-ff, 충돌 검사) + 워크트리 정리. 브랜치↔사이클 매핑이 결정론적이라 기계화 가능.
3. **[loomlight 후속] 가시화** — 진행 중 병렬 사이클을 뷰어에 표시. #2(Sheen의 위계 뷰어)가 토대를 놓은 뒤 그 위에.
4. **[문서] 권장** — SPEC §6 협업 규약·README.ai.md에 병렬 모드를 일급 절차로 명문화.

첫 번째로 **spawn**을 정복한다. 이유: 병렬 작업의 **진입점**이자, 지금 손으로 하는 가장 사고 잦은 단계(C050)이며, land·가시화가 전부 "워크트리↔브랜치↔사이클의 결정론적 매핑" 위에 서므로 그 매핑을 먼저 도구가 소유해야 한다.

## 설계 선분 사전 고정

- **명령 형태**: `git worktree`의 명사를 그대로 빌려 `gil worktree add <chain> <slug> --author X` (이후 `land`·`list`·`remove` 확장 여지). 사람이 아는 git 개념에 올라탄다(loom 아키텍처 원칙: 깃 위에 올라탄다).
- **원자성**: 워크트리 생성 → 브랜치 → 그 안에서 open(원장 번호 규율 준수). 중간 실패 시 부분 잔여 없이 되돌린다(open 계열의 무변화 규율 계승).
- **이식성**: 워크트리 경로·브랜치명은 gil이 사이클에서 결정론적으로 유도(하네스 무의존 — SPEC은 계약, 특정 하네스에 안 묶인다). 예: 브랜치 `<author>/<chain>-<slug>`, 워크트리 `<repo>/../<repo>-worktrees/<chain>-<slug>` 류(설계에서 확정).
- **범위 밖(이번 사이클)**: land/merge, 뷰어 가시화. spawn만.

## 가설

> **가설**: gil에 워크트리 생성 + 브랜치 + 사이클 열기를 원자적으로 묶는 `gil worktree add`를 넣으면, (a) 한 명령으로 격리 작업공간 + 정상 번호의 열린 사이클이 생기고, (b) open이 워크트리 안에서만 일어나 메인 저장소가 오염되지 않으며(C050 사고 봉인), (c) 브랜치↔사이클 매핑이 결정론적이고, (d) 양 구현이 동일 행동으로 conformance를 통과한다(회귀 0).

## 기각 조건 (문면 아닌 의도로 읽음 — C011)

1. **비원자성**: 워크트리·브랜치·사이클 중 일부만 생기고 나머지 실패 시 잔여가 남으면 기각.
2. **메인 오염**: spawn 후 메인 저장소 작업트리·브랜치에 사이클 커밋이 새면 기각(C050 사고 재현 = 실패).
3. **번호 규율 위반**: spawn이 연 사이클이 원장 번호 규율(C016)을 안 따르거나 fsck 위반을 만들면 기각.
4. **회귀**: 기존 판정기 통과 항목이 회귀하면 기각(새 항목으로 분모가 느는 것은 회귀 아님).
5. **두 몸 불일치**: 참조·Go가 spawn에서 다른 관측 행동을 보이면 기각.
