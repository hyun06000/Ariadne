# 5. 결과 보고

## 요약

상현님의 아크 발의 #1(병렬 사이클 모드)의 첫 조각. gil엔 워크트리 오케스트레이션 명령이 없어 병렬 존재 패턴이 전부 수동이었고 C050의 "메인에 잘못 open" 사고가 났다. **`gil worktree add`**를 양 구현에 신설 — 워크트리 생성+브랜치+사이클 열기를 원자적으로 묶고, open을 워크트리 안에서 self-invoke해 커밋이 그 브랜치에만 가게 했다. 판정기 `WORKTREE-SPAWN` 신설(77→78), 양 구현 78/78, 변이 격추, fsck 회귀 0. 판정 **채택(supported)**.

## 교훈

1. **규율을 도구로 승격하면 사고가 구조적으로 불가능해진다.** C050에서 Weft가 워크트리 아닌 메인으로 cd해 open이 main에 샌 사고는 "워크트리에서 일하라"는 훈계(소환 규약 v3)로만 막혔다 — 그리고 잊혀 사고가 났다(C038 재확인). `gil worktree add`는 open을 워크트리 안에서만 실행해 **그 사고를 물리적으로 불가능하게** 만든다. 훈계 → 계약.

2. **얇은 오케스트레이터 + self-invoke가 발산을 막는다.** spawn이 open을 복제했다면 두 구현이 open 계약을 각자 다시 구현해야 했다. "git worktree add + 자기 재호출"로 얇게 유지하니 open의 검증된 계약이 한 곳에만 살고, 양 구현 spawn이 자연히 대칭이 됐다. 도구가 자기를 재호출하되 자기 이름은 하드코딩 안 한다(§7).

3. **변이가 살아남는 것은 실패가 아니라 진단이다(C011).** cwd를 바꾼 첫 변이가 살아남아, 격리의 진짜 축이 cwd가 아니라 `--root`(open의 git-toplevel 해석을 워크트리로 고정)임을 가르쳐 줬다. 봉인의 축을 데이터로 확인하고서야 올바른 변이로 격추했다.

4. **재현성은 존재 조건 — 스텁으로는 닫지 않는다.** 설계 문서가 작업 중 스텁으로 되돌아갔고(원인 미상), 원 설계를 복원해 닫았다. (아래 미해결 참조.)
