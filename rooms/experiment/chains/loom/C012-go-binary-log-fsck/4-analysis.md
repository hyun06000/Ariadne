## 데이터 직접 관찰

- **구현 교체 가능성이 실물이 됐다**: 같은 저장소에 대해 두 언어의 gil이 같은 문장으로 답한다. 스위트는 무수정 — 판정 주입은 `--gil "<명령>"` 그대로였다. §7의 약속("바이너리는 `gil <명령>` — 계약은 동일")의 첫 실증.
- Go 구현은 461줄, 외부 모듈 0 — 참조 구현의 파싱 규칙(주석·따옴표·리스트·후행 주석)과 토폴로지 동순위 규칙(id 오름차순)을 그대로 옮기는 것으로 충분했다. **스펙과 참조 구현이 이식의 전 지식이었다** — 실험의 방 발굴 불필요.

## 예상과 달랐던 것 — 도그푸딩의 이중 수확

1. **스위트 v1은 부분 구현 앞에서 죽었다** (발견 1). OPEN 실패 산출물을 무가드로 읽고, log 검사가 impl의 open에 의존했다 — "부분집합 판정"이라는 이 사이클의 전제 자체가 스위트의 설계 밖이었다. v2에서 **판정 항목 간 독립**(상태는 스위트가 직접 구축)으로 수리. C011의 교훈("판정기도 검증 대상이다")이 하루 만에 재확인됐다.
2. **거부형 검사의 공허 통과**: 미구현 명령의 exit≠0는 REJECT류 4항목을 자동 통과시킨다. 거부형은 수락형과 짝일 때만 의미가 있다 — 스위트 문서에 명시.
3. **참조 구현의 SameFileError** (발견 2): 패키지의 gil.py 자신으로 릴리스하면 자기 파일 복사에서 죽는다. 지금까지 사이클 사본으로만 릴리스해 숨어 있었다. 수정 → 도구 변경 → 규칙대로 v0.5.0.
4. **승격 규칙의 공백 2건**: ① conformance.py 변경이 "문서"로 분류됨(규칙이 gil.py만 도구로 간주), ② 변경 감지 기준이 "실행 파일 vs 패키지 파일"이라 패키지 도구 직접 실행 시 변경 불가시 — 기준은 마지막 릴리스 태그여야 한다. RELEASE.md에 기록, 다음 스펙 개정 재료.
