# 2. 실험 설계

**정답을 도구보다 먼저 고정한다** (C001의 절차). 구현 전에 *무엇이 참이어야 하는가*를 못 박는다.

## 설계 원칙

### 원칙 1 — 봉인이 없으면 정정도 없다

`gil correct`는 **봉인된 사이클**(닫힘 + 태그 `cycle/<chain>/<id>` 존재)에만 적용된다.

- **열린 사이클**: 봉인되지 않았다. `cycle.yaml`을 직접 고쳐도 위조가 아니다 — 정정할 것이 없다.
- **태그 없이 닫힌 사이클**: 봉인이 없으니 `verify`가 보호하지도 않는다. **도구가 보호하지 않는 것을 도구가 정정할 이유도 없다.**

> **도구는 자기가 만든 문제만 푼다.** 회색지대는 *봉인 때문에* 생겼다. 봉인 없는 곳의 정정은 정정이 아니라 편의다.

### 원칙 2 — 봉인은 잠금이자 공증(公證)이다

증거 문서는 **작업 트리가 아니라 태그에서 읽는다** (`git show cycle/<chain>/<id>:<path>`).

작업 트리에서 읽으면 **아무 문장이나 새로 써 넣고 그것을 "증거"라 부를 수 있다.** 태그에서 읽으면 그 문장이 **닫힘 시점에 이미 거기 있었음이 증명된다.**

> 정정을 막던 그 봉인이 정정을 **허가하는 공증인**이 된다.

### 원칙 3 — 정정은 지우개가 아니라 각주다

거짓값은 **삭제되지 않는다.** 세 곳에 살아남는다:

| 어디 | 무엇 |
|---|---|
| `corrections.yaml` | `from: null` — 거짓값 그 자체 |
| 깃 히스토리 | `[correct]` 커밋 **이전**의 모든 커밋 |
| 태그 메시지 | 이전 커밋 해시 + 이동 사유 (§4 태그 이동 규약) |

C029(*되감기는 삭제가 아니라 분기다*)와 C035(*불변의 역사 + 후대의 주석*)가 같은 자리에서 만난다.

### 원칙 4 — 증거는 인용이 아니라 검사다

`--evidence`를 **문자열로 받아 적기만 하면** L2는 이름만 남는다 (*선언은 검증이 아니다* — C039). 도구는 인용된 문서를 **열어서** 새 값이 실제로 거기 있는지 **판정한다.** 없으면 거부.

> C040이 남긴 문장의 직접 적용 — **관측은 판정이 아니다.** 증거를 *기록*만 하는 것은 관측이고, **열어서 대조하는 것**이 판정이다.

## 스키마 v0.5

`cycle.yaml`에 스칼라 마커 하나를 더한다 — `deviations`(C030)의 형태를 그대로 계승한다.

```yaml
corrections: 0   # 출처 필드가 정정된 횟수. N>0이면 corrections.yaml 필수 (v0.5)
```

**왜 상세를 별도 파일로 빼는가**: `cycle.yaml`은 평탄한 `key: value` 문서다(§3.1). 중첩 레코드를 넣으면 두 구현의 파서 계약이 깨진다. C030이 `deviations.yaml`로 이미 푼 문제다.

### `corrections.yaml` (사이클 디렉토리 안)

```yaml
# 출처 필드 정정 기록 (스키마 v0.5) — 거짓은 지워지지 않는다. 덧쓰일 뿐이다.

- field: parent
  from: null
  to: C028-pages-command
  evidence: 1-hypothesis.md:5
  evidence_source: cycle/loom/C029-time-machine
  author: clew
  date: 2026-07-14
  reason: open 시 --parent 누락 — 도구가 저자의 침묵을 대필했다 (loom/C040)
```

**형식 제약**: 레코드는 `- field: <값>`으로 시작하고, 이어지는 키는 2칸 들여쓰기 `key: value` **한 줄씩**. 중첩·블록 스칼라 **금지**.

이유: **fsck가 일반 YAML 파서 없이 줄 단위로 판정할 수 있어야 한다.** `deviations.yaml`은 *사람이 읽는 문서*라 블록 스칼라를 허용했지만, `corrections.yaml`은 **도구가 판정하는 기록**이다 (R13).

## CLI 계약

```
gil correct <chain>/<id> --field <필드> --to <값> --evidence <파일>[:<줄>] --author <이름> [--reason "..."] [--push]
```

- 전역 표기 `<chain>/<id>` — `supersede`(C035)의 선례.
- `--field`/`--to`는 **반복 가능**(한 정정 커밋에 여러 필드). 순서대로 짝지어 소비한다.
- `lineage`의 값은 전역 표기(`<chain>/<id>`); 같은 필드에 `--to`를 여러 번 주면 리스트로 누적한다.
- **커밋은 선택이 아니다.** 정정의 본질이 `[correct]` 커밋 + 태그 이동이므로 `--no-commit`은 없다. `--push`만 선택.

### 정정의 의무 C1~C8 (§3.2의 `open` 의무표 O1~O5와 같은 형식)

| # | 조건 | 행동 |
|---|---|---|
| **C1** | 대상이 **봉인되지 않음** (열림 또는 태그 없음) | **거부** — "봉인되지 않은 사이클은 정정 대상이 아니다" |
| **C2** | `--author` 없음 | **거부** — 도구는 **정정의 출처도** 지어내지 않는다 (P1의 재귀) |
| **C3** | `--field`가 출처 필드(`author`·`parent`·`lineage`)가 아님 | **거부** — `verdict`·`status`·`title`·`step`·5스텝 문서는 **저자의 주장**이다 (**L1**) |
| **C4** | `--evidence` 없음 | **거부** (**L2**) |
| **C5** | 인용 문서(**태그에서 읽음**)가 새 값을 **증언하지 않음** | **거부** — 증거가 없는 값은 수리가 아니라 **수정**이다 (**L2**) |
| **C6** | 대상 사이클이 **이미 변조됨** (태그↔트리 불일치) | **거부** — 정정은 **무결한 사이클에만**. 변조를 정정으로 세탁하는 뒷문을 막는다 |
| **C7** | 새 값이 **스키마 위반** (해소 안 되는 parent, 자기 참조, 순환) | **거부** — 정정 후 fsck 위반이 될 값은 쓰지 않는다 |
| **C8** | 새 값이 **현재 값과 동일** | **거부** — 정정할 것이 없다 (빈 커밋·헛된 태그 이동 방지) |

거부의 계약면: **종료 코드 ≠ 0 + 저장소 무변화** (§3.2와 동일 — 스냅샷 해시로 검증).

### 성공 경로 (원자적)

1. C1~C8 **전부 선검사** — 하나라도 실패면 **아무것도 쓰지 않는다**
2. `cycle.yaml` 갱신: 대상 필드 = 새 값, `corrections` = 기존 + 이번 건수
3. `corrections.yaml`에 레코드 **덧붙임** (기존 레코드 뒤에 — 과거 정정도 지워지지 않는다)
4. `fsck` 재실행 — 위반이면 **롤백**
5. `[correct]` 커밋 — **그 두 파일만** (`cycle.yaml`, `corrections.yaml`)
6. **태그 이동** `git tag -f cycle/<chain>/<id>` → 태그 메시지에 **이전 커밋 해시 + 정정 사유** (§4)
7. `--push`면 커밋 + 태그 전파

> **5번이 결정적이다.** 커밋에 그 두 파일만 담으므로, 작업 트리에 다른 변조가 있어도 **태그의 트리 안으로 들어갈 수 없다.** C6(선검사)과 함께 **이중 방어** — 정정은 변조를 태그 안으로 밀반입하지 못한다.

## fsck 규칙 R13 (v0.5)

| 규칙 | 내용 |
|---|---|
| **R13** | `corrections`가 있으면 정수. `N > 0`이면 `corrections.yaml`이 존재하고 **레코드 수 = N**이며, 각 레코드는 `field`·`from`·`to`·`evidence`·`author`·`date`를 갖고 `field ∈ {author, parent, lineage}` — **위반** |

**왜 경고가 아니라 위반인가**: R10(`verdict` 없음 → 경고)은 **기존 사슬의 유예**였다. `corrections`는 v0.5에서 태어나므로 **유예할 과거가 없다.** 그리고 이 규칙이 지키는 것은 **L1과 L3 그 자체**다 — 손으로 쓴 정정 기록이 `from`을 빼먹으면 정정은 **각주가 아니라 지우개**가 된다.

## §4 확장 — 닫힌 사이클에 대한 **두 번째** 쓰기 등급

| 등급 | 마커 | 무엇을 바꾸나 | 의미 보존? |
|---|---|---|---|
| **이주** | `[migrate]` | 스키마 이주, `superseded_by` 각인 | **예** — 의미 무손실 |
| **정정** | `[correct]` | **출처 필드**를 문서가 증언하는 값으로 | **아니오** — 색인이 바뀐다. 다만 바뀌는 것은 **저자가 주장한 적 없는 값**이다 |

**왜 `[migrate]`를 재사용하지 않는가**: 이주의 정의가 *의미 무손실*이다. 둘을 같은 마커에 넣으면 **이 사이클이 그은 선(저자의 주장 ↔ 도구의 대필)이 커밋 로그에서 지워진다.**

부수 효과 하나가 값지다: `git log --grep="\[correct\]"`가 **저장소의 모든 정정 이력**이 된다. 원장 밖의 원장.

## 후대의 주석 (log·web·help)

| 표면 | 표시 |
|---|---|
| `gil log` | 정정된 사이클에 **`✎ corrected(N)`** — `↣ superseded`의 형태 (C035) |
| `gil web` JSON 훅 `gil-data` | 노드에 `"corrections": N` — **기계 검증의 훅** |
| `gil web` 표 | `정정` 열 |
| `gil help` | 명령 목록에 `correct` — **없으면 HELP-COMPLETE가 잡는다** (§7.2) |

## 실데이터 적용 — `loom/C029-time-machine`

도구가 서면 **우리 저장소의 거짓을 실제로 수술한다.**

```
gil correct loom/C029-time-machine \
  --field parent --to C028-pages-command \
  --evidence 1-hypothesis.md:5 \
  --author clew \
  --reason "open 시 --parent 누락 — 도구가 저자의 침묵을 대필했다 (loom/C040)"
```

기대: `fsck`의 `경고 [다중루트] loom` **소멸**, `verify` **OK 유지**, `corrections.yaml`에 `from: null` **영구 보존**.

> 이 사이클의 마지막 증거는 **저장소 자신**이다. C040이 진단서를 썼고, C041이 수술한다.

## 범위 밖 (의도적)

- **문서가 침묵하는 경우의 정정** — 증거가 없으면 거부다(C5). 사용자는 회색지대가 아니라 **정직한 막다른 골목**을 받는다. (*"그럼 어쩌나"* 의 답: **원장은 고칠 수 없어도 역사는 덧붙일 수 있다** — 새 사이클을 열어 그 사실을 기록하라.)
- **`verdict`·5스텝 문서의 정정** — 영원히 범위 밖. 그것이 L1이다.
