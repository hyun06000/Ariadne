# 5. 결과 보고

## 요약

도구를 `gil`(길 · GIt for Language model, 작명: 박상현)로 개명하고 스펙을 **구현 독립 계약**(§7)으로 승격했다 — 판정 5/5, **가설 채택**. v0.3.0(개명, 깃이 R098 rename으로 각인) + v0.3.1(잔존 언급 정정, 패치 경로).
스펙이 계약이고 `gil.py`는 참조 구현이다 — 장래의 단일 바이너리 배포(상현님 방향)에서 구현은 교체되고 계약·기록·체인은 살아남는다.

## 교훈

1. **이름은 계약 위에 세워야 갈아탈 수 있다.** 파일명 비의존화 + CLI 계약(명령·인자·종료 코드) 명문화로, gil은 이제 특정 파이썬 파일이 아니라 스펙을 이행하는 무엇이든 될 수 있다.
2. **개명 릴리스의 사전 절차에 전수 grep**: 치환 스크립트는 치환한 것만 안다 — 릴리스된 문서의 정독에서 잔존 3곳 발견, v0.3.1로 정정. 문서 제목의 고정 버전 표기 같은 이중 진실도 함께 청산.
3. 우리 규칙이 우리를 무는 것의 반복 확인: 개명은 도구 변경 → 마이너 강제(v0.3.0), 정정은 문서-only → 패치(v0.3.1). 승격 규칙의 양방향이 실전 연속 사용됐다.

## 다음 사이클을 위한 제안

- **(A) 바이너리 참조 구현의 첫걸음** (상현님 방향의 착수): 계약 준수 테스트 스위트를 구현 독립적으로 분리(어떤 gil 구현이든 같은 테스트로 검증) — 바이너리 이식의 전제 조건이자, "스펙이 계약"의 기계적 실현.
- **(B) 존재의 동시성 / 병렬 감사** (tapestry 이월): 병렬 소환의 명부 경합.
- **(C) 재현 문서 규약의 스펙 반영** (tapestry/C001 교훈 3): 불변 기준/가변 확인 분리 — 문서 릴리스 경로.
