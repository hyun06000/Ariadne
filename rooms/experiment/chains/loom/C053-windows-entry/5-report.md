# 5. 결과 보고

## 요약

C052(git 없이 우아하게)가 남긴 "A만으로는 Windows 친구가 end-to-end로 못 간다"를 스코프 B·C·D로 매듭졌다. **B**: 릴리스 워크플로에 `windows/amd64` 추가(`.exe`, SHA256SUMS 포함) — darwin 크로스컴파일이 유효 PE32+ 산출. **C**: PowerShell 설치 스니펫 + 체크섬 게이트(검증돼야만 `gil.exe`가 태어난다, C037 정신) — 알고리즘 양경로 검증. **D**: README·README.ko·README.ai에 Windows 블록·git-optional 안내, README.ai에 OS·git 인식 분기. 회귀 0(74/74×2). **채택(supported·이탈 1)** — Windows 런타임은 사전등록된 수신자 검증 경계.

## 교훈

1. **검증 도구가 없으면, 검증의 최소 핵심과 대리 불가한 부분을 갈라라.** pwsh를 못 깔아 게이트를 PowerShell로 실행 못 했지만, 게이트의 **안전 속성**(해시 일치 → 승격 / 불일치 → 미생성)은 이식 가능한 셸로 동치 검증하고, **cmdlet 런타임 + .exe 실행**은 진짜 수신 환경(Windows)으로 이관했다. pwsh-on-Mac 검증은 어차피 부분 대리였으니, 실패가 오히려 더 정직한 분업을 낳았다. **못 하는 검증을 흉내내지 말고, 할 수 있는 핵심을 확실히 하고 나머지는 명시 이관하라.**
2. **가장 먼 수신자를 향한 배포는 여러 사이클에 걸친다.** C052(git부재)→C053(Windows)은 "비개발자를 있는 자리에서 맞이하기"라는 한 아크의 두 매듭. A만으론 부족했고 B·C·D가 필요했다. **한 사이클이 스스로 남긴 정직한 한계가 다음 사이클의 정확한 스코프가 된다**(C009 "미완을 정직히 남기면 다음 가설의 재료"의 배포판).
3. **온보딩 문서의 숫자는 화석이 된다.** 세 대문 모두 `29/29`(v1.4)에 멈춰 에이전트에게 낡은 자격 기준을 먹이고 있었다 — C039의 정확한 재발("낡을 수 있는 목록은 고치지 말고 없애라"). 갱신 대신 "전 항목 통과(스위트가 개수 출력)"로 위임했다.
4. **상현님 질문이 인접 진실을 감사하게 했다 — 그리고 안심의 근거는 코드였다.** "GitHub 없으면 push 안 하지?"에 답하려 gil의 네트워크 호출을 전수 감사: **자체 호출 0, 텔레메트리 0, 외부는 `--push` 시 git push/fetch뿐.** 비개발자 친구의 보안 걱정에 "네 데이터는 네가 `--push`하기 전까진 기계를 안 떠난다"고 **코드로** 답할 수 있다. 신뢰는 주장이 아니라 감사에서 온다.

## 다음 사이클을 위한 제안

- **(A) [상현님 발의] GitHub-optional 우아화 + 보안 문서화** — `--push`인데 원격이 없을 때 gil이 git 날것 힌트를 뱉는다(비개발자에 혼란). C052의 git-부재 우아화처럼 "원격이 없어 push를 건너뛴다"는 안내로 다듬고, **보안 자세**(네트워크 호출 0·텔레메트리 없음·push는 opt-in·자기 원격만)를 README에 명문화. 같은 아크(비개발자 안심)의 다음 조각.
