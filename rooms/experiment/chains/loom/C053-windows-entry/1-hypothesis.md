# 1. 가설 수립

## 이전 사이클의 교훈

부모: **loom/C052** (git 없이도 우아하게, A). 그 사이클이 git 크래시를 고쳤으나 **정직한 한계**를 남겼다 — A만으로는 Windows 친구가 end-to-end로 가지 못한다. 실측:
- **설치 자체가 막힌다**: 사용자 설치 스니펫이 `curl│sha256sum│chmod` **Unix 전용** (README·QUICKSTART).
- **받을 바이너리가 없다**: 릴리스 워크플로가 `darwin/{arm64,amd64}`·`linux/{arm64,amd64}` **4타깃만** 굽는다 — Windows `.exe` 없음.

C052가 확인한 큰 교훈: **가장 먼 수신자가 가장 깊은 간극을 드러낸다.** 비개발자는 Windows일 확률이 높다(상현님 통찰). 이 사이클은 그 아크의 B·C·D를 하나로 — Windows 사용자가 설치하고 실행할 수 있게.

## 문제 분할

- **B 바이너리**: 릴리스 워크플로에 `windows/amd64` 추가 → `gil-windows-amd64.exe`. Go 크로스컴파일로 산출(실측: PE32+ 생성 확인). SHA256SUMS에 포함.
- **C 설치 경로**: PowerShell 스니펫(`Invoke-WebRequest`+`Get-FileHash`) + **체크섬 게이트**(C037의 문지기 정신 — 검증 안 되면 gil.exe가 태어나지 못한다).
- **D 온보딩**: README.ai.md가 LLM에게 OS·git 가용성을 확인하고 알맞은 설치 경로를 고르게. gateway/C001("에이전트에게 이 글을 읽히세요")의 환경-인식판.

셋은 하나의 사용자 스토리다 — 바이너리 없는 설치도, 설치 없는 바이너리도, 안내 없는 둘도 무용하다. 그래서 한 사이클.

## 검증의 정직한 경계 (사전 등록)

나는 darwin에 있고 **Windows 런타임을 실행 검증할 수 없다.** 내가 검증 가능한 것과 수신자(상현님/친구)에게 남는 것을 미리 가른다:
- **내가 검증**: (B) 크로스컴파일이 유효한 PE 바이너리를 만드는가 + SHA256SUMS 포함, 워크플로 build 블록이 5타깃을 굽는가(loom/C024식 추출 실행). (C) PowerShell 게이트 **로직**이 실제로 도는가 — `pwsh`(PowerShell Core)를 설치해 다운로드·해시·비교·게이트를 실행(pwsh의 Get-FileHash/Invoke-WebRequest/Select-String은 Windows와 동일 cmdlet). (D) 링크 해소·구조.
- **수신자에게 남음**: 실제 Windows PC에서 `.exe` 실행 + PowerShell 스니펫 end-to-end. (C024 선례: "상현님이 다른 맥에서 테스트 — 그 결과가 다음 사이클의 재료.")

## 가설

> **가설**: 릴리스가 `gil-windows-amd64.exe`를 굽고(B), 그 바이너리를 체크섬 게이트로 안전하게 받는 PowerShell 스니펫(C)과 OS·git을 인식해 경로를 고르는 온보딩(D)을 더하면 — Windows 사용자의 설치·실행 경로가 **내부적으로 일관되게**(유효 PE + 체크섬 일치 + 게이트 동작 + 링크 해소) 성립한다. Windows 실제 실행은 수신자 검증으로 남긴다.

## 기각 조건

- 크로스컴파일이 유효한 Windows PE를 못 만들거나 SHA256SUMS에서 빠지면 → B 기각.
- PowerShell 게이트 로직이 (pwsh 실측에서) 체크섬 불일치를 **통과시키거나** 일치를 거부하면 → C 기각 (C037의 "검증 안 된 바이너리는 태어나지 못한다"가 깨짐).
- 기존 4타깃/Unix 설치 경로에 회귀가 생기면 → 기각 (기존 사용자 불변식).
- 판정은 총계가 아니라 **내부 일관성 + 게이트 정확성 + 회귀 0**, Windows 런타임은 수신자-pending으로 명시(이탈 아님 — 사전 등록된 경계).
