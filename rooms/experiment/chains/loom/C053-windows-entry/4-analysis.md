## 데이터 직접 관찰

- **크로스컴파일 산출물이 진짜 Windows PE다.** `file gil-windows-amd64.exe` → `PE32+ executable (console) x86-64, for MS Windows`. darwin에서 만든 바이너리가 Windows 실행 형식임을 파일 시그니처로 확인 — 워크플로가 배포할 것과 동일 산출.
- **게이트의 안전 속성을 양경로로 봤다.** 정상: `want == got`(b7faf449…) → 승격. 위조: 1바이트 append → 해시 완전 변동 → 불일치 → `gil.exe` 미생성. **"검증된 것만 태어난다"가 알고리즘 수준에서 참**임을 확인. PowerShell은 이 알고리즘을 `if ($got -ne $want) { Remove-Item; throw }`로 인코딩한다.
- **상현님 질문이 드러낸 인접 진실 (감사로 확인)**: gil은 자체 네트워크 호출이 **0**이다(urllib/requests/socket/http 0건). 외부 통신은 `git push`/`fetch origin`뿐, `--push`일 때만. **GitHub 없으면 아무것도 안 나간다** — 로컬 커밋만. 이 사이클(Windows 진입)과 직교하지만 같은 아크(비개발자 안심)의 재료.

## 예상과 달랐던 것

- **pwsh를 못 깔았다 — 그리고 그게 검증을 더 정직하게 만들었다.** pwsh-on-Mac 검증은 어차피 부분 대리였다(Windows PowerShell ≠ pwsh-on-Mac, .exe 실행은 여전히 불가). 설치 실패가 나를 "게이트 알고리즘은 내가 확실히, cmdlet 런타임은 진짜 수신 환경에서"라는 더 깨끗한 분업으로 밀었다. **검증 도구가 없을 때, 검증의 최소 핵심(안전 속성)과 대리 불가한 부분(플랫폼 런타임)을 가르는 것이 정직한 대안이다.**
- **"29/29" 화석 3구.** D 작업 중 세 대문 모두 `29/29`(v1.4 시절)에 멈춰 있었다 — 온보딩 문서가 에이전트에게 낡은 숫자를 먹이고 있었다(C039의 정확한 재발). 숫자를 갱신하지 않고 **"전 항목 통과(스위트가 개수 출력)"로 위임** — 낡지 않는 표현으로.

## 판정

**가설 채택 (supported·이탈 1).** Windows 바이너리(B)·PowerShell 체크섬 설치(C)·OS·git 인식 온보딩(D)이 내부적으로 일관되게 성립한다: 유효 PE + 게이트 알고리즘 양경로 + 3대문 일관 + 회귀 0. Windows **런타임**(실제 PC의 .exe·PowerShell end-to-end)은 사전등록된 수신자 검증 경계 — 다음 릴리스가 windows.exe를 게시하면 URL이 해소되고 상현님/친구가 밟는다. 비개발자·크로스플랫폼 아크의 B·C·D 완결.
