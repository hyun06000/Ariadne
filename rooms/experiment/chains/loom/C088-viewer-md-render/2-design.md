# 2. 실험 설계 — 마크다운 렌더 토글 + 이미지 임베드

## 접점 (기존 구조 재사용)
- `_collect_cycle_docs`([~1766]) — 스텝 content를 gil-data에 담는 곳. 여기에 이미지 임베드를 얹는다.
- `_WEB_APP_JS`의 `build`([~1436]) — 스텝을 `<pre>esc(content)</pre>`로 그리는 곳. 토글 분기를 얹는다.
- 존재 문서 `buildBeing`([~1453])도 같은 렌더를 공유(대칭).

## 절차

### A. 인라인 마크다운 파서 (JS, 외부 CDN 0)
1. `_WEB_APP_JS`에 `renderMd(src, images)` 추가. **esc를 기반으로** 안전하게 — 먼저 전체 이스케이프 후, 화이트리스트 인라인/블록만 태그로 승격:
   - 블록: `#`~`######` 헤딩, ``` ``` 펜스 코드블록, `- `/`* `/`1. ` 리스트, `>` 인용, 빈 줄 = 문단, `|…|` 표(선택), `---` hr.
   - 인라인: `**bold**`, `*em*`/`_em_`, `` `code` ``, `[text](url)`(http/상대/앵커만, `javascript:` 차단), `![alt](path)` 이미지.
   - 이미지: `images[path]`가 있으면 `<img src="dataURI">`, 없으면 `<a>path</a>` + alt 폴백.
2. XSS: 원문은 항상 `esc()` 통과 후 승격 — 사용자 콘텐츠가 raw HTML로 안 들어간다. url은 스킴 화이트리스트.

### B. 이미지 임베드 (Python, 굽는 시점)
3. `_collect_cycle_docs`에서 각 스텝 content를 `![alt](path)` 정규식으로 스캔, 상대경로 이미지를 사이클 디렉토리 기준 해석.
4. **장당 상한**(`_WEB_IMG_MAX_BYTES`, 예 2MB) 내 파일만 base64 data URI로. 초과·부재·비이미지 확장자는 스킵(JS가 링크 폴백). MIME은 확장자로.
5. steps[i]에 `images: {path: dataURI}` 맵 추가(임베드된 것만 — 없으면 키 부재로 바이트 영향 0).

### C. 토글 UI (JS + CSS)
6. 각 사이클 body(또는 페이지) 상단에 **"원문 / 렌더" 토글**. 기본 = 원문(`<pre>`, 현행과 동일). 토글 상태는 전역(한 번 켜면 이후 여는 문서도 렌더) — localStorage 없이 JS 변수(자기완결).
7. 토글 전환 시 이미 그려진 문서를 다시 그린다(done 캐시 무효화 또는 재빌드).

## 준비물
- 정본 gil.py. 테스트 픽스처: 이미지가 든 스텝 문서 1개(작은 PNG data URI를 실제 파일로).

## 측정 방법
- M1 토글 기본 원문: 초기엔 `<pre>` 원문(현행 바이트 동일 경로) — 렌더 강제 아님.
- M2 렌더 정확: `# H`→`<h1>`, `**b**`→`<strong>`, `` `c` ``→`<code>`, `[t](u)`→`<a>`, 리스트→`<ul>`.
- M3 이미지 임베드: 상한 내 이미지가 `data:image/...;base64,`로 gil-data에 존재, 렌더 시 `<img>`.
- M4 상한 초과 폴백: 큰 이미지는 임베드 안 되고 링크로.
- M5 XSS 안전: `<script>`·`javascript:` 링크가 실행 태그로 안 새어나감(esc 통과).
- M6 자기완결: 렌더 경로에 외부 http(s) src/href 요청 0(이미지는 data URI).
- M7 회귀: WEB-* 등 conformance 회귀 0, flat/hierarchy 대칭.
- 신설 conformance: WEB-MD-RENDER(토글 존재 + 기본 원문 + 렌더 시 태그 승격 + XSS 이스케이프).
