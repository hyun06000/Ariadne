# 4. 결과 분석

## 통계적 결과
| 측정 | 기대 | 관측 | 판정 |
|---|---|---|---|
| M1 토글 기본 원문 | rendered=false, <pre> | 초기 원문, 클릭 시 렌더 | ✅ |
| M2 렌더 정확 | h/strong/em/code/list/link/table | 헤드리스 확인 | ✅ |
| M3 이미지 임베드 | 상한 내 data URI | pic.png만 | ✅ |
| M4 상한 폴백 | 큰/외부/부재 스킵 | 전부 스킵 | ✅ |
| M5 XSS | javascript:·raw script 차단 | safeUrl 차단, esc 통과 | ✅ |
| M6 자기완결 | 외부 http src 0 | data URI only | ✅ |
| M7 회귀 | 0 | baseline 동일, 무이미지 키부재 | ✅ |

## 데이터 직접 관찰
- **헤드리스가 blockquote 버그를 잡았다.** `esc()`가 `>`→`&gt;`로 바꿔 blockquote 정규식이 못 잡던 것을, 실제 브라우저 렌더 덤프(`<p>&gt; 인용문</p>`)가 드러냈다. 정적 grep이나 파서 코드 읽기로는 못 봤을 것 — **렌더는 렌더로 검증해야 한다.** verify 스킬의 정신("테스트가 아니라 실제 동작을 관찰")이 뷰어에도 적용됐다.
- **상현님 직관("필요할 때만 렌더")이 이미 구조였다.** C075의 클릭-시-마운트 덕에 렌더는 완전 lazy(연 문서만). 이미지 base64만 굽는 시점이나, 장당 상한+링크 폴백으로 비대화를 막았다 — 무게는 여전히 "연 5문서의 이미지"뿐.
- **esc 기반 파서 = XSS 안전.** 원문을 항상 esc한 뒤 화이트리스트 태그만 승격하므로 raw HTML(`<script>`)이 못 샌다. url은 스킴 화이트리스트. 안전을 나중에 덧댄 게 아니라 파서 구조 자체가 안전.

## 예상과 달랐던 것
- 순수 렌더가 아니라 **파서 버그(blockquote)**가 숨어 있었다 — 인라인 마크다운 파서는 esc와의 상호작용(엔티티화)을 반드시 고려해야 한다. `>`뿐 아니라 향후 `<`,`&`도 같은 함정.

## 판정
**채택(supported).** 기각 조건 다섯(토글 없음·상한 없음·CDN 요구·XSS·회귀) 어디에도 안 걸림. "정직한 원문"(기본)과 "보기 좋은 렌더"(토글)를 둘 다 살렸다.
