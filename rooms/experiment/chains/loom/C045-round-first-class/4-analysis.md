# 4. 결과 분석

## 통계적 결과

- **참조 구현 conformance 72/72** (기존 64 + 라운드 8), **회귀 0**.
- **Go 56/56** — `round` 미구현이나 HELP-COMPLETE가 정직한 부재(exit 3)를 판정.
- **변이**: M1(사전등록 위반)·M3(R15 검사 제거)·M4(하위호환 위반) 격추, M2(close 어휘검사 제거) 심층방어 생존 → M2-both가 계약 판정 증명.
- **하위호환**: 무라운드 저장소 web 두 구현 바이트 동일, 회귀 0.

기각 조건(선고정) 대조: (1) 사전등록 순서를 도구가 만들었다(fsck 무의존 유지) — 통과. (2) 평탄 파서 계약 무손상(rounds는 개수만, 상세는 round.yaml) — 통과. (3) 하위호환 유지(바이트 동일) — 통과. (4) 판정기가 새 계약을 본다(8항목) — 통과. **네 조건 모두 미충족 → 기각되지 않음.**

## 데이터 직접 관찰

- **사전등록의 물리적 증거를 커밋에서 확인했다.** `round --open --git` 직후 `git show --name-only HEAD`가 `rounds/R2/hypothesis.md`·`round.yaml`만 담고 `verification/`은 담지 않았다. verification 디렉토리를 만들어도(M1) 빈 디렉토리라 깃이 커밋하지 않아 ROUND-OPEN-GIT은 통과하고, **파일시스템 관찰**(ROUND-PREREG)이 잡았다 — 두 항목이 서로 다른 방어선이다.
- **M2를 손으로 실행해보고서야 심층 방어를 봤다.** close의 `if args.verdict not in _ROUND_VERDICTS` 를 지웠는데도 `bogus`가 거부됐다. 추적하니 `_fsck_or_report`가 R15의 round.yaml verdict 어휘 검사로 잡아 롤백하고 있었다. 두 방어선이 같은 계약을 지킨다.

## 예상과 달랐던 것

1. **M2의 생존 — C011의 동등 변이가 이 사이클에서 재현됐다.** 나는 close의 어휘 검사가 ROUND-REJECT-VOCAB의 단일 방어선일 거라 예상하고 변이를 설계했다. 아니었다. close가 쓴 뒤 `_fsck_or_report`가 R15로 잡아 롤백하므로, close 검사를 지워도 **최종 행동(거부+무변화)이 보존**된다. C011의 정확한 재현: *"스위트는 구현이 아니라 행동을 판정했고, 그게 §7의 의도다."* 그래서 나는 M2를 결함으로 처리하는 대신 **M2-both**(두 방어선을 다 제거)로 계약이 실제로 판정됨을 증명했다 — 그때 ROUND-REJECT-VOCAB이 FAIL(bogus 수용)했다. 한 조항을 시험하려면 다른 방어선이 침묵하는 입력을 골라라(C041)의 정확한 실천.

2. **디렉토리 경계의 양날이 C044의 정확한 반대편으로 나타났다.** C044에서 `chain.md`는 사이클 디렉토리 **밖**(체인 최상위)이라 `git add <사이클경로>`의 사정거리 밖 → 누락(저주). C043에서 예약 원장도 사이클 밖이라 `load_chain_records`가 안 봄 → 비침습(축복). **라운드는 사이클 디렉토리 안**(`rounds/R*/`)이라 `git add <사이클경로>`가 자동으로 커버 — 커밋 누락 걱정이 없다. 같은 "무엇이 그것을 보느냐"의 질문에서, 라운드는 **관측자(커밋 범위)가 보는 쪽**에 두어 안전을 얻었다. C044의 교훈("관측자가 커밋 범위면 격리가 누락")을 설계에 미리 반영한 셈이다.

3. **maru의 fsck 제안이 fsck 계약과 충돌했고, 그 충돌이 더 나은 설계를 낳았다.** maru(#10)는 "각 라운드는 verification보다 hypothesis가 먼저 커밋"을 **fsck**가 강제하자고 했다. 그러나 fsck는 깃 무의존(C007)이라 커밋 순서를 볼 수 없다. 이 충돌을 도구로 풀었다: `round --open`이 hypothesis만 먼저 각인한다(도구가 절차를, C003). 사전등록의 증거는 **도구가 만든 깃 히스토리**이지 fsck의 사후 검사가 아니다. 제안을 그대로 구현하지 않고 계약에 정합화하니 더 깨끗했다.

4. **여섯 번째 "테스트를 먼저 의심하라" — 같은 함정, 세 번째 재발.** 첫 conformance가 거의 전량 FAIL(`gil.py: No such file`)했다. 범인은 구현이 아니라 `--gil "python3 gil.py"`의 **상대 경로** — C028이 문서화하고 C043 회고가 "함정은 문서화됐다고 사라지지 않는다"고 적은 바로 그것이다. 절대 경로로 72/72. 실패 로그의 첫 줄이 한 번에 지목했다(C036 "실패 로그를 끝까지 읽어라").

## 판정

**채택(supported).** H1(사전등록은 도구가 보증)·H2(방법 무효 ≠ 가설 기각)·H3(하위호환 공짜) 모두 검증됨. 라운드가 검증 안의 (가설→검증) 반복을 문서 밖에서 데이터로 끌어올렸다. maru의 진단(#9)이 지목한 "산문 사후 고백"의 자리에, 이제 사전등록된 라운드가 선다.
