# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C080**(step 스텝 경계 스코프)에서 "**스코프의 굵기가 계약**"을 배웠다. 이슈 #19는 같은 계열의 close판 — close도 `git add -A -- <cycle_dir>`로 사이클 디렉토리 전체를 봉인한다(재현 확인: `3-verification/wrong-subdir/misplaced.txt` 오배치 파일이 불변 태그에 봉인됨).

## 문제 분할 (이슈 #19)

close가 봉인하는 것은 **불변**이다 — 태그로 고정돼 verify가 집행한다(C004). 그래서 오배치 파일이 봉인되면: 나중에 지우면 `verify`가 "변조 감지"로 뜨고, 태그를 못 고쳐 **중복 봉인이 영구화**된다. (이 세션의 C029 태그 drift가 바로 이 계열 — 봉인된 것을 나중에 정정하려다 verify 변조.)

### close의 딜레마 (step과 다른 점)
step은 "스텝 N 파일"이 명확했다(`_STEP_FILES[:n]`). 그러나 close는 **3-verification/ 아래에 사용자가 자유롭게 산출물을 둔다**(probe·스크립트·데이터·바이너리…). 화이트리스트로 엄격히 자르면 **정당한 검증 산출물이 누락**된다. 도구는 "표준 산출물"과 "오배치"를 내용으로 구별할 수 없다.

## 갈래 (설계 판단 필요)

- **(A) 봉인 대상 요약 + 예상 밖 경로 경고/확인**: close 전에 봉인될 파일 목록(특히 신규 untracked)을 출력하고, 예상 밖(하위 디렉토리·비표준 파일명)이 있으면 경고하거나 `--allow-extra` 확인 요구. 정당한 산출물은 막지 않되 사용자가 무엇을 봉인하는지 보게 한다.
- **(B) 표준 화이트리스트**: 1~5 문서 + 3-verification 표준 파일만 봉인, 그 외는 명시 플래그. 엄격하나 3-verification 자유 산출물과 충돌.
- **(C) 요약만 (차단 없음)**: 봉인 목록을 항상 출력해 투명성만 높이고 차단은 안 함. 가장 가볍고 정당 산출물 무해.

close의 불변성 + 3-verification 자유도를 감안하면 A(경고/확인) 또는 C(요약)가 유력. B는 정당 산출물을 막을 위험. 상현님께 갈래를 묻는다.

## 가설 (A 방향 잠정)

> **가설**: close가 봉인 전에 대상 파일(특히 신규 untracked)을 요약 출력하고, **사이클 디렉토리 바로 아래 표준 산출물이 아닌 예상 밖 경로**(예: 3-verification 하위의 비표준 디렉토리)가 있으면 경고하고 `--allow-extra` 없이는 거부하면, (a) 오배치 파일이 무심코 불변 봉인되는 것을 막고, (b) 정당한 검증 산출물은 (예상 경로면) 그대로 봉인되며, (c) 기존 정상 close는 불변이다.

## 기각 조건

- 경고/차단 후에도 오배치 파일이 무경고로 봉인되면 → 기각.
- 정당한 3-verification 산출물(README + 사용자 파일)이 막히거나 누락되면 → 기각.
- 기존 정상 사이클의 close가 깨지거나 봉인 내용이 바뀌면 → 회귀, 기각.
- "예상 밖"의 정의가 너무 넓어 정상 사용을 방해하면 → 재설계.
