# 1. 가설 수립 — 토폴로지 레이아웃

**부모**: loom/C031-web-lane-layout (web 레인 분리 — 형제 갈래를 다른 열에)
**발의**: 박상현 — "새 가지가 생기면 이전 가지의 맨 끝보다 더 아래에 노드가 잡히는데 굳이 그럴 필요가 없다. 부모가 같으면 서로 다른 가지끼리는 영향을 주지 않고 lineage로 지식을 전수하면 된다. 토폴로지만 잘 보이면 되고 노드 간 시간 순서는 무시하자. 너무 아래로 길어져 사람이 보기 힘들다."

## 이전 사이클의 교훈

C031이 형제 갈래를 **다른 열(레인)** 에 배치하는 버그를 고쳤다(`tracks.pop`이 인덱스를 당겨 둘째 갈래가 col0으로 흡수되던 것). 그때 확립된 것: 레인은 슬롯 인덱스이며 pop하지 않고 None으로 비운다. 하지만 C031은 **세로 위치(row)** 는 건드리지 않았다.

## 문제 분할

`_layout_columns`는 `for row, node in enumerate(order)` — **row가 토폴로지 순서(order 리스트)의 인덱스**다. order는 `_toposort`가 만든 선형 나열(동순위는 id 오름차순)이라, 모든 노드가 서로 다른 row를 갖는다. 그래서:

- 분기 P → {A갈래, B갈래}에서 A갈래를 다 그린 뒤 B갈래가 order상 뒤에 오면, B의 노드들이 A갈래 맨 끝보다 **아래 row**에 잡힌다.
- **세로 길이 = 노드 총수**에 비례해 자란다 — 형제 갈래가 나란히가 아니라 세로로 누적된다.

상현님의 관찰: 형제 갈래는 **서로 영향을 주지 않는다**(부모만 공유, 지식 전수는 lineage로). 그러니 시간 순서로 세로에 누적할 이유가 없다. **토폴로지(깊이)만 보이면 된다.**

첫 번째로 정복할 것: **row를 order 인덱스가 아니라 노드의 깊이(depth)로** 바꾼다. 형제는 같은 깊이 = 같은 row, 다른 col에 나란히. 세로 길이가 노드 수가 아니라 **최장 경로 길이**로 줄어든다.

## 가설

> **가설**: `_layout_columns`에서 `row = depth(node)`(루트로부터 최장 경로; 병합이면 `max(부모 깊이)+1`)로 바꾸면, 형제 갈래가 같은 높이에 나란히 놓여 **세로 길이가 노드 총수에서 최장 경로 길이로 줄어든다**. col(레인) 배치는 C031 로직 그대로 형제를 구분하고, 선형 체인은 depth가 순차 증가하므로 **기존과 동일**하다(하위호환).

- **H1 — 세로 압축**: 분기 있는 그래프에서 `max_rows`(세로 노드 수)가 `len(order)`에서 `max_depth+1`로 감소한다.
- **H2 — 선형 불변**: 분기 없는 체인(예: genesis)의 레이아웃은 변하지 않는다 — depth = order 인덱스.
- **H3 — 간선 무교차·무충돌**: 같은 (row, col)에 두 노드가 겹치지 않는다. 부모→자식 간선이 여전히 위→아래로 그려진다.
- **H4 — 두 구현 바이트 동일**: 참조와 Go의 web 산출물이 바이트 동일을 유지한다(SVG 좌표는 계약면이 아니나 두 구현 일치는 유지).

## 기각 조건 (선고정)

다음 중 하나라도 참이면 기각/부분 기각:

1. **선형 체인 레이아웃이 바뀐다** — depth 기반이 order 기반과 선형 그래프에서 다른 좌표를 낸다(H2 위반).
2. **(row, col) 충돌** — 서로 다른 노드가 같은 격자 칸에 놓여 겹친다(H3 위반).
3. **세로가 안 줄어든다** — 분기 그래프에서 max_rows가 그대로다(H1 위반).
4. **간선이 역행/교차** — 자식이 부모보다 위(row 작음)에 놓인다.

## 예상 산출물

- 참조 `_layout_columns`의 row = depth. Go 미러링(Weft 또는 직접). 픽스처(분기 그래프)로 height 감소 실측.
- conformance는 WEB-JSON(구조)만 보므로 회귀 없음(SVG 좌표는 §3.1 렌더=비계약). 두 구현 web 바이트 동일 별도 확인.
