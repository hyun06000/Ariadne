# 2. 실험 설계 — 토폴로지 레이아웃

가설(H1~H4)을 검증하기 위한 설계. **정답(기대 행동)을 구현보다 먼저 고정한다.**

## 설계 결정

### D1 — row = depth (루트로부터 최장 경로)

```python
depth = {}
for node in order:               # order는 토포순 — 부모가 항상 먼저 나온다
    ps = [p for p in cycles[node]["parents"] if p in depth]
    depth[node] = (max(depth[p] for p in ps) + 1) if ps else 0
```

- 루트: depth 0. 병합: `max(부모 깊이) + 1` (가장 긴 경로가 세로 위치를 정한다).
- `pos[node] = (depth[node], col)`. 기존은 `(order_index, col)`이었다.

### D2 — col(레인)은 C031 로직 그대로

`tracks` 슬롯 배정(첫째 자식이 레인 상속, 형제는 새/빈 레인, 병합은 흡수 레인 None)은 **손대지 않는다**. C031이 형제를 다른 열에 두는 것을 이미 보장한다. row만 depth로 바꾼다.

### D3 — (row, col) 충돌 방지

레인 재사용(한 갈래가 끝나 레인이 비고 다른 노드가 물려받음) 시, 새 노드의 depth가 이전 점유 노드와 같으면 같은 격자 칸에 겹칠 수 있다(주로 다중 루트). `occupied: set[(row,col)]`로 점유를 추적하고, 충돌 시 **새 free 레인**으로 민다. 단일 루트 그래프(우리 저장소)에선 발동하지 않아 하위호환 무영향.

## 기대 행동 (답의 고정)

| # | 항목 | 픽스처 | 기대 |
|---|---|---|---|
| T1 | H1 세로 압축 | P→{A→A1→A2, B} | max_row = 3(4행), **len(order)=5행이 아니라** |
| T2 | H2 선형 불변 | C001→C002→C003 | 각 노드 depth = order 인덱스 → 좌표 **기존과 동일** |
| T3 | H3 무충돌 | 위 픽스처 | 모든 (row,col) 유일 |
| T4 | H3 간선 방향 | 임의 그래프 | 모든 자식 depth > 부모 depth |
| T5 | H4 두 구현 동일 | 실 저장소 + 분기 픽스처 | 참조 web = Go web 바이트 동일 |

**정량 측정**: 분기 픽스처에서 SVG `height`(= `_TOP_PAD + 28 + (max_rows-1)*_ROW_H + 56`)가 개선 전보다 작다. `max_rows`: 개선 전 `len(order)`, 개선 후 `max_depth+1`.

## 검증 방법

1. 분기 픽스처(P·A·A1·A2·B)로 개선 전/후 SVG height 비교 → H1.
2. 선형 체인(실 저장소)의 web 개선 전/후 → H2(선형은 바이트 동일).
3. pos에서 (row,col) 유일성·간선 방향 검사 → H3·H4.
4. 참조 vs Go web 바이트 동일 → H4.
5. conformance 회귀 0 (WEB-JSON은 구조만; SVG 좌표는 §3.1 비계약).

- **성공**: T1~T5 전부.
- **부분 기각**: 충돌 방지가 tracks 안정성을 깨 선형 레이아웃이 변하면.
