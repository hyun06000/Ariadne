# 5. 결과 보고 — 토폴로지 레이아웃 (발의: 박상현)

## 요약

뷰어(gil web)의 SVG 레이아웃에서 노드의 세로 위치(row)가 **토폴로지 순서(order 인덱스)** 였던 것을 **깊이(루트로부터 최장 경로)** 로 바꿨다. 형제 갈래가 같은 깊이=같은 높이에 나란히 놓여, 세로가 노드 총수가 아니라 최장 경로로 압축된다. 참조·Go 양 구현, 바이트 동일, H1~H4 전부 검증, 회귀 0, 선형 체인 불변(하위호환). **채택(supported).**

## 교훈

1. **한 개념이 두 곳에 하드코딩되면 함께 고쳐야 한다 — 레이아웃과 캔버스.** row를 depth로 바꿔도 `_render_svg`의 `max_rows = len(order)`가 SVG height를 노드 총수로 잡고 있어, 한 곳만 고치면 노드는 겹쳐 올라가고 캔버스만 길게 남았을 것이다. 세로 크기가 레이아웃 함수와 캔버스 계산 **두 곳**에서 따로 노드 수를 가정했다. 실제 최대 depth로 둘 다 바꿔야 압축이 완성됐다. C038("목록형 규칙은 조용히 낡는다")의 좌표판.

2. **시그니처를 안 바꾸고 계약을 지킨다 — Go의 parents 역산.** Go `layoutColumns`는 `children`만 받는데 depth엔 parents가 필요했다. children의 역을 함수 안에서 구성했다. map 순회는 비결정적이나 **깊이는 max라 부모 순서에 무관**, pos는 order 슬라이스 순회라 결정적 — 두 구현 바이트 동일이 유지됐다.

3. **렌더는 계약이 아니므로 정확성을 다른 것으로 보증한다.** conformance는 SVG 좌표를 안 본다(§3.1). 이 개선의 정확성은 판정기가 아니라 **두 구현 바이트 동일 cross-check + height 실측**으로만 선다. C021의 자유(렌더=비계약)를 쓰되, 그 자유가 검증 공백이 되지 않게 두 구현 대조로 메웠다.

4. **압축의 값은 분기 밀도에 비례한다.** 우리 loom(대부분 선형)은 3074→2498px에 그쳤지만, 형제 갈래가 잦은 저장소일수록 극적이다. 상현님이 발의한 것은 실사용에서 세로가 감당 안 됐기 때문 — 수혜자는 분기를 많이 쓰는 사용자다.
