# 4. 결과 분석

## 통계적 결과

- **H1 세로 압축**: 분기 픽스처(5노드, 최장경로 depth 3) SVG height 386→**322**(5행→4행). 형제 A·B가 같은 행(depth 1).
- **H2 선형 불변**: genesis(선형) 개선 전/후 web **바이트 동일** — depth = order 인덱스라 좌표 불변.
- **H3 무충돌**: 픽스처 노드 5개 = 유일 좌표 5개, 고유 행 4개.
- **H4 두 구현 동일**: 실 저장소·분기 픽스처 모두 참조 = Go 바이트 동일.
- **회귀 0**: 참조 72/72, Go 64/64.

기각 조건 대조: (1) 선형 안 바뀜(H2) — 미충족. (2) 충돌 없음(H3) — 미충족. (3) 세로 줄어듦(386→322) — 미충족. (4) 간선 역행 없음(자식 depth > 부모) — 미충족. **네 조건 모두 미충족 → 기각되지 않음.**

## 데이터 직접 관찰

- **실 loom은 3074→2498px, 576px 압축.** 47사이클인데 대부분 선형이라 압축 폭이 크진 않다 — 분기·병합(C036 병합, C031·C044의 분기)이 있는 지점에서만 형제가 합쳐진다. **상현님 저장소처럼 분기가 잦으면 훨씬 극적이다** — 이 개선의 값은 그래프의 분기 밀도에 비례한다.
- **이 사이클이 자기 문제를 도그푸딩했다.** parent를 C031(레이아웃을 마지막으로 정한 사이클)로 잡아 loom 그래프에 C031→C032와 C031→C047 분기를 만들었다. 개선 후 뷰어에서 두 갈래가 **같은 높이에서 갈라진다** — 고치려던 문제를 자기 노드로 보여준다.

## 예상과 달랐던 것

1. **`max_rows`가 `len(order)`에 고정돼 있었다 — row만 바꿔선 안 됐다.** `_layout_columns`의 row를 depth로 바꿔도, `_render_svg`의 `max_rows = max(max_rows, len(chain["order"]))`가 SVG height를 노드 총수로 계산하고 있었다. **레이아웃과 캔버스 크기가 두 곳에서 따로 노드 수를 가정**했다 — 한 곳만 고치면 노드는 겹쳐 올라가는데 캔버스는 그대로 길었을 것이다. 실제 최대 row(depth)로 바꿔야 압축이 보였다. *한 개념(세로 크기)이 두 곳에 하드코딩되면 함께 고쳐야 한다*(C038 "목록형 규칙은 조용히 낡는다"의 좌표판).

2. **Go는 parents를 안 받아서 역산해야 했다.** 참조는 `cycles`(parents 포함)를 받지만 Go `layoutColumns`는 `children`만 받는다. depth 계산엔 parents가 필요해 children의 역(`parents[child]=[p]`)을 함수 안에서 구성했다. Go map 순회는 비결정적이지만 **깊이는 max라 부모 순서에 무관**하고 pos는 order 슬라이스 순회라 결정적 — 그래서 두 구현이 바이트 동일을 유지했다. 시그니처를 안 바꾸고 계약을 지켰다.

3. **SVG 좌표가 계약이 아니어서 conformance가 이 변경을 못 본다.** 두 구현 회귀 0인데, 판정기가 SVG 좌표를 안 보기 때문이다(§3.1 "렌더는 계약이 아니다", 계약면은 WEB-JSON·exit code). **이 개선의 정확성은 conformance가 아니라 두 구현 바이트 동일 cross-check와 height 실측으로만 보증된다** — 다만 여기선 의도된 비계약(렌더 자유)이라 C021의 정신 그대로다.

## 판정

**채택(supported).** H1~H4 전부 검증. 형제 갈래가 같은 깊이에 나란히 놓여 세로가 최장 경로로 압축된다. 토폴로지는 보존되고 시간 순서는 무시된다 — 상현님의 요청 그대로. 선형 체인 불변(하위호환), 두 구현 바이트 동일.
