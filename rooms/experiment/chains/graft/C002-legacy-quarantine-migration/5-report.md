# 5. 결과 보고

## 요약

상현님의 아이디어("레포 정리 → legacy/ 격리 → gil로 재시작")를 재현 가능한 절차(M0~M3, `git mv` 히스토리 보존)로 확정하고 실 Flask에 적용·검증했다. **히스토리는 완벽히 보존**(676→677, `--follow` 추적)되고 골격 마찰(F2/F4/F7)은 완화됐으나, **F3(releases 침묵)은 미완화** — git 태그가 파일이 아니라 ref라 legacy/ 격리가 못 건드리기 때문. **채택(supported), 가설 정련.**

## 교훈

1. **legacy/ 이주 절차가 검증됐다 (migration-procedure.md).** `mkdir legacy; git ls-tree HEAD | while read i; do git mv "$i" legacy/$i; done; commit` → 레포 루트에서 `gil open --new-chain --new-root`. Flask 236항목 전부 rename 인식, 히스토리 보존(`git log --follow legacy/<파일>`이 이주 이전 이력 그대로), 최상위가 `legacy rooms` 두 세계로 깨끗이 분리. **레거시 채택의 골격 문제(F2/F4)를 완전히 해결하는 실증된 처방.**

2. **git의 역사는 "파일 층"과 "ref 층" 두 개다. legacy/는 파일 층만 접는다.** 이 사이클 최대 수확. 나는 "격리하면 F3(태그 충돌) 표면이 사라진다"고 예측했지만 반증됐다 — `git mv`는 작업트리 파일만 옮기고 태그(`2.2.3`…)는 ref라 그대로 살아있다(이주 후에도 16개). CHANGES.rst조차 legacy/로 "이동"했을 뿐 저장소 안에 남는다. **격리는 "치우기"가 아니다.** F3은 legacy/ 전략의 명확한 사각지대이고, 별도 처방이 필요하다.

3. **F3의 심각도는 벽이 아니라 실명이다.** F2/F4는 채택을 아예 못 하게 막던 벽 → 이제 없다. F3은 조회 명령의 조용한 실명 → 채택은 되지만 레거시 배포 역사가 안 보인다. 절차의 유효성은 유지되되, F3은 반드시 후속 처방으로 다뤄야 한다.

4. **loom/C007의 교훈이 또 증명됐다 — "가정하지 말고 확인하라".** "표면이 사라진다"는 내 상상이 실 Flask 실측에서 반증됐다. C001이 이 교훈으로 "상상 금지"를 기각조건에 넣었고, C002도 실 레포 적용으로 가정을 잡았다. graft 체인은 매 사이클 실데이터로 자기 가정을 심판한다.

## 다음 사이클을 위한 제안

- **(A, 1순위) F3 releases 실명 수정 — 이제 정확히 특정된 표적.** gil이 비-semver 태그(`2.2.3`)·비-gil CHANGELOG(`CHANGES.rst`)를 "없음(0)"이 아니라 "external/미인식 릴리스"로 구별해 보고하게. C002가 "legacy/로는 못 없앤다"를 증명했으므로 이 처방은 회피 불가. loom 계열 수정 사이클(양 구현 + 판정기).
- **(B) `gil adopt`/`init` 포셀린 — 절차의 도구화.** migration-procedure.md의 M1~M3을 한 명령으로 원자화(+ 존재 방 스텁 자동 생성으로 F5 보완). gateway("한 문장 진입")의 레거시판. 절차가 손으로 검증됐으니(C002) 이제 도구화의 명세가 준비됨.
- **(C) legacy 태그 네임스페이싱(선택).** 정말 "ref 층까지 정리"하려면 기존 태그를 `legacy/2.2.3`류로 리네임하는 옵션 — 단 이는 파괴적이고 upstream 태그를 깨므로 신중히. B의 옵션으로 흡수 가능.

**추천 경로: A → B.** F3은 legacy/로 못 피한다는 게 증명됐으니 먼저 고치고(A), 그 위에서 절차 전체를 도구화(B)한다. C가 필요한지는 B 설계가 판정.
