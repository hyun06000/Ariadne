# 4. 결과 분석

## 통계적 결과

- **히스토리 보존 (기각조건 1)**: 커밋 676 → 677 (격리 +1만). `git mv` 236항목 전부 rename 인식. `git log --follow legacy/src/flask/app.py`가 M0 캡처(fbb6f0b, 7b00886, c17f379, c77a520)와 동일한 이주 이전 이력을 그대로 추적. → **보존 확인, 기각 안 됨.**
- **마찰 완화 (기각조건 2)**: F2/F4/F7 완화 + 파일 경계 분리 = **부분 완화**. 그러나 **F3(releases 침묵)은 미완화**. → 아래 직접 관찰 참조.
- **재현성 (기각조건 3)**: M0~M3이 Flask 고유 지식 0으로 일반 단계로 적힘(migration-procedure.md). `git ls-tree | while git mv`는 임의 레포에 그대로 적용 가능. → **재현 가능, 기각 안 됨.**

## 데이터 직접 관찰

수치가 아니라 **왜 F3이 안 완화됐는가**를 직접 들여다본다 — 이게 이 사이클의 핵심 교훈이다.

1. **`git mv`는 "파일 세계"만 옮긴다. git 태그·브랜치는 "ref 세계"라 손대지 않는다.** 나는 스텝 1 가설에서 "legacy/ 격리가 F3(태그 충돌)의 **표면 자체를 없앤다**"고 예측했다. 실측은 반쯤 틀렸다:
   - `git mv CHANGES.rst legacy/CHANGES.rst` → 파일은 legacy/로 갔지만 **여전히 저장소 안**에 있다. 격리는 "치우기"가 아니라 "하위 디렉토리로 이동"일 뿐.
   - **Flask의 16개 태그(`2.2.3`…)는 git ref다.** `git mv`가 건드리는 것은 작업트리 파일뿐 — 태그는 `.git/refs/tags`에 그대로 살아있다(measure-friction.txt: 이주 후에도 `git tag -l | wc -l` = 16). 그래서 `gil releases`는 여전히 이 태그들을 마주치고, 여전히 `v<semver>` 형식이 아니라며 0으로 침묵한다.
   - **교훈**: 상현님의 "레포 정리" 직관은 파일 레벨에선 완벽히 작동하지만(경계 혼란·골격 부재 해결), **git의 역사는 파일과 ref 두 층**이고 legacy/는 파일 층만 접는다. loom/C007의 교훈("가정하지 말고 확인하라")이 또 증명됐다 — 내 "표면이 사라진다"는 가정이 실측에서 반증됐다.

2. **그래도 legacy/ 격리는 채택의 핵심 절반을 해결한다.** F2/F4(골격 부재)는 채택을 **아예 못 하게** 막던 벽이었는데, M2로 완전히 사라졌다(fsck 위반 0). 그리고 `ls-tree` = `legacy rooms`라는 **깨끗한 두 세계 분리**는 사람과 AI 모두에게 "여기까지가 옛것, 여기부터가 gil"이라는 명확한 경계를 준다. F3은 남지만, F3은 애초에 "조회 명령의 침묵"이지 "채택을 막는 벽"은 아니다 — 심각도가 다르다.

3. **F5(존재 방 미검사)가 절차 설계에 반영됐다.** M2가 `--author maintainer`를 존재의 방 없이 통과시켰다(rooms/existence 부재). 절차는 이를 M3(존재 방 스텁 권장)로 감쌌지만 gil이 강제하지 않으므로, 도구화 시 "존재 스텁 자동 생성"이 채택 완결성을 높일 후보다.

## 예상과 달랐던 것

- **핵심 반전**: "legacy/ 격리가 F3을 없앤다"는 가설의 예측이 **틀렸다**. git 태그가 ref라는 사실을 절차 설계 때 간과했다. 이건 기각이 아니라 **가설의 정련** — 격리는 F3의 "파일 부분"(CHANGES.rst)조차 완전히 치우지 못하고, "ref 부분"(태그)은 아예 못 건드린다. F3은 legacy/ 전략의 사각지대이고, **별도 처방(gil이 비-semver 태그를 external로 인식)이 반드시 필요**하다는 것이 이 사이클의 최대 수확이다.
- `git mv`의 rename 인식이 236/236 완벽했다 — 도트파일(`.github/`, `.gitignore` 등)까지 `git ls-tree`가 다 잡아 누락 0. 절차의 `while read` 루프가 견고함을 실측으로 확인.

## 판정

**채택(supported), 단 가설은 정련됨.**
- 가설의 축 1(히스토리 보존)과 축 3(재현성): **완전 지지.** `git mv` legacy/ 절차는 히스토리를 보존하며(--follow 추적) 재현 가능한 일반 단계로 확정됐다(migration-procedure.md).
- 가설의 축 2(마찰 완화): **부분 지지.** F2/F4/F7·경계 혼란은 완화되나, **F3(releases 침묵)은 미완화** — "격리가 F3 표면을 없앤다"는 예측은 반증됐다(태그=ref). 이는 기각조건 2를 F3 하나에 대해 발동시키지만, 전체 마찰 지형에서 F3은 채택을 막는 벽이 아니라 조회의 실명이므로 절차 자체의 유효성은 유지된다.

**결론**: legacy/ 이주는 **레거시 채택의 골격 문제를 해결하는 검증된 절차**다. 남은 F3은 이 절차의 사각지대로 명확히 특정됐고, 다음 사이클의 정확한 표적이 된다.
