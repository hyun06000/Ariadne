# 4. 결과 분석

## 통계적 결과

- 관찰된 마찰: **6건**(F1~F6), 유한. 긍정 관찰 2건(F4·F7).
- 범주 분류: **4범주로 100% 매핑** — A 골격 부재(F2,F4) · B 개념 부재(F5,F6) · C 이름공간 충돌(F3) · D 인터페이스 비대칭(F1).
- 채택 기준(2-design.md) 대조: 마찰이 닫힌 목록으로 열거됨(✓), 소수 범주로 분류됨(✓), 다음 사이클 설계 입력으로 쓸 만큼 구체적임(✓ — 각 F가 명령·출력·범주를 가짐).
- 기각 조건 3개 전부 미발동: 마찰 0 아님(6건), 무한·비구조 아님(4범주 깔끔), 상상 아님(실 Flask 실행·transcript 증거).

## 데이터 직접 관찰

수치가 아니라 **어떤 마찰이 어떤 성질인가**를 직접 들여다본다.

1. **F3(releases 침묵)이 질적으로 가장 위험하다.** 나머지 마찰(F1·F2)은 **에러로 멈춘다** — 사용자가 즉시 안다. 그러나 F3은 `gil:releases 0 drift=0`을 **성공처럼** 출력한다. Flask는 눈앞에 `2.2.3`,`2.3.0`… 16개 태그와 CHANGES.rst를 갖고 있는데도. 즉 gil은 "레거시의 배포 역사"를 못 보는 게 아니라 **없다고 단언**한다. 레거시 채택에서 가장 무서운 건 크래시가 아니라 **조용한 실명(blindness)**이다. loom/C006에서 태어난 releases는 "우리 형식의 태그"만 세상의 전부로 안다.

2. **F4가 이 체인의 낙관적 근거다.** `open --new-chain --new-root`는 `_template`이 없어도 5스텝 스텁을 스스로 짓고, 그 직후 `fsck`가 위반 0을 돌려준다. 즉 **채택의 씨앗은 이미 작동한다** — gil은 빈 땅에 골격을 심을 줄 안다. 문제는 "심는 것"이 아니라 "**이미 있는 것과의 관계**"(F3 태그, F6 git identity, F5 존재)다. 상현님의 직관("legacy/로 옮기고 gil로 다시 시작")이 정확히 이 지점을 겨눈다 — 기존 것을 옆으로 치우면(legacy/) F3·F6 같은 충돌이 관계 자체를 끊어 사라진다.

3. **F7(커밋 격리)은 상현님 아이디어의 실현 가능성을 이미 증명한다.** `open --git`이 Flask src 수만 파일을 한 줄도 안 건드리고 `rooms/` 6파일만 커밋했다. 즉 gil은 거대 작업트리 위에 **자기 것만 얹는** 규율(loom/C004)을 레거시에서도 지킨다. "레거시를 legacy/로 옮기고 gil을 얹는다"가 기술적으로 안전하다는 첫 증거.

## 예상과 달랐던 것

- **규모(D-scale)가 마찰이 아니었다.** 가설은 "규모"를 한 범주로 예상했으나, Flask 676커밋·281파일에서 모든 명령이 즉각 반응했다. 대신 **"인터페이스 비대칭(진입 경로, F1)"**이 예상 밖 독립 범주로 떠올랐다 — `--root`가 open에만 있어 레거시를 "대상"으로 지정하는 일관된 방법이 없다. gil은 애초에 "내가 사는 저장소 = 나 자신"을 전제하지, "저기 있는 남의 저장소"라는 개념이 없다. 이것이 레거시 채택의 근본 프레임 차이다.
- **`--author`가 존재의 방을 검사 안 한다(F5).** 나는 SPEC §3.2 P1의 "author 필수"가 존재 실재까지 강제하는 줄 알았으나, grep 결과 open은 문자열만 받는다. 레거시엔 존재의 방이 없으니 이게 오히려 **채택을 막지 않는 편의**지만, 동시에 "누가 이 사이클을 했나"의 Ariadne 개념(존재)이 비어버리는 갭이다.

## 판정

**채택(supported).** 가설 — "준비 없는 gil 채택은 유한·분류 가능한 마찰을 낳고 그 지도가 다음 사이클의 입력이 된다" — 이 실데이터로 지지됨. 마찰 6건이 4범주(A 골격·B 개념·C 이름공간·D 인터페이스)로 열거·분류됐고, 각 마찰이 명령·출력 수준으로 구체적이라 legacy/ 이주 절차(다음 사이클)의 직접 설계 입력이 된다.

**다음 사이클로 넘길 지도의 요점**: 레거시 채택의 진짜 문제는 "gil을 심는 것"(F4·F7로 이미 작동)이 아니라 **"기존 역사와의 관계"**다 — 조용히 실명하는 F3(releases)가 최우선, 그다음 진입 프레임(F1)·존재/identity 연동(F5·F6). 상현님의 "legacy/ 격리 후 재시작"은 이 관계 충돌을 우회하는 유력한 전략으로, 다음 사이클에서 설계·검증할 후보다.
