# 2. 실험 설계

오직 1-hypothesis.md의 가설 — "준비 없는 gil 채택은 유한·분류 가능한 마찰을 낳는다" — 하나만 검증한다.
처방(legacy/ 이주)은 이 사이클의 범위가 아니다. **관찰만 한다.**

## 관찰 대상 (실데이터)

**Flask** (github.com/pallets/flask) — 실제 성숙한 레거시 파이썬 웹 프레임워크.
- gil / chain / cycle 개념이 전혀 없는, 우리 손 밖에서 자란 진짜 코드베이스.
- 자체 CHANGELOG(`CHANGES.rst`), 자체 git 태그(`x.y.z` 버전), 수천 커밋의 역사를 가짐 — gil이 쓰는 것과 **같은 이름공간**(태그·CHANGELOG)을 이미 점유하고 있어 충돌 관찰에 이상적.
- 재현성: `git clone --depth 200`으로 누구나 같은 대상을 얻는다(전체 히스토리까지는 이 지도 사이클에 불필요).

## 절차

각 단계에서 **일어난 일을 있는 그대로 기록**한다. 성공/실패 자체가 데이터다. 마찰이 관찰되면
`3-verification/friction-log.md`에 `[F##]` 번호를 붙여 즉시 적는다(증상·명령·출력·추정 범주).

1. **환경 격리**: 스크래치패드에 Flask를 클론(`--depth 200`). gil은 참조 구현(`gil.py`)을 절대경로로 호출 — Flask 레포 안에 gil을 설치하지 않고, `--root <flask경로>`로 gil이 Flask를 대상 저장소로 인식하게 시도한다. (백지 사용자는 이 경로를 모른다 → 그 자체가 마찰 후보 F.)

2. **`gil log` — 빈손 진입**: gil이 모르는 저장소에서 계보 조회를 시도. `rooms/experiment/chains/`가 없는 저장소에서 gil이 무엇을 하는가(빈 그래프? 에러? 크래시?)를 관찰.

3. **`gil fsck` — 규칙 검사**: chain/cycle이 하나도 없는 저장소에 스키마 규칙 검사를 걸면 무엇이 나오는가.

4. **`gil open` — 첫 사이클 시도**: 레거시 레포 안에서 첫 chain/cycle을 열어본다. `rooms/experiment/` 골격이 없는 상태에서:
   - 디렉토리 구조가 자동 생성되는가, 아니면 사용자가 먼저 만들어야 하는가?
   - `--git`이 Flask의 기존 히스토리 위에 커밋을 얹는가? 기존 작업 트리(수만 파일)를 어떻게 다루는가?
   - `--author`는 존재의 방(`rooms/existence/`)을 전제한다 — 그게 없으면?

5. **`gil release` / `gil releases` — 이름공간 충돌**: gil의 릴리스 개념(`v<semver>` 태그 + `RELEASE.md`/CHANGELOG)이 Flask가 **이미 가진** `x.y.z` 태그·`CHANGES.rst`와 어떻게 부딪히는가. `gil releases`의 drift 검사가 Flask의 기존 태그를 어떻게 읽는가.

6. **`gil web` — 뷰어**: 레거시 레포에서 뷰어를 생성하면 무엇이 그려지는가(빈 그래프? 크래시?).

7. **규모 관찰**: 위 명령들이 Flask 규모(수천 커밋·수만 파일)에서 시간·출력 면에서 어떻게 동작하는가(체감 마찰).

## 측정 방법

- **주 측정**: 관찰된 마찰의 **개수와 범주**. 가설이 예고한 범주(히스토리 충돌 / 개념 부재 / 도구 전제 위반 / 규모)에 실제 마찰이 매핑되는가, 아니면 예상 밖 범주가 나오는가.
- **채택 기준**: 마찰이 유한하게 열거되고(원장에 `[F##]`로 닫힌 목록), 각 마찰이 소수 범주로 분류되며, 그 지도가 다음 사이클(이주 절차)의 설계 입력으로 쓸 만큼 구체적이다.
- **기각 기준**: 1-hypothesis.md의 기각 조건 3가지(마찰 0 / 무한·비구조 / 상상에 그침).
- **재현성**: 모든 명령을 그대로 재실행할 수 있게 transcript에 명령+출력을 남긴다.
