## 요약

v3 스텝 트리를 **사이클 DAG와 같은 시각 언어**(가로 흐름·원형 노드·작은 원·kind 색·파선
backtrack)로 렌더하는 뷰어(`circletree.py`)를 지었다. 상현님과 **캡쳐→피드백→사이클**로
다듬었고, 그 티키타카 자체를 **gilv3 스텝 트리(37노드)로 각인**해 뷰어가 자기를 설계한
대화를 그리게 했다(자기증명·첫 도그푸딩). 상현님이 "닫자"로 방향 승인(M3) → 성공.

## 무엇을 했나 (시각 언어 확정)

1. **가로 흐름** (사이클 DAG처럼 왼→오른). depth=가로축, 형제=세로축.
2. **원형 노드 + 원 아래 이름 + 작은 원** (v2 DAG 실측 r=9·간격116 참고, R=11).
3. **kind 색 구분**: 문제정의 파랑·가설 보라·검증 청록·분석 회색.
4. **분석 다음 네 결과** (대화 중 상현님과 확정):
   - **문제정의(define)** = 앞으로 새 가지를 뻗는 노드. **루트든 분석 다음이든 하나로
     통합**(define/redefine 폐지 — 문제정의는 그냥 문제정의).
   - **실패(fail)** = 백트래킹(파선 아치)으로 조상 문제정의 복귀. 죽은 잎.
   - **성공(success)** = 산 잎, 끝.
   - **대기(pending)** = 사람에게 묻는 중. human-in-the-loop을 데이터로 인정.
5. **문제 영역 배경 리본**: 각 문제정의가 자기 하위 시도를 감싸 "백트래킹을 문제 단위로"를
   시각화 — 실패가 어디로 돌아갈지 찾을 필요 없이 자기 문제 리본으로.
6. **노드 클릭 → 그 스텝 본문만** 펼침(단일 열림, 자기완결 인라인 임베드).

## 핵심 발견 (⭐ 대부분 v3-build로 이월)

- **뷰어를 그리다 v3 모델의 근본 원리가 드러났다**: 결과 잎 노드화, 문제정의 통합,
  pending(human-in-loop), 문제 단위 백트래킹.
- **⭐ 깃/gil 역할 정립 (상현님 정정)**: 깃=**append-only, 앞으로만**(모든 스텝이 커밋으로
  영원히 쌓임, 죽은 가지=히스토리=벽의 지도, 롤백 없음). gil=**백트래킹·분기 지능**(실패서
  어디로 돌아갈지 판단 + 원하는 위치서 새 분기). 깃이 전진만 하면 "checkout하면 기록도
  롤백"이라는 모순이 소멸. v2 gil의 append-only 원장 정신을 v3 스텝 트리 층에 세운 것.
