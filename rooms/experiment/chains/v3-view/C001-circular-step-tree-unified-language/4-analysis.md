# 4. 결과 분석

## 통계적 결과

| 측정 | 기준 (2-design) | 결과 | 판정 |
|---|---|---|---|
| M1 위상 보존 (K1) | 원형 렌더가 노드·엣지·잎 전부 담음(정보 손실 0) | 원 10·parent 9·backtrack 2·죽은/산 잎 보존 | PASS |
| M2 시각 언어 (K2) | 원+실선+파선+kind 색이 캡쳐에 보임 | 전부 렌더, 자기완결 | PASS |
| M3 방향 판정 (K3) | 상현님이 캡쳐 보고 방향 승인 | "닫자, 다음으로" = 승인 | PASS |

세 kill 조건 미발동. 상현님 판정이 곧 M3 — 사람의 눈이 판정자였다.

## 데이터 직접 관찰

이 사이클은 **캡쳐→피드백→사이클**로 진행됐고, 그 티키타카 자체를 gilv3 스텝 트리로
각인했다(`design-steptree/`, 37노드). 그 트리를 직접 본다:

- **우리 대화가 스텝 트리의 모든 문법을 실동작시켰다**: 문제정의로 여러 번 앞으로
  (세로→가로, 결과잎, 작은원, 리본, 깃/gil) · s9에서 처음 **실패-백트래킹**(상호작용
  배선 가지가 실패, 문제 단위로 s9 복귀) · **대기(pending)**로 매번 상현님을 기다림 ·
  마지막 s37 **성공**. 한 트리가 문제정의·실패·성공·대기 네 결과와 백트래킹을 다 담았다.
- **뷰어가 자기를 설계한 대화를 그린다**: circletree.py가 그린 트리가 곧 circletree.py를
  만든 대화. 자기증명이자 gilv3 첫 실전 도그푸딩.
- **시각 언어가 v2 사이클 DAG에서 파생됐다**: DAG 실측(원 r=9·간격116)을 그대로 가져와
  스텝 트리를 원형·가로·작은 원으로. 세 층(체인·사이클·스텝)이 한 시각 언어로 이어질
  토대가 섰다.

## 예상과 달랐던 것

- **문법이 대화 중 진화했다**: 설계 땐 "원형으로 바꾼다"만 봤으나, 상현님 피드백이
  데이터 모델 층까지 밀고 갔다 — 결과 잎 노드화, 문제정의 통합(define/redefine→define),
  대기(pending) 노드 신설(human-in-the-loop), 문제 단위 백트래킹, 그리고 **깃/gil 역할
  정립**(깃=append-only 전진 기록, gil=백트래킹·분기 지능). 뷰어를 그리다 v3 모델의
  근본 원리가 여럿 드러났다 — 정직히 v3-build로 이월.
- **내 오해를 두 번 정정받았다**: (1) 실패 백트래킹을 s13으로 잘못 보냄 → 상현님이
  "두 종류 갈라진 지점을 찾아라"로 s9를 짚게 함. (2) 깃을 롤백 도구로 오해 → 상현님이
  "깃은 앞으로만, gil이 백트래킹"으로 역할을 뒤집음. 사람이 루프에 있어 방향이 교정됐다.

## 판정

**채택 (supported).** 스텝 트리를 사이클 DAG 시각 언어(가로·원형·작은 원·kind 색·파선
backtrack)로 렌더하니 위상 손실 0·자기완결로 담겼고(M1·M2), 상현님이 캡쳐를 보고 방향을
승인했다(M3, "닫자"). 세 kill 조건 미발동, 산 잎(s37 성공) 도달 → 닫는다. 3층 드릴다운
배선·본문 마크다운 렌더·Go 이식, 그리고 **깃/gil 역할·코드 저장·스텝=커밋 본격화**는
정직히 이월(v3-build).
