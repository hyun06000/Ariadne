# s4 · analyze/success

## 분석 — 3층 드릴다운 목업이 연속 시각 언어를 유지한다

5측정 전부 PASS. 목업 캡쳐가 가설을 뒷받침한다:

1. **층 1(사이클 DAG)이 스텝 트리와 같은 언어를 쓴다** — 원형 노드+실선 계보 엣지. C001이
   확정한 스텝 트리 언어를 사이클 층에 그대로 올리니, 층이 바뀌어도 "같은 종류의 그래프
   안으로 더 깊이"라는 연속감이 산다. K1(시각 불연속) 미발동.

2. **드릴다운이 자기완결로 된다** — 사이클 노드 클릭 → 그 사이클 스텝 트리를 iframe srcdoc로
   인라인 임베드. fetch 0, `file://`로 열림. v2 뷰어 계약(자기완결) 유지. K2 미발동.

3. **상태 색이 두 층에서 의미가 다르되 언어는 같다** — 스텝 층의 색=kind(문제정의·가설·검증
   ·분석·결과), 사이클 층의 색=status(루트·진행중·solved·대기·폐기). 같은 "원+색" 문법에
   다른 어휘를 실었다 — 연속감을 깨지 않으면서 층의 의미 차이를 표현.

## ⭐ 발견 — 도구엔 아직 pending outcome이 없다

C001 목업이 제안한 **pending(사람에게 묻는 중)** 을 목업 층 1의 사이클 status에도 넣었는데,
실 gil v3 도구의 analyze outcome은 `success|backtrack|fail` 셋뿐이다(gil.py OUTCOMES).
**뷰어가 모델보다 앞서 있다** — C001이 "결과 잎 노드화·pending을 v3-build로 이월"한 그
좌표가 아직 코드로 미실현. 이 사이클은 그걸 다시 확인했다(다음 v3-build 후보).

## 정직한 경계

- **이 사이클은 배선 목업까지** — 실 gil 원장 배선·본문 마크다운 렌더·넓은 트리 tidy는 이월.
- C004 하나만 실 스텝트리, 나머지 사이클은 status 색만.
- 산 잎(success) 도달 → 성공. **단 봉인(close)은 상현님 승인 후** — chain.md의 작업 방식
  (목업+캡쳐를 펜딩으로 올려 승인/리젝트)을 따른다. 상현님이 리젝트하면 그 지적을 새 가설로
  backtrack.
