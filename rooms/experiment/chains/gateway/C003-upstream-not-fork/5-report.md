# 5. 결과 보고 — 우회 대신 이슈: gil을 포크·래핑하지 말고 상류로 (발의: 박상현)

## 요약
필드 LLM이 gil을 즉석에서 고치거나 래퍼로 감싸는 경향을 온보딩에서 재지향했다. README.ai.md의 Iron rules(LLM 직독)에 **"Don't fork, patch, or wrap gil — file an issue instead."** 철칙을, QUICKSTART.md(사람용)에 대응 한 줄을 명령형으로 못 박았다. 둘 다 막기만 하지 않고 `gh issue create` 대안 + "필드 마찰이 상류가 가장 원하는 것"이라는 동기를 함께 준다. M1~M4 통과, 데모 스모크 불변. **채택(supported).**

## 교훈
1. **금지는 대안과 함께여야 작동한다.** "고치지 마라"만으로는 LLM이 다른 우회를 찾는다. 즉시 실행 가능한 정당한 경로(이슈)를 같이 줘야 충동이 재지향된다 — 기각 조건에서 미리 예측한 실패를 문안 설계로 회피했다.
2. **이 규율은 새 규범이 아니라 있던 원리의 번역이었다.** gil의 "계약이 구현을 정의한다"(conformance)를 필드 LLM 언어로 옮기면 "구현을 로컬에서 갈지 말고 계약=이슈로 바꿔라"가 된다. "닫힌 사이클 불변" 바로 뒤에 놓은 것도 같은 계열("봉인된 공유 진실을 로컬에서 손대지 마라") 때문.
3. **온보딩은 실증으로 말할 때 설득력을 얻는다.** "최근 기능 다수(`--live`·배포 버저닝)가 필드 이슈에서 태어났다"는 이번 세션 자체의 사실(#24·#25→C085·C086)이다 — 추상 규범이 아니라 증거로.
