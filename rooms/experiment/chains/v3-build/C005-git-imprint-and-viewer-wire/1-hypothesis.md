# 1. 가설 수립

## 이전 사이클의 교훈

부모: **C003-first-commands-on-step-tree** + **C004-v3-viewer-step-tree** (병합 노드,
v3-build의 첫 병합). 두 부모가 한 지점으로 수렴한다:

- **C003 (명령)**: `gilv3 open/step/close`가 커서 없이 트리를 짓는다(성장 팁 계산).
  **정직한 경계로 "깃 각인(스텝=커밋)은 범위 밖, 다음"**이라 명시. 읽기(파생)와
  쓰기(명령)가 같은 상태기계를 공유.
- **C004 (뷰어)**: 독립 생성기 `steptree.py`가 steps.yaml→자기완결 HTML/SVG로 트리를
  그린다(backtrack 엣지 실좌표). **정직한 경계로 "실 gil 명령에 배선은 다음"**이라 명시.
- 두 부모의 다음-제안이 겹친다: **"뷰어를 실 명령으로"** + **"스텝=커밋 각인"**. C005가
  그 수렴점이다.
- chain.md 원칙: **스텝 = LLM의 진짜 행동 단위, 이 행동이 끝나야 깃을 커밋한다.** 즉
  v3에서 "스텝을 밟는다"는 곧 "커밋한다"여야 한다 — 아직 gilv3엔 그게 없다.

## 문제 분할

C005 "각인 + 배선"을 가장 작은 단위로:

1. **G1 — 스텝=커밋 각인**: `gilv3 step`이 steps.yaml·본문을 쓴 뒤 **그 스텝을 깃 커밋
   하나로** 남긴다(커밋 메시지에 스텝 id·kind·outcome). open은 루트 커밋, close는 봉인 커밋.
2. **G2 — 뷰어 배선**: `gilv3 view <dir>`가 C004의 steptree 로직으로 그 사이클 디렉토리의
   steps.yaml을 읽어 out.html을 낸다(독립 스크립트가 아니라 명령 표면으로).
3. **G3 — 커밋 계보 = 스텝 트리 대응 확인**: 스텝마다 커밋을 남기면, 깃 로그의 커밋
   순서가 steps.yaml의 id 순서(시간순)와 일치하는가. chain.md "스텝=커밋" 대응 실증.
4. **G4 — 되돌아감의 커밋 표현**: backtrack 스텝(analyze/backtrack)도 커밋 하나. 깃엔
   선형 히스토리로 남지만 steps.yaml의 `backtrack` 포인터가 논리 구조를 담는다(깃은
   시간순, 트리는 데이터 — C002 불변식의 깃 층 재확인).

**지금 정복할 첫 문제 = G1+G2 (각인 + 배선)**을 한 덩어리로. 이유: 둘 다 "gilv3를
독립 프로토타입에서 실제 도구로" 올리는 같은 카브다. C003 rebuild-case를 **각인 켠 채**
재실행해 커밋이 스텝마다 나는지(G1)와, 그 디렉토리에서 `gilv3 view`가 C004와 바이트
동일한 뷰어를 내는지(G2)를 한 검증에서 본다. G3·G4는 G1이 서면 커밋 로그 관찰로 공짜.

## 가설

> **가설**: `gilv3 step`에 **스텝 단위 깃 커밋**을 붙이고 `gilv3 view`로 C004 steptree
> 로직을 배선하면, C003의 실사례 재구성을 **각인 켠 채** 돌렸을 때 (a) 스텝마다 커밋이
> 하나씩 나 커밋 순서가 steps.yaml id 순서와 일치하고, (b) `gilv3 view`가 낸 out.html이
> C004 생성기 산출물과 동등(트리·엣지·잎 구별 보존)하며, (c) steps.yaml은 여전히 C002
> 스키마 그대로(깃 각인이 데이터 모델을 안 오염)다.

## 기각 조건

- **K1**: 스텝=커밋이 1:1이 아니다 — 한 step 명령이 커밋을 0개 또는 2개 이상 내거나,
  커밋 순서가 id 순서와 어긋난다 (각인 실패 → G1 재정의).
- **K2**: `gilv3 view`가 C004 뷰어와 다른 트리를 낸다 — 엣지·잎 구별이 깨지거나 배선이
  독립 생성기를 재현 못 한다 (배선 실패 → G2 재정의).
- **K3**: 깃 각인을 붙이려니 steps.yaml에 깃 메타(커밋 해시 등)를 넣어야 한다 — C002
  불변식(트리는 포인터만, 깃은 별개 층)이 깨진다 (모델 오염 → 데이터 모델로 되돌아감).
