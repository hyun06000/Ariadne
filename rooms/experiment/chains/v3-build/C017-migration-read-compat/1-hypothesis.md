# 1. 가설 — v3의 눈이 지문 없는 v2 커밋을 유령으로 건너뛴다 (읽기 호환)

부모: v3-build/C016-leaf-is-a-tag (supported — 잎=태그, 쓰기 축 다섯 조각 전부 도구 강제).
lineage: v3-build/C009-reconstruct-tree-from-git-log (깃 로그로 트리 재구성 원리).

## 이전 사이클의 교훈

C014~C016으로 gil v3의 **쓰기 축**(스텝종결·백트래킹·위계·잎·lineage)이 전부 도구 강제로 닫혔다. 이제 v3-build 체인의 궁극 목적지 중 하나 — **v2→v3 마이그레이션**(상현님 "v2 유저가 v3로 업글하면?") — 의 선행조건(checkout·태그 못·머지가 도구에 섬)이 충족됐다.

이 세션 마이그레이션 노트의 핵심 진단: **원장이 깨지는 게 아니라 v3의 눈이 과거를 못 읽는다.** v2 커밋에는 v3의 trailer 지문(Step-Id·Kind·Parent…)이 없어 v3 재구성기에게 **유령**이다. 완충 = 데이터 변환이 아니라 과거 커밋에 지문을 소급 각인. "깃도 옛 커밋에 서명 없어도 그래프가 읽히듯, 지문 없으면 덜 읽힐 뿐 파괴가 아니다."

## 문제 분할

마이그레이션은 4단계다: ① 동결백업 → ② **읽기호환** → ③ 소급각인 → ④ 백트래킹 복원. 각 독립안전·opt-in.

**가장 작은 첫 카브 = 읽기호환.** 이유:
- **모든 것의 토대.** 소급각인(③)·복원(④) 전에, v3 도구가 지문 없는 커밋을 만났을 때 **파괴 없이 우아하게 덜 읽는지**가 먼저 참이어야 한다. "지문 없으면 유령"이 *안전한* 유령(무해)임을 증명하는 게 첫걸음.
- **현재 코드에선 이 명제가 거짓이다.** C009 rebuild.py는 알 수 없는 커밋 subject를 만나면 `sys.exit`(즉사)한다. 지문 없는 v2 커밋을 만나면 v3 재구성기가 덜 읽는 게 아니라 **죽는다.** 마이그레이션 노트가 약속한 "덜 읽힐 뿐 파괴 아님"이 아직 구현 안 됨 — 이 카브가 그것을 참으로 만든다.

## 가설

> **가설(H1)**: v3 재구성기가 **trailer(Step-Id) 유무를 pre-gil 판별식**으로 삼아, 지문 없는 커밋을 유령(pre-gil)으로 **건너뛰고**(즉사 안 함), 지문 있는 v3 커밋만으로 스텝 트리를 복원하면, v2 커밋이 섞인 원장에서도 v3의 눈이 **파괴 없이 덜 읽는다**(마이그레이션 노트의 "유령 무해" 약속을 코드가 지킨다).

- **H1a 무해한 유령:** 지문 없는 커밋(v2 스타일)이 원장 앞에 쌓여 있어도 재구성기가 죽지 않고, v3 커밋만으로 온전한 트리를 복원한다.
- **H1b 경계 보존:** 유령 커밋은 삭제·변조되지 않는다 — 그냥 안 읽힐 뿐(rev-list --all에 그대로 생존, 깃 그래프 불변).
- **H1c 순수 v3 무회귀:** 유령이 하나도 없는 순수 v3 원장(C014~C016 케이스)에서 재구성이 이전과 동일(스킵 로직이 순수 경로를 안 건드림).
- **H1d 유령 가시성:** 재구성기가 몇 개의 유령을 건너뛰었는지 보고한다(침묵 스킵 금지 — 사용자가 "얼마나 덜 읽혔나"를 안다).

## 기각 조건

- 지문 없는 커밋에서 재구성기가 즉사하거나 트리가 깨진다. → H1a 기각.
- 유령 커밋이 삭제·이동·변조된다. → H1b 기각(마이그레이션의 "파괴 아님" 위반).
- 순수 v3 원장에서 재구성 결과가 C009/C014와 달라진다. → H1c 기각(회귀).
- 유령을 침묵으로 건너뛰어 사용자가 못 안다. → H1d 기각.

## 검증 설계 (다음 스텝)

pre-gil 커밋(v2 스타일 — trailer 없는 일반 커밋)을 앞에 쌓은 뒤 그 위에 v3 스텝 트리를 각인한 **혼합 원장**을 만들어 재구성기를 돌린다.
- M1 무해한 유령(H1a): 혼합 원장에서 재구성이 안 죽고 v3 트리 온전.
- M2 경계 보존(H1b): 유령 커밋이 rev-list --all에 생존, 깃 그래프 불변.
- M3 순수 v3 무회귀(H1c): 유령 0 원장(C016 케이스)에서 재구성이 C009와 동형.
- M4 유령 가시성(H1d): 재구성기가 스킵한 유령 수를 정확히 보고.

## 정직한 범위 (선긋기)

- **읽기호환만.** 소급각인(③, 유령에 지문 박기)·복원(④)은 다음 카브로 이월. 이 카브는 "유령이 무해한가"만.
- pre-gil 커밋은 v2 스타일 일반 커밋으로 모사(실제 v2 원장 189 cycle.yaml 전체 적용은 규모 확장, 다음).
- trailer(Step-Id) 유무를 판별식으로 — subject 파싱(C009)보다 근본적(C010 이후 trailer가 진실원). subject 기반 유령 판별은 안 다룸.
