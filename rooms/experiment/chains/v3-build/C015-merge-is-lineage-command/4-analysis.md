# 4. 결과 분석

## 측정 결과 요약

5측정 ALL PASS (measure-out.txt):

- **M1 다중부모** — 도구가 만든 close 머지 커밋의 부모가 정확히 두 산 잎(s4·s7), 첫 부모=s4(HEAD). `gilv3 close --lineage s4,s7`이 생 `git merge` 0으로 다중부모 커밋을 각인했다.
- **M2 trailer 복원** — Kind=merge·Parent="s4, s7"·Merge=lineage. 부모 지문(`git %P`)만으로 lineage DAG 복원(cycle.yaml 안 봄).
- **M3 append-only** — 머지 후 두 갈래 커밋 전부 생존(총 8커밋). `_assert_append_only` 통과.
- **M4 squash 음성대조** — `--squash`는 부모 1개 → lineage 소실. 도구의 `--no-ff`가 이를 막음.
- **M5 회귀** — `--lineage` 없는 close = C014 동형(부모 1개 선형 봉인, gil/live·gil/sealed 못 불변).

## 가설 판정

**H1 (lineage=머지=git merge --no-ff 도구 강제) — supported.** H1a(다중부모, M1)·H1b(trailer, M2)·H1c(append-only, M3) 전부 확인. H2(squash 음성대조, M4)·H3(회귀, M5)도 통과. 기각 조건 어느 것도 발생 안 함.

## 교훈

1. **⭐⭐ 도구가 아는 충돌은 이월이 아니라 자동 해소다.** C012는 독립 코드 파일만 고쳐 steps.yaml 충돌을 우회했다. 실제 gilv3 도구화에서 두 산 갈래는 **논리 트리 전체를 담는 단일 steps.yaml**을 각자 써 구조적으로 충돌한다 — 순수 깃엔 없던 문제. 그러나 그 충돌의 정답은 **gil이 이미 안다**(메모리의 완전 트리 = 두 갈래 합집합). C012 교훈 "충돌 해소 자체가 지식 통합의 판단"이 여기서 실현됐다: `--no-commit`으로 당긴 뒤 gil이 자기 트리로 steps.yaml을 해소하고 머지를 완성한다. **도구가 판단을 아는 충돌과, 모르는 충돌(같은 코드 영역=진짜 지식 충돌, 정직히 멈춰 이월)을 가른다.** 이것이 "충돌=지식통합 판단면"(C012 제안)의 첫 구획.

2. **⭐ multi_solution 도달 가능성이 lineage의 선행조건이었다.** 두 산 잎을 만들려면 첫 산 잎 뒤 백트래킹이 필요한데 C014는 live_leaf step을 전면 거부했다 → `cycle_state`의 multi_solution 분기가 **도달 불가능한 죽은 코드**였다. C014에서 append-only 정정이 백트래킹의 선행조건이었듯, live_leaf 백트래킹이 lineage의 선행조건. **원리를 도구로 옮기면 종종 앞 사이클이 남긴 도달불가 분기를 먼저 살려야 한다** — C014("종종 개념 정정")의 lineage판. 진짜 계약: 산 잎 뒤 선형 전진 금지(정답 도달)하되 새 형제 가지(다른 정답 탐색)는 chain.md 그리디의 실현.

3. **⭐ append-only 계약이 머지를 공짜로 허용했다.** C014가 정정한 "커밋 불소멸"(HEAD 전진이 아니라 도달가능성 단조)이 여기서 배당을 냈다 — 머지는 HEAD를 앞으로 옮기며 새 커밋만 추가하니 `_assert_append_only`를 손대지 않고 통과(M3). C010판 `_assert_forward_only`였다면 머지의 첫 부모 아닌 둘째 부모 병합을 어떻게 판정했을지 불명. **계약을 정밀화하면 다음 연산이 공짜로 맞아떨어진다** — checkout(C014)과 merge(C015)가 같은 집행기 아래 나란히 선다.

4. **재사용이 얇은 껍질을 확인했다.** lineage 머지는 백트래킹의 인프라(`_commit_of_sid`·`_assert_append_only`·산 잎 못 `gil/live/`)를 그대로 재사용했다 — 새 헬퍼는 `git_merge_lineage` 하나. C012 교훈 "gil의 모든 구조가 깃 네이티브"가 도구 코드량으로도 나타난다: 합류를 얹는 데 든 코드는 분기 위의 얇은 층.

## 정직한 경계

- **2부모 머지만.** `--lineage`는 콤마 리스트로 3+ 부모를 자연 확장하나 측정은 2부모(C012 규모). 중첩 머지(머지 노드를 또 머지)·체인 간 머지 미검증.
- **충돌 해소는 steps.yaml만 자동.** 같은 코드 영역의 진짜 지식 충돌은 정직히 멈춰 이월(C012 제안 "충돌 해소 각인"). 이 사이클은 자동 병합되는 갈래(steps.yaml만 충돌)로 한정.
- **잎=태그는 여전히 ◐.** 브랜치 못(`gil/live/`·`gil/dead/`·`gil/sealed/`)이 태그 아님(C011 결론=태그). 이 사이클은 lineage에 집중(그리디).
- **multi_solution 봉인의 의미.** lineage 머지로 닫힌 사이클은 두 산 잎을 통합한 것 — chain.md는 "여러 정답 중 선택은 새 사이클"이라 하나, lineage 머지는 그 선택을 "둘 다 통합"으로 각인한다. 이 의미 구분(선택 vs 통합)은 후속 관찰 대상.
- **계측기 결함 0.** 이번엔 build_case가 두 번째 실행에서 바로 통과(첫 실행은 steps.yaml 충돌을 잡아 설계를 정정 — 반증 아니라 도구 갭 발견).
