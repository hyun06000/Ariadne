# 2. 실험 설계

## 상현님 확정 방향 (설계 2연속 컨펌)

- 범위 = **v2 21항목을 v3로 교체** (게이트 주입 아니라 계약 이전).
- 철학 = **"v2를 완전히 버린다. 다만 v2를 썼던 레포에서는 v3로 정당하고 스무스하게 시작할 수 있도록 안내한다. 모든 건 v3 위주로. 겁내지 말고."**

## 핵심 통찰 — "OPEN을 v3로 교체"는 대부분 "교체할 대응 계약이 없다" (코드 실측)

conformance OPEN-* 16항목과 v3 open의 실제 계약을 코드로 대조 (C033 s2):

| v2 OPEN-* 항목 | 검사하는 v2 계약 | v3 open 대응물 |
|---|---|---|
| OPEN-CREATE | cycle.yaml + 1-hypothesis 스캐폴딩 | **다름** — steps.yaml + 루트 define s1 |
| OPEN-INCREMENT | 번호 C001→C002 자동증가 | **없음** — 경로가 정체(번호 개념 없음) |
| OPEN-REJECT-SLUG | 슬러그 형식 규칙 | **없음** — dir 경로엔 슬러그 규칙 없음 |
| OPEN-AUTHOR-REQUIRED | --author 필수(§3.2 P1) | **없음** — v3 open은 저자를 안 받음 |
| OPEN-PARENT-REQUIRED / -CLOSED-GATE | 부모 닫힘 게이트(C097) | **없음** — v3 사이클 간 부모는 notes/cycle.yaml 층 |
| OPEN-NEW-ROOT / -ROOT-CONFLICT | --new-root 계약 | **없음** — v3 루트는 s1 define 자체 |

**v3 open은 "빈 디렉토리에 루트 define s1 생성" 하나뿐**(gil.py 6273–6286 실측): 저자·부모·번호·슬러그 계약이 없다. v2 open이 사이클 *간* 계보·번호를 다뤘다면, v3 open은 사이클 *내* 스텝 트리의 시작만 다룬다. 사이클 간 계보(부모·번호)는 v3에서 notes 층(migrate)·cycle.yaml로 이동했다.

**결론**: "16항목 v3 교체"는 대부분 **v3에 대응 계약이 없어 1:1 교체가 불가능**하다. 이는 C032 "인터페이스 정체성 전환"의 conformance 판(반증 아니라 분할 경계의 실증). "v2를 버린다"는 상현님 방향은 옳다 — 그러나 v2 계약 중 **v3에 대응물이 있는 것만 v3로 옮기고, 대응물이 없는 것은 v3 모델에서 그 계약이 어디로 갔는지 명시**하는 게 정직한 이전이다.

## C033이 실제로 검증하는 것 (가장 작은 조각)

전체 21항목 교체는 v3 쓰기 계약 전면 설계를 요구(백트래킹·죽은 잎·close·사이클 간 부모의 v3 표현). 한 사이클엔 크다. C033은:

- **A. V3-OPEN-CREATE 신규** — `gil v3 open <dir>` → steps.yaml + 루트 define s1 + steps/s1.md 생성. v3 open의 핵심 계약을 판정기에 세운다.
- **B. V3-OPEN-REJECT-EXISTING 신규** — 이미 steps.yaml 있는 dir에 v3 open → 거부(빈 사이클에만). v3 open의 유일한 거부 계약.
- **C. V3-RETIRE-GUIDANCE 신규** — v2 습관 `gil open <chain> <slug>`(게이트 없이) → 친절한 은퇴 안내(exit≠0, v3 방식 명시). C032 승격을 판정기가 검사.
- **D. 정직한 이월 명시** — OPEN-INCREMENT·AUTHOR·PARENT-GATE 등 사이클-간 계약이 v3에서 어디로 갔는지 문서화(교체 아니라 이전 경로 명시). GUARD 이전·전체 v3 쓰기 계약은 후속 카브.

즉 C033은 **v3 쓰기 계약의 첫 판정 항목 3개를 세우고**(v2 삭제가 아니라 v3 신규로 시작 — 안전한 순증), 나머지 v2 계약의 v3 행방을 정직히 매핑한다. "겁내지 않되(v3 계약을 실제로 세운다) 작은 확실함으로(순증부터, 삭제는 대응 검증 후)."

## 절차

1. **baseline** — 현 conformance 결과(게이트 상속 시 134/134) 기록.
2. **v3 계약 항목 신규 구현** — conformance.py에 V3-OPEN-CREATE·V3-OPEN-REJECT-EXISTING·V3-RETIRE-GUIDANCE 3항목 추가 (`impl.run(cwd, "v3", "open", dir, ...)` / `impl.run(cwd, "open", ...)` 은퇴 안내).
3. **v2→v3 계약 행방 매핑 문서** — 21항목 중 대응물 없는 것의 v3 위치를 4-analysis에 표로.
4. **측정** — 신규 3항목 PASS, 기존 134 무회귀(게이트 상속 시), 게이트 없이 v3 항목만이라도 초록.

## 준비물

- 배포판 `gil.py`(C032 승격됨)·`conformance.py`.
- Python3 stdlib. 격리 샌드박스(make_sandbox 재사용).

## 측정 방법

- **M1 (V3-OPEN-CREATE)**: 신규 항목이 `gil v3 open`을 호출해 steps.yaml·루트 define·s1.md 생성을 검사, PASS.
- **M2 (V3-OPEN-REJECT-EXISTING)**: 이미 있는 사이클에 v3 open → 거부 검사, PASS.
- **M3 (V3-RETIRE-GUIDANCE)**: v2 습관 open(게이트 없이) → 은퇴 안내(exit≠0) 검사, PASS. 기각조건 3: 게이트 환경서 안내 안 나오면 반증.
- **M4 (무회귀)**: 신규 3항목 추가로 기존 134 항목이 안 깨짐(게이트 상속 시 총 137/137). 기각조건 4: baseline보다 초록 줄면 회귀.
- **M5 (v3 항목 게이트-독립)**: 게이트 없이 돌려도 신규 v3 3항목은 초록(v3 계약은 게이트 불요). v2 계약 항목은 여전히 게이트 필요(정직히 이월).

## 사용자 컨펌

설계 단계에서 상현님께 **2연속 컨펌**:
- 형태 = v2 21항목을 v3로 교체
- 철학 = v2 완전 폐기 + 스무스한 v3 시작 안내, "겁내지 말고"

내가 정직히 좁힌 것: "16항목 v3 교체"는 대부분 v3에 대응 계약이 없다(코드 실측). C033은 v3 계약 항목을 **신규로 세우고**(안전한 순증) 나머지 v2 계약의 v3 행방을 매핑. v2 삭제는 대응 검증 후 후속 — 상현님 "겁내지 말고" 방향(v3 계약 실제 수립)을 실현하되 "작은 확실함"으로 순증부터.
