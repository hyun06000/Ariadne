# 1. 가설 수립

## 이전 사이클의 교훈

부모: **v3-build/C032** (gil open을 v3로 승격 — 버전리스 본체).

C032가 찍은 좌표: **conformance v3 재정의가 다음 필수 관문.** conformance 128항목 중 **21개(OPEN-\* 16 + GUARD-\* 5)가 v2 open 인터페이스(`gil open <chain> <slug>`)를 직접 실행**한다. 게이트(`GIL_V2_OPEN=1`) 없이는 첫 open에서 crash — v2 open을 은퇴시키면 판정기가 그 자리에서 깨진다. 완전 버전리스(게이트 없이 초록)는 conformance를 v3 계약으로 재정의해야 완성.

## 상현님 방향 (설계 2연속 컨펌)

- **범위**: v2 21항목을 **v3로 교체** (게이트 주입이 아니라 계약 이전).
- **철학**: **"v2를 완전히 버린다. 다만 v2를 썼던 레포에서는 v3로 정당하고 스무스하게 시작할 수 있도록 안내한다. 모든 건 v3 위주로. 겁내지 말고."**

이 방향의 정확한 해석:
- "v2를 버린다" = v2 open **계약 검사**를 판정기에서 버린다 (OPEN-* 16항목).
- "스무스한 시작 안내" = v2에서 온 레포는 migrate(눈)+온보딩(C032 은퇴 안내)으로 v3로 부드럽게. 판정기가 그 **안내 자체**를 검사(V3-RETIRE-GUIDANCE).
- **GUARD-\* 5항목은 v2 계약이 아니라 C050 병렬 안전 가드**(주 체크아웃 오염 방지) — 버전 무관하게 살아야 한다. v3 open에 가드가 뚫리는지 검증으로 **이전**(삭제 아님). 상현님 "스무스한 시작 안내"가 곧 안전의 존치.

## 문제 분할

"conformance v3 재정의"를 가장 작은 단위로:

1. **OPEN-\* 16항목을 v3 계약으로 교체** — v2 `gil open <chain> <slug>`(번호·cycle.yaml) 검사를 v3 `gil v3 open <dir>`(steps.yaml·루트 define s1) 검사로.
2. **V3 쓰기 계약 신규** — step kind 순환(define→hypothesis→verify→analyze)·백트래킹(--outcome backtrack --to)·죽은 잎·close(산 잎 solved) 검사.
3. **V3-RETIRE-GUIDANCE** — v2 습관 `gil open`이 친절한 은퇴 안내를 내는지(C032 승격) 검사.
4. **GUARD-\* 5항목 v3 이전** — C050 안전 가드가 v3 open에도 살아 뚫리지 않는지.
5. **게이트 없이 초록** — 재정의 후 conformance가 `GIL_V2_OPEN` 게이트 없이 완전 통과.

**C033이 정복할 첫 번째 문제**: 이 재정의는 크다. 한 사이클에 21항목 전부 + v3 쓰기 계약 신규를 다 밀면 검증이 흐려진다. C033은 **가장 작은 검증 가능한 조각** — OPEN-* 핵심(생성·루트 define)을 v3로 교체 + V3-RETIRE-GUIDANCE 신규 + 게이트 없이 초록 회복을 실증한다. 전체 v3 쓰기 계약(백트래킹·죽은 잎)·GUARD 이전 완결은 정직히 이월. 겁내지 않되(상현님) 작은 확실함으로(Clew).

## 가설

> **가설**: conformance의 OPEN-\* v2 계약 항목을 v3 계약(`gil v3 open` → steps.yaml 루트 define)으로 교체하고 v2 은퇴 안내 검사(V3-RETIRE-GUIDANCE)를 신규하면 — 그리고 C050 안전 가드를 v3 맥락에 존치하면 — 판정기가 **게이트(`GIL_V2_OPEN`) 없이** v3 계약을 판정하는 방향으로 이동한다. "v2를 버린다"는 v2 계약 검사를 버리는 것이지 병렬 안전을 버리는 게 아니다.

## 기각 조건

1. OPEN-* 교체 후에도 다른 항목(close·step·fsck 등)이 여전히 v2 open 산출물(cycle.yaml·5문서)에 암묵 의존해 게이트 없이 crash → OPEN만 교체로 불충분, 판정기의 v2 결합이 open보다 깊음(재설계·범위 재조정).
2. v3 open이 C050 안전 가드를 아예 안 가져 GUARD 이전이 "검증"이 아니라 "신규 구현"이 됨 → 이 조각의 경계를 넘어섬(GUARD 이전을 별도 카브로).
3. V3-RETIRE-GUIDANCE가 C032 은퇴 안내를 검사하는데 게이트 환경에서 안내가 안 나옴(게이트 상호작용) → 승격과 판정의 결합 오류.
4. OPEN 교체가 오히려 다른 v2 항목을 깨 총 초록 수가 baseline(134)보다 줄고 회복 불가 → 재정의가 순증이 아니라 회귀.
