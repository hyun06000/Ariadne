# 4. 분석

## 판정: supported (채택)

5측정 ALL PASS. 백트래킹=checkout이 순수 깃 원리(C011)에서 **gilv3.py 명령 동작**으로 승격됐다. 도구가 open/step/close만으로 C011 build_branches.sh가 손으로 짠 3층 분기를 자동 생성하고, C008의 forward-only 집행이 C011 모델로 정직하게 정정됐다.

## 무엇이 증명됐나

1. **도구가 진짜 분기를 그린다** (M1) — 루트 s1 커밋이 자식 3개(세 형제 가지). C010판은 선형으로 쌓아 이걸 못 했다. 이제 `git log --all --graph`가 셸 스크립트 없이 gilv3 명령 산출물로 분기를 그린다.
2. **논리 트리 == 깃 그래프** (M2) — steps.yaml의 parent/backtrack 포인터와 깃 커밋 부모 그래프가 동형. 백트래킹 노드는 checkout으로 깃 부모가 조상 define이 된다 — steps.yaml이 담은 논리가 깃 위상으로 정확히 번역된다.
3. **trailer 복원 무손상** (M3) — checkout 분기가 생겨도 자기완결 trailer(C010)가 `--all`로 복원되면 원 트리와 동형. 백트래킹이 C009·C010 복원 계약을 안 깬다.
4. **C008이 C011로 정정됐다** (M4) — append-only의 진짜 계약이 "HEAD 전진"이 아니라 "커밋 불소멸"임을 코드가 집행한다. checkout(HEAD 뒤로) 통과, reset --hard(커밋 삭제) 거부.
5. **회귀 0** (M5) — 선형 사이클은 C010과 동일. 백트래킹 없는 경로에 checkout이 안 끼어든다.

## ⭐ 핵심 교훈

1. **append-only의 진짜 계약은 커밋 불소멸이지 HEAD 전진이 아니다.** 이것이 이 사이클의 정점이고, 왜 자동화가 두 사이클 이월됐는지의 답이다. C008이 "깃=전진기록"을 세울 때 forward-only 집행이 **checkout(HEAD 뒤로, 커밋 보존)** 과 **reset(커밋 삭제)** 을 구별 못 했다. 그래서 C011의 백트래킹=checkout이 자기 자신의 집행기에 막혔다. 정정의 열쇠는 기능 추가가 아니라 **계약을 정밀화**하는 것이었다 — "무엇을 지키려 했나"(커밋을 안 지움)로 돌아가니, checkout은 그 계약을 지키고 있었다. **원리를 도구로 옮기는 일은 종종 코드 작성이 아니라 개념 정정이다.**

2. **잎에는 반드시 못이 필요하다 — 산 잎도(C011 재확인, 도구가 재발견).** checkout으로 분기하면 떠나는 가지의 팁이 ref 없으면 dangling→gc된다. C011은 이걸 "죽은 가지" 관점에서 발견했는데, 이 사이클은 **산 가지와 close 커밋도 같은 운명**임을 도구 구현 중 재발견했다(M4 초기 FAIL). 셋 다 못(`gil/dead/`·`gil/live/`·`gil/sealed/`)이 필요하다 — checkout이 결국 그 자리를 떠나기 때문. **"모든 잎에 못"은 C011 build_branches.sh가 모든 잎에 태그를 박은 것의 필연을 도구가 재유도한 것.**

3. **닫힌 사이클의 산출물 불변이 재사용을 강제한다 — 그래서 정직하게 우회한다.** C010 rebuild_trailer가 `--all` 없어 분기를 못 읽었지만, C010은 닫힌 사이클이라 못 고친다. 정직한 길은 그 파서 로직을 `--all`로 **재호출**(C010 원본 불변, 로직 재사용)이었다 — 이것이 C011·C012가 예고한 "재구성기를 git log --all 기반으로 재배선" 이월의 첫 국소 실현. **불변 규율이 코드 복제가 아니라 로직 재사용을 강제한다.**

## 도구화 진척 — 깃 ≅ gil 완성표의 실장 상태

| gil 개념 | 깃 네이티브 | 원리 증명 | **도구 실장** |
|---|---|---|---|
| 스텝 종결 | 1 커밋 | C005·C011 | ✅ C005~ |
| 위계 | trailer 지문 | C010·C011 | ✅ C010 |
| **백트래킹** | **checkout + detached** | **C011** | **✅ C014 (이 사이클)** |
| 잎 | 태그(못) | C011 | ◐ C014(브랜치 못, 최소형) |
| lineage | 머지 다중부모 | C012 | ⬜ 다음 |
| 뷰어 | git log --all --graph | C011·C012 | ⬜ 다음(재배선) |

**백트래킹이 실장되며 도구가 처음으로 깃 그래프에 진짜 분기를 냈다.** 이게 나머지(잎 태그 정식화·lineage 머지)의 토대 — 분기가 있어야 잎과 통합이 의미를 갖는다.

## 정직한 경계

- **잎 못이 태그 아니라 브랜치다.** C011은 잎=**태그**로 결론지었다(불변 시점 = 못박음, push 생존). 이 사이클은 최소 형태로 **브랜치 ref**(`gil/dead/`·`gil/live/`·`gil/sealed/`)를 썼다 — "어떤 ref든 못이면 가지가 산다"는 C011 M3 실측의 최소 충족. **태그로의 정식화, 해시 이름 규칙(C011 상현님 규칙: 스텝 분기=해시), push 생존은 잎=태그 자동화 사이클(다음)의 몫.** 브랜치는 움직이는 포인터라 잎(불변)에 의미상 부정확 — 정식판은 태그여야.
- **rebuild `--all` 재배선은 측정 안 국소 적용**이지 rebuild_trailer.py 자체 갱신이 아니다. 정식 v3 재구성기/뷰어를 `--all` 기반으로 만드는 건 이월(C011·C012 예고).
- **lineage=머지·잎=태그 미실장** — 이 사이클은 백트래킹 하나에 집중(철학 충돌이 거기 있었으므로). 그리디.
- **실사례 최소** — C011과 동일(3층·백트래킹 2·잎 3). 중첩 백트래킹, BFS(가지 병행), 다중 루트, 대형 트리 성능 미검증.
- **detached HEAD 잔여** — close 후 HEAD가 detached(사이클 종결 못은 있으나). 실제 gil 워크플로에서 다음 사이클 open이 이 상태를 어떻게 잇는지는 미탐구(체인 연속성).
- **파일명 gilv3.py 유지** — C010 규율(프로토타입 단계). 배포 시 gil 본체 통합 명명은 나중.
