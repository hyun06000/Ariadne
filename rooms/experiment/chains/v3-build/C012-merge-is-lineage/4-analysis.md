# 4. 분석

## 판정: supported (채택)

4측정 ALL PASS. 가설의 네 주장이 모두 순수 깃으로 실증됐다:

1. **깃이 통합을 다중부모 엣지로 담는다** (M1) — 머지 커밋 `cat-file -p`가 `parent` 줄 2개를 담고, 그 순서가 gil `[C020, C016]`과 대응(첫 부모=HEAD=C020, trailer `Parent: C020, C016`). **gil의 lineage 리스트 = 깃 머지 커밋의 부모 리스트.** 별도 필드 불필요.
2. **`git log --graph`가 병합 노드로 그린다** (M2) — in-degree 2 합류. gil log `◀ 병합:`과 위상 동형. 뷰어는 아무것도 새로 안 그린다.
3. **부모 지문만으로 lineage DAG를 재구성한다** (M3) — `%H %P`만 읽어 cycle.yaml 없이 복원, C036이 C016·C020 두 부모임을 정확히. C009(분기 재구성)의 합류판.
4. **두 계보가 실제로 합류한다** (M4) — 공통조상 s0, 두 갈래 코드 모두 생존, 진짜 머지, 그리고 **squash로는 안 됨**(음성대조).

## 무엇이 증명됐나 — 깃 ≅ gil의 마지막 조각

C011이 위계·잎을 깃(지문·태그)에 녹였을 때, gil에 남은 마지막 "별도 메타데이터"가 **다중부모 lineage**였다. C012가 그것을 깃 머지 커밋에 녹였다. 이제 gil의 모든 구조가 깃 네이티브다:

| gil 개념 | 깃 네이티브 표현 | 증명 |
|---|---|---|
| 스텝 종결 | 1 커밋 | C011 |
| 백트래킹 | `git checkout <조상>` + detached 커밋 | C011 |
| 위계(체인⊃사이클⊃스텝) | 커밋 trailer 지문 | C010·C011 |
| 잎(죽은/산) | 태그(못) | C011 |
| **lineage(지식 통합)** | **머지 커밋의 다중 부모** | **C012 (이 사이클)** |
| 뷰어 | `git log --all --graph` + 메시지 | C011·C012 |

**깃의 두 기본 연산이 gil의 두 기본 연산을 완전히 덮는다:**
- 깃 **분기**(갈라짐) = gil **백트래킹**(C011)
- 깃 **합류**(머지) = gil **지식 통합**(C012)

깃 ≅ gil이 분기·합류 **양방향으로 닫혔다.** 상현님이 C011을 닫으며 던진 씨앗이 정확히 이것이었다.

## 핵심 교훈

1. **lineage는 별도 필드가 아니라 깃 머지 커밋이다.** gil v2는 `parent: [A, B]`를 cycle.yaml 리스트로 관리하고 loom/C001이 그 필드로 재구성을 증명했다. 하지만 그것은 gil이 **떠받쳐야 하는** 메타데이터였다. 깃 머지 커밋의 다중 부모는 gil이 아무것도 안 해도 깃이 공짜로 담고, 그리고, 재구성 가능하게 한다. **C011의 "얇은 껍질" 논리의 완성** — lineage조차 껍질 밖(깃 안)에 있다.

2. **다중부모여야 lineage가 보존된다** (M4 음성대조의 핵심). squash 머지는 두 갈래의 *코드*는 합치지만 *계보*를 잃는다(부모 1개). 지식 통합의 본질은 결과물이 아니라 **"어느 두 계보에서 왔는가"의 추적 가능성**이다. 그래서 gil lineage는 반드시 `--no-ff`/다중부모 머지여야 한다 — fast-forward나 squash는 통합의 결과만 남기고 통합의 *역사*를 지운다. **이것이 코드 병합(결과 중심)과 지식 병합(계보 중심)의 결정적 차이다.**

3. **분기와 합류는 한 데이터 모델의 두 방향이다.** C011의 backtrack(하나가 여럿으로)과 C012의 merge(여럿이 하나로)는 같은 커밋 DAG의 역방향 연산이다. C009의 재구성이 분기에 성립했듯(M3) 합류에도 그대로 성립한 것은 우연이 아니다 — 부모 지문(`%P`)이 두 방향을 대칭으로 담기 때문이다. **한 데이터 모델(커밋 DAG + trailer)이 뿌리면 분기·합류·재구성·뷰어가 한 규칙에서 파생된다** (C009의 "읽기·쓰기가 한 상태기계"의 lineage판).

## 정직한 경계

- **순수 깃으로 원리만 증명** — gilv3.py를 안 고쳤다(C011의 그리디 리듬 계승). 실제 `gil close --lineage A,B`가 머지 커밋을 만드는 자동화는 다음 사이클 몫.
- **실사례 최소** — 2부모 머지 1개. 3+ 부모(옥토퍼스 머지, 실제 gil엔 드묾), 체인 간 머지(여기선 단일 체인 v3-demo 안 두 사이클), 중첩 머지(머지 노드를 또 머지)는 미검증.
- **머지 충돌 미검증** — 두 갈래가 다른 파일(ledger.py·web.py)을 고쳐 자동 병합됐다. 같은 파일 같은 줄을 고친 두 계보의 통합(실제 지식 통합에서 흔할 수 있음)은 충돌 해소가 필요하고, 그 해소 자체가 통합의 판단을 담는다 — 미탐구.
- **loom/C036 대칭은 축약** — 실제 C036은 각 부모가 여러 사이클을 거친 깊은 계보다. 여기선 각 갈래 2스텝으로 축약. 위상은 동형이나 규모는 최소.
- **코드 아티팩트 통합의 깊이** — M4는 "두 파일이 모두 존재"까지 봤다. 실제 지식 통합은 두 계보의 *교훈*이 코드에서 상호작용하는 것(예: C020의 web이 C016의 ledger를 호출)까지 갈 수 있다. build_merge.sh의 integrated.py가 그 방향을 살짝 보였으나(둘 다 호출) 깊은 통합은 미탐구.
