# 1. 가설 수립

부모: v3-build/C011-everything-is-a-branch (supported). lineage: loom/C036-go-open-git-ledger (실제 병합 노드 `[C020, C016]` — 이 사이클의 실증 표적).

## 이전 사이클의 교훈

C011은 **깃 데이터 모델 ≅ gil 위계 모델**을 확정했다. 그 매핑:

- 백트래킹 = `git checkout <조상>` + detached HEAD 커밋 (분기 자연 발생, 명시적 브랜치 0)
- 1스텝 종결 = 1커밋
- 위계(체인⊃사이클⊃스텝) = 커밋 메시지 trailer 지문 (`Chain`·`Cycle`·`Step-Id`·`Parent`)
- 잎 = 태그(못) — 죽은 가지도 태그가 있어야 `git log --all`에서 생존
- 뷰어 = `git log --all --graph` + 커밋 메시지 (아무것도 새로 안 그림)

**그러나 C011은 깃의 두 기본 연산 중 하나만 증명했다 — 분기(갈라짐).** 상현님이 닫으며 던진 씨앗이 나머지 반쪽이다:

> "깃 머지로 lineage까지 다시 생각. 사이클간·체인간 지식 통합."

깃의 두 번째 기본 연산은 **합류(머지)** 다. 머지 커밋은 **다중 부모**를 가진다 — 이것이 정확히 gil의 **lineage**(지식 통합)다.

## 문제 분할

C011의 매핑에서 아직 깃 네이티브로 녹지 않은 마지막 gil 필드가 하나 있다: **`parent: [C020, C016]` 같은 다중부모 lineage.** 지금 gil은 이것을 cycle.yaml의 리스트 필드로 관리하고, `gil log`가 `◀ 병합:` 으로 그린다(loom/C036 실증). 즉 lineage는 여전히 **gil이 별도로 관리하는 메타데이터**다.

C011의 논리를 끝까지 밀면 — 위계가 커밋 지문에 살고 잎이 태그에 살듯 — **lineage도 깃이 이미 가진 것(머지 커밋의 다중 부모)에 살아야 한다.** 별도 필드가 아니라.

풀 문제를 분할하면:

1. **[이번 정복] gil의 다중부모 lineage가 깃 머지 커밋과 동형인가** — `parent: [A, B]`를 `git merge`의 다중부모로 표현하면, 별도 lineage 필드 없이 깃이 그것을 담고 재구성 가능한가.
2. lineage 재구성이 깃 머지 그래프만으로(cycle.yaml 안 보고) 되는가 — C009가 "깃 로그로 트리 재구성"을 분기에 대해 증명했듯, 합류에도 성립하는가.
3. (이월) 실제 gilv3 명령 자동화 — `gil close --lineage A,B`가 머지 커밋을 만들게.

**1을 먼저 정복하는 이유**: 이것이 상현님이 지목한 1순위이고, C011의 정확한 대칭이다. C011이 분기를 증명했으니 합류를 증명하면 **깃 ≅ gil이 분기·합류 양방향으로 닫힌다** — 깃의 두 기본 연산이 gil의 두 기본 연산(백트래킹·지식통합)을 완전히 덮는다. 순수 깃으로 원리만 증명(gilv3.py 안 고침, 자동화는 다음 — C011의 그리디 리듬 계승).

## 가설

> **가설**: gil의 다중부모 lineage(사이클 간·체인 간 지식 통합, 예: loom/C036의 `parent: [C020, C016]`)를 **깃 머지 커밋(다중 부모)** 으로 표현하면 — 별도 lineage 필드 없이 — ① 깃이 그 통합을 다중부모 엣지로 담고, ② `git log --graph`가 그것을 병합 노드로 그리며(뷰어 재구현 0), ③ 커밋 부모 지문만으로(cycle.yaml 안 보고) lineage 그래프를 정확히 재구성할 수 있고, ④ 통합된 두 계보의 코드/교훈이 머지 커밋에서 실제로 합류한다(git merge-base로 공통 조상 추적 가능). 그리하여 **깃 ≅ gil이 분기(C011)와 합류(C012) 양방향으로 닫힌다.**

## 기각 조건

이 중 하나라도 나오면 가설이 틀렸다:

- **M1 기각**: 다중부모 머지 커밋이 `git cat-file -p`에서 두 부모를 모두 담지 못하거나, gil의 `[A, B]` 순서/의미와 깃 부모 순서가 대응되지 않는다.
- **M2 기각**: `git log --all --graph`가 병합 노드를 다중부모 합류로 그리지 못한다(gil log의 `◀ 병합:` 표현과 위상이 다르다).
- **M3 기각**: 커밋 부모 지문만으로 lineage 그래프를 재구성했을 때 실제 lineage와 위상이 다르다(다중부모를 놓치거나 없는 엣지를 만든다). 또는 재구성이 cycle.yaml 같은 gil 메타데이터를 읽어야만 성립한다.
- **M4 기각**: 두 계보의 합류가 이름뿐 — `git merge-base`가 공통 조상을 못 찾거나, 머지 커밋이 두 부모 중 한쪽 코드만 담아 "통합"이 허상이다. 또는 fast-forward로 접혀 다중부모 커밋 자체가 안 생긴다.
