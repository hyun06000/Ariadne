## 데이터 직접 관찰

수치 뒤로 들어가 out.html의 실제 마크업을 들여다봤다.

- **backtrack 엣지가 진짜 진실을 담는다.** s7 노드의 SVG:
  `<path class="edge-backtrack" ... data-from="s7" data-to="s1" data-y-from="587.0"
  data-y-to="215.0" marker-end="url(#bt-arrow)"/>`. y가 587→215로 **위로** 간다 —
  잎(depth 3)에서 뿌리 define s1(depth 0)로 되돌아감. C002가 "v2엔 이 엣지가 아예
  없었다"고 한 바로 그 엣지가, 실제 렌더 좌표로 화면에 존재한다. s7 본문이 스스로
  "벽의 지도: 폴링 이분은 실브라우저 계측기 필요 — 이 길도 막힘"이라 부른 그 되돌아감을,
  뷰어가 곡선 한 가닥으로 비춘다.

- **죽은 잎과 산 잎이 클래스로 갈린다.** s4·s7 `<g class="node kind-analyze leaf-dead">`
  (점선 테두리·바랜 opacity·"✕ 죽은 잎 · 벽의 지도" 배지), s10 `<g class="node
  kind-analyze leaf-live">`(굵은 초록 테두리·채운 바탕·"✓ 산 잎 (success)" 배지). 셋 다
  kind=analyze지만 outcome이 운명을 갈랐다 — v2 고정 5스텝은 outcome 개념 자체가 없어
  이 구별이 불가능했다.

- **세 형제 가지가 부채꼴로 갈린다.** s1 아래 s2(col 0)·s5(col 1)·s8(col 2)가 depth 1에
  나란히, 각자 아래로 verify→analyze로 선형 하강. col 배치가 leaf 순서 기반이라 세 가지가
  겹치지 않고 한눈에 들어온다.

## 예상과 달랐던 것

- **범례의 정직함이 측정을 한 번 깨뜨렸다.** 나는 범례 스와치에 실제 엣지와 **같은
  class**를 일부러 썼다 — 사람이 범례에서 본 선과 그래프의 선이 동일해야 하니까. 그런데
  그 정직함이 measure.py를 속였다(범례 line을 3번째 backtrack 엣지로 셈). 교훈: **뷰어의
  "사람에게 정직한 표현"과 "측정자의 데이터 필터"는 다른 축이다.** 데이터 엣지는
  `data-from`을 갖고 장식 엣지는 안 갖게 해 둘을 갈랐다. 뷰어를 고친 게 아니라 측정자를
  고쳤다 — 출력은 처음부터 옳았다. 이건 내 정체성("보이는 것의 정직함")과 규율(실측)이
  충돌한 게 아니라, 실측이 "무엇을 세는지"까지 정직해야 함을 드러낸 것(C010의 귀속 정직
  재현: 실측은 "무엇의 탓/무엇을 셈"까지 판정한다).

- **정적 파서 실측이 CDP보다 나았다.** v2 뷰어는 라이브 폴링 상태를 CDP(실브라우저)로
  쟀지만, v3 스텝 트리의 최소 계약은 **정적 구조**라 stdlib html.parser로 충분했고 오히려
  재현성이 높았다(브라우저·드라이버 의존 0). 계약이 정적이면 측정도 정적으로 — 도구를
  문제에 맞춘다.

## 판정

**가설 채택 (supported).** 네 측정 전부 통과, 네 기각 조건 전부 미발동. v3 뷰어가 한
사이클 안의 스텝 트리를 그려 — (a) parent 가지 3개, (b) backtrack 되돌아감 2개(v2엔
없던 엣지), (c) 죽은 잎 2·산 잎 1의 구별 — 을 자기완결 SVG로 동시에 실현하고 헤드리스로
실증했다. 검증의 최소 기준(임무 명세)을 정확히 충족한다. 산 잎(가설 채택)에 도달 →
사이클을 닫을 수 있다.
