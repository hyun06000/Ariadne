# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C022**(마이그레이션 실제 원장 적용)가 마이그레이션 4단계를 전부 세우고, 절차를 실제 우리 원장에 되돌림 가능하게 적용·원격 push까지 완료했다. 그 실증된 절차는 지금 **사이클 산출물 스크립트**(`apply_migration.py`·`rollback_migration.py`·`full_ledger_migrate.py`·`splice_topology.py`)로 흩어져 있다.

C022의 이월 제안 4번: **"마이그레이션의 gil 명령화 — 쓰기 축 다섯 조각처럼 도구 강제로 올릴 수 있다(다른 사용자·다른 원장의 v2→v3 이주에 재사용)."**

핵심 배경: C014~C016이 쓰기 축(백트래킹·머지·잎=태그)을 흩어진 원리에서 **gil 명령으로 도구 강제**했다. 그때마다 교훈은 "원리를 도구로 옮기면 재사용 가능·판정 가능해진다"였다. 마이그레이션도 같은 승격을 기다린다.

## 문제 분할

`gil migrate` 명령군을 다음으로 나눈다:
1. **`gil migrate --dry`** — 드라이런: 도출·접합 카운트만 보고(각인 안 함). C020 full_ledger_migrate 드라이런의 명령판.
2. **`gil migrate`** — 적용: 동결백업 + 노드 소급 + 위상 접합. C022 apply_migration의 명령판.
3. **`gil migrate --rollback`** — 되돌림: 백업 ref로 리셋 / notes 삭제. C022 rollback_migration의 명령판.

**첫 번째로 정복할 것: 세 서브명령을 한 카브로.** 이유 — 마이그레이션은 왕복(dry→apply→rollback)이 한 계약이다. C022가 이미 세 절차를 실증했으니, 명령화는 "흩어진 검증된 스크립트를 한 도구 표면으로 통합"하는 작업이다. 분할하면 반쪽 명령이 된다.

## 무엇을 어느 몸에 넣는가 (설계 결정)

- **대상 도구: gilv3** (v3-build 실험의 v3 도구). 이유: 마이그레이션은 "v3의 눈으로 v2 원장을 읽는 것"(C017~C022)이므로 v3 도구의 명령이 자연스럽다. v2 gil.py(실사이클 운영)는 마이그레이션 대상이지 수행 주체가 아니다.
- **로직 재사용**: C020/C022의 검증된 함수(`derive_fingerprint`·`retro_imprint`·`splice_topology`·백업/되돌림)를 gilv3 안으로 통합하거나 import. 새 로직을 쓰지 않는다 — C022가 실증한 것을 그대로 명령 표면에 올린다.
- **안전 계약 보존**: C022의 불변 게이트(커밋·트리·cycle.yaml 불변)와 되돌림 가능성을 명령이 그대로 집행한다.

## 가설

> **가설**: C022가 실증한 마이그레이션 절차(동결백업 + 노드 소급 + 위상 접합 + 되돌림)를 gilv3의 `migrate` 서브명령군(`--dry`·적용·`--rollback`)으로 승격하면, 격리 복제본에서 (a) `gil migrate --dry`가 각인 없이 도출·접합 수를 보고하고, (b) `gil migrate`가 C022와 동일하게 노드·엣지를 refs/notes에 각인하며 커밋·트리·cycle.yaml을 불변으로 유지하고, (c) `gil migrate --rollback`이 notes를 완전 제거(잔재 0)한다 — 즉 흩어진 스크립트가 한 도구 표면으로 통합되면서 C022의 결과·안전 계약이 바이트 수준으로 보존된다.

## 기각 조건

- `gil migrate` 적용 결과가 C022 스크립트 결과와 다르면(노드 수·엣지 수·DAG 구조) → **기각**(명령화가 로직을 바꿈).
- `gil migrate` 후 커밋 SHA·cycle.yaml이 바뀌면 → **기각**(안전 계약 파괴).
- `gil migrate --rollback` 후 notes 잔재가 남으면 → **기각**(되돌림 계약 파괴).
- `gil migrate --dry`가 각인을 하면(notes 생김) → **기각**(드라이런 계약 파괴).
