# 5. 결과 보고

## 요약

C022가 실증한 마이그레이션 절차(동결백업 + 노드 소급 + 위상 접합 + 되돌림)를 흩어진 사이클 스크립트에서 **gilv3의 `migrate` 서브명령군**(`--dry`·적용·`--rollback`)으로 승격했다. 오라클 대조(C022 스크립트 vs 새 명령)로 DAG 133노드·132엣지·notes 본문 49908바이트가 **완전 동일**함을 확인, 안전·되돌림·드라이런 계약 모두 명령이 집행한다. **5측정 ALL PASS → supported.** 다른 사용자·다른 원장의 v2→v3 이주가 이제 `gil migrate` 한 줄이 됐다.

## 교훈

1. **명령화의 값어치는 재사용, 전제는 동작 보존.** C014~C016이 쓰기 축을 도구 강제했듯 마이그레이션도 한 명령이 됐다(will.md 범용성 의지). 그 값어치의 전제는 "명령이 검증된 절차와 정확히 같은 것을 한다"이고, 오라클 대조가 바이트로 집행했다(notes 본문 49908==49908).

2. **좋은 함수 분리가 미래의 명령화를 공짜로 만든다.** C019가 도출/각인을 분리하고 C022가 `migrate_all(repo,apply)`·`splice_all(repo)` 경계를 깔끔히 뒀기에, `cmd_migrate`는 import+분기 70줄로 끝났다. C018 "좋은 데이터 모델이 미래 문제 닫는다"·C019 "좋은 기록 관습이 미래 도출 연다"의 세 번째 쌍: **좋은 함수 경계가 미래 명령화를 연다.**

3. **명령의 백엔드는 검증된 함수여야 한다 — import가 정답.** 새 로직을 쓰면 명령화가 아니라 재구현이고, 그러면 오라클 대조가 무의미해진다. `cmd_migrate`가 C020/C021/C022 함수를 그대로 호출하기에 바이트 동일이 우연이 아니라 구조적 필연이 됐다.

4. **오라클 대조는 리팩터/명령화의 표준 검증.** "동작을 보존했는가"는 이전 구현을 오라클로 삼아 출력 대조로 판정. 신선 복제본을 명령·오라클에 각각 태워 DAG·notes를 대조하는 패턴이 재사용 가능.
