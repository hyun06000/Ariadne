# 5. 보고 — v3-build/C001-define-v3-target

부모: 없음 (v3-build 체인의 루트). 저자: Clew. 소환자: 없음 (main 단독).
판정: **supported (채택)**.

## 한 줄

v3-build의 **첫 정복 대상 = 데이터 모델(스텝 트리 표현)**임을 확정했다. 실제
사례 loomlight/C012→C013→C014(백트래킹 포함 리젝트 체인)를 최소 스텝 트리 모델로
재표현해 **v2의 왜곡(백트래킹을 선형으로 뭉갬)이 사라짐**을 실증하고, 그 모델이
open/step/close를 파생하는 뿌리임을 확인했다.

## 무엇을 했나

1. **v3-build 체인 목표 확정** (chain.md): v2로 v3를 짓는다 — 상현님과 확정한
   세 층 위상(체인=DAG, 사이클=스텝의 트리, 스텝=행동단위 with 백트래킹)을
   처음부터 담는 도구. 배포는 체인 끝에서 굵직하게.
2. **최소 스텝 트리 모델 스케치** (4-analysis): 스텝 노드 =
   `{id, kind(문제정의*/가설/검증/분석), parent, outcome(실패/되돌아감/성공),
   backtrack}`. 백트래킹 = 분석 노드의 `outcome=되돌아감` + `backtrack`이 가리키는
   조상 문제정의*에서 새 형제 가지.
3. **실사례 재표현 (M1)**: v2의 3 사이클(C012·C013·C014)이 v3에서 **1 사이클,
   하나의 스텝 트리**가 된다. P0(카드닫힘) 아래 P1·P2·P3 세 형제 가지 —
   P1·P2는 죽은 잎(`되돌아감`), P3만 산 잎(`성공`).

## 핵심 발견

- **v2 `parent`의 죄**: "시간 순서상 다음"과 "논리적 자식"을 구별 못 해,
  C013→뿌리→C014의 **백트래킹을 선형 계승 C013→C014로 뭉갰다.** v3 모델에서는
  C014(=P3)의 부모가 P3 자신의 문제정의이고, `backtrack=P0`가 "C013에서 자란 게
  아니라 뿌리로 되돌아가 낸 형제"라는 진실을 **명시적 데이터**로 담는다.
  *v2가 못 담던 것을 v3 모델이 담는다 — 이것이 첫 대상이 데이터 모델인 이유.*
- **죽은 잎은 벽의 지도**: C012·C013의 기각 사유(상호작용·아코디언·detKey 배제 /
  폴링 이분은 실브라우저 필요)가 `되돌아감` 잎에 보존돼, 후대가 그 길을 안 헤맨다.
  리젝트가 자산이 되는 상현님 방법론이 **데이터 구조로 각인**된다.
- **모델이 명령을 강제한다 (M2)**: `kind` 순환(문제정의→가설→검증→분석)과
  `outcome` 분기가 open/step/close를 파생한다 — 분석 없이 다음 문제정의로 못 가고,
  되돌아감은 조상 문제정의*를 가리켜야 한다. v2가 C090에서 애써 얹은 step-by-step
  강제가 **트리 구조 자체에서 자연히 나온다.**

- **이 사이클은 v2 고정 5스텝으로 진행됐다** — v3 스텝 트리는 아직 도구가 없다.

  첫 발이자, "도구는 아직 선형이지만 존재는 이미 트리로 생각한다"는 C001 가설의
  자기실증(직전 세션 C012→C014에서 내가 이미 스텝 트리로 사고한 것과 같은 결).

  스텝 DAG?)는 의도적으로 안 건드렸다(그리디 — C002의 몫). "더 나은 모델?"은
  되돌아가 새 사이클(최적화).
