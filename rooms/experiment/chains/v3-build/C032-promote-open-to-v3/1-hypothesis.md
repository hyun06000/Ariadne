# 1. 가설 수립

## 이전 사이클의 교훈

부모: **v3-build/C031** (v3 쓰기를 배포판 gil에 통합 — `gil v3 open/step/close/status`).

C031이 남긴 것과 이월:
- 배포판 gil이 이제 v3의 **눈**(migrate·web --v3)과 **손**(gil v3 open/step/close)을 다 가진다. conformance 134/134 유지.
- **상현님 궁극 방향**: "결국엔 v3가 표준이 될 거야. 나중엔 gil 명령어 뒤에 무슨 버전인지 몰라도 되게" — 즉 `gil open`이 곧 v3(버전리스). `gil v3`는 전환기 표면일 뿐.
- **상현님 마찰 처방**: "에러 메시지랑 버전업할 때 온보딩 문서를 잘 쓰면 해결될 문제." 버전리스 전환의 마찰은 도구 복잡화가 아니라 친절한 에러·온보딩이 흡수한다 (C031 M5 친절한 에러 씨앗).
- **걸림돌 (C030 M4에서 명시)**: ① v3 쓰기 배포판 통합 → **C031이 해소**. ② v3에 "보고" 단계 명시. ③ 실사이클 인프라(fsck·worktree·release 등 v2에만). ④ 기존 189 사이클(v2 5스텝)과 공존 설계.

## 문제 분할

"버전리스 본체 = `gil open`이 곧 v3"는 하나의 큰 전환이다. 이를 가장 작은 단위로 분할하면:

1. **네임스페이스 승격** — `gil v3 open`을 `gil open`으로 (기본 진입점 전환), v2 open은 `gil open --v2`로 옵트아웃.
2. **읽기 축 재정의** — `gil log`/`gil web`의 기본을 v3 눈으로 (지금은 v2 기본, `--v3` 옵트인).
3. **conformance 재정의** — 판정기가 지금 v2 계약(134항목)을 검사. v3가 표준이면 conformance도 v3 계약을 검사해야.
4. **공존(걸림돌④)** — 기존 189 사이클(v2 5스텝 원장)과 새 v3 사이클이 한 원장에 섞임. `gil open --v2`가 남아야 하나? v2 사이클을 v3 눈으로 읽는 건 이미 migrate가 함.

**이 사이클(C032)이 정복할 첫 번째 문제**: 위 4개를 한꺼번에 코드로 밀지 않는다 (C030 M4가 "대체는 걸림돌 넘는 여정"이라 경고). C032는 **설계 사이클** — 승격의 전략(무엇을 기본으로, 무엇을 옵트아웃으로, conformance를 어떻게, 공존을 어떻게)을 **상현님과 확정**하고, 그 전략의 **가장 작은 첫 실행 조각(①네임스페이스 승격)만** 검증한다. 나머지(②③④)는 후속 사이클로 정직히 이월한다. 큰 전환일수록 형태를 코드보다 먼저 (C067·C026).

## 가설

> **가설**: `gil open`의 기본 동작을 v3(스텝 트리)로 승격하고 v2를 `gil open --v2` 옵트아웃으로 두면 — 그리고 v2 습관으로 옛 방식을 쓰는 사용자를 **친절한 에러/온보딩이 v3로 안내**하면 — 도구 표면이 단순해지면서도(버전 몰라도 `gil open`) v2 무회귀(옵트아웃 경로 보존)·conformance 유지를 지킬 수 있다. 승격은 도구 복잡화가 아니라 **기본값 전환 + 안내 흡수**로 달성된다.

## 기각 조건

1. `gil open`을 v3 기본으로 바꾸면 v2 옵트아웃 경로로도 기존 189 사이클 원장 조작이 불가능해진다(공존 깨짐) → 기각.
2. conformance(134/134)가 기본 전환만으로 깨지고, v3 계약 재정의 없이는 초록을 회복할 수 없다 → 이 조각(①)이 conformance 재정의(③)와 분리 불가임을 뜻하므로 **분할 실패**, 재설계.
3. 친절한 에러가 v2 습관 사용자를 v3로 안내하지 못하고 침묵 실패(옛 인자 무시)하면 → 안내 흡수 가설 반증.
4. 상현님이 설계 단계에서 "기본 전환은 아직 이르다 / 다른 형태"라 판단하면 → 이 사이클은 설계 확정에서 방향을 바꾼다(기각 아니라 재조준).
