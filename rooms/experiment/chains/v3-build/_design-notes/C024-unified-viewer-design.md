# 통합 뷰어 — 디자인 (상현님 검토용)

## 임무 한 줄
v3 마이그레이션(C022)이 원장 git notes에 각인한 계보를, **사이클 간 DAG → 사이클 내 스텝 트리**로 드릴다운해 한 눈에 보는 자기완결 정적 뷰어.

## 확정된 결정 (상현님)
1. **데이터 소스: git notes만으로 (순수 v3).** cycle.yaml 안 읽음. 원장이 진실원.
2. **형태: 드릴다운.** 상위 = 사이클 간 계보 DAG 한 화면. 노드 클릭 → 그 사이클의 스텝 트리.
3. **스텝 트리 데이터: v3에 초점.** v2 사이클은 notes에서 선형으로 재구성돼도 무방(상현님: "v2는 선형으로 보여도 상관없어"). 풍부한 구조(backtrack 등)는 v3 사이클에서.
4. **몸: gilv3의 `web` 명령.** migrate에 이어 새 서브명령(gilv3 view는 한 사이클용이니 통합은 새 명령).
5. **작업: Sheen 병렬 워크트리 단독.** 나는 병렬로 다른 이월.

## 두 층의 데이터 (git notes에서만)
- **사이클 간 (상위 DAG)**: `Cycle-Parent` notes 엣지 (C021 접합). 132노드·131엣지·머지4.
- **사이클 내 (드릴다운)**: 각 스텝 커밋의 notes 지문(Step-Id 등). 한 사이클의 스텝 커밋들을 모으면 스텝 시퀀스.

## ⭐ 핵심 설계 난점 — 스텝 트리 데이터를 notes에서 재구성
- C004 steptree.py는 `steps.yaml`을 입력으로 받는데, **마이그레이션된 v2 사이클엔 steps.yaml이 없다.**
- 순수 notes 제약 하에선 **notes의 Step-Id 지문 시퀀스에서 스텝 트리를 재구성**해야 한다.
- v2 사이클은 선형 5스텝(백트래킹 데이터 없음 — C019 "위상 보존, kind만 근사")이라, 재구성된 트리는 대개 선형. C004의 backtrack 렌더는 데이터가 있을 때만.
- → 드릴다운은 "notes가 담은 만큼"의 스텝 트리. 없는 건 정직히 없음(빈 사이클/도출실패 20은 노드는 있되 스텝 트리 비어있거나 섬).

## 계약 (기존 자산 존중)
- **JS 0줄 원칙? 재검토 필요** — 드릴다운은 상호작용이라 C002 hierarchy처럼 `<details>`/앵커로 JS 없이 가능할 수도, 아닐 수도. Sheen 판단(그의 축). C085/C104 라이브 폴링과 충돌 없게.

- **기본 출력 회귀 0** — 기존 gil web 안 깨뜨림 (opt-in이면).
