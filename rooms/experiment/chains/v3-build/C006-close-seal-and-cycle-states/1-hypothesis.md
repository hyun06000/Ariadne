# 1. 가설 수립

## 이전 사이클의 교훈

부모: **v3-build/C005-git-imprint-and-viewer-wire** (supported). C005가 남긴 씨앗:

- **close가 각인에서 특수했다**: 파일 변경이 없어 첫 커밋이 죽고 `--allow-empty` 빈
  봉인 커밋으로 냈다. C005 5-report의 제안: *"close에 verdict를 파일로 남기면 빈 커밋이
  아니게 되고, 그건 v2 gil의 close가 cycle.yaml을 갱신·태그하는 것과 같은 방향."*
- **깃은 시간순, steps.yaml은 논리 트리 — 안 섞인다** (C005 핵심). verdict도 이 원칙을
  따라야 한다: 봉인 메타를 어디에 두든 스텝 트리(노드 포인터)를 오염시키면 안 된다.
- **close 판정은 산 잎(success analyze) 유무** (C002·C003). 그런데 아직 미정: 산 잎이
  **없는**(진행 중) 사이클과 **여럿인**(최적화) 사이클의 상태 표현.

## 문제 분할

C006 "봉인 표현 + 사이클 상태"를 가장 작은 단위로:

1. **S1 — 사이클 상태 분류기**: 스텝 트리에서 사이클 상태를 계산한다:
   - `in_progress`: 산 잎 0 (죽은 잎만 있거나 아직 analyze 전) — 못 닫음.
   - `solved`: 산 잎 1 — 닫을 수 있음 (C003의 현재 판정).
   - `multi_solution`: 산 잎 ≥2 — 최적화 사이클(chain.md "다른 정답도?"). 닫되 표시.
   - **트리에서 순수 계산** (C002 불변식 — 상태 필드 저장 안 함).
2. **S2 — close 봉인 표현**: close가 verdict를 어디에 남기나. 후보:
   (a) 빈 커밋(현재 C005) (b) `cycle.yaml` 같은 봉인 메타 파일 (c) 깃 태그.
   **결정 대상**: steps.yaml을 안 건드리면서 봉인을 데이터로 남기는 최소 형식.
3. **S3 — close 게이트**: `in_progress`면 close 거부, `solved`/`multi_solution`이면 허용.
   S1 분류기가 게이트를 파생.

**지금 정복할 첫 문제 = S1+S3 (상태 분류 + close 게이트)**을 먼저. 이유: close 봉인
표현(S2)은 "닫을 수 있는가"(S3)가 정해져야 의미가 있고, S3은 S1 분류기에서 파생된다.
즉 **상태 분류기가 뿌리** — 그것이 서면 게이트가 공짜, 봉인 표현은 그 위에 얹는다.
S2는 S1이 통과한 뒤 최소 형식만 확정(그리디 — 태그 vs 파일 논쟁은 필요한 만큼만).

## 가설

> **가설**: 스텝 트리에서 **사이클 상태(in_progress / solved / multi_solution)를 산 잎
> 개수로 순수 계산**하는 분류기를 만들고 close를 그 분류에 게이트하면, (a) C002·C003의
> 세 트리 변형(진행 중=죽은 잎만 / 정답=산 잎 1 / 최적화=산 잎 2)이 정확히 세 상태로
> 분류되고, (b) `in_progress`에서 close가 거부되며, (c) 이 분류가 steps.yaml에 상태
> 필드를 안 더한다(트리에서만 계산).

## 기각 조건

- **K1**: 산 잎 개수만으로 세 상태를 못 가른다 — 예를 들어 "죽은 잎만 있는데 아직
  진행 중인가 포기인가"를 트리가 구별 못 한다 (분류 불충분 → S1 재정의).
- **K2**: close 게이트가 분류기와 별개 규칙을 요구한다 — 분류에서 게이트가 파생 안 됨
  (S1이 뿌리가 아님 → 되돌아가 상태 정의 재검토).
- **K3**: 상태를 계산하려니 steps.yaml에 상태·verdict 필드를 넣어야 한다 — C002 불변식
  (트리는 포인터만) 위반 (모델 오염 → 데이터 모델로 되돌아감).
