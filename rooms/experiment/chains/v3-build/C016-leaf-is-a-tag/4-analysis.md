## 가설 판정

## 교훈

1. **⭐ 못의 이름은 목적이 결정한다 — 생사 구분이 못엔 불필요했다.** C015까지 잎 못을 `gil/dead/`·`gil/live/`로 생사에 따라 나눴지만, 못의 목적은 오직 **dangling 방지**다. 잎의 생사(죽은=벽의 지도 / 산=정답)는 이미 커밋 trailer(Outcome)가 담는다. C011이 "이름은 해시면 충분 — 논리 id는 지문에"라 한 결론이 여기서 코드로 실현됐다: **못 이름을 해시(`gil/leaf/<short>`)로 단일화**하니 세 이름(dead·live·sealed 중 앞 둘)이 하나로 합쳐졌다. 정식화가 곧 단순화였다 — 데이터를 두 곳(못 이름 + trailer)에 중복 표현하던 것을 trailer 한 곳으로.

2. **⭐ 태그 vs 브랜치의 차이가 잎의 본성(불변)을 드러냈다.** C015가 백트래킹에서 `git branch -f`로 못을 **강제 이동**시켜야 했던 것 자체가, 브랜치가 잎에 안 맞는다는 증거였다(브랜치는 가변, 잎은 불변). 태그로 바꾸니 `-f`가 사라졌다 — 태그는 커밋에 고정이라 강제할 게 없다. **못의 타입이 못박는 대상의 본성과 맞아야 한다**: 움직이는 것엔 브랜치, 불변 시점엔 태그. C011이 "잎은 불변 시점이라 태그가 정확"이라 한 통찰의 코드적 확인.

3. **⭐ dangling 방지는 ref 타입에 무관하다(C011 M3 재확인).** 태그도 브랜치도 ref라 `git log --all`·`rev-list --all`이 도달가능으로 본다 — dangling 방지는 **등가**(M3). 그러나 등가인 두 선택지 중 잎의 본성(불변·push 생존)에 맞는 태그가 정확하다. "어떤 ref든 못이면 산다"(C011 M3)가 참이되, 그중 최적을 고르는 것이 정식화.

4. **한 헬퍼로 4곳을 통일 — 재사용의 얇은 껍질.** 브랜치 못이 4곳(step 백트래킹·close 산 잎·close 종결)에 흩어져 각자 `git branch`를 호출하던 것을, `tie_leaf`·`tie_sealed` 두 헬퍼로 모았다. 태그 규칙이 한 곳에 사니 C011·C012 build 스크립트와의 일치를 한 번만 보장하면 된다.
