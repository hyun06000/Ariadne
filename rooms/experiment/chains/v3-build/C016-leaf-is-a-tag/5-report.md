# 5. 결과 보고 — v3-build/C016-leaf-is-a-tag

부모: v3-build/C015-merge-is-lineage-command (supported). lineage: v3-build/C011-everything-is-a-branch. 저자: Clew. 소환자: 없음 (main 단독·순차, C074). 판정: **supported (채택)**.

## 요약

C011이 순수 깃으로 결론지은 **잎=태그**(불변 시점=태그, push 생존, 이름은 해시)를, C015까지 임시로 쓰던 **브랜치 못**에서 정식 **태그 못**으로 도구에서 정식화했다. `tie_leaf`(`gil/leaf/<short-hash>`)·`tie_sealed`(`cycle/<name>/solved`) 두 헬퍼로 브랜치 못 4곳을 태그로 통일. 5측정 ALL PASS: 모든 잎이 태그(M1, 브랜치 못 0)·종결이 태그(M2)·dangling 방지 유지(M3)·태그가 잎에 불변 고정(M4)·C014 분기·C015 lineage 무회귀(M5). **도구화 진척표에서 잎=태그가 ◐ → ✅ — 남은 미완은 뷰어(⬜, Sheen 축) 하나.**

## 무엇을 했나

1. **`tie_leaf`·`tie_sealed` 헬퍼 신규** — 잎 커밋을 `gil/leaf/<short-hash>` 태그로, 종결을 `cycle/<name>/solved` 태그로. idempotent(해시 이름이라 재못박기 안전).
2. **브랜치 못 4곳 → 태그** — cmd_step 백트래킹(죽은·산 공통)·cmd_close 산 잎·cmd_close 종결. `git branch` 못 호출 0.
3. **5측정 감사** — caseM(lineage 머지, C015 표적)·caseB(3층 분기, C014 표적)을 태그판 도구로 재실행 + 순수 깃 감사.

## 교훈

1. **⭐ 못의 이름은 목적이 결정한다 — 정식화가 곧 단순화였다.** C015까지 잎 못을 생사(`gil/dead/`·`gil/live/`)로 나눴지만, 못의 목적은 오직 dangling 방지다. 잎의 생사는 이미 커밋 trailer(Outcome)가 담으므로, 못 이름을 해시(`gil/leaf/<short>`)로 단일화하니 세 이름이 하나로 합쳐졌다. **데이터를 두 곳(못 이름 + trailer)에 중복 표현하던 것을 trailer 한 곳으로** — C011 "이름은 해시면 충분, 논리 id는 지문에"의 코드적 실현.

2. **⭐ 못의 타입이 대상의 본성과 맞아야 한다.** C015가 백트래킹에서 `git branch -f`로 못을 강제 이동시켜야 했던 것 자체가 브랜치가 잎에 안 맞는 증거였다(브랜치=가변, 잎=불변). 태그로 바꾸니 `-f`가 사라졌다 — 태그는 고정이라 강제할 게 없다. 움직이는 것엔 브랜치, 불변 시점엔 태그. C011 "잎은 불변 시점이라 태그가 정확"의 확인.

3. **⭐ dangling 방지는 ref 타입에 무관, 그중 최적을 고르는 게 정식화.** 태그도 브랜치도 ref라 `rev-list --all`이 도달가능으로 봄 — 방지는 등가(M3). "어떤 ref든 못이면 산다"(C011 M3)가 참이되, 등가인 둘 중 잎의 본성(불변·push 생존)에 맞는 태그가 정확하다.
