# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C027**이 v3 눈의 뿌리(`gil migrate`)를 배포판 gil.py에 통합했다 — 인라인 이식, 오라클 대조 바이트 동일(743fc56a), conformance 134/134. 그 사이클의 이월 1순위:

> **`gil web` v3 뷰어를 배포판에 통합** — Sheen C025 통합 뷰어(notes 두 층 드릴다운)를 배포판 gil.py에. 이걸로 읽기 축도 하나의 gil이 되어 **"v3만 쓴다"의 도구가 완성**(쓰기=migrate 각인, 읽기=web 뷰어).

## 문제 — 배포판엔 이미 cmd_web(v2)이 있다 (이름 충돌)

배포판 gil.py엔 이미 `cmd_web`(2765줄, v2 뷰어: 사이클 그래프·hierarchy·라이브 폴링)이 있다. Sheen의 v3 통합 뷰어(C025)는 **다른 것** — 순수 notes 두 층(사이클 간 DAG × 사이클 내 스텝 트리) 드릴다운. 통합 시 이름·표면 충돌을 풀어야 한다.

## 문제 분할 — v3 뷰어를 어떻게 배포판 표면에 얹나

후보:
1. **`gil web --v3` 플래그** — 기존 web에 옵트인 플래그. 하위호환(기본은 v2), C026 형태("v3 = v2 위의 눈")에 정합.
2. **`gil web3` / `gil view` 별도 서브명령** — 표면 분리. 명확하나 명령 수 증가.
3. **v3가 기본, v2는 --v2 옵트아웃** — v3 우선이나 하위호환 깨짐(기존 스크립트).

**첫 번째로 정복할 것: `gil web --v3` 플래그(후보 1).** 이유:
- **하위호환**: 기본 `gil web`은 v2 그대로 — 기존 스크립트·conformance 무회귀(C027 M4 정신).
- **C026 형태 정합**: v3는 v2 위에 얹힌 눈이니, 뷰어도 v2 web에 옵트인으로 얹는 게 개념적으로 맞다.
- **점진적**: v3가 상시화되면 나중에 기본 전환 가능(지금은 옵트인).

## ⭐ 통합의 형태 — C027 이식 패턴 재사용

C027이 확립: 검증 백엔드를 gil.py에 인라인 이식(재구현 아님), 오라클 대조로 바이트 보존 집행, 자기완결 계약(import 0). web 통합도 동일:
- Sheen C025 백엔드(notes_reconstruct 160 + web_render 308 + steptree 308 ≈ 776줄)를 gil.py에 인라인.
- 오라클 = C025 gilv3 web 출력 HTML. 배포판 `gil web --v3`가 바이트 동일이면 이식 보존.
- 이름 충돌 방지: v3 함수는 접두어(`_v3web_`) 또는 명확한 이름으로.

## 가설

> **가설**: Sheen C025가 검증한 v3 뷰어 백엔드(notes_reconstruct·web_render·steptree)를 배포판 gil.py에 `gil web --v3` 플래그로 인라인 통합하면, (a) `gil web --v3`가 C025 gilv3 web과 바이트 동일 HTML을 내고, (b) 기본 `gil web`(v2)은 무회귀(conformance 통과·바이트 불변)이며, (c) gil.py가 자기완결(외부 모듈 import 0)을 유지한다 — 즉 v3 눈의 읽기 축이 하나의 gil로 하위호환하며 들어와 "v3만 쓴다"의 도구(쓰기 migrate + 읽기 web)가 완성된다.

## 기각 조건

- `gil web --v3` HTML이 C025 gilv3 web과 다르면 → **기각**(이식이 렌더 바꿈).
- 기본 `gil web`(v2)이 바뀌거나 conformance 회귀하면 → **기각**(하위호환 파괴).
- 통합에 새 렌더 로직을 발명해야 하면 → **조사**(이식이 아니라 재구현).
- gil.py가 외부 모듈에 의존하면 → **기각**(자기완결 배포 계약 위반).
