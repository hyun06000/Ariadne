# s2 · hypothesis (가설)

## 가설

> load_chain_records의 v3 분기(cycle.yaml 없는 열린 v3 사이클)에서 `cycle_state(load(
> entry_dir))`로 상태를 도출해 record.status에 넣으면, gil log의 render_graph 뱃지가
> `[?]` 대신 `[in_progress]`/`[solved]`/`[multi_solution]`로 표시되고, 닫힌 v3(cycle.yaml
> 있음)·v2 사이클·fsck·conformance는 무회귀한다.

## 근거 (실측 기반)

- render_graph는 `meta.get("status") or "?"`(gil.py 263)로 뱃지를 만든다 — status 채우면
  뱃지 채워짐.
- `cycle_state(nodes)`(6300)가 이미 열린 v3 상태를 순수 계산(in_progress/solved/
  multi_solution) — 새 로직 불필요, 재사용.
- 닫힌 v3는 v3 close가 cycle.yaml을 써서("S2 봉인") 이미 v2 로더가 잡음 → 이번 변경은
  **열린 v3에만** 적용(cycle.yaml 없는 분기).

## 기각 조건

1. status 도출을 넣어도 log 뱃지가 여전히 `[?]`면 → render_graph에 안 닿음(기각).
2. cycle_state 호출이 열린 v3에서 crash·오분류하면 → 도출기 부적합(기각).
3. 닫힌 v3·v2·fsck·conformance가 깨지면 → 무회귀 실패(기각).
