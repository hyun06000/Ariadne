# s1 · define (루트 · 문제 정의)

## 이 사이클의 특별함 — 첫 v3 네이티브 도그푸딩

부모 **C042**. 상현님 "지금 바로 gil v3로 전환" → 이 C043은 **v3-build 체인이 처음으로
자기 자신을 gil v3로 여는 사이클**이다. steps.yaml 스텝 트리·계보 trailer(Cycle-Author:
clew, Cycle-Parent: C042)로 열렸고, fsck·log가 이 사이클을 인식·계보 표현한다. 우리
도구가 자기 자신을 v3로 쓴다 — 대체(replace)를 몸으로.

## 문제 정의

C042까지 gil v3 실사이클의 네 전제(격리·원장인식·계보 각인·계보 표현)가 완성됐으나,
정직한 경계 하나: **v3 사이클의 상태 뱃지가 아직 `[?]`**. 실측 — `gil log`에서 이 C043
사이클이 `● C043-v3-status-badge [?]`로 그려진다. render_graph(gil.py 263)가
`meta.get("status") or "?"`를 쓰는데, 열린 v3 record엔 status가 없어 `?`.

**핵심 실측 — v3에 이미 상태 도출기가 있다.** `cycle_state(nodes)`(gil.py 6300)가
steps.yaml 트리에서 상태를 순수 계산한다: in_progress(산 잎 0)·solved(산 잎 1)·
multi_solution(산 잎 ≥2). 그리고 v3 close는 cycle.yaml을 쓴다("S2 봉인") — 즉:
- **닫힌 v3 사이클**: cycle.yaml 있음 → 기존 v2 로더가 status/verdict 잡음(뱃지 이미 됨).
- **열린 v3 사이클**: cycle.yaml 없음 → C040/C042 v3 분기가 status 없이 record → `[?]`.

**정복할 문제**: 열린 v3 사이클의 상태를 `cycle_state`로 도출해 record.status에 넣어,
log 뱃지가 `[?]` 대신 `[in_progress]`/`[solved]`/`[multi_solution]`로 표시되게 한다.
이로써 관전자가 v3 실사이클의 진행 상태를 v2처럼 읽는다 — 도그푸딩의 마지막 표현 조각.
