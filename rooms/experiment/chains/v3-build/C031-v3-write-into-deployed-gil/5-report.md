# 5. 결과 보고

## 요약

C030이 명시한 "v3가 v2를 대체" 걸림돌 ①(v3 쓰기 배포판 미통합)을 넘었다 — gilv3의 v3 쓰기(open/step/close/status + 22 헬퍼)를 배포판 gil.py에 **`gil v3 <cmd>` 서브명령 그룹**으로 인라인 통합. 오라클 대조로 gil v3가 gilv3와 steps.yaml 바이트 동일(262d3523), v2 무회귀(conformance 134/134), 자기완결. 상현님 방향 반영 — 궁극은 버전리스(gil open=v3), 전환 마찰은 **친절한 에러**(v2 습관에 v3 방식 안내)가 흡수. **5측정 ALL PASS → supported. 이제 실사이클을 배포판 gil로 v3 네이티브로 연다.**

## 교훈

1. **버전리스는 안내가 흡수하고, 네임스페이스가 지탱한다.** 상현님 두 방향(궁극 버전리스 + 안내로 해결)의 답: 지금은 `gil v3`로 안전하게 얹고(중첩 서브파서로 v2 무손상), 전환 마찰은 친절한 에러가 흡수. 나중 gil open=v3 승격 시도 v2 습관 사용자는 에러가 가르친다. **버전리스는 도구 단순화가 아니라 안내가 전환을 부드럽게 해서 달성.** genesis 이래 저장소 방식(복잡성을 안내 한 줄로).

2. **인자 충돌은 네임스페이스 분리로.** v2 step(번호)·v3 step(--kind)의 충돌을 `gil v3` 중첩 서브파서가 원천 해소 — 각자 파서라 안 섞인다. 옵트인 플래그(후보 A)보다 깨끗.

3. **이식 규율이 확립되면 반복이 기계적.** C027(migrate)·C028(web)·C031(write) 세 이식이 같은 패턴(자동 추출·ALREADY 제외·접두어·오라클 대조). 세 번째는 헬퍼 충돌 0·스캐폴딩 함정 미리 회피로 기계적. 좋은 패턴이 반복 비용을 낮춘다.

4. **친절한 에러가 상현님 통찰의 실현.** `gil v3 step <dir>`(kind 없이)이 "번호가 아니라 kind", v2→v3 대응, 순환·백트래킹을 안내. 버전 난제를 도구 로직이 아니라 안내 텍스트가 흡수 — C069 "안내가 모델 전달"의 도구 전환판.

## 다음 사이클을 위한 제안 (버전리스로, 걸림돌 계속)

1. **gil open을 v3로 승격 검토 (버전리스 본체)** — 상현님 궁극 방향. gil open/step/close 기본을 v3로, v2는 --v2 옵트아웃 또는 은퇴. conformance를 v3 기준으로 재정의하는 큰 전환 — 신중한 별도 사이클(v2 189 사이클 공존 계획 포함, 걸림돌 ④와 함께).

2. **v2→v3 온보딩 문서 (전환 안내 전면화)** — 친절한 에러(씨앗)를 SPEC/README의 "v2에서 온 사람에게" 대응표로. 5스텝(번호)↔스텝 트리(kind·백트래킹) 매핑. "문서가 곧 테스트".

3. **v3에 "보고" 단계 명시 (걸림돌②)** — v3 kind 순환에 report/close가 v2 5-report만큼 명시적이지 않다. 방법론 규율(재현 보고)을 v3에 강제.

4. **실사이클을 gil v3로 열어보기 (도그푸딩 심화)** — 이제 배포판 gil v3가 있으니, 실제 사이클을 gilv3(실험)가 아니라 배포판 gil v3로 굴려 대체를 몸으로.
