## 데이터 직접 관찰

통계(6/6)에 안주하지 않고 아티팩트를 눈으로 확인했다.

### H4가 이 사이클의 진짜 기여 — `same-file-three-versions.txt`

`calc/core.py`는 저장소에 **경로가 하나**다. 그 한 경로에 세 개의 완전히 다른 `add()`가 그래프에 공존한다:
- **가지 A (s4, 죽음)**: `return int(str(a) + str(b))` — 문자열 연결(add(12,3)=123, 틀림).
- **가지 B (s7, 죽음)**: `return add(a + 1, b - 1)` 재귀 — 큰 b에서 스택 폭발, float 깨짐.
- **가지 C (s10, 산 잎)**: `return a + b` — 올바름. **작업트리에 물리적으로 존재하는 유일한 버전.**

세 버전은 pairwise 상이하고 서로 오염되지 않았다(M5). 작업트리의 core.py는 C 하나뿐인데(`git ls-files` 경로 1개), 죽은 A·B는 `git show <hash>:calc/core.py`로 온전히 되살아난다.

**이것이 디렉토리 복사 방식이 원리적으로 못 하는 것이다.** 사이클 디렉토리를 통째 복사해 코드를 격리하면 각 가지가 코드 스냅샷 "한 벌"을 물리적으로 따로 들어야 해서 "공유 한 벌(single source of truth)"이라는 개념 자체가 사라진다. 깃 방식은 물리 파일 한 벌 + 커밋 그래프 = 무한 버전. 죽은 가지 코드도 별도 사본 없이 태그 하나로 영구 접근.

### H1 — "덮어쓰기"가 아니라 진짜 "증분"

C011의 artifact.py는 매 스텝 파일 전체를 재기록했다(diff=항상 전체). 이 사이클은 부분 수정이 diff로 정확히 잡히는지 검사: s2~s8은 `core.py`만, **s9는 `core.py`+`util.py` 두 파일 증분**, s10은 `__init__.py`만. 10 스텝 변경파일집합이 100% 일치. 스텝 커밋 diff가 그 스텝의 코드 변경분을 정확히 담는다 → 커밋 grep·`git log -- <경로>`만으로 트레이싱, 뷰어 재구현 0.

### H2 — git show 롤백은 진짜 무손상

죽은 잎 s4·s7의 core.py를 꺼내는 동안 HEAD·`status --porcelain`·인덱스·디스크 파일이 **네 가지 모두 불변**. 롤백은 "조회"이지 "체크아웃"이 아니다 — 작업트리는 산 잎(C)에 머문다. 디렉토리 복사(cp 덮어쓰면 작업 상태 오염) 대비 우위.

### H3 — 못(태그)의 필요성 음성대조 재확인

s7 태그를 지우니 `git log --all`에서 s7이 사라지고(gc 위험), 되박으니 부활. 죽은 가지 코드의 영구 생존엔 태그(못)가 필수임이 코드 맥락에서도 성립(C011 발견의 재현).

## 예상과 달랐던 것 — 계측기 결함 vs 가설 반증 (정직 구분)

첫 측정에서 M1이 `KeyError: 's2'`로 죽었다. 이는 **가설 반증이 아니라 계측기 결함**이었다: 빌드가 `remember`(커밋 해시 기록)를 잎 커밋에만 호출해 commit-index에 중간 스텝 해시가 비었다. `step` 헬퍼가 매 커밋 자동 `remember` 하도록 한 줄 고쳐 재빌드하니 ALL PASS. 깃 동작은 처음부터 옳았고 기록 도구의 누락이었다(3-verification/README.md 실행기록에 명시). 계측기 결함과 반증을 뭉뚱그리지 않는 것이 이 저장소 문화.

## 판정

**H1·H2·H3·H4 전부 채택 (supported).** 기각 조건(diff 오염·워킹트리 변경·음성대조 실패·분기 오염) 어느 것도 발동하지 않았다. 코드 아티팩트는 사이클 디렉토리 복사 없이 공유 경로 한 벌 위에서 스텝별로 트레이싱되고, git show로 죽은 가지까지 무손상 롤백되며, 태그로 영구 생존하고, 같은 파일의 여러 분기가 독립 보존된다 — 상현님 원질문의 마감.
