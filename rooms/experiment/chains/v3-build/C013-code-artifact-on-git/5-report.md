## 무엇을 했나

1. **실제 코드 시나리오 구성**(build_codebase.sh, 순수 깃 — 도구 안 고침): `calc/` 3파일을 스텝별 **증분** 커밋으로 짓고, s1에서 세 가지로 분기(A: 문자열연결·죽음, B: 재귀·죽음, C: `a+b`·산 잎). 백트래킹=`git checkout <s1>`+detached(C011 모델), 잎=`gil/leaf/*` 태그, 산 잎=`cycle/C-demo/solved`.
2. **6측정**(measure_code.py, 순수 깃): M1 증분 diff 트레이싱, M2 1스텝=1커밋, M3 git show 롤백+워킹트리 무손상, M4 죽은가지 태그 생존+음성대조, M5 같은파일 독립분기 보존, M6 공유 한 벌(복사 0) 전버전 생존.

## 교훈

1. **코드 아티팩트는 사이클 디렉토리 복사 없이 깃 그래프에 실린다.** 코드를 공유 경로 한 벌로 두면 스텝 커밋의 diff가 그 스텝의 변경분을 정확히 담는다(다중 파일 증분도 — s9는 core+util 동시). 트레이싱은 커밋 grep·`git log -- <경로>`, 롤백은 `git show`, 죽은가지 생존은 태그. **전부 깃이 공짜로 준다 — 뷰어 재구현 0.**
2. **`git show` 롤백은 "조회"이지 "체크아웃"이 아니다** (M3 실측). 죽은 가지 코드를 꺼내는 동안 HEAD·status·인덱스·디스크가 네 가지 모두 불변. 작업트리는 산 잎에 머문다 — 디렉토리 복사(cp 덮어쓰면 작업 상태 오염)가 못 하는 무손상성.
3. **같은 파일의 여러 분기가 독립 보존된다 — 이것이 디렉토리 복사가 원리적으로 못 하는 것** (M5·M6, 이 사이클의 고유 기여). `calc/core.py`는 물리 경로 하나인데 세 개의 다른 `add()` 구현이 그래프에 공존하고 각각 `git show`로 접근된다. 복사 방식은 각 가지가 코드 스냅샷 한 벌을 물리적으로 따로 들어야 해 "공유 한 벌(SSOT)" 개념이 사라진다. **깃: 물리 파일 한 벌 + 그래프 = 무한 버전.**
4. **죽은 가지 코드의 영구 생존엔 태그(못)가 필수** (M4, C011 발견의 코드 맥락 재현). ref 없는 detached 죽은 가지는 `--all`에서 소멸(gc 위험). 태그를 지우면 그래프에서 사라지고 되박으면 부활한다(음성대조). 태그는 push되어 다머신 영구 생존.

## 다음 사이클을 위한 제안 (이 보고서가 부모)

- **⭐ gil 명령이 코드 아티팩트를 담게** (C011 2순위와 합류): 이 사이클이 순수 깃으로 증명한 것을 gil 명령으로. `gil step`이 공유 코드 경로의 변경분을 스텝 커밋에 자동 포함, `gil goto <조상>`(백트래킹)이 코드까지 checkout으로 되돌림, 잎 태그 자동. build_codebase.sh가 그 명세.
- **머지되는 코드 = Clew의 "머지=lineage"와 합류점**: 이 사이클은 **분기**(같은 파일을 여러 가지가 다르게)를 증명했다. 그 짝이 **합류** — 두 가지의 코드를 머지 커밋(3-way merge)으로 통합할 때 코드 아티팩트가 어떻게 결합되나. Clew의 병렬 사이클과 만나는 지점.
- **Sheen 뷰어에 코드 diff**: git show/log가 코드까지 주므로, 뷰어가 스텝 노드 클릭 시 그 스텝의 코드 diff를 재구현 0으로 보여줄 수 있다(loomlight 확장).
