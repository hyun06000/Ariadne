# 1. 가설 수립 — v3-build/C013-code-artifact-on-git

부모: v3-build/C011-everything-is-a-branch (supported). 저자: Bobbin. 소환자: Clew (병렬 — Clew는 main에서 "머지=lineage" 사이클 진행 중).

## 이전 사이클의 교훈 (C011에서 이어받음)

C011이 **깃 데이터 모델 ≅ gil 위계 모델**을 확정했다:
- 백트래킹 = `git checkout <조상>` + detached HEAD 커밋 → 분기 자연 발생.
- 1스텝 = 1커밋.
- 위계(체인⊃사이클⊃스텝) = 커밋 메시지 trailer 지문.
- 잎 = 태그(못). 죽은 가지는 ref가 없으면 `git log --all`에서 사라지지만(dangling→gc), 태그를 박으면 영구 생존 + push되어 다머신 공유.

C011은 이를 `artifact.py` **한 파일**로 원리만 건드렸다(각 스텝이 파일 전체를 덮어씀). C011의 5-report가 남긴 1순위 제안:

> **코드 아티팩트를 이 위에 실음**: 코드를 공유 경로 한 벌로 두고 스텝 커밋이 변경분을 담으면, 사이클 디렉토리 복사 없이 `git show`로 롤백·태그로 죽은 가지 코드까지 생존. 실제 코드 아티팩트 시나리오로 재현 가능하게 실증하라.

## 문제 분할

C011이 남긴 마감을, "artifact.py 한 파일 덮어쓰기"에서 **실제 코드 아티팩트 시나리오**로 끌어올린다. 실제 시나리오의 요건:
1. **다중 파일 코드베이스** (한 파일 덮어쓰기가 아니라, 여러 모듈이 있는 실제 코드).
2. **증분 변경** (스텝 커밋의 diff가 그 스텝의 변경분만 담는다 — 파일 전체 재기록이 아니라).
3. **같은 파일을 서로 다른 가지가 다르게 수정** — 이것이 디렉토리 복사 방식이 흉내낼 수 없는 핵심. (디렉토리 복사는 각 사이클이 코드 스냅샷 한 벌을 통째로 들고 있어, 공유 코드의 "한 벌" 개념이 없다.)

가장 작은 첫 정복 대상: **"복사 없는 한 벌(single shared tree) 위에서 코드가 스텝별로 트레이싱·롤백·생존하는가"** 를, 네 측정으로 한 번에 실증한다.

## 가설

> **가설**: 코드를 **공유 경로 한 벌**로 두고 스텝 커밋이 변경분(diff)을 담으면 — 사이클 디렉토리 복사 없이 — (H1) 각 스텝 커밋의 diff가 그 스텝의 코드 변경분을 정확히 담아 스텝별 트레이싱이 되고, (H2) `git show <스텝>:<경로>`로 죽은 가지 포함 임의 스텝의 코드를 워킹트리 무손상으로 롤백/복원할 수 있으며, (H3) 죽은 가지 코드는 태그(못) 덕에 영구 생존한다(음성대조: 태그 지우면 `--all` 그래프에서 소멸, 되박으면 부활), (H4) 여러 가지가 **같은 파일을 다르게 고쳐도** 각 가지의 코드가 독립적으로 보존된다(디렉토리 복사가 못 하는 것).

## 기각 조건

- **H1 기각**: 어떤 스텝 커밋의 diff가 그 스텝의 의도한 코드 변경분과 불일치하거나(변경분 누락·오염), 스텝별 트레이싱이 커밋 grep 만으로 안 될 때.
- **H2 기각**: `git show`가 죽은 가지 코드를 못 꺼내거나, 롤백 조회가 워킹트리/HEAD/인덱스를 변경할 때.
- **H3 기각**: 태그가 있어도 죽은 가지가 `--all`에서 안 보이거나, 태그를 지워도 그래프에 남거나(음성대조 실패), 되박아도 안 돌아올 때.
- **H4 기각**: 두 가지가 같은 파일을 다르게 고쳤을 때 한 가지의 버전이 다른 가지 버전으로 오염되거나, 각 가지에서 그 가지 고유 코드를 독립적으로 못 꺼낼 때.
- **계측기 결함 vs 가설 반증 구별**: 스크립트 버그로 인한 실패는 계측기 결함으로 분류하고 수정 후 재측정한다. 깃의 실제 동작이 가설과 다르면 반증으로 정직히 보고한다.
