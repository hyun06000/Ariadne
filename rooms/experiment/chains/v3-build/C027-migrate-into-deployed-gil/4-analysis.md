## 데이터 직접 관찰

### notes digest 743fc56a가 이식의 정확성을 봉인

M2가 가장 강한 증거다. 오라클(C023 gilv3)과 배포판(gil.py 인라인)이 각인한 **모든 notes 본문 digest가 743fc56a로 일치**했다. 이는 노드 지문·Cycle-Parent 엣지가 한 글자도 안 다르다는 뜻 — 인라인이 알고리즘을 재설계하지 않고 함수를 그대로 옮겼음을 바이트가 증언한다. C023이 gilv3에서 오라클 대조로 확인한 것을, 이 사이클은 배포 층에서 재확인.

### hashlib·glob 누락 — 이식의 숨은 의존

첫 실행이 `NameError: hashlib`로 죽었다. migrate 백엔드(snapshot의 cycle_yaml_hash·full_ledger_migrate의 glob)가 쓰는 import가 배포판 gil.py 헤더에 없었다. **이식의 함정: 함수만 옮기면 그 함수가 의존하는 import는 안 따라온다.** 오라클 대조 전에 잡혀(문법 OK였으나 런타임 실패) 수리 — 계측기/이식 결함이지 로직 반증 아니다. 자기완결 배포 계약(M5)이 이걸 강제로 드러냈다 — 외부 모듈에 안 기대니 모든 의존이 gil.py 안에 있어야 한다.

### conformance 134/134가 배포판 무결성을 지켰다

M4에서 배포판 기존 명령(open/step/close/web/fsck/deploy…)이 134/134 통과. migrate 백엔드 321줄을 삽입했는데도 회귀 0 — 인라인이 기존 네임스페이스를 안 오염시켰다(함수명 충돌 0, `_mig_` 접두어·언더스코어 구분). 5020줄 본체에 큰 블록을 넣는 위험을 conformance가 게이트했다.

## 예상과 달랐던 것

### 통합이 예상보다 기계적이었다 (C023 함수 경계의 배당)

C023이 백엔드를 깔끔한 함수(`migrate_all(repo,apply)`·`splice_all(repo)`)로 나눠뒀기에, 인라인은 함수 블록 자동 추출 + 접두어 제거로 끝났다. 새 글루 코드가 거의 없었다. **C023 "좋은 함수 경계가 미래 명령화를 연다"의 연장 — 좋은 함수 경계가 미래 이식도 연다.** cmd_migrate조차 C023판을 접두어만 바꿔 재사용.

## 판정

**채택 (supported).** `gil migrate`가 배포판 gil.py에 통합됐다. 오라클 대조 바이트 보존(743fc56a), conformance 134/134 회귀 0, 자기완결 배포 계약 유지. v3 눈의 뿌리가 하나의 gil로 들어왔다.

## 정점 통찰 — 인라인은 재구현이 아니라 이식, 오라클이 그 구분을 집행

자기완결 배포 계약(SPEC §7) 때문에 백엔드를 gil.py에 인라인했으나, 함수를 옮겼을 뿐 알고리즘을 재설계하지 않았다. 이 구분이 중요하다 — 재구현이면 미묘한 버그가 들어오고 오라클이 안 맞았을 것. **743fc56a 동일이 "이식했지 재구현 안 했다"를 증명한다.** 자동 추출(함수 블록 + 접두어 제거)이 이식의 기계적 정확성을 보장했다.

## 방법론에 귀속되는 교훈

- **인라인 이식은 오라클 대조로 검증한다.** 이전 구현(gilv3)을 오라클로, 새 구현(gil.py)이 바이트 동일이면 이식 성공. 리팩터/명령화(C023)·이식(C027) 공통 패턴.
- **이식의 함정은 숨은 의존(import).** 함수만 옮기면 그 함수의 import는 안 따라온다. 자기완결 계약(M5)이 모든 의존을 드러내게 강제 — 배포 계약이 이식 완전성을 집행.
- **좋은 함수 경계가 미래 이식을 연다(C023 연장).** 깔끔한 함수 분리 → 자동 추출 가능 → 기계적 정확 이식. C018·C019·C023·C027로 이어지는 "좋은 구조가 미래를 연다" 사슬.
