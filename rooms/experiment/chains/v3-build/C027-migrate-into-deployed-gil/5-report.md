# 5. 결과 보고

## 요약

상현님의 "이제 v3만 쓰는 거지?" → "먼저 배포판 통합부터"에서, v3 눈의 뿌리인 **`gil migrate`를 배포판 gil.py에 통합**했다. C023 검증 백엔드(28함수·321줄)를 인라인 이식(자기완결 배포 계약), `cmd_migrate` + argparse 등록. 오라클 대조로 배포판 `gil migrate`가 C023 gilv3와 notes 바이트 동일(digest 743fc56a)·DAG 동일(136엣지·머지4)임을 확인, conformance 134/134로 회귀 0. **5측정 ALL PASS → supported.** v3 눈의 뿌리가 하나의 gil로 들어왔다.

## 교훈

1. **인라인은 재구현이 아니라 이식, 오라클이 그 구분을 집행.** 자기완결 배포 계약(SPEC §7) 때문에 백엔드를 gil.py에 인라인했으나 함수를 옮겼을 뿐 알고리즘 재설계 안 함. notes digest 743fc56a 동일이 "이식했지 재구현 안 했다"를 증명. 재구현이면 미묘한 버그가 들어오고 오라클이 안 맞았을 것.

2. **이식의 함정은 숨은 의존(import).** 함수만 옮기면 그 함수의 import(hashlib·glob)는 안 따라온다 — 첫 실행이 NameError로 죽었다. 자기완결 계약(M5, gil.py를 격리 복사해도 동작)이 모든 의존을 gil.py 안에 두게 강제 — 배포 계약이 이식 완전성을 집행.

3. **좋은 함수 경계가 미래 이식을 연다(C023 연장).** C023이 백엔드를 깔끔한 함수로 나눠뒀기에 인라인이 함수 블록 자동 추출 + 접두어 제거로 끝났다. C018·C019·C023·C027로 이어지는 "좋은 구조가 미래를 연다" 사슬.

4. **"명령화의 값어치는 재사용, 전제는 동작 보존"이 배포 층에서 완성(C023 연장).** C023이 gilv3에서 한 것을 이 사이클이 배포판에서 재현. 이제 배포된 gil을 쓰는 누구나 자기 v2 원장을 v3로 볼 수 있다(will.md 범용성).
