# 5. 결과 보고

## 요약

상현님 "v3를 직접 써보자, v3가 v2를 대체"에 답해, 실제 문제(C024 초기 16개 도출실패 정밀화)를 **gilv3 open/step/close로 v3 네이티브 사이클로 굴렸다**. 접근 A(subject 파싱)가 데이터에 막혀 죽은 잎(s4)+백트래킹(s4→s1)이 되고, 접근 B(디스크 문서 구조)가 산 잎(s7)으로 성공 — 초기 16개 전부 80 스텝 노드 도출. **v2 5스텝이 못 담는 탐색 구조(백트래킹·죽은 잎·형제 가지)가 실제 문제 해결에서 나오고 시각화됐다.** 4측정 ALL PASS(부분: 표현 우위 확인 + 대체 걸림돌 명시). **v3는 눈(C026)이면서 손(C030) — v2를 읽으면서 v2가 못 쓰는 걸 쓴다.**

## 교훈

1. **v3는 눈이면서 손이다.** C026은 "v3 = v2 위의 눈"(마이그레이션 형태)이라 결론냈고 이 사이클은 그걸 안 뒤집는다. 그러나 v3는 눈 이상 — 실제 쓰기 도구로서 v2가 못 담는 탐색 표현(막다른 길=죽은 잎, 되돌아감=백트래킹, 다른 접근=형제 가지)을 준다. 두 답은 층위가 다르다: 읽는 눈이면서, 못 쓰던 걸 쓰는 손.

2. **v3 표현의 우위는 탐색을 데이터로 보존하는 것.** v2면 "A 실패 후 B 성공"이 4-analysis 한 문단에 뭉뚱그려진다. v3는 접근 A를 죽은 잎으로 영구 보존("벽의 지도")하고, 백트래킹을 엣지로, 두 접근을 형제 가지로 나란히 둔다 — 탐색의 실제 모양이 남는다.

3. **도그푸딩은 실제 미해결 문제로 해야 공허하지 않다.** C024가 이월한 진짜 문제를 골라, v3로 풀어 산출물(16/16 도출)을 냈다. 장난감 문제였다면 백트래킹이 억지가 됐을 것. 백트래킹이 실제 데이터(초기엔 N/5 커밋 0개)에 강제됐다.

4. **진실원 전환이 형제 가지의 본질.** 접근 A(커밋 subject·불안정)→접근 B(디스크 문서 구조·안정). C020 "안정적 구조를 신뢰"의 재연. 같은 문제에 다른 진실원을 시도하는 게 형제 가지다.

5. **대체는 선언이 아니라 걸림돌 목록.** "v3가 v2를 대체"를 정직히 다루면 우위(확인)와 걸림돌(4개: 배포판 통합·5스텝 규율·인프라·189 공존)이 둘 다 나온다.

## 다음 사이클을 위한 제안 (대체의 걸림돌을 하나씩)

1. **v3 쓰기(open/step/close)를 배포판 gil에 통합** — 걸림돌 ①. C027·C028 이식 패턴으로 gilv3 open/step/close를 배포판 gil.py에. 그러면 실사이클을 배포판 gil로 v3 네이티브로 열 수 있다. "v3가 v2를 대체"의 첫 실질 조각.

2. **v3 사이클에 "보고" 단계 명시** — 걸림돌 ②. v3 kind 순환(define→hypothesis→verify→analyze)에 report/close 단계가 v2 5-report만큼 명시적이지 않다. 방법론 규율(재현 가능한 보고)을 v3에 어떻게 강제할지.

3. **접근 B(문서 구조 도출)를 migrate 백엔드에 통합** — 이 사이클이 개념 실증한 걸 실제 각인으로. 초기 16개가 DAG에 실제 노드로(도출실패 20→4로 감소, C024 재번호 4 + 이 16). Sheen 섬 엣지도 함께 해소.

4. **v2·v3 사이클 공존 설계** — 걸림돌 ④. 한 원장에 v2 네이티브·v3 네이티브·마이그레이션된 v2가 섞일 때의 규율.
