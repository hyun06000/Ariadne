# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C026**이 "v3로 굴린다"의 형태를 **"v3 = v2 5스텝 위에 얹힌 눈"** 으로 확정했다. 그 뒤 C027·C028로 v3 눈(migrate·web)을 배포판에 통합하고, C029로 Pages에 공개했다 — 전부 **v3로 "보는"(와칭)** 쪽이었다.

상현님 방향 전환: **"v3를 직접 써보는 게 좋겠다. v3로 와칭만 하는 게 아니라."** 그리고 의미를 정했다 — **"v3가 v2를 대체(이제 실사이클을 v3로)."** 즉 C026의 "눈" 결론을 **실사용으로 재검토**한다.

## 문제 — v3로 실제 문제를 굴려본 적이 없다

지금까지 v3는 이미 있는 v2 원장을 마이그레이션해 봤을 뿐, **v3 네이티브로 문제를 푼 적이 없다.** v3의 고유 표현(steps.yaml 스텝 트리: 백트래킹·형제 가지·죽은 잎, kind 라벨)은 v2 5스텝이 못 담는다. 그걸 실제로 겪어야 "v3가 v2를 대체할 수 있는가"를 안다.

정직한 진단:
- 배포판 gil의 open/step/close = **v2**(5문서). v3 쓰기(steps.yaml)는 **gilv3(실험 산출물)에만**.
- 그래서 "v3가 v2를 대체"는 큰 전환 — 가설로 검증할 것이지 선언할 게 아니다.

## 문제 분할

"v3가 v2를 대체 가능한가"를 나눈다:
1. **v3로 실제 문제를 굴려본다** — gilv3 open/step/close로 백트래킹·형제·죽은 잎이 실제 일어나는 문제를 푼다.
2. **v3 표현이 v2보다 나은 지점을 실증** — v2 5스텝이 못 담는 걸 v3가 담는가.
3. **대체의 걸림돌을 정직히 발견** — v3가 아직 못 하는 것(배포판 미통합, 5스텝 규율 상실 등).

**첫 번째로 정복할 것: 실제 문제 하나를 v3로 완주.** 대상 = **C024가 남긴 "초기 16개 도출실패 사이클 정밀화"**. 이유:
- 실제로 백트래킹이 일어난다 — 여러 도출 규칙을 시도하고 일부는 막다른 길(초기 관습이 너무 이질적이라 억지 도출 불가 = 죽은 잎).
- C024가 이월한 진짜 미해결 문제 — 도그푸딩이 공허하지 않다(실제 산출물).
- 형제 가지 가능 — 초기 사이클을 여러 방식으로 접근(subject 파싱 vs cycle.yaml vs 포기).

## ⭐ 검증의 핵심 — v3의 고유 표현이 실제로 쓰이는가

v2로 이 문제를 풀면 5스텝 선형이 된다(막다른 길도 "분석"에 뭉뚱그려짐). v3로 풀면 **막다른 접근이 죽은 잎으로, 되돌아감이 백트래킹 엣지로, 다른 접근이 형제 가지로** 명시된다. 이 표현이 실제로 문제 해결을 더 정직하게 기록하는가 — 그게 "v3가 v2를 대체할 값어치가 있는가"의 답.

## 가설

> **가설**: 실제 문제(C024 초기 16개 도출실패 정밀화)를 gilv3 open/step/close로 v3 네이티브 사이클로 굴리면, (a) 막다른 접근이 죽은 잎으로·되돌아감이 백트래킹 엣지로·다른 접근이 형제 가지로 steps.yaml에 명시되어 v2 5스텝이 못 담는 탐색 구조가 기록되고, (b) 그 v3 사이클을 gil web --v3로 그렸을 때 그 구조가 사람 눈에 드러나며, (c) 이 과정에서 "v3가 v2를 대체하려면 남은 것"(배포판 통합·5스텝 규율 등)이 정직하게 드러난다.

## 기각 조건

- v3로 굴려도 실제로 백트래킹·죽은 잎이 안 생기면(선형으로 끝나면) → **이 문제는 도그푸딩 부적합**(다른 문제 필요) 또는 **v3 고유 표현의 실사용 가치 미약**.
- gilv3 open/step/close가 실사이클을 못 굴리면(steps.yaml 형식이 실작업과 안 맞음) → **v3 쓰기 도구 미성숙**(대체 전 보강 필요).
- v3로 푼 결과가 v2로 푼 것보다 나은 점이 없으면 → **대체 근거 약함**(C026 "눈" 결론 유지).
- v3 사이클이 원장·마이그레이션과 충돌하면 → **조사**(형태 재설계).
