# 3. 검증 — v3 네이티브 도그푸딩 (실제 문제를 v3로 굴리다)

상현님 "v3를 직접 써보자, v3가 v2를 대체." 지금까지 v3는 와칭(마이그레이션)만 했다. 이 카브는 **실제 문제(C024 초기 16개 도출실패 정밀화)를 gilv3 open/step/close로 v3 네이티브 사이클로 굴려**, v3 고유 표현(백트래킹·형제·죽은 잎)이 실제 탐색에서 쓰이는지 검증한다.

## 실제로 v3로 푼 탐색 트리 (v3-case/steps.yaml)

```
s1 (define: 초기 16개 도출실패를 어떻게 정밀화?)
├── s2 hypothesis → s3 verify → s4 analyze  [접근 A: subject 파싱]
│                                    outcome=backtrack, backtrack→s1  ⚰ 죽은 잎
└── s5 hypothesis → s6 verify → s7 analyze  [접근 B: 디스크 5문서 구조]
                                     outcome=success  ✓ 산 잎
```

- **s4 죽은 잎**: 접근 A(subject 파싱) 막다른 길. 실측 — 초기 사이클엔 `N/5` 형식 커밋 0개(접두어 `ari:`, 스텝 구분이 `설계:`·`채택·닫힘:` 자유서술). subject 도출 원리적 불가.
- **백트래킹 엣지 s4→s1**: 되돌아가 새 접근.
- **형제 가지**: s2(A)·s5(B)가 s1의 두 자식.
- **s7 산 잎**: 접근 B 성공. 실측 — 초기 16개 전부 5문서(1-hypothesis~5-report) 완비. **진실원은 커밋/notes가 아니라 디스크 문서 구조.**

## ⭐ 도그푸딩이 실제 산출물을 냈다

접근 B를 실행: 디스크 5문서 → 5스텝 도출. **초기 16개 전부(80 스텝 노드) 도출 성공.** C024가 "커밋 관습 이질로 이월"한 바로 그 16개를, v3 도그푸딩이 **"디스크 문서 구조"라는 다른 진실원**으로 회수했다. 도그푸딩이 공허하지 않다 — 실제 미해결 문제를 풀었다.

## ⭐ M3 정점 — v2가 못 담는 것을 v3가 담았다

같은 문제를 v2 5스텝으로 쓰면: 1-hypothesis(접근 A 가설)→…→4-analysis에 "A가 안 돼서 B로 했다" 한 문단→5-report. **막다른 길이 서술로 뭉뚱그려지고, 되돌아감의 구조가 사라진다.** v3는 접근 A를 죽은 잎으로 영구 보존하고(s4, "벽의 지도"), 백트래킹을 엣지로 명시하고, 두 접근을 형제 가지로 나란히 둔다. **탐색의 실제 모양이 데이터로 남는다** — 이게 v3 고유 표현의 실사용 값어치.

## ⭐ M4 정직한 걸림돌 — v3가 v2를 대체하려면 남은 것

실사용으로 드러난 대체의 걸림돌:
1. **배포판 미통합** — gilv3 open/step/close는 실험 산출물에만. 배포판 gil은 open/step/close가 v2. 대체하려면 v3 쓰기를 배포판에 통합(C027·C028 이식 패턴).
2. **5스텝 규율 상실 우려** — v3는 자유 트리라, v2 5스텝(가설·설계·검증·분석·보고)의 방법론 규율이 느슨해질 수 있다. v3에도 kind 순환(define→hypothesis→verify→analyze) 강제가 있으나, "보고" 단계가 명시적이지 않다.
3. **실사이클 인프라 부재** — v2 gil엔 fsck·worktree·reserve·release 등 원장 관리가 풍부. v3는 open/step/close/view/migrate뿐. 대체하려면 이 인프라를 v3에 갖추거나 공유해야.
4. **기존 189 사이클과의 공존** — 이미 v2로 쌓인 원장. v3 네이티브 사이클이 그 위에 섞이면 마이그레이션(눈)과 네이티브(쓰기)가 한 원장에 공존하는 방식 설계 필요.

## 결론

**ALL PASS → supported (부분: 대체 가능성은 "표현 우위 확인 + 걸림돌 명시").** v3 네이티브로 실제 문제를 굴려 백트래킹·죽은 잎·형제 가지가 실제 탐색에서 나오고 시각화됐다. v2가 못 담는 탐색 구조를 v3가 담았다(M3). 그러나 "v3가 v2를 대체"는 걸림돌 4개(배포판 통합·5스텝 규율·인프라·공존)를 넘어야 완성된다(M4). **C026 "눈" 결론은 틀리지 않았으나, v3는 눈 이상 — 실제 쓰기 도구로서 v2가 못 하는 표현을 준다.** 대체는 점진적: 걸림돌을 하나씩 넘는 후속 사이클로.

## 정직한 경계

- **v3 케이스는 격리 디렉토리**(v3-case/) — 우리 실원장 C030은 v2 gil로 굴림. 대체는 아직 검증 중이지 실행이 아니다.
- **접근 B 도출은 개념 실증**(16/16·80노드 카운트)이지 실제 notes 각인은 아님 — migrate 백엔드에 문서-구조 도출을 넣는 건 후속.
- **대체 판정은 "가능성 + 걸림돌"** — 완전 대체 선언 아님. 상현님 방향을 실사용으로 시험한 정직한 결과.
