## 데이터 직접 관찰

수치 뒤로 들어가 실측 트리를 직접 본다:

```
s1 [define]                        ← P0 뿌리
  s2→s3→s4 [analyze/backtrack→s1]  ← C012 가지 (죽은 잎)
  s5→s6→s7 [analyze/backtrack→s1]  ← C013 가지 (죽은 잎)
  s8→s9→s10 [analyze/success]      ← C014 가지 (산 잎)
```

- **v2의 죄가 실제로 사라졌는지 노드 단위 확인**: `s10.parent = s9`(steps.yaml에 직접
  확인). v2 원장에서는 C014의 `parent`가 C013이었다 — 시간순 "다음"을 논리적 "자식"으로
  뭉갠 것. v3 steps.yaml에서 s10의 부모는 **자기 가지의 검증 노드 s9**이고, C013(=s7)과는
  형제 관계다. 둘을 잇는 유일한 논리 엣지는 s7의 `backtrack→s1` — "s7이 벽에 부딪혀
  뿌리로 되돌아갔고, 거기서 s8 가지가 새로 났다"는 진실. **이 엣지가 v2엔 아예 없었다.**
- **죽은 잎이 벽의 지도로 보존됨**: s4.md·s7.md 본문이 기각 사유(상호작용·detKey 배제 /
  실브라우저 계측기 필요)를 그대로 담는다. roundtrip.py가 이 둘을 `outcome=backtrack`으로
  식별하고 "새 형제 가지 아래 s1"로 파생 — 후대가 이 두 길을 다시 안 걷는다.
- **명령 파생이 관습 없이 됨**: derive_action이 kind만 보고 define→hypothesis→verify→
  analyze를 파생하고, analyze는 outcome으로 close(s10)·backtrack(s4,s7)을 갈랐다. **어느
  노드에서도 `???`(스키마 밖 관습 필요)가 안 나왔다** — 모델이 명령을 완전히 강제한다.

## 예상과 달랐던 것

- **`fail`을 결국 안 썼다**: 설계에서 outcome을 {fail, backtrack, success}로 뒀으나,
  실사례를 담아보니 모든 되돌아감을 **명시적 `backtrack` 포인터**로 적는 게 자연스러웠다
  (목적지가 데이터에 남으니 벽의 지도가 더 선명). `fail`(목적지 자동 해소)은 이 사례에서
  잉여였다. → **다음 사이클의 씨앗**: fail을 스키마에서 뺄지(backtrack로 일원화), 아니면
  "가장 가까운 조상으로 자동" 편의로 남길지는 실사용에서 결정(그리디 이월).
- **디렉토리 중첩을 안 쓰길 잘했다**: 세 가지가 s1의 형제라 디렉토리 계층(v2식 중첩)으로는
  형제 백트래킹을 못 담았을 것. 평면 리스트 + 포인터가 트리를 온전히 담았다 — 설계 시
  "v2 5-.md가 깨진다"고 예측한 지점이 실측으로 확인됐다.

## 판정

**채택 (supported).** 가설대로 C001 스키마를 `steps.yaml`+노드 .md 디스크 표현으로
구체화하니 실사례가 (a) 왜곡 0 (b) 왕복 무손실 (c) 명령 파생 가능으로 담겼다. 세 kill
조건 미발동, 산 잎(s10 성공) 도달. **이 사이클의 스텝 트리 자체가 산 잎에서 성공** —
되돌아감 없이 첫 정답에 닿았으므로 닫는다(그리디). fail 필드 정리·id 발급 엣지케이스·
스텝 DAG(다중 부모) 가능성은 정직히 이월.
