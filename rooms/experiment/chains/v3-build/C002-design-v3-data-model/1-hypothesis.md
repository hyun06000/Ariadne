# 1. 가설 수립

## 이전 사이클의 교훈

부모: **v3-build/C001-define-v3-target** (supported). C001이 확정한 것:

- v3의 첫 정복 대상 = **데이터 모델(스텝 트리 표현)**. 다른 모든 것(명령·뷰어·백업)이
  이 모델에서 파생하므로 뿌리부터 정한다.
- **최소 모델 스케치** (종이): 스텝 노드 =
  `{id, kind(문제정의*/가설/검증/분석), parent, outcome(실패/되돌아감/성공), backtrack}`.
  백트래킹 = 분석 노드가 `outcome=되돌아감` + `backtrack`이 가리키는 조상 문제정의*에서
  새 형제 가지가 자란다.
- **v2 `parent`의 죄**: "시간 순서상 다음"과 "논리적 자식"을 구별 못 해 백트래킹을
  선형 계승으로 뭉갰다. v3는 `backtrack`을 **명시적 데이터**로 담아 이를 없앤다.
- C001은 **종이 스케치**에서 멈췄다(그리디). 디스크 표현·id 발급·상태기계·backtrack
  해소·close 판정은 전부 C002의 몫으로 정직히 이월됐다.

## 문제 분할

C002가 풀어야 할 "완전 설계"를 가장 작은 단위로 분할한다:

1. **D1 — 노드 스키마 확정**: C001의 5개 필드가 충분한가? 부족·과잉 필드를 실사례로 검증.
2. **D2 — 디스크 표현**: 스텝 트리를 파일/디렉토리/yaml 중 무엇으로 담나. v2는 사이클당
   디렉토리 + 5개 고정 .md였다. v3는 스텝 수가 가변(트리)이므로 이 표현이 깨진다.
3. **D3 — id 발급 규칙**: 스텝 id를 어떻게 매기나(트리라서 v2의 단조 증가로는 형제·백트래킹
   순서를 못 담을 수 있다).
4. **D4 — kind 상태기계**: `문제정의*→가설→검증→분석→[분기]`가 어떤 전이만 허용하는지
   명시(모델이 명령을 강제하는 근거).
5. **D5 — backtrack 목적지 해소 + close 산 잎 판정**: `되돌아감`이 가리킬 수 있는 조상은
   어디까지인가, 사이클은 언제 닫히는가.

**지금 정복할 첫 문제 = D1+D2 (스키마 + 디스크 표현)**을 한 덩어리로 잡는다. 이유:
스키마는 디스크 표현과 분리해서 검증할 수 없다 — C001의 검증법("실제 스키마 파일로 써서
왕복 무손실")이 곧 "디스크에 쓴 것을 다시 읽어 트리를 복원"이므로, 스키마와 그 직렬화
형식은 한 검증에서 함께 선다. D3·D4·D5는 이 왕복이 통과하는 데 필요한 만큼만 이번에
확정하고, 남는 엣지 케이스는 그리디하게 이월한다.

## 가설

> **가설**: C001의 스텝 노드 스키마를 **디스크 표현(사이클 디렉토리 안의 단일
> `steps.yaml` = 스텝 노드 리스트 + 노드별 본문 .md)**으로 구체화하면, 실사례
> loomlight/C012→C013→C014를 **하나의 스텝 트리로 디스크에 쓰고 다시 읽어 복원**했을
> 때 (a) 왜곡 0(백트래킹·죽은 잎·산 잎이 전부 보존)이고 (b) 왕복 무손실(모델→디스크
> →재구성이 동형)이며 (c) 그 표현만으로 open/step/close가 파생된다.

## 기각 조건

- **K1**: 실사례를 디스크 표현으로 쓸 때 C001의 5필드로 담기지 않는 정보가 나온다
  (필드 부족 → 스키마 미완성, 되돌아가 D1 재정의).
- **K2**: 디스크→재구성 왕복이 무손실이 아니다 — 쓴 것과 읽은 것이 트리로서 다르다
  (직렬화 형식 실패 → D2 재정의).
- **K3**: 표현만으로 open/step/close 전이 규칙이 결정되지 않는다 — 명령을 파생하려면
  스키마 밖 관습이 추가로 필요하다 (모델이 명령을 강제하지 못함 → D4 재정의).
