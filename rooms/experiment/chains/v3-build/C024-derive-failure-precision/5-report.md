# 5. 결과 보고

## 요약

C020 이래 이월된 20 도출실패 사이클을 원인별로 갈라, **재번호 부류 4개**(loom/C083·C106, v3-build/C004·C007)를 provenance 추적으로 회수했다. 재번호는 번호만 바꾸고 slug은 유지한다는 불변식을 이용해 옛 full id(`C{old}-{현재 slug}`)로 스텝 커밋을 회수, 도출실패 20→16. slug 포함 매칭으로 오염 0(같은 번호 다른 사이클 분리), 잔여 16은 전부 초기 커밋 관습 이질로 정직히 남겼다. **4측정 ALL PASS → supported.**

## 교훈

1. **정밀화의 정직은 "무엇이 회수 가능한가"를 가르는 것.** 20을 억지로 0 만들 수도 있었으나(초기 관습에 특수 규칙), 그건 C020이 경계한 억지 도출. 명확한 단일 원인(재번호)에 명확한 데이터(slug 보존 매핑)가 있는 4개만 회수하고, 근본적으로 이질적인 초기 16개는 남겼다. **정밀화는 유령을 다 없애는 게 아니라 없앨 것과 남길 것을 정확히 가르는 것**(C017·C020 연장).

2. **전역 유일 식별자를 잊으면 데이터가 조용히 뭉개진다(C021 재연).** 첫 도출기가 옛 번호만으로 매칭하니 같은 번호 C003의 두 사이클(v3-viewer / first-commands)을 다 긁어 회수가 10커밋으로 오염됐다. slug 포함 full id가 유일 키 — 부분 식별자로 매칭하면 조용한 오염. C021 short_id 함정의 세 번째 실증.

3. **재번호의 불변식: 번호만 바뀌고 slug은 유지.** git·5-report 전수 확인. 이 규율이 데이터로 확인돼 옛 id 재구성을 결정적으로 만들었다 — 좋은 재번호 관습이 미래 추적을 열었다(C019 "좋은 기록 관습이 미래 도출 연다"의 쌍대).

4. **진실원 다중화의 통합.** 재번호 매핑이 git 커밋(`재번호 C<old>→C<new>`)과 5-report 주석 두 곳에 흩어져, 둘을 합쳐야 완전(C083은 git만, C004·C007은 5-report). 안정적(git) 1순위 + 보완(report). C020 "안정적 구조를 신뢰"의 연장.

1. **회수 4건을 실제 원장에 소급 (선택)** — 이 카브는 회수 가능성 검증(조회)까지. 실제 notes 각인은 `gil migrate` 재실행 시 정밀화 도출기를 태우면 자동 반영. 도출기 정밀화를 `full_ledger_migrate`에 통합할지 결정(그러면 `gil migrate`가 4건 더 각인 → DAG 132→136).
