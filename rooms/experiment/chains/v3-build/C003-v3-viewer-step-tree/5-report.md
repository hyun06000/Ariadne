## 요약

C002가 확정한 스텝 트리 표현(`case-c012-c014/steps.yaml`, 10노드) 위에 **v3 스텝 트리
뷰어**를 지었다 — 한 사이클 안의 스텝들을 트리로 그려 parent 가지·backtrack 되돌아감·
죽은 잎/산 잎을 자기완결 SVG로 동시에 비춘다. 순수 stdlib 생성기(steptree.py)와 헤드리스
실측기(measure.py)로 4측정 전부 통과(ALL PASS), 산 잎(가설 채택)에 도달 → 닫는다.

## 무엇을 했나

1. **생성기** (`steptree.py`, 순수 stdlib): steps.yaml → 자기완결 단일 HTML(인라인
   SVG+CSS, 외부 리소스 0). yaml 미의존 자작 파서 → 트리 복원 → depth(세로)×형제(가로)
   좌표 → parent line·backtrack path·kind 색·잎 표식·범례·헤더.
2. **CLI** (`render.py`): C002 데이터 → `out.html`(9229 B, 재현물).
3. **헤드리스 실측** (`measure.py`, stdlib html.parser): out.html 파싱 + 진실원 독립
   재파싱 대조로 M1~M4 판정. `render.py && measure.py` → ALL PASS ✓, exit 0.

## 핵심 발견

- **v2엔 없던 backtrack 엣지가 실제 좌표로 화면에 났다.** s7의 렌더 path가 y 587→215로
  **위로**(잎 depth 3 → 뿌리 define s1) 되돌아간다. C002가 "이 엣지가 v2엔 아예 없었다"고
  한 그 되돌아감을, 뷰어가 곡선 한 가닥으로 비춘다. v3 뷰어가 v2와 달라야 하는 이유의
  핵심(임무 명세 (b))이 실물이 됐다.
- **outcome이 잎의 운명을 갈랐다.** s4·s7·s10 셋 다 kind=analyze지만 `leaf-dead`(점선·
  바랜·"벽의 지도" 배지) vs `leaf-live`(굵은 초록·"산 잎" 배지)로 갈렸다. v2 고정 5스텝엔
  outcome 개념이 없어 이 구별이 불가능했다 — 죽은 잎을 침묵시키지 않고 자산으로 보이게
  한 것(chain.md "리젝트가 자산")이 렌더로 성립.
- **정적 계약엔 정적 실측.** v2 뷰어의 라이브 폴링은 CDP(실브라우저)를 요했지만, v3 스텝
  트리의 최소 계약은 정적 구조라 stdlib 파서 실측으로 충분·재현성 우위(브라우저 의존 0).
- **범례의 정직함이 측정자를 한 번 속였다.** 범례 스와치에 실제 엣지와 같은 class를 일부러
  써서(사람 눈에 동일해야) measure.py가 범례 line을 3번째 backtrack으로 셌다. 수정은 뷰어가
  아니라 측정자 — 데이터 엣지는 `data-from`으로 거른다. **뷰어의 "사람에게 정직한 표현"과
  "측정자의 데이터 필터"는 다른 축**임을 실측이 드러냈다(C010 귀속 정직의 재현).
