# 2. 실험 설계

오직 1-hypothesis.md의 가설 — **v3로 굴린다 = v2 5스텝 위에 얹힌 눈(notes)** — 을 검증한다. 검증 대상은 **이 사이클(C026) 자신**이다(도그푸딩).

## 정답을 도구보다 먼저 고정한다

정답: v2 실사이클(5스텝 문서·커밋)은 원장에 그대로 각인되고, v3 눈(gil migrate가 만든 notes)이 그 커밋들을 읽어 (a) 스텝 트리, (b) 사이클 DAG를 steps.yaml 없이 재구성한다. C026 자신이 이 성립의 증거가 된다.

## 절차

1. **C026을 v2 gil로 5스텝 완주** — 지금 하고 있는 그대로(1~5 문서 + gil step 커밋). 원장이 진실.
2. **v3 눈을 C026에 적용** — `gil migrate`(격리 복제본)로 C026의 5스텝 커밋에 notes 지문 소급 + Cycle-Parent 엣지 접합. C022/C023 파이프라인 재사용.
3. **눈이 C026을 읽는지 실측**:
   - **스텝 트리**: C026의 notes에서 도출된 스텝 시퀀스(s1~s5, kind=define/hypothesis/verify/analyze/report)가 5스텝을 담는가.
   - **사이클 DAG**: C026이 부모 C023으로 이어지는 엣지가 notes로 접합됐는가.
4. **원장 불변 확인** — v3 눈 적용이 C026의 커밋·문서·cycle.yaml을 하나도 안 바꾸는가(C018 계약, 기각 조건 4 겨냥).
5. **문서 갱신**: 이 형태를 SPEC/README에 반영할 초안. "v3로 굴린다 = v2 5스텝 + v3 notes 눈"을 문서화(도그푸딩→문서, "문서가 곧 테스트"). 단 이 사이클은 **형태 결정 + 최소 실증**까지; 전면 문서 재작성은 이월(범위 절제).

## 준비물

- C023판 마이그레이션 파이프라인(`gil migrate`) — C026을 대상으로 격리 실행.
- C024 정밀화 도출기(선택) — C026은 재번호 아니라 불필요.
- 격리 복제본(우리 원장 조회만).

## 측정 방법 (4측정)

| 측정 | 확인 | 통과 기준 |
|---|---|---|
| **M1 스텝 트리 읽힘** | C026의 notes에서 5스텝(s1~s5) 도출, kind가 5스텝(가설~보고)에 대응 | 5스텝 노드 도출 |
| **M2 사이클 DAG 읽힘** | C026→C023 부모 엣지가 Cycle-Parent notes로 접합 | 엣지 복원 |
| **M3 원장 불변** | v3 눈 적용이 C026 커밋·문서·cycle.yaml 불변 | 바이트 동일 |
| **M4 steps.yaml 불요** | C026 디렉토리에 steps.yaml 없이도 M1·M2 성립 | steps.yaml 부재 확인 |

## 이 형태가 답이면 따라오는 것 (설계 함의)

- **도그푸딩**: v3로 굴린다 = 지금처럼 v2 5스텝 + `gil migrate`로 눈 얹기. 특별한 새 워크플로 불요.
- **문서**: SPEC에 "v3 = v2 위의 notes 눈" 절 추가. steps.yaml은 v3 표현의 한 선택지(탐색적 사이클용)이지 실사이클 필수 아님.
- **배포**: gil 배포판에 migrate + 뷰어(읽기 축)를 얹으면 v3 정식. gilv3 실험 산출물을 gil.py에 통합(이름은 gil).
