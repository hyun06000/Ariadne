# 1. 가설 수립

## 이전 사이클의 교훈

부모 **C021**(위상 접합)까지 v3 마이그레이션 왕복이 **격리 복제본에서 전부 안전 실증**됐다:

- ②읽기호환(C017) — v2 유령은 무해하게 건너뛰어진다(파괴 아님).
- ③소급각인(C018/C019/C020) — `git notes`가 커밋을 안 건드리고 v3 지문을 소급 각인한다. C018 정점: **append-only는 소급을 금지하지 않는다, '커밋을 고치는 소급'만 금지한다.**
- ④위상접합(C021) — cycle.yaml parent를 사이클 간 `Cycle-Parent` notes 엣지로 소급, 131 노드·130 엣지 DAG 복원.

lineage **C018**: 소급각인의 근간이 `git notes`(커밋 SHA 불변, refs/notes에만 append)라는 원리. 이 사이클은 그 원리를 **실제 우리 원장에 적용**한다.

C020/C021의 정직한 경계가 한결같이 못 박은 것: **"격리 복제본(드라이런). 실제 적용은 별도 결정."** 그 별도 결정이 이 사이클이다.

## 문제 분할

마이그레이션 4단계 중 남은 것:
- ①**동결백업** ⬜ — 실제 적용의 선행조건이자 되돌림의 토대. 유일한 미완.
- ②③④ ✅ — 격리 실증 완료.

실제 원장 적용은 다음 최소 카브들로 나뉜다:
1. **동결백업(①)** — 적용 직전 refs 상태를 되돌릴 수 있게 못박는다.
2. **실제 적용** — notes를 복제본이 아닌 **우리 진짜 refs/notes**에 각인한다.
3. **되돌림 실증** — 백업 지점으로 되돌리면 적용 이전 상태로 정확히 복귀함을 실측한다.

**첫 번째로 정복할 것: 셋을 한 카브로 묶되, 되돌림 가능성이 골격이다.** 이유 — 실제 원장을 건드리는 유일한 문턱이므로, "적용"과 "되돌림"은 분리하면 반쪽이다. 동결백업 없는 적용은 무책임하고, 적용 없는 동결백업은 공허하다. 셋이 한 왕복(백업→적용→되돌림 실증)으로 닫혀야 "실제 적용이 안전하다"가 증명된다.

## 핵심 통찰 — 무엇이 "동결백업"인가

마이그레이션 노트의 4단계는 원래 v2 데이터 변환을 상정했다. 그러나 C018이 밝힌 대로 **원장 커밋은 애초에 안 바뀐다**(notes는 refs/notes에만 append). 따라서 실제 "되돌림 대상"은 커밋이 아니라 **refs/notes/\* 뿐**이다.

→ **동결백업 = 적용 직전 refs/notes 상태를 태그(또는 백업 ref)로 못박는 것.** 되돌림 = 그 백업 ref로 refs/notes를 리셋(또는 refs/notes 전체 삭제). 커밋·cycle.yaml·작업 트리는 애초에 건드리지 않으니 백업 대상이 아니다.

## 가설

> **가설**: 적용 직전 refs/notes 상태를 백업 ref로 동결(①)한 뒤 C020/C021 파이프라인(노드 소급 + 위상 접합)을 **우리 실제 원장의 refs/notes에 각인**하면, (a) 원장 커밋 SHA·cycle.yaml·작업 트리는 전부 불변이고, (b) 재구성기가 우리 실제 원장에서 v3 사이클 DAG를 복원하며, (c) 백업 ref로 되돌리면 적용 이전 상태(notes 부재)로 정확히 복귀한다.

## 기각 조건

- 적용 후 원장 커밋 SHA가 하나라도 바뀌면 → **기각**(notes가 append-only가 아님, C018 반증).
- cycle.yaml·작업 트리 파일이 하나라도 바뀌면 → **기각**.
- 되돌림(백업 ref로 리셋) 후 notes가 완전히 제거되지 않고 잔재가 남으면 → **기각**(동결백업이 되돌림을 보장 못 함).
- 실제 원장 재구성 결과가 C021 격리 실측(131 노드·130 엣지, 머지 4)과 유의하게 다르면 → **조사**(복제본≠원장이라는 숨은 차이).
