## 데이터 직접 관찰

**번호 중복이 공짜로 잡혔다.** M2에서 C001-native·C001-dup 두 v3 사이클을 만드니
`R1 demo: 번호 001 중복: C001-dup, C001-native`. 이 위반 메시지는 내가 v3용으로 새로
쓴 게 아니라 **기존 R1 번호 중복 검사(gil.py 495)**가 낸 것 — v3 record를 `numbers`
딕셔너리에 편입하기만 하니 자동 발화. 코드로 확인: v3 record도 `numbers.setdefault(
m.group(1), []).append(cid)`(397)를 타 C001에 두 cid가 쌓이고, 495의 `len(dupes)>1`이
잡는다. **records 형태 통일의 핵심 이득 — v3 전용 검사는 V3-ROOT 하나뿐, 번호·id·dir은
전부 재사용.**

**거짓위반 0의 메커니즘을 확인.** M3에서 정상 v3 사이클에 R8(status closed인데 closed
없음) 같은 거짓위반이 안 났다. 원인 두 겹: (1) v3 record에 status를 안 넣어(None) R8의
`if status == "closed"` 조건이 자연 미발화, (2) 그래도 확실히 하려 `_v3`면 루트 검사 후
`continue`로 뒤 R규칙 전체를 건너뜀. 방어 이중이라 v3가 cycle.yaml 전용 규칙에 안 걸린다.

**"사이클 0개→1개"가 사각지대 소멸의 직접 증거.** C039 말미 실측에서 v3 사이클은
"사이클 0개"로 투명했다. C040 후 같은 저장소가 "사이클 1개". fsck·log가 v3를 원장
구성원으로 셈 — 실사이클을 v3로 열어도 이제 무결성 검사 안에 있다.

## 예상과 달랐던 것

- **V3-ROOT 하나로 충분했다.** 설계 때 "v3 전용 검사(루트 define·번호 유일성·트리 정합)"
  셋을 예상했으나, 번호 유일성은 R1이 이미 하고, 트리 정합은 이번 범위(루트 존재)로
  최소화 가능해 **새 검사는 V3-ROOT 하나**. records 통일이 나머지를 흡수. 최소 표면.
- **길3이 길2의 이득까지 가져왔다.** 길2(steps.yaml 직접 수집)의 걱정은 "R규칙 전면
  재매핑·거짓위반"이었는데, records 통일 + `continue` 가드로 **재매핑 없이** v3를 수집
  (길2의 네이티브성)하면서 R규칙은 v2 전용 유지(길3의 안전). 두 길의 좋은 합.

## 판정

**채택 (supported).** 가설대로 fsck·load_chain_records가 v3 사이클을 인식하고(사이클
0→1개), 번호 중복이 R1로 검출되며(기각조건 3 해소), v2 인식·conformance 121/121
무회귀(기각조건 1 불충족). 기각조건 2(R규칙 근본 충돌)도 불충족 — `continue` 가드로
거짓위반 0. 세 기각조건 모두 회피.

**정직한 경계**: v3 전용 무결성은 **루트 define 존재(V3-ROOT)까지만** — 트리 전체 정합
(모든 노드 parent 유효·백트래킹 정합·죽은 잎 규칙)은 이월. author/parent 계보(C039의
두 번째 경계)는 여전히 v3 사이클에 미기록 — 이번은 "인식"이지 "계보 복원"이 아님.
log/graph의 v3 표현(사이클로 세지만 계보 그래프 노드로는 아직)도 이월.
