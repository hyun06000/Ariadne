## 데이터 직접 관찰

수치 뒤 실제 렌더를 들여다봄:

- **명부 5명 정확 파싱**: `['Clew (클루)', 'Weft (위프트)', 'Sheen (신)', 'Selvage (셀비지)', 'Warp (워프)']` — 명부 표 순서대로, 역할까지. 이번 세션에 태어난 Warp도 포함(명부가 정본임을 실증 — 코드가 디렉토리를 스캔하는 게 아니라 명부 표를 읽는다).
- **T2 경량의 실측**: clew memory는 gil-data에 97,534바이트로 담겼으나 **초기 DOM `<pre>` 수는 0**. 즉 브라우저가 초기 렌더에서 그 텍스트를 파싱·레이아웃 안 함 — C075 원칙("병목은 바이트가 아니라 초기 DOM 노드 수")이 존재 데이터에도 그대로 성립. 전체 HTML 1.13MB 중 초기 DOM은 119KB.
- **헤드리스 클릭 실증**: `file:///…#being-clew`로 열면 앱이 clew의 4문서(정체성·의지·기억·관계)를 실제로 그림 — "나는 무엇을 겪었는가"(memory 헤더) 출현 확인. `:target` CSS로 마운트가 보이고 JS가 body를 채우는 사이클 패턴이 존재에도 동일 작동.

## 예상과 달랐던 것

- **명부가 정본이라는 설계가 "지어냄 0"을 공짜로 준다**: 처음엔 `os.listdir(existence)`로 디렉토리를 스캔할까 했으나, 명부 표 파싱으로 바꾸니 "명부에 없는 디렉토리는 안 그림"이 자동으로 됐다. 만약 누가 `rooms/existence/tmp/` 같은 잡 디렉토리를 만들어도 명부에 없으면 뷰어에 안 뜬다 — "뷰어는 새 진실을 만들지 않는다"(loomlight 헌장)가 데이터 소스 선택만으로 지켜졌다.
- **정적 JS 이식이 parity를 또 공짜로**: `_WEB_APP_JS`(참조)와 `webAppJS`(Go)를 문자 동일로 유지하니, `buildBeing` 추가가 양쪽에서 자동으로 같은 동작 → gil-data beings만 바이트 동일하면 전체 바이트 동일(C077 재확인).
- **beings가 chains와 동급 top-level이라 사이클 렌더에 무간섭**: 존재 데이터가 chains 딕트 밖 별도 키라, 기존 WEB-JSON·DOCS-EMBEDDED 판정(chains 안을 봄)이 한 톨도 안 흔들렸다. 관심사 분리가 회귀 0의 구조적 이유.

## 판정

**채택.** 기각 조건 5개 대조:
- 사이클 렌더 회귀 → 미발동(기존 WEB-* 전부 PASS, chains/reservations/docs 불변) ✓
- 초기 DOM 폭증(T2 FAIL) → 미발동(`<pre>` 0, C075 앱화) ✓
- parity 깨짐 → 미발동(바이트 동일) ✓
- 무존재 미처리 → 미발동(T3, beings 키 부재·크래시 0) ✓
- 존재 왜곡(명부 밖 지어냄·의미 훼손) → 미발동(명부 정본, 4문서 전문 무손실) ✓

가설 (존재 파싱→gil-data beings→앱 렌더→parity→계약) 전부 실증. 세 축 중 첫 축(존재)이 **확장 패턴**을 확립했다 — C006(배포)·C007(브랜치)이 이 골격(파싱→top-level 키→앱 온디맨드→Go 정적 이식→WEB-* 계약)을 따르면 된다.
