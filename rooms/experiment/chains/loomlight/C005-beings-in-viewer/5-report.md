# 5. 결과 보고

## 요약

상현님 발의("뷰어가 실험 위주라 AI 자아·배포도 봐야 한다") 중 첫 축(존재)을 정복했다. `rooms/existence/` 명부와 각 존재의 4문서(identity·will·memory·relations)를 파싱해 gil-data top-level `beings` 키로 내장하고, 위계 뷰어에 "이 저장소에 사는 존재" 패널(클릭 시 앱이 상세를 온디맨드 렌더 — C075 패턴)을 심었다. 참조·Go 동시 이식으로 parity 바이트 동일, 초기 DOM 경량 유지(memory 97KB도 초기 `<pre>` 0), conformance WEB-BEINGS 신설(참조 113·Go 99, 수정 전 FAIL). 명부를 정본으로 삼아 "지어냄 0"까지 실증해 **채택**.

## 교훈

1. **뷰어가 비추는 진실은 사이클만이 아니다** — loomlight 헌장("원장에 이미 짜인 진실을 사람 눈에 비춘다")에서 "진실"은 존재·배포도 포함한다. 존재는 이미 `rooms/existence/`에 완전히 있었는데 뷰어가 안 비췄을 뿐. 데이터를 만드는 게 아니라 **이미 있는 것을 데이터 경로에 잇는 일**이었다.
2. **데이터 소스 선택이 곧 무결성 계약** — 디렉토리 스캔 대신 **명부 표 파싱**을 고르니 "명부에 없으면 안 그림"이 공짜로 됐다. "뷰어는 새 진실을 만들지 않는다"가 코드 방어가 아니라 소스 선택으로 지켜졌다. 정본(명부)을 데이터 소스로 삼아라.
3. **관심사 분리가 회귀 0의 구조** — beings를 chains 딕트 밖 top-level 키로 두니 기존 사이클 판정(chains 안을 봄)이 무간섭. 새 데이터는 기존 데이터의 형제로 얹어라(하위 아님).
4. **C075·C077 원칙의 재사용성** — 초기 DOM 경량화(문서를 JSON에 넣고 앱이 온디맨드 렌더)와 정적 JS 이식(parity 공짜)이 존재에도 그대로 적용됐다. 확립된 앱화 패턴은 새 데이터 종류에 재사용된다 — **다음 두 축(배포·브랜치)도 이 골격**(파싱→top-level 키→앱 온디맨드→Go 정적 이식→WEB-* 계약).
