## 데이터 직접 관찰

- **바이트 동일의 실물**: `cmp py-hier.html go-hier.html` 무출력(동일). 두 몸이 서로 다른
  언어(Python f-string vs Go `fmt.Sprintf`/`strings.Builder`)로, 서로 다른 JSON 직렬화기
  (`json.dumps(ensure_ascii=False)` vs 손으로 짠 `webJSONPayload`)로 짰는데도 1.16MB가 한 바이트도
  안 갈렸다. 특히 `bake`에 `"hierarchy": true`가 `refresh` 뒤 조건부로 붙는 키 순서까지 일치.
- **가장 미묘했던 지점 — CSS 이스케이프**: 참조 `_WEB_HIER_CSS`의 `content:"\\25B8"`(파이썬 소스)는
  런타임 문자열 `content:"\25B8"`(백슬래시 1개). Go 원문자열(백틱) 안의 `\25B8`가 그대로 같은 바이트가 돼
  일치했다. `\2212`(−, minus)도 동일. 만약 Go에서 일반 문자열("...")로 썼다면 `\2`가 이스케이프로
  해석돼 갈렸을 것 — 원문자열 선택이 바이트 동일의 열쇠였다.
- **_dirs 이식**: 참조는 cid→디렉토리명 매핑을 JSON에 직렬화하지 않아 기본 출력을 지킨다. Go도
  `webChain.dirs`를 더하되 `webJSONPayload`가 참조하지 않게 해, M2b(기본 불변)가 그대로 통과했다.
- **미지 verdict 순서**: `_verdict_tally`의 "알려지지 않은 verdict는 삽입 순서로"를 Go에서 map
  랜덤 순회로 재현하면 갈렸을 것 — `firstSeen` 슬라이스로 삽입 순서를 보존해 막았다(실 저장소엔
  미지 verdict가 없어 이 경로는 방어적이지만, 픽스처가 생기면 바로 실증된다).

## 예상과 달랐던 것

- **stdout 정렬이 참조와 이미 갈려 있었다.** 변경 전 Go의 `--refresh` 로그는 `(체인 N개 · 자동
  리로드 N초)`로 괄호 **안**, 참조는 괄호 **밖**(`(체인 N개) · 자동 리로드`). conformance가 이
  문자열을 계약으로 안 봤기에 78/78이 유지돼 왔다. 이번에 위계 로그를 참조 형식(괄호 밖 `· 위계`)으로
  맞추며 Go의 stdout을 참조에 수렴시켰다 — 산출물(HTML)이 아닌 로그라 회귀 위험 없음(M3 재확인).
- **공유 `/tmp`가 병렬 세션의 함정이었다.** 검증 중 `/tmp/gil-go`가 스테일 바이너리를 잡아
  `--hierarchy`를 "알 수 없는 플래그"로 뱉는 현상을 겪었다. 재빌드 후 정상. 원인을 병렬 세션의
  `/tmp` 충돌로 짚고 산출물을 세션-로컬 경로로 옮겨 재현성을 되찾았다. C002에서 겪은 "공유 상태 =
  병렬의 위험"이 이번엔 워크트리가 아니라 `/tmp`에서 재발한 것 — 격리의 원칙은 빌드 산출물에도 든다.

## 판정

**채택(supported).** 참조의 위계 렌더 경로를 Go에 동형 이식하니 `gil web --hierarchy` 산출물이
참조와 **바이트 동일**해졌고(C020 "두 몸 한 계약"을 위계로 복원), 기본 출력·conformance는 회귀 0을
지켰다(opt-in 계약). C002가 정직히 이월한 미완을 이 사이클이 되찾았다. 릴리스·병합은 소환자 Clew의 몫.
