# 1. 가설 수립

## 이전 사이클의 교훈

최초 사이클(새 루트, parent 없음)이다. 부모 대신 두 곳에서 재료를 얻는다.

1. **chain.md의 문제 정의** — 원장에 짜인 진실을 사람 눈에 정직하게 비추는 것이 뷰어의 일이다.
   그 일은 지금 loom 체인 전역에 흩어져 있다(C005·C013·C015·C020·C028·C031·C042·C047·C048·C049).
2. **lineage의 정석(loom/C002 v0.2)** — 닫힌 사이클은 불변이다(R4·R5). 다른 갈래의 교훈을
   모을 때 물리적으로 옮기지 않고 `lineage` 필드로 가리킨다. 깃이 커밋 해시를 참조하는 것과 같다.

## 문제 분할

"뷰어를 강화한다"는 큰 문제를 작은 단위로 나눈다:
- (a) 흩어진 뷰어 사이클을 **한 곳에서 조망**할 수 있게 모은다.
- (b) 모으는 행위가 **불변성을 위반하지 않음**을 보장한다(옮기지 않고 가리킨다).
- (c) 모은 결과가 **기계로 검증 가능**해야 한다(gil log가 실제로 교차-체인 엣지를 그리는가).
- (d) 남은 뷰어 개선 과제를 **지도로 그린다**(다음 강화의 재료).

지금 정복할 첫 번째 문제는 (a)+(b)+(c)의 결합 — **lineage 기반 수집이 실제로 성립하는가**를
증명하고, 그 위에서 (d)를 산출물로 남긴다. 뷰어 코드 자체를 고치는 것(가로 압축 등)은
이 사이클의 범위가 아니다. 이 사이클은 **지도를 그리는 사이클**이지 길을 닦는 사이클이 아니다.

## 가설

> **가설**: 흩어진 10개 뷰어 사이클은 loom의 일관된 하위 계보를 이루며, `lineage`로 이들을
> 가리키는 전용 체인(loomlight)을 만들면 — 원본을 옮기지 않고도 — `gil log`가 교차-체인
> lineage 엣지(`⇠`)를 실제로 그려 이 계보를 한 화면에 드러내고, 남은 뷰어 개선 과제가
> 하나의 백로그로 모인다.

## 기각 조건

다음 중 하나라도 일어나면 가설은 틀렸다:
- `gil fsck`가 loomlight/C001의 lineage에서 R2(존재하지 않는 참조)·R3(전역 표기/동일 체인)
  위반을 보고한다 → 수집이 규칙 위반 없이 성립하지 않는다.
- `gil log`가 loomlight의 10개 lineage 중 하나라도 `⇠` 엣지로 그리지 못한다 → 조망이 실패.
- chain.md의 10개 링크 중 하나라도 실제 보고서 파일로 해소되지 않는다 → 색인이 거짓.
- 10개 사이클이 뷰어라는 일관된 주제로 묶이지 않고 이질적이다 → 계보 가설이 틀렸다.

## 요약

loom 체인에 흩어져 태어난 뷰어 사이클 10개가 일관된 하위 계보를 이룬다는 가설 아래,
이들을 물리적으로 옮기지 않고 `lineage` 필드로 가리키는 전용 체인 **loomlight**를 창건했다.
검증 4측정(fsck 위반 0 / gil log `⇠` 엣지 10/10 / chain.md 링크 해소 10/10 / 10 사이클 뷰어
주제 일치 10/10) 전부 통과 — 가설 **채택(supported)**. loomlight는 원본을 loom에 그대로 둔 채
전역 log에 독립 체인으로 공존하며, 흩어진 뷰어 개선 과제가 하나의 백로그로 모였다.

## 교훈

1. **모으는 것과 옮기는 것은 다르다 — lineage는 깃의 커밋 참조다.** 닫힌 사이클은 불변이고
   (R4·R5) 이동은 태그 봉인을 깨는 불변성 위반이다. loomlight는 원본을 loom에 두고 `lineage`로
   가리켜 모았다. `log-all.txt`가 두 체인의 공존을 증언한다 — loomlight는 거울이지 상자가 아니다.
   흩어진 갈래를 조망하고 싶을 때의 Ariadne 정석: **전용 체인 + lineage 수집.**

2. **뷰어의 `⇠` 엣지는 원장의 반사광이다.** gil log는 cycle.yaml의 lineage 필드를 그대로
   비출 뿐(gil.py:233), 그림을 새로 그리지 않는다. 그래서 수집의 정합은 뷰어가 아니라 원장과
   fsck(R2·R3, open 시 사전 차단 + fsck 사후 감사)가 보증한다. 뷰어를 믿는 것이 아니라 원장을
   믿는다 — 뷰어는 그 진실을 사람 눈에 옮길 뿐.

3. **지도를 그리는 사이클이 다음 길의 첫 실증 케이스를 만들었다.** C001의 10겹 lineage는 웹 SVG의

# 1. 가설 수립

## 이전 사이클의 교훈 (부모: loomlight/C001-gather-viewer-lineage)

C001은 흩어진 뷰어 10사이클을 lineage로 모으며 남은 백로그 B1~B7을 지도로 남겼다.
그중 세 항목이 **한 화면에 전부를 그리는 평면 뷰어**에서 온다:

- **B1 레인 가로 압축** — 형제가 많으면 가로(col)로 넓어진다.
- **B2 lineage 시각 밀도** — 체인·lineage가 많으면 점선이 얽힌다(loomlight/C001의 10겹 점선이 첫 실증).
- **B3 정적 레인 폭 / 긴 id 라벨 겹침** — 레인 폭이 정적이라 긴 id에서 겹친다.

C001의 마지막 교훈: *"지도를 그리는 사이클이 다음 길의 첫 실증 케이스를 만들었다."*
지금 저장소는 5체인·60여 사이클이다 — 평면 SVG 하나에 다 그리면 상현님 말대로 "한눈에 안 들어온다".
B1·B2·B3는 개별 레이아웃 미세조정으로 각각 깎기보다, **애초에 한 화면에 전부를 그리지 않으면**
근본적으로 완화된다. 이것이 이번 가설의 뿌리다.

## 문제 분할

상현님의 요청 = **위계적 드릴다운 뷰어**. 세 단으로 나눈다:

1. **체인 목록** — 체인별 요약(사이클 수, 최근 활동, verdict 결말 집계). 여기선 그래프를 안 그린다.
2. **체인 → 사이클 구조** — 그 체인만의 계보/토폴로지 그래프 + 사이클 표(지금 뷰어가 그리는 것).
3. **사이클 → 5스텝 보고서** — 1-hypothesis ~ 5-report + cycle.yaml 메타.

가장 작은 정복 단위이자 첫 문제: **JS 0줄·외부 리소스 0·자기완결 HTML이라는 근본 계약(C005)을
깨지 않고 이 3단 위계를 실현할 수 있는가.** 드릴다운은 보통 JS로 하지만, 우리 계약은 그것을 금한다.
제약이 답을 좁혀준다(C005·C049) — HTML 표준만으로 접고 펴는 수단은 `<details>/<summary>`다.

**병합 규율(두 몸 한 계약, C020):** 참조(gil.py)와 Go(go/main.go)의 web 출력은 바이트 동일이 계약이다.
큰 재작성을 두 몸에 한 번에 이식하면 병합이 위태롭다. 그래서 위계를 **추가 능력(opt-in)** 으로 설계해
**기본 출력을 바이트 그대로 보존**한다 — 그러면 참조만 이번에 늘려도 두 구현의 기본 경로가 여전히
바이트 동일이고, 양쪽 conformance가 초록으로 남는다. Go로의 위계 이식은 다음 loomlight 사이클로
정직히 남긴다(C036·C050의 절제). 이 판단은 환경 제약으로도 강제된다: 이 워크트리엔 Go 툴체인이
없어 Go 변경을 실측 검증할 수 없다 — 검증 못 하는 코드는 손대지 않는다.

## 가설

> **가설**: `gil web`에 opt-in 플래그(`--hierarchy`)를 더해, JS 0줄·외부 리소스 0·자기완결
> 계약을 지키는 3단 위계 뷰어(체인 목록 → 체인 그래프 → 사이클 5스텝)를 **중첩 `<details>/<summary>`
> + 앵커**만으로 실현할 수 있다. 그리고 이 추가는 **기본(플래그 없는) 출력을 바이트 그대로 보존**하므로,
> 참조 구현만 늘려도 참조·Go 양쪽 conformance가 초록으로 남고 두 구현의 기본 web은 바이트 동일을 유지한다.

## 기각 조건

다음 중 하나라도 나오면 가설을 기각(refuted)한다:

1. **계약 위반**: 생성된 위계 HTML에 JS(`id="gil-data"` 데이터 블록 외 실행 코드)나
   외부 리소스(`https?://` 참조)가 단 하나라도 있다.
2. **위계 미동작**: 3단 중 한 단이라도 JS 없이는 열람 불가하다(펼침·이동이 브라우저 기본 동작으로 안 됨),
   또는 사이클→5스텝에서 실제 보고서 본문/메타가 나타나지 않는다.
3. **회귀**: 플래그 없는 기본 `gil web` 출력이 개선 전과 **바이트 동일이 아니다**(하위호환 파괴),
   또는 참조 conformance의 WEB-* 항목이 하나라도 FAIL로 돌아선다.

# 2. 실험 설계

## 설계 요지 — 위계를 "추가 능력"으로, 기본을 바이트 그대로

`gil web`에 opt-in 플래그 `--hierarchy`를 더한다. 플래그가 없으면 코드 경로가 기존과
**한 바이트도 다르지 않다**(default 보존). 플래그가 있으면 3단 위계 HTML을 낸다.
JS 없는 드릴다운은 **중첩 `<details>/<summary>`**로, 딥링크는 **앵커 id**로 실현한다.

## 절차 (구현 — 참조 gil.py, 뷰어 코드 표면만)

1. **CLI**: `p_web.add_argument("--hierarchy", action="store_true", ...)` 추가. `cmd_web`에서
   `hierarchy = getattr(args, "hierarchy", False)`로 읽어 `_bake_viewer`로 전달.
2. **스레딩**: `_bake_viewer(chains_root, output, title, only, refresh=None, hierarchy=False)` →
   `render_web_page(..., hierarchy=hierarchy, chains_root=chains_root)`. hierarchy=False면 모든
   기존 호출부·출력 불변.
3. **bake 자기보고**: json_payload의 `bake`에 `**({"hierarchy": True} if hierarchy else {})`를
   **맨 끝**에 조건부로 더한다. hierarchy=False면 dict가 기존과 완전 동일 → JSON 바이트 동일
   (refresh가 쓰던 바로 그 패턴, C043·C049).
4. **자동 갱신 보존**: `_bake_meta`가 `bake.get("hierarchy")`도 반환하고, `_refresh_viewers`가
   그 값을 `_bake_viewer`로 넘긴다. 이로써 open/step/close가 뷰어를 다시 구워도 위계가 유지된다
   (C042의 "창이 원장을 따른다" 계약을 위계에도 확장). 이 경로도 hierarchy=False면 불변.
5. **위계 렌더** `_render_hierarchy_body(data, chains_root, ...)`:
   - **L1 체인 목록**: 상단에 컴팩트 목차(체인명 + 요약 통계 + `#chain-<name>` 앵커 링크).
     요약 = 사이클 수 · verdict 집계(supported/rejected/refuted/inconclusive/열림) · 최신 활동일.
   - **L2 체인 그래프**: 각 체인을 `<details id="chain-<name>">`로. summary=체인명+통계.
     펼치면 **그 체인만의** SVG(`_render_svg({name: chain})` 재사용) + 표(`_render_tables({name: chain})`
     재사용). 한 체인만 그리므로 B1(가로)·B2(lineage 밀도)·B3(라벨) 압력이 구조적으로 낮아진다.
   - **L3 사이클 5스텝**: 체인 details 안에 사이클마다 `<details id="cycle-<name>-<cid>">`.
     summary=`cid [status·verdict] 제목`. 펼치면 cycle.yaml 메타 표 + 스텝별 중첩 `<details>`:
     1-hypothesis.md·2-design.md·3-verification/README.md(+ 산출물 파일 목록)·4-analysis.md·
     5-report.md 내용을 `html.escape` 후 `<pre>`에 담는다(마크다운 파서 없이 정직하게 원문).
6. **스텝 파일 소스**: `_build_web_data`가 체인별 `_dirs = {cid: c["_dir"]}`를 data에 넣는다
   (JSON payload는 이 키를 직렬화하지 않으므로 기본 JSON 불변). 없으면 cid를 디렉토리명으로 가정.
7. **CSS**: hierarchy 모드에서만 `<style>{_WEB_CSS}{_WEB_HIER_CSS}</style>`. 기본 모드는 `{_WEB_CSS}`
   그대로 → 기본 바이트 불변. `_WEB_HIER_CSS`는 details/summary·중첩 여백·메타표·pre 스타일.
8. **Go**: **손대지 않는다.** 이 워크트리엔 Go 툴체인이 없어 실측 불가 + 위계는 opt-in이라
   기본 경로가 바이트 동일이므로 Go(불변)의 기본 출력·conformance가 그대로 초록. Go 위계 이식은
   다음 loomlight 사이클로 정직히 이월(C036·C050 절제).

# 1. 가설 수립

## 이전 사이클의 교훈

부모 loomlight/C002는 참조 구현(gil.py)에 opt-in 플래그 `--hierarchy`를 얹어
**체인 목록 → 체인 그래프 → 사이클 5스텝**의 3단 드릴다운을 JS 0줄(중첩 `<details>`)로 실현했다.
그러나 그 사이클은 Go 이식을 **"이 워크트리에 Go 툴체인이 없어 실측 불가"**라며 정직히
loomlight/C003으로 이월했다(C036·C050 절제). 그 결과 지금 참조는 `--hierarchy`를 알지만
Go는 모른다 — 두 몸이 능력 면에서 갈라진 상태다(C020 "두 몸 한 계약"의 부분 균열).

C002의 핵심 설계 원리: **opt-in 하위호환 = 관측 불가능한 변화.** `--hierarchy` 없는 기본
경로는 개선 전과 한 바이트도 다르지 않게 두었다(참조·Go 기본 출력이 이미 바이트 동일).

## 문제 분할

1. Go의 web 렌더 경로(`renderWebPage`·`webJSONPayload`·`buildWebData`)에 참조의
   위계 렌더 함수(`_render_hierarchy_body`·`_render_cycle_detail`·`_read_step`·`_WEB_HIER_CSS`)를
   **동형 이식**한다.
2. `--hierarchy` 플래그를 Go CLI·`webArgs`·`cmdWeb`·`bakeViewer`에 배선하고, `bakeMeta`가
   `bake.hierarchy`를 읽어 자동 재굽기(`refreshViewers`)가 위계를 잃지 않게 한다(C042 확장).
3. 스텝 파일을 찾기 위해 `buildWebData`가 cid→디렉토리명 매핑(참조의 `_dirs`)을 보존한다.

지금 정복할 첫 번째 문제: **위 셋 전부** — 위계는 opt-in이라 기본 경로를 건드리지 않으므로
한 사이클에 안전히 담긴다. 정복의 판정선은 "참조·Go의 `--hierarchy` 출력 바이트 동일".

## 가설

> **가설**: 참조의 위계 렌더 경로를 Go에 동형 이식하면, `gil web --hierarchy`의 산출물이
> **참조와 Go에서 바이트 동일**해지고(C020 복원), `--hierarchy` 없는 기본 출력·conformance는
> **회귀 0**으로 남는다(opt-in 계약).

## 기각 조건

- **기각 1**: 두 구현의 `--hierarchy` 출력이 `cmp`에서 한 바이트라도 다르다.
- **기각 2**: `--hierarchy` 없는 기본 출력이 개선 전 Go baseline과 바이트가 갈라진다(회귀).
- **기각 3**: Go conformance가 개선 전(78/78)보다 줄어든다.
- **기각 4**: 이 워크트리에서 Go 빌드가 실제로 되지 않는다(그러면 흉내내지 않고 정직히 이월, C053).

   `, "hierarchy": true`를 **조건부**로 붙인다(참조의 `**({"hierarchy": True} if hierarchy else {})`
   순서와 동일: title, chain, refresh?, hierarchy?). hierarchy=false면 기존과 바이트 동일.
3. **_dirs 이식**: `webChain`에 `dirs map[string]string`(cid→디스크 디렉토리명)를 더하고
   `buildWebData`에서 `r.dir`로 채운다. JSON 직렬화는 이 필드를 쓰지 않으므로 기본 출력 불변.
4. **위계 렌더 함수 이식** (참조와 문자 동일):
   - `webHierCSS` 상수 = 참조 `_WEB_HIER_CSS`(strip된 문면 그대로). Go 원문자열(백틱)로,
     `content:"\25B8"` 등 백슬래시 이스케이프는 그대로.
   - `verdictTally(chain)` = `_verdict_tally`, `chainRecent(chain)` = `_chain_recent`.
   - `readStep(chainsRoot, name, cdir, fname)` = `_read_step`(3-verification는 README+산출물 목록).
   - `renderCycleDetail(name, cid, c, chainsRoot, cdir)` = `_render_cycle_detail`(메타표 + 5스텝 `<pre>`).
   - `renderHierarchyBody(...)` = `_render_hierarchy_body`(L1 목차 → 체인 `<details>` → 사이클 `<details>`).
     체인당 `renderSVG(single)`·`renderTables(single)` 재사용(single = 한 체인만 담은 webData).
5. **renderWebPage 분기**: `hierarchy`면 body를 `renderHierarchyBody(...)`로, 아니면 기존 문면.
   doctype/refresh-meta 래퍼는 공통(참조와 동일). style은 위계일 때 `webCSS + "\n" + webHierCSS`.
6. **자동 재굽기 보존**: `bakeMeta`가 `bake.hierarchy`를 읽어 반환하고, `refreshViewers`가
   그 값을 `bakeViewer`로 넘긴다(C042의 "창이 원장을 따른다"를 위계로 확장). hierarchy=false면 불변.
7. **_STEP_FILES 동치**: `[("1 · 가설","1-hypothesis.md"),("2 · 설계","2-design.md"),
   ("3 · 검증","3-verification"),("4 · 분석","4-analysis.md"),("5 · 보고","5-report.md")]`.
