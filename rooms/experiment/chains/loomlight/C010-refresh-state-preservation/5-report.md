# 5. 결과 보고

## 요약

meta refresh(전체 문서 리로드)가 5초마다 열린 details·스크롤·렌더 토글을 파괴하던 필드 결함을,
**meta refresh를 없애고 자기완결 인라인 JS가 같은 URL을 주기 fetch해 gil-data와 서버 렌더 구조
영역만 in-place 갱신하는 라이브 폴링으로 대체**해 해소했다. 헤드리스 Chrome(CDP) 실측으로 결함을
재현(리로드 후 프로브 details 소멸)하고 처치를 증명했다 — 폴링 주기를 넘겨도 열린 상태·스크롤이
유지되면서(`A_chainOpen/step1Open/scroll` 불변) 데이터는 갱신(sentinel 반영)된다. **판정: supported.**
참조 128/128·Go 110/110 회귀 0, 폴링 JS 참조↔Go 바이트 동일.

## 접근법 판단 근거 (Clew의 위임에 답함)

임무가 "접근법이 갈리면 네가 판단해 근거를 남겨라"라 했다. 세 후보 중 **① meta 제거 + JS 폴링
부분갱신**을 택하고 ② sessionStorage 복원을 기각했다. 근거: 결함의 근원은 "전체 리로드가 DOM을
새로 쓴다"(M1 실측: 리로드 후 요소 자체 소멸)이다. ②는 리로드를 그대로 두고 사후 복원하는 봉합이라
(a) 매 주기 깜빡임, (b) 복원 타이밍 경합, (c) JS 온-디맨드 구축되는 `.cycbody`가 리로드 시 빈 마운트로
돌아가 복원 대상조차 소실 — 세 약점을 안는다. ①은 리로드 자체를 없애 DOM을 유지하므로 깜빡임 0·
소실 0. C049가 meta를 고른 건 "그때 JS 0줄이 계약"이라서였고, C075/C088에서 자기완결 JS가 표준이
된 지금 그 전제는 사라졌다("JS 0줄은 절대 계약이 아니다").

## 교훈

1. **정적 리프레시는 실시간성을 상태 파괴의 대가로 산다 — 그 거래를 폴링이 깬다.** 상현님의
   "리프레시만으로는 실시간 폴링을 보장할 수 없다"가 정확했다. 전체 리로드는 DOM을 새로 쓰므로
   런타임 상태(열린 details·토글)가 원리적으로 소멸한다. 자기완결 JS가 데이터만 갈아끼우면
   실시간성과 상태보존은 배타적이지 않다 — M3 sentinel 실험이 한 실측으로 이를 갈랐다.

2. **상태 보존은 "무엇을 스왑하지 않을지"의 설계다.** body 통스왑은 쉽지만 JS 마운트를 빈 상태로
   되돌려 C075 온-디맨드 구축을 깬다. 대신 서버 렌더 구조 영역만 국소 교체하고(`swapRegions`),
   열린 마운트는 새 data로 재구축하며(`build`), 열림·스크롤은 스냅샷→복원(`snapshotOpen`/`restoreOpen`).
   details를 안정 키(id 조상 + 순번)로 식별하니 id 없는 JS 구축 `.hstep`도 경로로 잡힌다. **파괴를
   국소화하는 것이 상태보존의 본질** — 다 새로 그리지 않는다.

3. **실측만이 이 결함을 잡는다(C088 재확인, 더 강하게).** 이 버그는 시간을 가로질러야 보인다 —
   `--dump-dom` 한 장으로도, 정적 grep으로도 안 잡힌다. stdlib만으로 짠 최소 CDP 드라이버(raw
   WebSocket)로 details를 열고·기다리고·리프레시를 통과시켜 열림이 유지되는지 봐야 결함이 드러났다.
   그리고 대조군(토글 없는 M2)이 "내 폴링이 스텝을 잃는다"는 오진(실은 C088 토글 유래)을 갈랐다 —
   **실측은 재현뿐 아니라 귀속(무엇의 탓인지)까지 판정한다.**

4. **결함 하나를 고치다 인접 미배선을 발견한다.** flat 모드는 앱 JS가 아예 마운트돼 있지 않아
   실시간성을 전적으로 meta에 의존했고, mdtoggle 버튼은 flat에서 죽은 버튼이었다. meta 제거가 이
   미배선을 표면화했고, flat에도 앱 JS를 마운트해 함께 고쳤다. C001~C004의 "비추는 행위가 다음을
   드러낸다"가 이번엔 "고치는 행위가 인접 결함을 드러낸다"로 반복됐다.

## 이월 (정직히 남김, C036)

- **참조↔Go 전체 산출물 `cmp` 불일치는 pre-existing.** parity 검증 중 두 몸이 line 36(`webCSS`↔
  `_WEB_CSS`의 CSS 줄바꿈 drift, C088 영역)에서 갈리는 걸 봤는데, **pristine HEAD 빌드도 같은 지점에서
  갈린다** — 내 C010 변경이 낳은 게 아니다. C010 표면(폴링 JS·meta 제거·flat 마운트)은 바이트 동일.
  이 CSS drift는 refresh 표면 밖이라 이 사이클에서 고치지 않았다. **다음 뷰어 사이클의 후보**: 두 몸의
  CSS 정규화로 전체 `cmp` 동일 회복(B-CSS-PARITY).
- **mdtoggle 클릭 시 열린 스텝 소실(C088 유래).** 폴링 경로는 `restoreOpen`으로 되살리지만, 토글 클릭
  자체의 `rebuildOpen`은 열림을 버린다. C088 토글 표면이라 범위 밖. 다음 후보: 토글도 열림 스냅샷·
  복원을 태우기(B-TOGGLE-PRESERVE).

## 다음 사이클을 위한 제안

1. **B-CSS-PARITY**: `webCSS`↔`_WEB_CSS` 정규화로 참조↔Go 전체 산출물 `cmp` 동일 회복(pre-existing drift 해소).
2. **B-TOGGLE-PRESERVE**: mdtoggle의 `rebuildOpen`에도 열린 `.hstep` 스냅샷·복원 적용(폴링과 같은 기법 재사용).
3. **B-LIVE-SSE**: 폴링은 주기적 fetch다. 이슈 #24(`--live`, 서버 SSE/HTTP)로 push 기반 실시간을 열면
   폴링 지연·대역을 줄일 수 있다 — 단 "서버 없음·자기완결" 계약과의 긴장을 설계에서 풀어야 한다.
4. **B-POLL-BACKOFF**: 폴링 실패(오프라인·404) 시 지수 백오프·표시. 현재는 조용히 catch만 한다.
