# 1. 가설 수립

## 이전 사이클의 교훈

부모 loomlight/C009-panel-order는 뷰어의 패널 순서를 다뤘다. 그 계보에서 이어지는
뷰어 표면 작업이다. 관련해 짚어야 할 선행 계약:
- **loom/C049**: `--refresh N`으로 `<meta http-equiv="refresh">`를 도입 — 새로고침 없는 실시간 관찰.
- **loom/C085**: refresh를 **기본화**(기본 5초). 옵트아웃은 `--refresh 0`.
- **loom/C075·C088**: 이미 자기완결 인라인 JS(`_WEB_APP_JS`, 외부 CDN 0)를 쓴다 — JS 0줄은 절대 계약이 아니다.
- **loom/C020·C047 (두 몸 한 계약)**: 참조(gil.py)와 Go(main.go)는 바이트/계약 parity를 지킨다.

## 문제 분할

필드 발견 결함: 로컬에서 뷰어를 앱으로 띄우고 기본 폴링(meta refresh 5초)을 켜두면,
**5초마다 페이지가 통째로 리로드되며 사용자가 열어둔 상태(열린 `<details>`·스크롤·마크다운
렌더 토글)가 전부 초기 상태로 리셋된다.**

정적 페이지 리프레시는 문서를 새로 쓴다 — 그래서 런타임 상태가 소멸한다. 이 결함을
가장 작은 단위로 나누면:
1. **DOM 파괴의 근원 제거**: meta refresh(전체 문서 리로드)를 실시간성의 수단에서 뺀다.
2. **데이터는 갱신**: 리로드 없이도 원장 변화가 화면에 반영되어야 한다(실시간성 유지).
3. **상태 보존**: 열린 details·스크롤·렌더 토글이 갱신을 가로질러 유지된다.

첫 번째로 정복할 것: **1·2·3을 한 기제로 — meta refresh를 자기완결 JS 폴링(같은 URL을
fetch → 새 gil-data JSON 추출 → DOM 부분 갱신)으로 대체**한다. 이것이 근원(전체 리로드)을
제거하면서 데이터 갱신과 상태 보존을 동시에 준다. sessionStorage 복원(후보 ②)은 리로드를
전제로 상태를 되살리는 사후 봉합이라, 리로드 자체를 없애는 ①보다 깜빡임·경합에 취약하다.

## 재현 (착수 전 실측)

meta-refresh 뷰어(`--refresh 3`)를 HTTP로 서빙하고 헤드리스 Chrome(CDP)으로 구동:
details 하나를 열고 `data-probe`를 심은 뒤 4.5초 대기(리프레시 통과) → 프로브가 사라지고
(`hasProbe:false`) 첫 details가 닫힘(`firstDetailsOpen:false`). **DOM이 통째로 재구축됨을
실측으로 확인.** 정적 grep으로는 못 잡는 런타임 결함이다(C088 교훈).

## 가설

> **가설**: meta refresh를 제거하고, 자기완결 인라인 JS가 주기적으로 같은 URL을 fetch해
> 새 `#gil-data` JSON과 동적 콘텐츠만 in-place 갱신하도록 바꾸면, 5초마다 데이터는 최신으로
> 유지되면서도 사용자가 열어둔 `<details>`·스크롤·마크다운 렌더 토글 상태가 갱신을 가로질러
> 보존될 것이다.

## 기각 조건

다음 중 하나라도면 가설 기각:
1. 폴링 갱신 후 헤드리스 실측에서 열어둔 details가 닫히거나 프로브가 소멸한다.
2. 데이터가 실제로 갱신되지 않는다(원장 변화가 폴링으로 화면에 안 나타남).
3. 외부 리소스가 0이 아니게 되거나(자기완결 위반), 서버가 필요해진다(정적 HTML 계약 위반).
4. 참조↔Go parity 또는 기존 conformance에 회귀가 생긴다(회귀 0 원칙).
