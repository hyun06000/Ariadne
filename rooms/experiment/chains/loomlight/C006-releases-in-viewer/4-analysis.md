## 데이터 직접 관찰

- **실저장소 배포 데이터**: current=**2.33.0**(도구 자기 상수), entries=**65개**(CHANGELOG ∪ 태그, SemVer 역순), **drift=0**. 최신 항목이 v2.33.0 "바이너리 자족성…"(이 세션 배포) — 뷰어가 방금 한 배포를 비춘다.
- **drift 0의 의미**: 모든 릴리스가 태그·CHANGELOG 양쪽에 있음(in_tag && in_changelog). 즉 배포 무결성이 뷰어로 확인된다 — 상현님 질문 "배포가 성공적으로 됐는지"에 대한 시각적 답. 만약 누가 태그만 만들고 CHANGELOG를 안 고치면 그 항목에 ⚠ 배지가 뜬다.
- **current의 자기보고성**: current는 `_GIL_VERSION`(참조)/`gilVersion`(Go)이라 **지어낼 수 없다** — 이 뷰어를 구운 도구가 자기 버전을 정직히 말한다. 배포 시 version 표면이 동기화되므로(release 명령) 두 구현이 같은 값.

## 예상과 달랐던 것

- **Go 이식이 생각보다 작았다**: C005가 "releases 명령이 Go에 없다"고 경고해 큰 이식을 예상했으나, 뷰어에 필요한 건 **명령 전체가 아니라 파싱 3함수**(CHANGELOG·태그·조립)뿐이었다. `cmd_releases`의 출력·drift 게이트 로직은 뷰어와 무관 — **필요한 최소 표면만 이식**하면 됐다. 교훈: "명령이 미이식"이 "그 데이터가 불가"를 뜻하진 않는다.
- **drift를 별도 배열 대신 원소 플래그로**: 처음엔 `drift: [버전들]` 배열을 생각했으나, 각 entry에 `in_tag`·`in_changelog`를 넣으니 뷰어가 그 항목 옆에 바로 배지를 그릴 수 있고 데이터도 자기설명적이 됐다. 관계를 별도 목록이 아니라 원소 속성으로 표현.
- **릴리스 노트는 앱화 불요**: 존재 memory(97KB)와 달리 릴리스 노트는 짧아(한 줄) 초기 DOM에 직접 넣어도 경량. C005 앱화 패턴을 기계적으로 따르지 않고 **데이터 무게에 맞춰** 판단(존재=온디맨드, 배포=직접).

## 판정

**채택.** 기각 조건 5개 대조:
- 사이클/존재 렌더 회귀 → 미발동(기존 WEB-* 전부 PASS) ✓
- parity 깨짐(drift 비결정) → 미발동(바이트 동일 — 같은 repo·태그·자기버전 결정론) ✓
- current 오류 → 미발동(2.33.0 정확, 자기보고) ✓
- 무CHANGELOG 미처리 → 미발동(T2, releases 키 부재·크래시 0) ✓
- 배포 왜곡 → 미발동(entries=CHANGELOG∪태그, 지어냄 0, drift 정확) ✓

가설 전부 실증. 세 축 중 둘째(배포) 완성. C005 골격(파싱→top-level 키→패널→Go 정적 이식→WEB-* 계약)이 두 번째로 작동 — 패턴이 재현 가능함을 확증. C005가 이월한 Go 비대칭도 이 사이클이 해소.
