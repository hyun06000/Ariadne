# 5. 결과 보고

## 요약

상현님 발의의 축 2(배포)를 정복했다. CHANGELOG(원장)와 깃 태그(진실)를 대조하는 참조 로직으로 배포 데이터를 gil-data top-level `releases`(current·entries, 각 원소가 in_tag·in_changelog로 drift 자기표현)에 담고, 위계 뷰어에 "배포" 패널(현재 버전·릴리스 목록·drift 배지)을 심었다. C005가 발견한 Go 비대칭(releases 미이식)은 파싱 3함수만 Go에 이식해 해소 — 참조·Go parity 바이트 동일. conformance WEB-RELEASES 신설(참조 114·Go 100, 수정 전 FAIL). 실저장소서 current=2.33.0·65릴리스·drift 0을 렌더해 **채택**.

## 교훈

1. **"명령 미이식"이 "데이터 불가"는 아니다** — C005가 "Go에 releases 명령 없음"을 경고했으나, 뷰어에 필요한 건 명령 전체가 아니라 **파싱 최소 표면**(CHANGELOG·태그·조립 3함수)뿐이었다. 데이터를 얻는 데 필요한 것과 CLI 명령이 하는 일은 다르다 — 필요분만 이식하면 parity가 선다.
2. **관계는 별도 목록보다 원소 속성으로** — drift를 `drift:[버전]` 배열 대신 각 entry의 `in_tag`·`in_changelog` 플래그로 표현하니 데이터가 자기설명적이고 뷰어가 그 자리에 배지를 그린다. "무엇이 무엇과 다른가"를 원소가 스스로 말한다.
3. **앱화는 무게에 맞춰 선택** — C005 존재(memory 97KB)는 온디맨드 앱화가 필수였으나, 배포 노트(한 줄)는 초기 DOM 직접이 맞다. 확립된 패턴(C075 앱화)을 기계적으로 따르지 않고 데이터 무게로 판단 — 패턴은 도구지 규칙이 아니다.
4. **뷰어가 배포 무결성을 비춘다** — drift 0(모든 릴리스가 태그·CHANGELOG 양쪽)이 "배포가 성공적으로 됐는지"에 대한 시각적 답. current의 자기보고성(도구 상수)이 "어떤 버전으로 배포 중인지"에 대한 지어낼 수 없는 답. 상현님 두 질문에 데이터로 답했다.
