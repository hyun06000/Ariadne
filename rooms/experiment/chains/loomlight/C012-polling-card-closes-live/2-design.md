# 2. 실험 설계

가설: 내 이전 재현(JS `.open=true`)이 실제 클릭과 달라 착시를 냈다. 실제 클릭으로 재현하면 카드 닫힘이 드러나고, `name="hchain"` 아코디언·replaceChild가 원인일 것이다.

## 핵심 설계 원칙 — 실제에 맞춘 재현

이전 실패의 교훈: **재현이 실제와 다르면 진단이 틀린다.** 그래서 이번엔:
1. **실제 github.io를 대상으로** 한다(로컬 재굽기 아님) — 상현님이 보는 바로 그것.
2. **마우스 클릭으로 연다** — `summary.click()` 또는 CDP `Input.dispatchMouseEvent`로 `<summary>`를 실제 클릭. `.open=true` JS 설정 금지(이게 착시의 원인이었을 수 있음).
3. **폴링을 실제로 기다린다** — 5초 주기를 넘겨(6.5초+) 자연 폴링이 돌게. 파일 교체 모사 없이, github.io의 실제 폴링(같은 URL 재fetch)만으로.
4. **DOM을 여러 시점에 관찰** — 열기 직후 / 폴링 1회 후 / 2회 후. 무엇이 언제 닫히는지.

## 절차

1. **github.io 로드 + 폴링 확인**: CDP로 `https://hyun06000.github.io/Ariadne/` 열고 3초 대기(2.4MB 로드). `gil-data`의 `bake.refresh` 확인(=5).
2. **실제 클릭으로 카드 열기**: `document.querySelector('#chain-loom > summary').click()` — JS `.open=true`가 아니라 클릭 이벤트. 열림 확인.
3. **폴링 전 상태 스냅샷**: 열린 details 수, chain-loom.open, 스크롤.
4. **자연 폴링 대기**: 6.5초(1주기+여유). 파일 안 건드림 — github.io 실제 폴링만.
5. **폴링 후 관찰**: chain-loom.open, 열린 details 수. 닫혔으면 **언제**(1주기 후) 닫혔는지.
6. **원인 격리 실험** (닫히면):
   a. `name="hchain"` 아코디언 가설: 로컬에서 name 속성 제거한 뷰어로 같은 재현 → 안 닫히면 아코디언이 원인.
   b. replaceChild 가설: 폴링의 swapRegions가 chain-loom 노드를 실제로 교체하는지, restoreOpen이 그 후 호출되는지 DOM 관찰.
   c. detKey 가설: 폴링 직전/직후 detKey 값 비교(단 IIFE 내부라 노출 안 됨 → 로컬 뷰어에 임시 노출 빌드로).
7. **원인 특정**: 위 격리로 "무엇이 열림을 잃게 하는가"를 한 문장으로.

## 준비물

- 실제 github.io (상현님이 보는 것). CDP 드라이버(C010 cdp.py 재사용).
- 로컬 격리 실험용: 현재 gil.py로 구운 뷰어 + name 제거/detKey 노출 변형본(진단용, 배포 아님).
- **권한**: 상현님이 "권한 다 준다" 했으므로 CDP·로컬 빌드·변형본 자유.

## 측정 방법

| # | 측정 | 판정 |
|---|---|---|
| M1 | 실제 클릭으로 연 카드, 폴링 1주기 후 | 닫히면 재현 성공(가설 지지 방향), 안 닫히면 재현 방식 재검(기각1) |
| M2 | 닫힌 시점 | 1주기(5s) 정확히 후인가 |
| M3 | name 제거 변형 | 안 닫히면 아코디언 원인(보조가정2) |
| M4 | swapRegions가 chain 노드 교체하나 | replaceChild 대상에 chain-loom 포함되나 |
| M5 | restoreOpen이 실패하는 지점 | detKey 불일치? 아코디언? 타이밍? |

## 검증분석 분기 (미리 정함)

- **재현되고 원인 특정되면**: 수정이 작으면 이 사이클에서 수정+검증까지, 크면 원인만 확정하고 수정을 자식 사이클로(진단/수정 분리 — 상현님 "작은 문제 하나에 사이클").
- **재현 안 되면**: 가설(상호작용 차이) 기각. 되돌아가 다른 축(폴링 타이밍·뷰 조건·환경)을 다음 가설로. 이 사이클은 rejected 또는 재정의.
