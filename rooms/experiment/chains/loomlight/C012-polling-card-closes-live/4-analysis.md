# 4. 결과 분석

## 가설 판정: **기각(rejected)** — 되돌아간다

가설("상호작용 차이·name 아코디언·detKey가 카드 닫힘의 원인, 실제 클릭으로 재현하면 드러난다")이 검증에서 **지지되지 않았다.** 헤드리스 CDP로 실제 github.io를 대상으로 실제 클릭·hash·장시간(3주기+)·실제 폴링(fetch 발생 확인) 모든 조건에서 재현했으나, **카드도 사이클 상세 내용도 폴링을 가로질러 유지됐다.** 상현님이 겪는 "5초마다 닫힘"이 이 환경에서 안 나타난다.

기각 조건 1("실제 클릭으로도 안 닫히면 가설 틀림 — 되돌아간다")에 정확히 걸렸다.

## 데이터가 말하는 것 — 되돌아가되, 정보를 얻었다

이 기각은 빈손이 아니다. 배제로 좁혔다:
1. **폴링은 실제로 돈다**(selfFetches 2/7초) — "폴링 미작동으로 유지된 착시"가 아니다.
2. **최신 뷰어(detKey 수정 포함)에서 헤드리스는 안 닫힌다** — C011 수정이 헤드리스 재현 범위는 실제로 고쳤다.
3. **그럼에도 상현님 실제 브라우저는 닫힌다 + 하드리프레시 무효** — 남은 유일한 축은 **환경(헤드리스 Chrome ≠ 실제 브라우저)**.

즉 결함은 **헤드리스가 재현 못 하는 무언가**에 있다. 후보(다음 가설 재료):
- (a) 실제 브라우저 특유의 렌더/타이밍(헤드리스는 requestAnimationFrame·페인트 타이밍이 다름).
- (b) 내가 못 본 상현님 특유의 상호작용(스크롤 중 폴링, 특정 카드 종류, 브라우저 확장).
- (c) **폴링 자체가 유일한 닫힘 경로** — 폴링을 끄면(`--refresh 0`) 닫힘이 멈추는지가 결정적 이분 격리다. 이건 헤드리스 무관하게 원인이 폴링인지 아닌지를 가른다.

## 왜 되돌아가는가 (상현님 설득)

헤드리스 CDP로 더 파는 것은 **벽**이다 — 재현이 안 되는 환경에서 원인을 특정할 수 없다. 여기서 두 갈래로 되돌아가는 게 합당하다:

1. **폴링 원인 이분 격리(가장 값싸고 결정적)**: 폴링이 유일한 닫힘 경로라면, `--refresh 0`(폴링 끔) 뷰어에서는 절대 안 닫혀야 한다. 상현님이 실제 브라우저에서 이걸 확인하면(또는 내가 github.io에 폴링 끈 변형을 잠깐 배포하면) 원인이 폴링인지 아닌지 한 번에 갈린다. **닫힘이 멈추면 → 폴링이 범인, 헤드리스가 못 잡는 폴링 부작용을 다음 사이클에서.** 안 멈추면 → 폴링 무관, 완전히 다른 원인(CSS·브라우저).

2. **실제 브라우저 진단 데이터**: 헤드리스로 안 되면 실제 브라우저 콘솔 로그가 필요하다 — 이건 상현님 손이 필요한 부분(내 권한 밖). 뷰어에 진단 로깅을 잠깐 넣어 상현님이 콘솔을 보고하는 경로.

## 판정 결론

**이 사이클은 rejected로 닫는다** — 가설이 틀렸으니 정직하게 죽은 가지로 그래프에 남긴다(C098 rejected close, step 보존). 진단 자체는 성과다: **원인이 헤드리스 재현 범위 밖 = 환경 의존**임을 확정했고, 다음 사이클의 정확한 첫 수(폴링 이분 격리)를 얻었다. C011의 애드혹 진단이 착시로 헤맨 것과 달리, 이번엔 사이클 안에서 각 배제가 기록돼 "무엇이 원인이 아닌지"가 데이터로 남았다.

**⭐ 방법론 자기증명**: 진단을 사이클로 하니, 가설 기각조차 다음 사이클의 재료가 됐다. "물어볼 것을 가설로" — 상현님께 "실제 브라우저냐/폴링이냐"를 물으려던 것을, 가설(상호작용 차이)로 세워 검증했고, 그게 기각되며 진짜 물어야 할 것(폴링 이분 격리)이 드러났다.
