# 1. 가설 수립

## 이전 사이클의 교훈

부모 C011(detKey idx=-1 수정, supported). C011이 detKey 버그를 고쳐 **내 CDP 재현에서는** 카드가 폴링을 견뎠다(hchain·cycdoc·hstep 전부 `POST_open:true`). v2.48.0 배포됨. **그런데 상현님이 github.io에서 여전히 "가만히 둬도 5초마다 카드가 닫힌다"고 보고했고, 하드리프레시로도 해결 안 됐다.** 즉 캐시가 아니라 진짜 잔존 결함이고, **내 CDP 재현이 실제 사용자 상호작용을 재현하지 못했다**(C011 착시 패턴 세 번째).

## 방법론 전환 (상현님 지침, 2026-07-20)

이 사이클부터 **진단을 사이클 안에서** 한다. 지금까지 진단을 사이클 밖 애드혹으로 하다 세 번 헤맸다. 상현님이 5스텝을 다시 정의해줬다:
- **문제정의**: 문제는 내가 안다.
- **가설수립**: 사용자에게 물어볼 것을 **가정하고 출발** = 그게 가설이다.
- **가설검증**: 어떤 방식이든 검증한다(권한 다 있다).
- **검증분석**: 가설이 합당한지 분석해 나아갈지/되돌아갈지 스스로 안다.
- **분석보고**: 왜 그랬는지 설득한다.

→ 물어볼 것을 가설로 세운다.

## 문제 분할

github.io 최신 뷰어(v2.48.0, detKey 수정 포함, meta refresh 없음, 폴링 5초)에서 **사용자가 연 카드가 가만히 둬도 5초마다 닫힌다.** 하드리프레시 무효 = 캐시 아님, 실제 코드 결함. 지금 정복할 첫 문제: **실제 사용자 상호작용을 재현해 "왜 닫히는지"를 특정**한다. 내 이전 재현(JS `.open=true`)은 실제(마우스 클릭)와 달라 착시를 냈으니, 재현 방식부터 실제에 맞춘다.

## 가설 (물어볼 것을 가정으로)

> **가설**: 내 CDP 재현이 실제와 달랐던 이유는 **실제 사용자 상호작용(마우스로 `<summary>` 클릭)과 내가 JS로 `.open=true`를 설정한 것이 다르기 때문**이다. `<details class="hchain" name="hchain">`의 **`name` 속성(HTML 아코디언 그룹 — 같은 name은 하나만 열림)** 때문에, 폴링의 `swapRegions`가 `.mapchains`를 통째 교체하고 `restoreOpen`이 열림을 복원할 때 아코디언 의미나 클릭-연 상태 복원이 실패해 카드가 닫힌다. 실제 클릭으로 재현하면 이 결함이 드러난다.

**보조 가정** (검증에서 하나씩 확인):
1. CDP `.open=true`(JS)가 실제 클릭과 다른 경로를 탔다 → 실제 클릭(summary dispatch)으로 재현.
2. `name="hchain"` 아코디언이 restoreOpen과 충돌한다.
3. 폴링 replaceChild 시 사용자가 연 노드가 대체되며 열림이 소실되는데 detKey가 여전히 못 잡는 케이스.

## 기각 조건

1. **재현 실패 시 되돌아간다**: 실제 클릭 시뮬레이션으로도 카드가 안 닫히면 가설(상호작용 차이)이 틀렸다 — 다른 원인(폴링 타이밍·특정 뷰 조건·별개 코드)을 봐야 한다.
2. **name 아코디언이 원인 아니면 부분기각**: name 제거해도 닫히면 아코디언은 원인 아님.
3. **원인 특정이 근거 없으면 기각**: 실측(CDP DOM 관찰)으로 "언제·왜 닫히는지"를 짚지 못하면 기각. 추측 금지.

## 성공 정의

github.io와 동일 조건(전체 체인, 폴링 5초)에서 **실제 클릭으로 연 카드가 폴링을 가로질러 닫히는 것을 재현**하고, DOM에서 무엇이 열림을 잃는지 특정한다 — 다음 단계(수정)의 정확한 재료. 이 사이클은 **진단 사이클**이다: 원인 특정 후 수정까지 갈지, 원인 특정에서 닫고 수정을 자식 사이클로 넘길지는 검증분석에서 정한다.
