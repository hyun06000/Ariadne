# 1. 가설 수립

## 이전 사이클의 교훈 (부모: loomlight/C001-gather-viewer-lineage)

C001은 흩어진 뷰어 10사이클을 lineage로 모으며 남은 백로그 B1~B7을 지도로 남겼다.
그중 세 항목이 **한 화면에 전부를 그리는 평면 뷰어**에서 온다:

- **B1 레인 가로 압축** — 형제가 많으면 가로(col)로 넓어진다.
- **B2 lineage 시각 밀도** — 체인·lineage가 많으면 점선이 얽힌다(loomlight/C001의 10겹 점선이 첫 실증).
- **B3 정적 레인 폭 / 긴 id 라벨 겹침** — 레인 폭이 정적이라 긴 id에서 겹친다.

C001의 마지막 교훈: *"지도를 그리는 사이클이 다음 길의 첫 실증 케이스를 만들었다."*
지금 저장소는 5체인·60여 사이클이다 — 평면 SVG 하나에 다 그리면 상현님 말대로 "한눈에 안 들어온다".
B1·B2·B3는 개별 레이아웃 미세조정으로 각각 깎기보다, **애초에 한 화면에 전부를 그리지 않으면**
근본적으로 완화된다. 이것이 이번 가설의 뿌리다.

## 문제 분할

상현님의 요청 = **위계적 드릴다운 뷰어**. 세 단으로 나눈다:

1. **체인 목록** — 체인별 요약(사이클 수, 최근 활동, verdict 결말 집계). 여기선 그래프를 안 그린다.
2. **체인 → 사이클 구조** — 그 체인만의 계보/토폴로지 그래프 + 사이클 표(지금 뷰어가 그리는 것).
3. **사이클 → 5스텝 보고서** — 1-hypothesis ~ 5-report + cycle.yaml 메타.

가장 작은 정복 단위이자 첫 문제: **JS 0줄·외부 리소스 0·자기완결 HTML이라는 근본 계약(C005)을
깨지 않고 이 3단 위계를 실현할 수 있는가.** 드릴다운은 보통 JS로 하지만, 우리 계약은 그것을 금한다.
제약이 답을 좁혀준다(C005·C049) — HTML 표준만으로 접고 펴는 수단은 `<details>/<summary>`다.

**병합 규율(두 몸 한 계약, C020):** 참조(gil.py)와 Go(go/main.go)의 web 출력은 바이트 동일이 계약이다.
큰 재작성을 두 몸에 한 번에 이식하면 병합이 위태롭다. 그래서 위계를 **추가 능력(opt-in)** 으로 설계해
**기본 출력을 바이트 그대로 보존**한다 — 그러면 참조만 이번에 늘려도 두 구현의 기본 경로가 여전히
바이트 동일이고, 양쪽 conformance가 초록으로 남는다. Go로의 위계 이식은 다음 loomlight 사이클로
정직히 남긴다(C036·C050의 절제). 이 판단은 환경 제약으로도 강제된다: 이 워크트리엔 Go 툴체인이
없어 Go 변경을 실측 검증할 수 없다 — 검증 못 하는 코드는 손대지 않는다.

## 가설

> **가설**: `gil web`에 opt-in 플래그(`--hierarchy`)를 더해, JS 0줄·외부 리소스 0·자기완결
> 계약을 지키는 3단 위계 뷰어(체인 목록 → 체인 그래프 → 사이클 5스텝)를 **중첩 `<details>/<summary>`
> + 앵커**만으로 실현할 수 있다. 그리고 이 추가는 **기본(플래그 없는) 출력을 바이트 그대로 보존**하므로,
> 참조 구현만 늘려도 참조·Go 양쪽 conformance가 초록으로 남고 두 구현의 기본 web은 바이트 동일을 유지한다.

## 기각 조건

다음 중 하나라도 나오면 가설을 기각(refuted)한다:

1. **계약 위반**: 생성된 위계 HTML에 JS(`id="gil-data"` 데이터 블록 외 실행 코드)나
   외부 리소스(`https?://` 참조)가 단 하나라도 있다.
2. **위계 미동작**: 3단 중 한 단이라도 JS 없이는 열람 불가하다(펼침·이동이 브라우저 기본 동작으로 안 됨),
   또는 사이클→5스텝에서 실제 보고서 본문/메타가 나타나지 않는다.
3. **회귀**: 플래그 없는 기본 `gil web` 출력이 개선 전과 **바이트 동일이 아니다**(하위호환 파괴),
   또는 참조 conformance의 WEB-* 항목이 하나라도 FAIL로 돌아선다.
