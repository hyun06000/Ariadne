# 2. 실험 설계

가설: 카드 닫힘의 원인은 폴링이다. `--refresh 0`이면 안 닫힌다(이분).

## 설계의 핵심 제약 (C012에서 배운 것)

헤드리스 CDP는 원래 닫힘을 재현 못 한다. 그러니 "헤드리스에서 `--refresh 0` 뷰어가 안 닫힌다"는 무의미하다(폴링 켠 것도 헤드리스선 안 닫혔으니). **이분의 실제 판정은 상현님 실브라우저에서만 가능하다.** 따라서 검증을 두 층으로 나누고, 내가 할 수 있는 것(구조 확인)과 상현님이 할 수 있는 것(실판정)을 분리한다.

또한 이 검증은 gil을 바꾸지 않는다(상현님: "gil 막 바꿀 필요 없다"). 기존 `--refresh 0` 플래그(C085에 이미 있음)를 쓴다.

## 절차

### 층 (a) — 내가 하는 것: `--refresh 0` 뷰어가 폴링을 실제로 끄는지 구조 확인

1. `gil web --refresh 0`으로 뷰어를 굽는다.
2. 헤드리스로 열어 **fetch가 0회**인지 확인(performance resource) — 폴링 부재 실증. (C012에서 refresh 5는 fetch 2회였다. 0이어야 격리 유효.)
3. 굽힌 HTML에 `setInterval`이 안 걸리는지(bake.refresh=0 → startPolling이 setInterval 안 함) 코드/데이터 확인.
4. 이로써 "`--refresh 0`은 폴링을 실제로 끈다"를 확정(기각조건 2 방어).

### 층 (b) — 상현님이 하는 것: 실브라우저 이분 판정

상현님이 실브라우저에서 폴링 끈 뷰어로 카드가 닫히는지 확인해야 이분이 갈린다. 상현님께 드릴 것 두 옵션(설계 선택):
- **옵션 1 (권장, gil 무변경)**: 상현님이 로컬에서 `python3 gil.py web --refresh 0 -o /tmp/noref.html && open /tmp/noref.html`로 폴링 끈 뷰어를 직접 열어 카드가 닫히는지 본다. 이게 github.io와 동일 코드(v2.48.0)에 폴링만 끈 것.
- **옵션 2 (github.io 임시)**: ariadne-pages를 `--refresh 0`으로 잠깐 배포. 되돌리기 번거롭고 다른 관전자 영향 → 옵션 1이 깔끔.

→ **옵션 1 채택.** 상현님께 명령 한 줄을 드리고 결과(닫힘 멈춤 여부)를 받는다.

### 판정
- 상현님 실브라우저에서 `--refresh 0`이 **닫힘을 멈추면** → 폴링 범인 확정(억셉트). 다음 사이클: 폴링의 어느 부작용인지.
- **여전히 닫히면** → 폴링 배제(기각). 다음 사이클: CSS·브라우저·다른 축.

## 준비물

- gil v2.48.0. `python3 rooms/deployment/ariadne-spec/gil.py`. 저장소 루트.
- 헤드리스 CDP(C010 cdp.py). Chrome.
- 상현님 실브라우저(층 b) — "권한 다 준다" 했으나 실브라우저 관찰은 상현님 손이 필요한 유일한 데이터.

## 측정 방법

| # | 측정 | 판정 |
|---|---|---|
| M1 (a) | `--refresh 0` 뷰어 헤드리스 fetch 수 | 0이어야 폴링 실제 꺼짐(격리 유효) |
| M2 (a) | 굽힌 HTML의 setInterval/bake.refresh | refresh 0 → 폴링 미장착 확인 |
| M3 (b) | 상현님 실브라우저, `--refresh 0`에서 카드 닫힘 | 멈추면 폴링 범인(억셉트)·닫히면 폴링 배제(기각) |

## 검증분석 분기 (미리 정함)

- M3에서 닫힘 멈춤 → **억셉트**: 폴링 범인. 5-report에 "다음 사이클 = 폴링 부작용 정밀 추적(실브라우저 콘솔 로깅)".
- M3에서 여전히 닫힘 → **기각**: 폴링 배제. rejected close, 다음 가지 = CSS/브라우저 축.
- M3 판정 못 얻음(상현님 확인 지연) → **부분/이월**: 층 a(폴링 꺼짐 확인)까지만 supported, 실판정은 이월.
