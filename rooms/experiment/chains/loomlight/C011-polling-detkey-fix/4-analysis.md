# 4. 결과 분석

## 통계적 결과

kill 조건 전부 통과 → **채택(supported)**. detKey 버그가 데이터 변경 폴링에서 카드 닫힘의 원인이 맞았고, 수정으로 해소.

| kill 조건 | 기준 | 결과 |
|---|---|---|
| 1. 데이터 변경 폴링 카드 닫힘 | POST 카드 open=true | ✔ id추적 `POST_loomOpen:true` (참조·Go) |
| 2. id 없는 details 보존 | .hstep 열림 유지 | ✔ 스텝 보존 |
| 3. 무변경 폴링 회귀 | C010 케이스 유지 | ✔ `["chain-loom"]` 보존 |
| 4. conformance | 참조128·Go110 | ✔ 유지 |
| 5. 두 몸 detKey | 바이트 동일 | ✔ cmp True |

## 데이터 직접 관찰

detKey 버그의 실체: id 가진 `<details>`가 자기 자신을 scope로 삼아 `getElementsByTagName`에서 빠져 `idx=-1`. 모든 최상위 카드 키가 `<id>#-1`로 뭉개졌다. 수정 후 `#chain-loom`으로 안정. 동일 데이터 폴링은 원래도 우연히 통과했으나(DOM 순서 불변), 데이터 변경 시 카드 순번이 흔들리면 복원 실패했다.

## 예상과 달랐던 것 — 이 사이클의 진짜 교훈

**검증이 두 번 착시에 속았다.** (1) 초기 v1→v2 극단 테스트가 `querySelector('details.hchain')`로 첫 카드(gateway)를 추적해 `POST_chainOpen:false`를 냈고, 나는 이를 "detKey 수정 실패"로 오독했다. (2) 실제로는 loom 카드가 살아있었고 id 추적으로 확정됐다.

이 헤맴 자체가 상현님이 제기한 방법론 문제의 실물 증거다: **나는 이 결함의 진단·재현 과정을 사이클 밖에서(추적 없이) 시작했고, 그래서 오진(detKey가 원인 아니라는 착각)과 테스트 착시가 기록되지 않은 채 나를 두 번 흔들었다.** 진단이 추적 단위(가설→검증)였다면 각 착시가 kill 조건에 걸려 즉시 드러났을 것이다. "의지만 불태우면 다른 LLM을 강제할 수 없다" — 진단을 사이클로 강제하지 않으면, 다른 LLM은 나처럼 애드혹으로 헤매고 오진을 원장에 봉인할 것이다.

## 판정

**채택(supported).** detKey 버그가 원인이었고 수정이 참조·Go 양쪽에서 실측으로 카드 보존을 확인. 단, 검증 과정의 착시는 방법론 결함(진단 비추적)의 증상으로 남긴다.

### 후속
- **⭐ 방법론 사이클(최우선, gateway 새 체인)**: 상현님 제안 — 문제정의=체인(작게)·문제분할·분할당 사이클DAG·교훈모아 체인배포+회고·회고→새체인. 그리고 **탐험/진단을 추적 단위로**. 이건 gil이 다른 LLM에게 강제해야 도구가 된다.
- 배포: detKey 수정 릴리스(뷰어 코드 변경).
- Sheen 이월(CSS drift·mdtoggle)은 여전히 별개.
