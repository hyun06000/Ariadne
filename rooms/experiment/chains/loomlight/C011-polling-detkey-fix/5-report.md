# 5. 결과 보고

## 요약

Sheen의 C010 폴링이 남긴 필드 결함("자꾸 열었던 카드가 닫힌다")의 근본 원인이 `detKey`의 idx=-1 뭉갬 버그(id 가진 details가 자기 자신을 scope로 삼음)임을 재현·특정하고, 참조·Go에 바이트 동일하게 고쳤다. 데이터 변경 폴링에서 카드·상세 보존을 헤드리스 CDP 실측(id 추적)으로 확인, 참조 128·Go 110 무회귀로 **채택**.

## 교훈

1. **id 가진 요소는 id가 곧 안정 키 — 순번을 매기지 마라.** `detKey`가 id 가진 details까지 순번을 매기려다 자기 자신을 scope로 삼아 idx=-1로 뭉갰다. id가 있으면 그걸 쓰는 게 옳다.
2. **재현 스크립트도 실험의 일부다.** querySelector로 첫 카드를 추적한 착시가 잘못된 반증을 냈다. 선택자가 틀리면 반증이 틀린다 — 판정 대상을 id로 콕 집어야 한다.
3. **⭐ 진단이 추적 밖이면 오진을 낳는다(상현님 지적의 실증).** 이 결함의 진단을 사이클 밖 애드혹으로 시작해 두 번 착시에 흔들렸고, detKey를 "원인 아님"으로 오판할 뻔했다. 진단이 사이클(가설→검증)이었다면 각 착시가 kill 조건에 걸렸을 것이다. **의지만으로는 다른 LLM을 강제 못 한다 — 진단을 추적 단위로 강제해야 gil이 도구가 된다.**

## 다음 사이클을 위한 제안

- **⭐ 방법론 재설계(최우선, gateway 새 체인)**: 상현님 제안 구조 —
  - **문제 정의 = 체인 생성**(작게. 지금 loom 100+ 사이클은 "체인을 너무 큰 단위로" 쓴 것)
  - **문제 분할** → 분할된 작은 문제 하나마다 **사이클 DAG**
  - 사이클 DAG의 교훈·결과를 모아 **체인 답안 배포 + 회고**
  - **회고 → 새 체인 생성**
  - 그리고 **탐험·진단을 추적 가능한 동작 단위로**(이 C011이 그게 없어서 헤맴).
  이건 gil이 다른 LLM에게 **강제**해야 도구다. 설계 사이클로 깊이 다룬다.
