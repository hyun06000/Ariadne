# 1. 가설 수립

## 이전 사이클의 교훈

부모 loomlight/C006(배포를 뷰어에)이 배포 데이터(current·릴리스 이력·drift)를 gil-data `releases`에 담았다. C006 교훈 ④"뷰어가 배포 무결성을 비춘다", 그리고 제안 A: **축 3 = 사이클→배포 연결** — "실험이 끝난 브랜치는 배포가 성공적으로 됐는지"(상현님). C006이 짚은 핵심 설계 질문: "사이클이 어느 릴리스에 들어갔나"는 현재 데이터에 직접 없다.

**설계 질문의 해답(실측)**: 사이클 태그(`cycle/<chain>/<id>`)와 릴리스 태그(`v<semver>`)가 둘 다 있으니, `git tag --contains <cycle-tag>`로 그 사이클 close 커밋을 포함하는 릴리스 태그들을 얻고, **가장 낮은 버전**이 "이 사이클이 처음 배포된 릴리스"다. 실측 확인: C082→v2.33.0, C081→v2.32.0(최초), C005(이번 세션)→미배포(빈 결과). **연결이 데이터로 가능하다.**

## 문제 분할

C005·C006 골격을 사이클→배포 연결에 적용:
1. **판정**: 각 닫힌 사이클에 대해 `git tag --contains cycle/<chain>/<id>`에서 v<semver> 최소 = released_in. 열린 사이클/미배포는 null.
2. **데이터**: 사이클 메타(gil-data chains.cycles[cid])에 **`released_in` 필드** 추가(있을 때만). 별도 top-level 아니라 사이클 원소 속성(C006 교훈 ②: 관계는 원소 속성으로).
3. **렌더**: 사이클 노드/마운트에 배포 배지 — "vX.Y 배포됨" 또는 "미배포". 뷰어에서 끝난 실험이 배포까지 갔는지 한눈에.
4. **parity**: `git tag --contains`는 git 호출 — Go에도 이식. 사이클마다 호출은 비쌀 수 있으니 **한 번에 역인덱스**(릴리스 태그별 포함 사이클) 구축 고려.
5. **계약**: WEB-CYCLE-RELEASE 신설(gil-data 자기보고).

첫 정복: **1~5를 묶어 사이클의 배포 상태를 뷰어에.** 성능(사이클 95개 × git 호출)은 역인덱스로 관리.

## 가설

> **가설**: 각 닫힌 사이클의 태그를 포함하는 릴리스 태그(`git tag --contains`)의 최소 버전을 `released_in`으로 판정해 gil-data 사이클 메타에 담고 뷰어 노드에 배포 배지를 렌더하면, 사람이 **끝난 실험(사이클)이 어느 버전으로 배포됐는지, 아직 미배포인지**를 뷰어에서 보고, 참조·Go가 동일 판정하며, 기존 사이클·존재·배포 렌더는 회귀 0일 것이다.

## 기각 조건

- 배포 상태 판정이 **기존 렌더를 회귀**시킨다(WEB-JSON·BEINGS·RELEASES FAIL).
- `released_in`이 **틀리게** 판정된다(포함하는데 미배포로, 또는 미배포인데 배포로 — git tag --contains 오용).
- 참조와 Go의 `released_in`이 **다르다**(같은 저장소·태그인데 판정 불일치 → parity 깨짐).
- 사이클 95개 git 호출이 **뷰어 생성을 실용 불가로 느리게** 한다(역인덱스로 해소 못 하면).
- 태그가 없는 환경(신규 저장소, 릴리스 0)에서 **깨지거나** released_in을 정직히 null로 못 둔다.
- 배포 상태를 **왜곡**한다(태그 대조 없이 지어냄).
