# 5. 결과 보고

## 요약

상현님 발의의 마지막 축 3(사이클→배포 연결)을 정복했다. 사이클 태그(`cycle/<chain>/<id>`)가 릴리스 태그(`v<semver>`)의 조상인지를 `git tag --merged` 역인덱스(낮은 버전부터, O(릴리스) 호출)로 판정해 각 사이클의 **최초 배포 릴리스**를 gil-data 사이클 메타 `released_in`에 담고, 뷰어 노드에 "vX.Y 배포"/"미배포" 배지를 렌더했다. 참조·Go 동형(parity 바이트 동일), conformance WEB-CYCLE-RELEASE 신설(참조 115·Go 101, 수정 전 FAIL). 실저장소서 92/96 배포·성능 0.7초로 **채택**. 상현님 "실험 위주 뷰어" 지적의 세 결(존재·배포·사이클→배포)이 모두 뷰어에 들어왔다.

## 교훈

1. **두 태그 체계의 교차가 새 데이터를 만든다** — 사이클 태그와 릴리스 태그가 이미 각각 있었는데, 그 **조상 관계**(`git tag --merged`)가 "이 실험이 배포됐나"라는 새 진실이었다. 데이터를 만든 게 아니라 이미 있는 두 데이터의 관계를 계산. 깃의 DAG가 답을 이미 갖고 있었다(C016 "깃의 답을 훔쳐라").
2. **역인덱스로 O(N×M)을 O(M)으로** — 사이클마다 `git tag --contains`(96회) 대신 릴리스마다 `git tag --merged`(65회) + 낮은 버전부터 미배정분 배정. "각 사이클의 최초 릴리스"를 릴리스 수만큼의 호출로. 방향을 뒤집으면 비용이 준다.
3. **세 축, 한 골격** — C005(존재)가 확립한 패턴(파싱→gil-data 키→패널/배지→Go 정적 이식→WEB-* 계약)이 C006(배포)·C007(사이클→배포)에서 그대로 재현됐다. 세 번 작동한 패턴은 신뢰할 수 있는 방법이다 — 다음 뷰어 확장도 이 골격.
4. **데이터 위치가 flat/hierarchy 노출을 정한다** — beings·releases는 top-level(hierarchy 전용)이라 flat 무영향이었으나, released_in은 사이클 메타라 flat에도 노출. 키 조건부(배포된 것만)로 무릴리스 저장소 바이트 동일을 지켜 회귀 아닌 성장으로.
