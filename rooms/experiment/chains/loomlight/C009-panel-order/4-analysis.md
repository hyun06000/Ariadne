# 4. 결과 분석

## 통계적 결과

| 확인 | 기대 | 결과 |
|---|---|---|
| 순서 | 배너 → 체인 → 배포 → 존재 | ✓ 배너(209) < 체인(10927) < 배포(126293) < 존재(142972) |
| 데이터 불변 | WEB-BEINGS·RELEASES 등 | ✓ 참조 115·Go 101 회귀 0 |
| parity | 바이트 동일 | ✓ hierarchy·flat |

## 데이터 직접 관찰

- **문자 위치로 순서 실측**: 체인지도(card hmap)가 배포(card releases)보다 앞, 배포가 존재(card beings)보다 앞. 병렬배너(⟳)는 헤더 직후 최상단 유지 — "지금 진행 중"의 자리. 상현님 지시 "체인 아래 배포 아래 존재"가 정확히 실현.
- **데이터 계약 불변 확인**: gil-data의 beings·releases·chains 키는 위치·내용 그대로(WEB-* 판정이 데이터를 검사하므로 렌더 순서 변경에 불변). 순서는 렌더(표현)일 뿐 계약(데이터)이 아니다(§7).

## 예상과 달랐던 것

- **순서 재배치가 순수 문자열 이동**: 참조는 f-string 두 줄 위치만, Go는 format 인자 순서만 바꿨다. 데이터·CSS·JS 무관 — "렌더는 계약 아님"이라 이런 재배치가 회귀 0으로 가능. C008(패널 접기)에 이어 표현/계약 분리의 실용성 재확인.

## 판정

**채택.** 기각 조건 5개 미발동:
- 순서 틀림 → 미발동(체인<배포<존재) ✓ / 데이터 변경 → 미발동(WEB-* PASS) ✓ / parity 깨짐 → 미발동(바이트 동일) ✓ / 요소 깨짐 → 미발동(배너·아코디언·toc 정상) ✓

상현님 지시 반영. 뷰어는 이제 작업(체인)을 먼저, 그 성과(배포)를, 그 주체(존재)를 순서로 비춘다.
