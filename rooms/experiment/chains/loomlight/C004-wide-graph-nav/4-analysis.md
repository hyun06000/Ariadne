## 데이터 직접 관찰

- **미니맵 치수(loom)**: `viewBox="0 0 489 34"`. width = miniX0(6) + maxDepth(53)·miniColW(9) + miniX0(6) = 489.
  height = miniY0(8) + maxLane(2)·miniRowH(9) + miniY0(8) = 34. loom은 69사이클이나 최장경로 깊이는 53 —
  즉 형제 갈래(merge/branch)로 깊이<개수. maxLane=2라 미니맵이 **본선 + 2개의 형제 레인**까지 한 폭에 담았다.
  스크린샷에서 C003 부근 노드 하나가 아래로 살짝 내려간 것이 그 형제 갈래(C003b류)다 — 전체 형상을 왜곡 없이 압축.
- **폭맞춤의 실제**: 489px 자연 폭 미니맵이 카드(≈1040px)에선 자연 크기로, 더 좁은 뷰포트에선
  `max-width:100%`로 축소돼 **가로 스크롤 없이** 전부 보인다. 반면 본 그래프는 hX0(50)+53·hColW(116)+90 =
  6288px로 여전히 스크롤이 필요 — 미니맵이 정확히 그 간극(전체 조망)을 메운다.
- **링크 동형**: loom 미니맵의 href 69개 집합이 본 그래프 href 69개 집합과 **완전 일치**했고, 각 href가
  실제 `id="cycdoc-loom-*"`와 매칭. 미니맵 클릭은 본 그래프 클릭과 문자 그대로 같은 :target을 연다 —
  새 상호작용 기제를 만들지 않고 C067의 것을 재사용(중복 0).
- **좁은 체인 `cmp`**: genesis(깊이 2)·weft·selvage가 개선 전 출력과 바이트 동일. loomlight만 1곳
  “활동 4분 전→1분 전”이 달랐는데, 이는 두 렌더 사이 실제 경과분(라이브 필드, C042)일 뿐 구조 변화 아님.
  → 미니맵은 **넓은 체인에만 순수 추가**되고 좁은 체인엔 한 바이트도 안 닿았다.

## 예상과 달랐던 것

1. **깊이(53) ≠ 사이클 수(69)**. 임계를 “사이클 수”가 아니라 “깊이(maxDepth)”로 잡은 것이 결과적으로 옳았다 —
   미니맵 폭은 개수가 아니라 최장경로에 비례하므로 깊이 기준이 폭 압력과 정확히 일치. (설계 때 둘을 구분해
   `_MINI_MIN_DEPTH`로 명명해 둔 것이 맞아떨어졌다.)
2. **`max-width:100%`가 SVG를 정말로 균일 축소**한다는 것(viewBox 있으면 종횡비 유지 축소). 좌표 바이트는
   불변이라 parity에 영향 0 — “표시는 CSS, 계약은 바이트”라는 C047/C020 원칙이 축소에도 그대로 성립했다.

## 판정

**채택(supported).** 미니맵은 JS 0줄·기본 레이아웃 불변·정수 좌표로 (a) loom 69노드 전체 형상을 한 화면에
폭맞춤으로 보이고 (b) 클릭으로 임의 사이클 문서에 도달하며 (c) 참조·Go 바이트 동일 (d) 좁은 체인은 출력 불변 —
가설의 네 절(a~d)이 모두 성립하고 기각 조건은 하나도 발생하지 않았다.
