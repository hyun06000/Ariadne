# 5. 결과 보고

## 요약

넓은 체인(loom, 최장경로 깊이 53·69사이클)의 자연 크기 가로 그래프(~6288px)가 한 화면에 안 담기던
네비게이션 문제를, 그래프 위에 **폭맞춤 미니맵**(전체 사이클을 작은 정수 좌표로 압축한 SVG, 각 노드 =
본 그래프와 같은 `#cycdoc-*` 앵커)을 얹어 풀었다(loomlight/C004). 깊이 < 12 인 좁은 체인은 미니맵을
안 그려 출력이 개선 전과 바이트 동일. JS 0줄·본 그래프 불변·정수 좌표라 참조·Go **바이트 동일**
(기본·flat), conformance 참조 90/90·Go 83/83 — 가설 채택(supported).

## 무엇을 강화했나 (목업 결정 포함)

- **미니맵 하나만**(절제). C067/C068이 이월한 넓은-그래프 네비게이션 세 후보(미니맵·현재위치·깊이접기)
  중, 현재위치는 스크롤 추적=JS 필요(불변 계약 위반)라, 깊이접기는 본 그래프 레이아웃을 변형해 회귀
  위험이라 기각하고, JS 0·기본레이아웃 불변으로 유일하게 가능한 **미니맵**만 골랐다. lineage 호버-전체는
  **이미 노드 `<title>`이 제공**(C067)하므로 다시 만들지 않았다(중복 회피).
- **목업 먼저(C067 철칙)**: 진행 전 `2-design.md`에 전/후 ASCII + `design-mockup-before-after.{html,png}`
  (다크테마 SVG)를 그려 “가장 보수적 선택(순수 추가·기본레이아웃 불변·좁은체인 무변화·JS 0)”을 시각화하고
  근거 5항을 명시했다. 병렬 세션이라 실시간 컨펌 대신, 목업이 코드보다 먼저 존재하게 해 원칙을 지켰다 —
  land 시 Clew/상현님이 이 목업과 실물 스크린샷(`3-verification/minimap-loom.png`)을 대조해 검토.
- **기본 출력 바이트 동일**: 참조↔Go가 기본·`--flat` 모두 `cmp` 무출력(1,367,385 B 동일).
- **Go 이식 완료**: `renderCycleMinimap` + hbody 배선 + `webHierCSS` 동형 이식. 정수 좌표(삼각함수·나눗셈
  없음, C064)라 이식이 곧 바이트 동일. CSS는 원문자열(백틱)로 이스케이프 보존(C003).

## 교훈

1. **임계는 “보이는 폭”을 만드는 축으로 잡아라.** 미니맵을 켤 기준을 사이클 “개수”가 아니라 “깊이(최장경로)”로
   잡은 것이 옳았다 — 그래프 폭은 개수가 아니라 깊이에 비례하기 때문(loom: 깊이 53 ≠ 개수 69). 문제가
   압박하는 그 축으로 임계를 정의하면 넓은 것만 정확히 골라내고 좁은 것은 건드리지 않는다(회귀 최소).
2. **표시는 CSS, 계약은 바이트 — 축소에도 성립.** `max-width:100%`가 viewBox SVG를 종횡비 유지로 균일
   축소하지만 좌표 바이트는 불변이라 parity에 0 영향. C068이 “넓은 그래프는 자연 크기”로 풀었던 것을,
   미니맵엔 **반대로(폭맞춤)** 적용해 같은 원칙(C020/C047)으로 상반된 목적을 달성했다. 자연크기 vs 폭맞춤은
   요소마다 CSS로 고르는 표시 선택일 뿐, 바이트 계약과 독립이다.
3. **새 기제를 만들지 말고 기존 앵커를 재사용하라.** 미니맵 노드가 본 그래프와 같은 `#cycdoc-*`를 가리키게
   해, C067의 `:target` 기제를 그대로 물려받았다(href 집합 69==69 일치). 상호작용 검증도 “새 클릭 동작”이
   아니라 “링크 동형”으로 환원돼 간단해졌다. 절제는 검증 비용도 줄인다.
