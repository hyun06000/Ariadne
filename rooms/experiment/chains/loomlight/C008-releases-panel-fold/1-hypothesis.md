# 1. 가설 수립

## 이전 사이클의 교훈

부모 loomlight/C007이 세 축(존재·배포·사이클→배포)을 완성하며, C006이 만든 **배포 패널의 UX 문제**를 이월했다: 상현님 실시간 피드백 **"배포가 리스트로 65개 다 뜨니 너무 많다 — 체인처럼 보이고 관리될 필요."** C006 배포 패널은 릴리스 전부(65개)를 평평한 `<ul>`로 나열해 무겁다. C006은 닫혔으므로(불변) 새 사이클에서 패널을 개선한다.

## 문제 분할

상현님 지시 = **접기/펼치기**(AskUserQuestion 확정). "체인처럼": 기본은 요약(최신 몇 개), 펼치면 전체. loomlight의 기존 아코디언 패턴(C065 `<details>`, JS 0)과 일관.

- 최신 N개(N=5)는 항상 보임 → "지금 배포가 어떤 버전인지"에 즉답.
- 나머지(60개)는 `<details>`로 접기 → 펼치면 전체 이력. JS 0(네이티브 HTML).
- 무릴리스/소수(≤N) 저장소는 details 없이 전부 보임(바이트 동일 관성).

## 가설

> **가설**: 배포 패널의 릴리스 목록을 최신 N개(5)는 항상 표시하고 나머지를 `<details>`(접기)로 감싸면, 사람이 현재 배포 버전과 최근 릴리스는 즉시 보되 전체 이력은 펼쳐서 관리할 수 있고(무거움 해소), JS 0·참조 Go parity가 유지되며, 릴리스 ≤5인 저장소는 기존과 동일하게 전부 표시될 것이다.

## 기각 조건

- 접기가 **현재 버전/최신 릴리스를 가린다**(요약에 최신이 안 보임).
- 릴리스 ≤5 저장소에서 **불필요한 details가 생기거나** 깨진다.
- JS를 쓴다(자기완결 계약 — `<details>` 네이티브만).
- 참조·Go 배포 패널 산출물이 **다르다**(parity 깨짐).
- 기존 배포 데이터(gil-data releases)가 **바뀐다**(이건 렌더 변경일 뿐 — 데이터·다른 패널 회귀 0이어야).
