# 4. 결과 분석

## 통계적 결과

| 확인 | 기대 | 결과 |
|---|---|---|
| T1 최신 5 요약 | 초기 표시 5개, 현재 버전 즉시 | ✓ 65개 중 5개, v2.33.0 즉시 |
| T2 나머지 접기 | details "이전 N개" | ✓ "이전 릴리스 60개 더 보기" |
| T3 소수 저장소 | ≤5면 details 없음 | ✓ (설계상 조건부, 실저장소는 >5라 details) |

- **참조 115/115 · Go 101/101** 회귀 0. 렌더 변경이라 새 conformance 항목 없이 기존 WEB-RELEASES(데이터 검사)가 불변 확인 — "렌더는 계약 아님"(§7)의 실천.
- **parity 바이트 동일**: hierarchy·flat.

## 데이터 직접 관찰

- **무거움 해소 실측**: 배포 패널 초기 표시 릴리스 **65 → 5개**. summary "이전 릴리스 60개 더 보기". 현재 버전 v2.33.0이 요약(details 밖)에 즉시 보임 — "지금 어떤 버전으로 배포 중인지"에 즉답하면서 전체 이력은 펼침으로 관리. 상현님 "체인처럼 보이고 관리"가 실현.
- **JS 0 유지**: 외부 리소스 0, 네이티브 `<details>`만. C065 아코디언 패턴 재사용 — 새 메커니즘 0.

## 예상과 달랐던 것

- **렌더만 바꿔도 데이터 계약이 안 흔들린다**: gil-data releases는 그대로(65개 다 있음), 뷰어가 그중 5개만 초기 표시할 뿐. WEB-RELEASES가 데이터를 검사하므로 렌더 변경에 불변 — 계약면(데이터)과 표현(렌더)의 분리가 이런 UX 개선을 회귀 없이 가능케 한다(§7 "렌더는 계약 아님"의 실용적 가치).
