# 5. 결과 보고

## 요약

C006 배포 패널이 릴리스 65개를 다 나열해 무겁다는 상현님 피드백을, 최신 5개는 항상 표시하고 나머지를 `<details>`(접기)로 감싸 해소한다는 가설을 세웠다. 참조·Go 동형 수정(JS 0, 네이티브 details), 실저장소서 초기 표시 65→5개·"이전 릴리스 60개 더 보기"·현재 버전 즉시 확인. 렌더만 바꿔 데이터 계약(WEB-RELEASES) 불변, parity 바이트 동일로 **채택**.

## 교훈

1. **계약면과 표현의 분리가 UX 자유를 준다** — gil-data releases(데이터)는 65개 다 두되 뷰어(표현)가 5개만 초기 표시. "렌더는 계약 아님"(§7)이라 WEB-RELEASES를 안 건드리고 UX를 고칠 수 있었다. 데이터를 완전히 담고 표현을 선택적으로 — 이게 회귀 없는 개선의 구조.
2. **기존 패턴 재사용이 새 메커니즘을 없앤다** — 접기는 C065 아코디언(`<details>`, JS 0)을 그대로 썼다. 새 것을 발명하지 않고 확립된 패턴을 재사용 — 자기완결 계약도 공짜로 지켜진다.
3. **실시간 피드백을 사이클 경계로 다룬다** — 상현님 피드백이 C007 구현 중 도착했으나, C007을 중단하지 않고 완주 후 C008로 분리했다. C006(피드백 대상)이 닫혀 불변이므로 새 사이클에서 개선 — 방법론이 "끝난 것은 안 고치고, 새 가설로"를 강제해 기록이 깨끗하다.
