# 실험의 방 (Room of Experiment)

풀어야 하는 문제를 **사이클**이라는 가장 작은 달성 단위로 분할해 진행하는 방이다.
모든 동작은 목적 달성을 위한 것이어야 하며, 각 스텝에서 발생하는 모든 스크립트와 산출물은
문서 형태로 저장되어 **사이클 단위에서 재현 가능**해야 한다.

## 사이클의 5스텝

| # | 스텝 | 산출물 | 내용 |
|---|---|---|---|
| 1 | **가설 수립** | `1-hypothesis.md` | 이전 사이클의 보고서에서 교훈을 얻고 새 가설을 세운다. 현재 풀어야 할 문제를 가장 작은 단위로 분할해, 정복할 첫 번째 문제를 정의한다. |
| 2 | **실험 설계** | `2-design.md` | 수립된 가설에만 초점을 맞춰 검증 실험을 절차 중심으로 설계한다. 필요시 사용자에게 컨펌·수정을 받는다. |
| 3 | **가설 검증** | `3-verification/` | 설계를 코드와 도구로 직접 실행한다. 재현 가능성에 초점을 두고, 모든 아티팩트·코드·연산 결과를 저장한다. |
| 4 | **결과 분석** | `4-analysis.md` | 결과를 면밀히 분석한다. 통계 수치에서만 인사이트를 얻지 말고, 데이터와 아티팩트를 직접 들여다보며 교훈을 얻는다. |
| 5 | **결과 보고** | `5-report.md` | 분석에서 얻은 교훈을 작성하고 보고서를 퍼블리시한다. 이 보고서가 다음 사이클의 부모가 된다. |

## 체인과 계보

깃 커밋 그래프처럼, 모든 사이클은 자신이 어느 보고서의 교훈에서 태어났는지를 명시한다.

- 각 사이클의 `cycle.yaml`에 `parent` 필드로 부모 사이클 ID를 기록한다. (최초 사이클은 `parent: null`)
- 하나의 보고서에서 여러 가설이 나오면 **분기**가 일어난다 — 여러 사이클이 같은 부모를 가지면 된다.
- 서로 다른 갈래의 교훈을 합쳐 새 가설을 세우면 **병합**이다 — `parent`에 여러 ID를 나열한다.

```
C001 ──▶ C002 ──▶ C003 ──▶ C005   (병합: C005의 parent는 C003, C004)
           └────▶ C004 ────┘
```

## 디렉토리 구조

```
rooms/experiment/
├── _template/            # 새 사이클의 원형. 복사해서 시작한다.
└── chains/
    └── <체인이름>/        # 하나의 문제(목적)당 하나의 체인
        ├── chain.md      # 이 체인이 정복하려는 문제의 정의
        └── C001-<슬러그>/ # 사이클. ID는 체인 안에서 순차 증가.
            ├── cycle.yaml
            ├── 1-hypothesis.md
            ├── 2-design.md
            ├── 3-verification/
            ├── 4-analysis.md
            └── 5-report.md
```

## 새 사이클 시작하기

1. 부모가 될 사이클의 `5-report.md`를 읽는다. (최초라면 체인의 `chain.md`를 읽는다.)
2. `_template/`을 `chains/<체인>/C###-<슬러그>/`로 복사한다.
3. `cycle.yaml`의 `id`, `parent`, `chain`을 채운다.
4. 스텝 1부터 순서대로 진행한다. 스텝을 건너뛰지 않는다.
5. 보고서 퍼블리시로 사이클을 닫고 커밋한다. 닫힌 사이클의 문서는 수정하지 않는다 — 잘못이 발견되면 다음 사이클에서 다룬다.
