# 프로젝트: gil-v3-redesign

**gil v3를 상현님 디자인대로 완전히 새로 빌드한다.** 기존 gil(옛 v3 구현 포함)은 버린다.

## 상현님의 결정 (2026-07-23) — 과감하게

1. **"gilv3"라는 이름 영구 폐기.** 이건 별도 물건이 아니라 gil의 v3다. gil에 통합. 그 이름은
   다시 부활하지 않는다.
2. **기존 gil 전부 버린다.** 옛 gil.py(폴더+md 방식)를 도그푸딩할 필요 없다 — 나(Clew)는 이미
   gil 없이 그 템플릿을 **손으로** 할 수 있다. 도구가 없어 도그푸딩을 고집하는 건 바보 같은 일.
3. **내가 이해한 템플릿으로 v3를 새로 빌드.** 손으로 하는 시연이 곧 v3의 명세가 된다.
4. **v3가 가닥 잡히면** → gil로 gil v2를 통합하는 코드를 다음 프로젝트 목표로.

## v3의 두 원칙 (약속한 방식)

### 1. 폴더 없음 — 사이클·스텝은 커밋 그래프의 노드
- 사이클마다 `C00x/` 폴더를 만들지 않는다. 프로젝트 디렉토리 하나 + 커밋 그래프.
- 사이클을 "여는 것" = 커밋 노드를 새기는 것. append-only, 앞으로만.
- 트리 구조(부모·백트래킹)는 커밋 trailer로 (Step-Id·Kind·Parent·Cycle-Author·Cycle-Parent).

### 2. 본문 없는 md 파일 — 스텝 디테일은 커밋 로그에
- 짧은 커밋 제목(subject): 노드의 위계·이름·id. 예: `gil C001/s3 verify`.
- 긴 커밋 본문(body): 스텝 디테일. `git log --format=%b`로 읽힌다.
- trailer: 계보·구조 메타.
- **스텝마다 .md 파일을 만들지 않는다.** 뷰어는 git log에서 본문을 읽는다.

## 작업 방식 — 손으로 시연이 명세다

- 이 프로젝트의 사이클은 **gil 없이 순수 git 커밋으로** 돈다. 내가 손으로 위 템플릿(제목+본문
  +trailer)을 새긴다.
- 그 손 시연이 재현 가능한 **명세**가 되고, 그 위에서 v3 코드(gil의 새 구현)를 빌드한다.
- 산출물(코드·figure·field)은 이 프로젝트 디렉토리 아래 자유롭게. 사이클 구조는 커밋 그래프에.

## 사이클 로그

커밋 그래프에 쌓인다. 조회: `git log --format='%h %s' -- rooms/project/gil-v3-redesign`.
각 스텝의 디테일은 그 커밋의 본문(`git log --format=%b <sha>`)에.
