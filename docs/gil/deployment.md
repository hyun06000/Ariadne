# 배포와 체인 전환

체인이 배포 순환의 한 국면을 이루는 방식과, 한 국면이 끝나면 사이클을 늘리지 않고 체인을 전환하는 규칙을 다룬다.

## 체인 = 배포 순환의 한 국면

gil 에는 정적인 "방(room)" 분류가 없다. 지금 상태는 **어느 체인이 열려 있나**로 표현된다. 체인들은 배포 순환을 이룬다:

```
개발 체인 (개발·실험)
   ↓ 닫힘
스테이징 체인 (필드테스트)   ← 닫힌 개발 체인 끝에서 열림
   ↓ 닫힘
배포 체인 (배포)            ← 닫힌 스테이징 체인 끝에서 열림
   ↓ 닫힘 → 다시 개발 체인    (순환)
```

각 체인은 맡는 **검증 깊이가 다르다**:

- **개발(dev) 체인의 verify = 기능 검사(smoke)만.** "이 기능이 동작하는가"까지. 전 주기가 도는지, 명령이 커밋을 새기는지 정도. 빠르게 짓고 돌린다.
- **스테이징 체인 = 기능 검사 이상의 엄밀한 테스트.** 엣지 케이스·필드테스트를 여기서. 닫힌 개발 체인 끝에서 열린다.
- **배포 체인 = 봉인·릴리스.** 닫힌 스테이징 체인 끝에서 열린다.

검증을 두 체인으로 나누면, 개발은 가볍게 전진하고 엄밀함은 스테이징이 맡는다.

## ⭐ 체인 전환 — 사이클을 늘리지 말고 체인을 바꿔라

한 국면이 완결되면 다음처럼 국면을 전환한다:

```
gil chain-close <chain>          # 그 체인의 모든 사이클이 닫힌 뒤에만 허용
gil chain <새이름> --purpose <P>  # 그 닫힌 끝에서 새 체인을 연다
```

1. `gil chain-close <chain>` — 체인을 완결로 봉인한다. **모든 사이클이 닫힌 뒤에만** 허용된다. 이것은 사이클을 닫는 `gil close` 와 다르다(그 위 국면 전체를 닫는다).
2. 그 **닫힌 끝에서** `gil chain <새이름>` 으로 새 체인을 연다.

**핵심 규율**: 한 국면이 완결됐으면 사이클만 계속 늘리지 말고 체인을 전환한다. 국면(개발→스테이징→배포)이 바뀌는 지점은 새 사이클이 아니라 새 체인으로 표현되어야 배포 계보가 정확히 남는다. `gil handoff` 가 이 전환을 안내한다.

## 닫힌 끝에서 조상을 이어받는다 (orphan 아님)

닫힌 체인 끝에서 연 새 체인은 **대문·존재·교훈을 조상으로 이어받는다.** 새 체인은 orphan 이 아니라 닫힌 부모 체인 위에 뻗으므로, 이전 국면의 코드·판정·교훈이 그대로 물려진다. 분기 지점(어느 체인에서 뻗느냐)이 곧 상속받는 코드를 정한다.

역으로, **닫힌 부모 체인 안에서 새 사이클을 여는 것은 거부**된다 — 새 자식 체인을 열어야 한다. 이렇게 gil 이 배포 계보가 꼬이지 않도록 강제한다. 이 거부 규칙의 개념적 배경은 [개념](concepts.md)을 보라.
