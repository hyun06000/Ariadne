# 체인·사이클·스텝: 세 위계

gil이 git 위에 얹혀 사고의 역사를 다루는 방식과, 체인 > 사이클 > 스텝 3위계가 git의 무엇에 매핑되는지를 다룬다.

## gil은 git 위에 산다

gil은 git을 대체하지 않는다. **git 위에 얹힌다.** gil이 만드는 커밋도 그냥 git 커밋이다 — 다만 제목·본문·`Gil-*` 트레일러에 사고의 구조가 새겨져 있을 뿐이다. `git log`로도 읽힌다. 별도 DB도, 폴더도, md 파일도 없다 — **모든 것이 git 커밋과 트레일러로 산다.** gil 명령은 그 커밋을 규율 있게 만들고 읽어주는 얇은 껍질이다.

핵심 매핑:
- **노드 = 커밋** — 한 번의 생각(= 스텝)이 커밋 하나.
- **분기 = git 브랜치** — 모든 갈래는 진짜 git 브랜치로 표현된다.

이것이 왜 강력한가: **사고 역사가 코드 역사와 같은 그래프에 있으니**, 어떤 코드가 어떤 가설·실패·되돌아감에서 태어났는지 실을 따라 언제든 처음으로 되돌아갈 수 있다.

## git과의 대조

| | **git** | **gil** |
|---|---|---|
| 다루는 것 | 소스 코드의 역사 | **사고(추론)의 역사** |
| 노드 | 커밋(코드 스냅샷) | 커밋(**한 번의 생각 = 스텝**) |
| 브랜치 | 코드 갈래 | **체인**(작업의 큰 줄기) |
| 무엇을 기록하나 | "코드가 이렇게 바뀌었다" | "이런 가설을 세워 검증했고, 막혀서 되돌아갔다" |
| 진실원 | 커밋 그래프 | **같은 커밋 그래프** (gil은 git 위에 산다) |

## 세 위계

gil의 사고 구조는 3층이다. 큰 것부터 작은 것으로, 각각이 git의 무엇에 매핑되는지:

```
체인 (chain)   ─ 작업의 큰 줄기.  git 브랜치 <name> 에 매핑.        예: gil-v3-unified
  └ 사이클 (cycle) ─ 하나의 문제를 푸는 한 바퀴.  git 브랜치 <chain>-<cycle>.  예: c001, c002
      └ 스텝 (step)  ─ 한 번의 생각.  커밋 하나.                     예: s1, s2, s3
```

- **체인**: 작업의 큰 줄기. 새 체인을 열면 `git 브랜치 <name>`을 판다.
- **사이클**: 하나의 문제를 푸는 한 바퀴. 열면 `git 브랜치 <chain>-<cycle>`을 파고 `s1` define 스텝을 자동 생성한다.
- **스텝**: 한 번의 생각 = 커밋 하나. 종류(kind)를 가지며 정해진 순서로 흐른다. 스텝 흐름의 상세는 [사고의 생애](lifecycle.md)를 보라.

## 순서 규칙

- **다음 사이클은 닫힌 사이클 끝에서** 연다. (`c001` 닫음 → `c002` open OK)
- **체인은 닫힌 체인 끝에서만** 새로 연다.
- **닫힌 부모 체인 안에서 새 사이클을 여는 건 거부**된다 — 새 자식 체인을 열어야 한다. 배포 계보가 꼬이지 않도록 gil이 이걸 강제한다. 자세한 국면 전환은 [배포와 체인 전환](deployment.md)을 보라.
- 분기(갈래)는 전부 git 브랜치로 표현된다. **분기 지점은 코드 계보로 정한다** — 필요한 코드가 쌓인 체인에서 뻗는다.
