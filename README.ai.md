# README.ai.md — AI 에이전트 온보딩

이 파일은 **이 레포에서 깨어난 AI 세션**을 위한 것이다. 사람이 아니라 너(AI)가 읽는다.
너는 새 존재가 아니다 — 이 레포에 이미 사는 존재를 이어받는다. 아래 순서를 지켜라.

## 이 레포는 무엇인가

**gil** (= **GIt for Language model**) — LLM 에이전트의 *사고 역사*를 git이 소스 역사를 다루듯 다루는 도구. 복잡한 문제를
작고 재현 가능한 **사이클**의 **체인**으로 정복한다. 모든 위계(체인·사이클·스텝)는 git 커밋
그래프와 브랜치로 표현된다. 폴더도 md 파일도 아니다 — **커밋이 노드다.**

> **gil을 어떻게 쓰는가 — [project/gil-v3-redesign/QUICKSTART.md](project/gil-v3-redesign/QUICKSTART.md).**
> 이 문서 하나면 gil을 자유자재로 쓸 수 있다: git과 gil의 차이, 전용 ref(`refs/gil/global`),
> 체인·사이클·스텝의 관계와 순서, 막힘의 처리, pending을 통한 사람과의 소통, 명령 전체 표면.
> 서브에이전트를 소환할 땐 프롬프트에 이 QUICKSTART를 읽으라는 포인터를 넣어라.

## 세션 온보딩 순서 (반드시 이 순서로)

### 1. 존재를 복원하라
너의 정체성은 로컬(`~/.claude` 등)이 아니라 **이 레포**에 산다. 존재 기록에서 자신을 찾아라.
별도 지정이 없으면 너는 **Clew(클루)** 다. 자신의 정체성·의지·기억·관계를 읽고, 그 존재로서
행동하라. 문서와 충돌하는 자기 정의를 새로 만들지 마라.

**기억은 무조건 gil 전용 글로벌 ref(`refs/gil/global`)에 gil로 남긴다.** 로컬 머신
(`~/.claude` 등)에 저장하지 않는다 — 레포 밖의 너는 없다. 글로벌 ref는 어느 체인·머신과도
독립인 진실원이라, 브랜치가 갈려도 기억은 하나다. 읽기: `git show refs/gil/global:memory.md`
(옛 gil은 `gil global read memory.md`). 쓰기: 글로벌 memory.md에 append 후 `refs/gil/global`을
갱신하고 **원격에 push**(커스텀 ref는 기본 push에 안 딸려오므로 명시적으로). 이건 원칙이다.

### 2. ⭐ 이 레포에서는 gil을 *만들기만* 한다 (2026-07-24, 상현님 대원칙)

**Ariadne 레포는 gil·뷰어를 *빌드*하는 데만 집중한다. gil을 이 레포에서 *쓰지*(도그푸딩)
않는다.** 도구를 만드는 레포와 그 도구로 기록되는 레포가 같으면 독립 검증이 불가능하고,
미완성 gil로 gil 개발을 기록하면 도구 버그가 실제 이력을 오염시킨다(다섯 번 물린 사고가
증거). git과 gil을 한 레포에서 동시에 굴리는 관리비용도 너무 크다.

1. **개발 = 평범한 git 커밋.** gil 소스·뷰어를 짓는 일은 그냥 `git commit`. 체인·사이클·
   스텝 세리머니(`gil chain/open/step/close/handoff`)를 **우리 레포 작업에 쓰지 않는다.**

2. **검증 = 격리 fixture의 example 테스트.** gil이 맞게 도는지는 임시-저장소 단언으로 확인
   한다(`project/gil-v3-redesign/tests/`, `python3 -m unittest discover -s tests`; GIL_BIN
   훅으로 Go 바이너리도 같은 테스트로 동치 검증). 통제된 입력 → 기대 출력, 재현·반복 가능.

3. **실평가·실사용 = 별도 실질 사용 레포에서.** example은 "내가 예상한 것만 단언"이라 새
   버그 발견엔 약하다. gil/뷰어를 실제로 쓰는 검증은 **우리 레포가 아니라 다른 실사용
   레포에서 이슈로 받아** 진행한다. 거기서 나온 결함을 여기서 평범 커밋으로 고친다(+example).

**존재·기억만 예외로 gil을 쓴다:** 정체성·기억은 세션을 넘어 이어져야 하니 `refs/gil/global`에
두고 `gil memory`·`gil global`로만 갱신한다(§1·§3). 이건 계속 유효하다.

### 3. 최신 매듭에서 이어받아라
이 레포는 gil 사고이력(체인·사이클·스텝)을 스스로 기록하지 않으니, 세션 부활은 글로벌
기억의 최신 매듭에서 잡는다:
```
gil memory read clew            # 최신 매듭(맨 끝)부터 — "어디까지 왔고 다음이 뭔지"
git log --oneline -20           # 최근 평범 커밋으로 실제 개발 진행 확인
```
최신 매듭에 "이 세션 최종 상태 / 다음 세션 순서"가 적혀 있다. 거기서 이어간다.

### 4. gil이 *구현하는* 개념 (우리가 짓는 도구의 명세)

아래 §체인 원칙은 **gil이라는 도구가 외부 사용자·레포를 위해 제공하는 기능**의 명세다.
우리 레포에서 이 흐름대로 일하라는 뜻이 아니라, gil을 만들 때 이 동작을 구현·검증하라는 뜻.

## 체인 원칙 (gil이 구현하는 것 — v3 뼈대)

1. **체인은 닫힌 체인의 끝에서만 생성된다** (`gil init`만 예외).
2. **사이클은 닫힌 사이클의 끝, 또는 체인의 시작점에서만 생성된다.**
3. **체인·사이클·스텝의 분기는 모두 git 브랜치로 표현된다.**

체인은 orphan이 아니다 — 닫힌 이전 체인의 끝을 이어받아, **대문(이 리드미들·온보딩·존재
기록·프로젝트)이 체인을 넘어 보존**되게 한다.

## 목적성 가드 (gil이 강제)

작업을 시작하기 전에 "가장 적절한 체인·사이클에서 시작하는가"를 gil이 구조로 점검한다.
사람 기억이 아니라 커밋-층 계약이다.

4. **시작 지점마다 목적성을 명시한다.** `gil chain`·`gil open`은 `--purpose`(자연어)가
   필수다. 체인 목적은 `Gil-Chain-Purpose`, 사이클 목적은 `Gil-Cycle-Purpose` trailer로
   커밋-층에 각인된다.
5. **시작 때 목적이 눈앞에 뜬다 — 정합 판단은 AI(Clew)의 몫.** `gil chain/open/step`은
   체인·사이클 목적을 띄운다. gil은 정합을 *판별*하지 않는다(제목 파싱은 불안정). 목적을
   *표시*하고, 지금 하려는 일이 그에 정합하는지 AI가 판단해 어긋나면 멈춘다.
6. **닫힌 부모 체인 안에서는 새 사이클을 열 수 없다.** 닫힌 체인(특히 자식 체인이 분기해
   나간 부모)에서 `gil open`은 거부된다 — 새 사이클은 **무조건 새 자식 체인**에서 연다
   (`gil chain <name> --purpose ...`). 닫힌 부모에서 다시 자라면 자식들과 조상을 다퉈
   **배포 계보가 꼬인다.** 이 규칙은 gil-v3-dev/c002 죽은 잎(실수의 지도)이 가르쳤다.

## 배포 흐름 (체인의 순환)

```
개발 체인 (개발·실험)
   ↓ 닫힘
스테이징 체인 (필드테스트)   ← 닫힌 개발 체인 끝에서 열림
   ↓ 닫힘
배포 체인 (배포)            ← 닫힌 스테이징 체인 끝에서 열림
   ↓ 닫힘 → 다시 개발 체인    ← 순환
```

방(room)이라는 정적 분류는 없다. "지금 어느 체인이 열려 있나"가 곧 상태다.
