# gil v3 명세 — 커밋 그래프 위의 체인·사이클·스텝

**손 시연이 이 명세를 만든다.** 이 문서는 gil v3 코드가 구현할 청사진이다. 폴더도 md 파일도
없다 — 모든 위계는 커밋의 trailer로, 모든 본문은 커밋 로그로.

## 근본 원칙 — 무조건 gil로 모든 흔적을 남긴다 (상현님, 2026-07-23)

**모든 작업의 흔적은 gil 스텝으로 남긴다.** 조사·공부·구현·판단·실험 — 무엇이든 gil의
스텝 커밋으로 기록된다. gil 밖에서 이뤄진 작업(예: 스텝 없이 도구를 돌리거나 조사하는 것)은
흔적이 없어 재현·계보 추적이 끊긴다. 침묵은 관전자에게 멈춤과 구별되지 않는다. **작업하기
전에 그 작업을 담을 스텝이 열려 있어야 하고, 작업의 결과는 그 스텝의 본문·산출물로 남는다.**

## 0. 브랜치 개념 3층 (상현님 확정, 2026-07-23) — git 브랜치에 매핑

gil의 위계 3층은 **git 브랜치**로 표현된다. 이게 v3의 뼈대다.

| gil 층 | git 표현 | 정의 |
|---|---|---|
| **체인** | 브랜치 | 이전 **닫힌 체인의 끝에서 분기**한 git 브랜치. orphan이 아니다 — 대문을 이어받는다. |
| **사이클** | 사이클 가지 | 체인 안에서 분기한 가지. 닫힌 사이클 끝 또는 체인 시작점에서 생성. |
| **스텝** | 스텝 가지 | 사이클 안에서 분기한 가지. 스텝 트리(백트래킹·형제)를 git 가지로 표현. |

### 체인 원칙 (상현님 확정, 2026-07-23 — v2 틀을 뒤집음)

1. **체인은 닫힌 체인의 끝에서만 생성된다** (`gil init`만 예외).
2. **사이클은 닫힌 사이클의 끝, 또는 체인의 시작점에서만 생성된다.**
3. **체인·사이클·스텝의 분기는 모두 git 브랜치로 표현된다.**

- **체인이 orphan이 아닌 이유**: 각 체인은 이전 닫힌 체인의 끝을 이어받는다. 그래야 **대문**
  (리드미 3종·온보딩·존재 기록·프로젝트)이 체인을 넘어 보존된다. `gil init`만이 유일한 예외적
  시작점(orphan으로 구현되나 대문을 심는다).
- **⭐ orphan chain 개념 신설 (상현님, 2026-07-24 — 미구현 메모)**: 위 원칙은 *대문 체인*에
  대한 것이다. 그러나 **버려질 수 있는 실작업**(예: gil로 짓는 뷰어)은 대문·gil소스 뿌리에
  이력이 박히면 곤란하다 — 그래서 `git checkout --orphan`으로 **조상 0의 독립 뿌리**에서
  짓는다(뒤엎기 = `branch -D`, 무손실). 같은 레포이므로 gil 바이너리·`refs/gil/global`(존재·
  기억)·CI에 그대로 접근하고, `git log --branches`로 다른 브랜치의 gil 그래프를 읽어 그린다.
  익으면 `gil chain-merge`로 대문에 병합. **git의 `checkout --orphan`이 이미 이 격리를
  제공하니, gil이 이를 일급 개념(`gil chain --orphan` 등)으로 이어받아야 한다** — 지금은
  손으로 `git checkout --orphan` + 인덱스 비우기로 하고, gil이 이를 감싸는 건 미구현.
  - **닦아낼 거친 점 (2026-07-24 실사용 발견)**: (1) orphan 실작업 브랜치는 `./gil` 래퍼
    같은 *추적 파일*을 잃는다 → gil은 PATH의 설치 바이너리여야 한다(래퍼 삭제로 해소).
    (2) 스텝 없는 새 체인(chain-root만)은 `gil log`가 **빈 출력** — chain-root도 노드로
    보이게 하거나 "이 체인에 아직 스텝 없음"을 알려야 한다(LLM 프롬프트 원칙). `gil handoff`는
    이미 chain-root 트레일러로 체인을 감지하니 이 갭은 `gil log` 국소.
- **체인 간 관계**: 필요하면 **머지**로 잇는다(Gil-Merge → 두 조상 상속). git 머지가 정확히
  이 모델이다.
- **분기 지점은 코드 계보로 정한다 (상현님, 2026-07-23)**: 새 체인을 어느 닫힌 체인에서
  분기할지는 **개념적 독립성이 아니라 "필요한 코드가 어디 쌓였나"**로 정한다. 뷰어와 무관한
  기능이라도 그 기능이 뷰어 체인에 쌓인 코어 개선(예: `gil step --body`)을 필요로 하면,
  뷰어 체인에서 분기해야 그 개선을 물려받는다. git 브랜치가 코드를 물려주므로, 분기 지점이
  곧 상속받는 코드를 정한다.

### 배포 순환 (방을 대체하는 체인 흐름)

정적인 "방(room)" 분류는 없다. 상태는 **지금 어느 체인이 열려 있나**로 표현된다:

```
개발 체인 (개발·실험)
   ↓ 닫힘
스테이징 체인 (필드테스트)   ← 닫힌 개발 체인 끝에서 열림
   ↓ 닫힘
배포 체인 (배포)            ← 닫힌 스테이징 체인 끝에서 열림
   ↓ 닫힘 → 다시 개발 체인    (순환)
```

- **로드맵 (2026-07-24 완료)**: `gil-v3`(gil init 대문) → 첫 개발 체인 → 스테이징 → 배포.
  **이 로드맵은 완주됐다** — v3를 `main`으로 승격(옛 main은 `legacy`·`legacy-main` 브랜치로
  보존), `v3.0.0` 정식 릴리스(latest), `gil migrate`(v2→v3 계보 변환) 신설·검증까지 완료.

### 체인 모드 — 자율 완주 vs 승인 (상현님, 2026-07-23)

체인을 **열 때 한 번** 모드를 정한다. 이후 그 체인의 사이클·스텝은 그 모드로 진행된다.

- **자율 완주(autonomous) — 기본값.** 체인을 완성할 때까지 사람에게 묻지 않고 스스로
  진행한다. 사이클을 열고·스텝을 밟고·닫고, 다음 사이클로 이어간다. 사람은 관전하되
  개입하지 않는다(원할 때 끼어들 수는 있다).
- **승인(approval) — 예외.** 스텝·사이클 결과를 사람이 승인/리젝트해야 넘어간다. 디자인
  판단이 사람 몫일 때 쓴다(예: **뷰어** — 스크린샷·분석 보고서를 올려 승인받기). 리젝트는
  새 가설로 backtrack.

#### pending = human-in-the-loop 데이터화 (상현님, 2026-07-23)

**철칙 (반드시): approval 체인에서는 pending 스텝으로 사람의 승인/기각을 명시적으로 받는다.**
승인/기각 없이 산 잎·죽은 잎을 임의로 확정하지 않는다 — 사람 판단이 필요한 지점은 언제나
pending으로 멈추고 묻는다. 사람 판단이 필요한 지점은 **pending 스텝**으로 명시한다:

1. 에이전트가 접근을 verify까지 밟고, 판단이 필요하면 `--kind pending` 스텝을 새긴다.
2. 에이전트가 사람에게 **승인/기각 두 보기**로 묻는다(사람은 뷰어로 확인 후 답).
3. 답에 따라 가지를 뻗는다:
   - **승인** → `analyze --outcome success` (산 잎, 이 접근 채택).
   - **기각** → `analyze --outcome backtrack --to <조상 define>` (죽은 잎) → 조상 define로
     되돌아가 새 형제 가지(hypothesis)로 다른 접근.

pending은 "진행 중, 사람 대기"를 그래프에 남긴다 — 답이 오면 산 잎 또는 죽은 잎으로 확정된다.
사람의 판단이 스텝 트리에 데이터로 기록되어, **가지를 어디로 뻗을지가 정확히 포착**된다
(v3-view C001의 pending 개념 실현).

체인 루트에 `Gil-Mode: autonomous | approval` 트레일러로 각인한다. 열 때 선택하고, 그
체인 동안 유지된다. **기본은 자율** — 명시 없으면 autonomous.

### 체인별 검증 깊이 (상현님, 2026-07-23)

각 체인은 맡는 검증 깊이가 다르다:

- **개발(dev) 체인의 verify = 기능 검사(smoke)만.** "이 기능이 동작하는가"까지. 전 주기가
  도는지, 명령이 커밋을 새기는지 정도. 빠르게 짓고 돌린다.
- **스테이징 체인 = 기능 검사 이상의 테스트.** 엄밀한 검증·엣지 케이스·필드테스트를 여기서.
  개발 체인이 닫힌 끝에서 열린다.
- **배포 체인 = 봉인·릴리스.** 스테이징이 닫힌 끝에서 열린다.

검증을 두 체인으로 나누면, 개발은 가볍게 전진하고 엄밀함은 스테이징이 맡는다.

## 1. 노드 = 커밋

모든 스텝은 커밋 하나다.

- **제목(subject)**: `gil <chain>/<cycle>/<step> <kind>: <사람이 읽는 요약>`
  예: `gil gil-v3-redesign/C001/s3 verify: trailer 파싱 실증`
- **본문(body)**: 스텝 디테일 (자유 markdown). `git log --format=%b <sha>`로 읽는다.
- **트레일러(trailer)**: 구조 메타 (아래 §2). `git log --format='%(trailers:key=K,valueonly)'`.

산출물(코드·figure·field)이 있으면 그 커밋에 파일로 포함한다. 없으면 `--allow-empty` 노드.

## 2. 트레일러 키셋 (`Gil-` 네임스페이스, 원자적 분리)

키마다 값 하나. 나중에 키를 추가할 여지를 위해 최대한 분리한다.

### 위계 — 모든 스텝 커밋 필수
| 키 | 값 | 뜻 |
|---|---|---|
| `Gil-Chain` | 체인명 | 어느 체인. **없거나 미선언 체인이면 gil이 거부** |
| `Gil-Cycle` | 사이클 id | 어느 사이클 |
| `Gil-Step` | 스텝 id (s1, s2…) | 어느 스텝 |
| `Gil-Kind` | define\|hypothesis\|verify\|analyze\|success\|fail\|pending | 스텝 종류 |
| `Gil-Parent` | 부모 스텝 id \| null | 스텝 트리 부모 |

### 사이클 루트 스텝(s1)에만
| 키 | 값 | 뜻 |
|---|---|---|
| `Gil-Cycle-Author` | 존재명 | 이 사이클을 연 존재 |
| `Gil-Cycle-Parent` | 부모 사이클 참조 \| null | 사이클 계보. **여러 줄 = 사이클 머지(두 조상)** |

### 분석 결과 스텝에만
| 키 | 값 | 뜻 |
|---|---|---|
| `Gil-Outcome` | success \| backtrack \| fail | 분석 결말 |
| `Gil-Backtrack` | 조상 define 스텝 id | backtrack일 때 되돌아갈 곳 |

### 체인/사이클 머지에만 (git 머지 개념 차용)
| 키 | 값 | 뜻 |
|---|---|---|
| `Gil-Merge` | 합류하는 다른 체인/사이클 참조 | **여러 줄. 이 커밋 이후 노드는 양쪽 조상을 상속** |

> **머지 = 두 조상 (상현님, 2026-07-23)**: git 커밋이 머지되면 두 부모를 갖듯, gil 노드도
> 머지되는 순간부터 그 다음 노드들이 두 갈래의 조상을 물려받는다. 체인 간·사이클 간 관계는
> 이 머지로 표현한다. (옛 loom/C015 lineage 머지의 v3 네이티브 실현.)

### 머지는 역순으로 잡힌다 (상현님, 2026-07-23) — 아래 층이 완성돼야 위 층 머지

머지는 **완성된 것들끼리 위 층으로 올라가며** 성립한다. 아래에서 위로:

1. **스텝 머지** — 한 사이클 안 **산 잎들**(analyze/success)이 머지. 여러 산 갈래가 하나로.
   `Gil-Merge`에 합류하는 산 잎 스텝 id들. 이후 스텝은 두 갈래를 조상으로.
2. **사이클 머지** — 닫힌 **산 사이클들**(solved)이 머지. `Gil-Cycle-Parent` 여러 줄.
3. **체인 머지** — 닫힌 **체인들**이 머지. `Gil-Merge`에 체인 참조 여러 줄.

핵심: 산 잎이 모여 사이클을 이루고, 산 사이클이 모여 체인을 이룬다. **완성(산 잎·산
사이클·닫힌 체인)만이 위 층에서 머지 대상**이 된다 — 죽은 잎은 머지되지 않는다(벽의 지도로
남을 뿐). 배포 순환과 맞물린다.

## 3. gil이 방어하는 무결성 (fsck)

gil은 커밋 그래프를 훑어 아래를 검사한다. 위반이면 새 노드 선언을 거부하거나 fsck가 보고한다.

1. **위계 무결성** (상현님 핵심):
   - 체인 없는(=`Gil-Chain` 미선언·미존재 체인) 사이클 선언 거부.
   - 사이클 없는 스텝 선언 거부.
   - `Gil-Parent`가 가리키는 부모 스텝이 실재해야 함 (dangling parent 거부).
   - 스텝 트리에 순환 없음.
2. **id 문법·중복**: chain·cycle·step id는 **소문자**·숫자·하이픈만(git ref 안전, 마침표 금지,
   대문자 금지). 사이클 id는 `c001`처럼 소문자 c + 숫자. 같은 체인에서 사이클 id 중복 금지.
   (옛 R1 계승. 2026-07-23: 손 시연 fsck가 대문자 C001을 잡아 소문자로 통일.)
3. **스텝 순환 규칙**: define→hypothesis→verify→analyze 순환. analyze는 `Gil-Outcome` 강제.
   backtrack이면 `Gil-Backtrack`가 조상 define을 가리켜야 함.
4. **계보 참조 무결성**: `Gil-Cycle-Parent`·`Gil-Merge`가 실재하는 사이클/체인을 가리켜야 함
   (옛 R6의 v3판, C041 이월).

## 4. 명령 표면 (gil이 구현할 것)

손 시연으로 규약을 확립한 뒤 이 명령들을 빌드했다 (구현 완료 — 전체 표면은
[docs/gil/commands.md](../../docs/gil/commands.md)):

- `gil init` — 무에서 세팅(대문 + refs/gil/global + 존재의 방).
- `gil chain <name> --purpose <P>` / `gil chain-close <chain>` — 체인 개설/완결.
- `gil open <chain>/<cycle> --author <who> [--parent <cyc>...]` — 루트 define 커밋 새김.
- `gil step --kind <k> [--outcome <o>] [--to <define>]` — 스텝 전이 커밋.
- `gil approve` / `gil reject` — pending 에 대한 사람의 명시적 승인/기각.
- `gil close [--verdict <v>]` — 산 잎 봉인. `gil chain-merge` — 두 갈래 합류(양쪽 조상).
- `gil log [<chain>] [--all]` — 커밋 그래프 조회. `gil fsck` — §3 무결성 검사.
- `gil global` / `gil memory` — 존재·기억(refs/gil/global) 갱신. `gil handoff` — 세션 부활.
- `gil migrate --from <v2-ref> [--prefix <접두>]` — 옛 v2(폴더·cycle.yaml) 이력을 v3 커밋
  그래프로 이주(도구 레벨·범용).

각 명령은 **git 커밋을 새기고 trailer를 붙이는 얇은 래퍼**다. 진실원은 언제나 커밋 그래프.
